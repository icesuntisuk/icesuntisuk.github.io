# CH-13: การรักษาความปลอดภัยสำหรับระบบคลาวด์ และ Software-Defined Networking (SDN)

## วัตถุประสงค์การเรียนรู้ (Learning Objectives)

เมื่อจบบทนี้แล้ว ผู้เรียนสามารถ:
1. อธิบายแบบจำลองความรับผิดชอบร่วม (Shared Responsibility Model) ในระบบคลาวด์ได้
2. กำหนดค่า Security Group, NACL, และ Cloud Firewall เพื่อรักษาความปลอดภัยเครือข่ายในคลาวด์ได้
3. อธิบายแนวคิด SASE (Secure Access Service Edge) และ Zero Trust Network Access (ZTNA) ได้
4. อธิบายความท้าทายด้านความปลอดภัยของ SDN และ NFV ได้
5. ออกแบบสถาปัตยกรรมเครือข่ายที่ปลอดภัยบน Hybrid Cloud และ Multi-Cloud Environment ได้
6. วิเคราะห์กรณีศึกษาการออกแบบเครือข่ายที่ปลอดภัยบน AWS/Azure/GCP ได้

# ส่วนที่ 1: Cloud Network Security

## 1. Shared Responsibility Model

### 1.1 หลักการ

Cloud Security เป็นความรับผิดชอบร่วมกันระหว่าง **Cloud Provider** และ **Customer** — แต่ละฝ่ายรับผิดชอบคนละส่วนขึ้นอยู่กับ Service Model:

```
Shared Responsibility Model:
────────────────────────────
                    ┌────────────────────────────┐
                    │        ON-PREMISES          │
                    │  (คุณรับผิดชอบทุกอย่าง)      │
                    └────────────────────────────┘

                    ┌────────────────────────────┐
                    │            IaaS             │
                    │  ┌──────────────────────┐  │
                    │  │  คุณรับผิดชอบ:         │  │
                    │  │  Data, App, OS         │  │
                    │  │  Network Config       │  │
                    │  │  Firewall Rules        │  │
                    │  ├──────────────────────┤  │
                    │  │ Provider รับผิดชอบ:    │  │
                    │  │ Hypervisor            │  │
                    │  │ Physical Network      │  │
                    │  │ Physical Security     │  │
                    │  └──────────────────────┘  │
                    └────────────────────────────┘

                    ┌────────────────────────────┐
                    │            PaaS             │
                    │  ┌──────────────────────┐  │
                    │  │  คุณรับผิดชอบ:         │  │
                    │  │  Data, App Code       │  │
                    │  ├──────────────────────┤  │
                    │  │ Provider รับผิดชอบ:    │  │
                    │  │ Runtime, OS, Network  │  │
                    │  │ Middleware, Database   │  │
                    │  └──────────────────────┘  │
                    └────────────────────────────┘

                    ┌────────────────────────────┐
                    │            SaaS             │
                    │  ┌──────────────────────┐  │
                    │  │  คุณรับผิดชอบ:         │  │
                    │  │  Data Access          │  │
                    │  │  User Permissions     │  │
                    │  ├──────────────────────┤  │
                    │  │ Provider รับผิดชอบ:    │  │
                    │  │ ทุกอย่างด้านล่าง       │  │
                    │  └──────────────────────┘  │
                    └────────────────────────────┘
```

### 1.2 ความรับผิดชอบตาม Service Model

| ชั้น | บน-premises | IaaS | PaaS | SaaS |
|:-----|:----------:|:----:|:----:|:----:|
| **ข้อมูล (Data)** | คุณ | คุณ | คุณ | คุณ |
| **แอปพลิเคชัน** | คุณ | คุณ | คุณ | Provider |
| **Runtime** | คุณ | คุณ | Provider | Provider |
| **OS** | คุณ | คุณ | Provider | Provider |
| **Middleware** | คุณ | คุณ | Provider | Provider |
| **Virtualization** | คุณ | Provider | Provider | Provider |
| **Physical Server** | คุณ | Provider | Provider | Provider |
| **Network** | คุณ | Provider | Provider | Provider |
| **Physical Security** | คุณ | Provider | Provider | Provider |

### 1.3 ข้อผิดพลาดที่พบบ่อย (Cloud Misconfiguration)

| ปัญหา | รายละเอียด | ผลกระทบ |
|:------|:-----------|:--------|
| **S3 Bucket Public Read** | Bucket เปิด Public โดยไม่ตั้งใจ | Data Breach (Accenture, Dow Jones) |
| **Security Group เปิด 0.0.0.0/0** | เปิด RDP/SSH จากอินเทอร์เน็ต | Ransomware, Bitcoin Miner |
| **IAM Key รั่วไหล** | Access Key ถูก Commit ใน GitHub | Cloud Account Compromise |
| **Public RDS** | Database เปิด Public Access | Data Theft |
| **No Encryption at Rest** | ข้อมูลใน Cloud Storage ไม่เข้ารหัส | Data Breach |

## 2. Cloud Network Security Components

### 2.1 Security Group (AWS) / Network Security Group (Azure)

Security Group เป็น **Stateful Firewall** ที่ทำงานในระดับ Instance (VM Level):

```
AWS Security Group — Stateful:
───────────────────────────────
┌──────────────────────────────┐
│  Security Group: Web_SG       │
│  Inbound Rules:               │
│  ┌──────┬────┬────────┬─────┐ │
│  │ Type│Port│ Source  │ Desc│ │
│  ├──────┼────┼────────┼─────┤ │
│  │ HTTP │ 80 │ 0.0.0.0│ Web │ │
│  │HTTPS │443 │ 0.0.0.0│ Web │ │
│  │ SSH  │ 22 │10.0.0.0│ Mgt │ │
│  └──────┴────┴────────┴─────┘ │
│  Outbound Rules:              │
│  ┌──────┬────┬────────┬─────┐ │
│  │ ALL  │ ALL│ 0.0.0.0│     │ │
│  └──────┴────┴────────┴─────┘ │
└──────────────────────────────┘
```

**ลักษณะสำคัญของ Security Group:**
- **Stateful:** Return Traffic ผ่านโดยอัตโนมัติ (ไม่ต้องเขียน Outbound Rule)
- **Allow Rules Only:** ไม่มี Deny Rule — ถ้าไม่มี Allow = Deny
- **Instance Level:** ใช้กับ EC2, RDS, ELB, Lambda
- **สามารถมีหลาย SG ต่อ Instance ได้**

### 2.2 Network ACL (NACL)

NACL เป็น **Stateless Firewall** ที่ทำงานในระดับ Subnet:

```
AWS NACL — Stateless:
──────────────────────
┌──────────────────────────────────┐
│  Network ACL: Public_Subnet_ACL  │
│  Inbound Rules (เรียงลำดับ):      │
│  ┌────┬──────┬──────┬─────────┐ │
│  │ #  │ Type │ Port │ Source  │ │
│  ├────┼──────┼──────┼─────────┤ │
│  │ 100│ HTTP │  80  │0.0.0.0 │ │
│  │ 110│ HTTPS│ 443  │0.0.0.0 │ │
│  │ 120│ Ephem│ 1024-│0.0.0.0 │ │  ← ต้องเขียน Return Traffic
│  │ *  │ ALL  │ ALL  │0.0.0.0 │ │  ← Implicit Deny
│  └────┴──────┴──────┴─────────┘ │
│  Outbound Rules (เรียงลำดับ):    │
│  ┌────┬──────┬──────┬─────────┐ │
│  │ #  │ Type │ Port │ Dest    │ │
│  ├────┼──────┼──────┼─────────┤ │
│  │ 100│ HTTP │  80  │0.0.0.0 │ │
│  │ 110│ HTTPS│ 443  │0.0.0.0 │ │
│  │ 120│ Ephem│ 1024-│0.0.0.0 │ │
│  │ *  │ ALL  │ ALL  │0.0.0.0 │ │
│  └────┴──────┴──────┴─────────┘ │
└──────────────────────────────────┘
```

**Security Group vs NACL:**

| คุณสมบัติ | Security Group | Network ACL |
|:----------|:-------------:|:-----------:|
| Stateful/Stateless | Stateful | Stateless |
| ระดับ | Instance | Subnet |
| Allow/Deny | Allow Only | Allow + Deny |
| Rule Order | All Rules (ไม่เรียง) | Numbered Order |
| Return Traffic | อัตโนมัติ | ต้องเขียน Rule |
| เมื่อใดใช้ | Instance Protection | Subnet Protection Layer |

### 2.3 Cloud Firewall (WAF + Firewall Manager)

```
Defense in Depth on Cloud:
───────────────────────────
[Internet] ──▶ [Cloud WAF] ──▶ [Cloud Firewall] ──▶ [NACL] ──▶ [SG] ──▶ [Instance]
                   │                   │                  │         │
              Application           Network Layer      Subnet     Instance
              Layer 7               Layer 3-4           Layer 3-4  Layer 3-4
```

**Cloud WAF (Web Application Firewall):**
- AWS WAF, Azure WAF, Cloud Armor (GCP)
- ป้องกัน SQLi, XSS, OWASP Top 10
- Rate-based Rules (DDoS Protection)

**Cloud Firewall (NGFW):**
- AWS Network Firewall, Azure Firewall, GCP Cloud Firewall
- Stateful Inspection, IPS, DNS Filtering
- Central Policy Management

### 2.4 ตัวอย่างการออกแบบ Network Security บน AWS

```
3-Tier Web Application on AWS:
───────────────────────────────
                     [Internet]
                         │
                    [CloudFront CDN]
                    + AWS WAF (OWASP CRS)
                         │
                  [ALB (Load Balancer)]
                    SG: Allow 80,443 from CloudFront
                         │
            ┌────────────┴────────────┐
            │                         │
      [Web Tier]                [App Tier]
      Auto Scaling Group        Auto Scaling Group
      SG: Allow 80,443          SG: Allow 8080
      from ALB                  from Web SG only
            │                         │
            └────────────┬────────────┘
                         │
                   [DB Tier]
                    RDS Multi-AZ
                    SG: Allow 3306
                    from App SG only
                         │
                    [S3 Bucket]
                    (Encrypted at Rest)
                    Block Public Access
```

**Security Group Rules:**

```
Web Tier SG:
├── Inbound:  TCP 80,443 from ALB SG
├── Inbound:  TCP 22 from Bastion SG (SSH)
└── Outbound: ALL (default — แต่ควรจำกัด)

App Tier SG:
├── Inbound:  TCP 8080 from Web SG only
├── Inbound:  TCP 22 from Bastion SG
└── Outbound: TCP 443 to Internet (API calls)

DB Tier SG:
├── Inbound:  TCP 3306 from App SG only
└── Outbound: ❌ No outbound needed

Bastion SG:
├── Inbound:  TCP 22 from Admin IP (บริษัท) only
└── Outbound: TCP 22 to Web, App SG
```

## 3. SASE และ ZTNA

### 3.1 SASE (Secure Access Service Edge)

SASE (อ่านว่า "sassy") ถูกนิยามโดย Gartner ในปี 2019 — รวม Network และ Security Functions เป็น Cloud-based Service:

```
SASE Architecture:
───────────────────
[User] ──▶ [SASE POP] ──▶ [Internet / SaaS / Cloud / DC]
                │
           SASE Functions:
           ├── SD-WAN
           ├── SWG (Secure Web Gateway)
           ├── CASB (Cloud Access Security Broker)
           ├── ZTNA (Zero Trust Network Access)
           ├── FWaaS (Firewall as a Service)
           └── DLP (Data Loss Prevention)
```

**หลักการของ SASE:**
1. **Identity-based Access:** Policy ขึ้นอยู่กับ Identity ไม่ใช่ IP
2. **Cloud-native:** Service ทำงานจาก Cloud POP ทั่วโลก
3. **Edge:** ตรวจสอบ Traffic ใกล้ผู้ใช้ที่สุด
4. **Converged:** รวม Network + Security Functions ในบริการเดียว

| Traditional | SASE |
|:------------|:-----|
| MPLS + VPN + Firewall ใน Data Center | SD-WAN + SWG + FWaaS จาก Cloud |
| Branch → HQ → Internet | Branch → SASE POP → Internet |
| Hardware Appliance ที่ Branch | Software Agent หรือ uCPE |
| Security Backhaul (Branch→HQ→Internet) | Local Breakout (ตรงถึง Cloud/Internet) |

**ผู้ให้บริการ SASE:**
- **Zscaler** (Cloud Security) + **Cloudflare**
- **Palo Alto Prisma Access**
- **Cisco Umbrella + Meraki**
- **Fortinet FortiSASE**
- **Netskope** (SASE + CASB)

### 3.2 ZTNA (Zero Trust Network Access)

ZTNA เป็นหัวใจของ SASE — เข้าใกล้ทรัพยากรโดยไม่ต้องต่อ VPN:

```
ZTNA vs VPN:
─────────────
Traditional VPN:
[User] ──▶ [VPN Gateway] ──▶ [Corporate Network] ──▶ [App]
                │                        │
          Full Network Access      Once in = access ALL

ZTNA:
[User] ──▶ [ZTNA Broker] ──▶ [App]
                │
          App-specific Access
          User → [เฉพาะ App A] เท่านั้น
          ไม่มี Network-level Access
```

**หลักการ ZTNA:**
1. **Never Trust, Always Verify** — ไม่เชื่อถือผู้ใช้หรืออุปกรณ์โดยอัตโนมัติ
2. **Least Privilege** — เข้าถึงเฉพาะ App ที่จำเป็น
3. **App-specific Connection** — ไม่ใช่ Network Connection
4. **Continuous Verification** — ตรวจสอบตลอดเวลา

**ZTNA Models:**

| Model | คำอธิบาย | ตัวอย่าง |
|:------|:---------|:---------|
| **Endpoint-initiated (Client)** | ติดตั้ง Agent บนเครื่อง — เชื่อมต่อไปยัง ZTNA Gateway | Zscaler ZPA, Cloudflare Access |
| **Service-initiated (Agentless)** | ZTNA Gateway เชื่อมต่อไปยัง Private App — ไม่ต้องติดตั้ง Agent | Tailscale, Twingate |
| **Browser-based** | ใช้ Browser — Remote Browser Isolation | Cloudflare Browser Isolation |

# ส่วนที่ 2: SDN และ NFV Security

## 4. Software-Defined Networking (SDN)

### 4.1 โมเดล SDN

SDN แยก **Control Plane** (การตัดสินใจ) ออกจาก **Data Plane** (การส่งข้อมูล):

```
Traditional Network (Distributed Control):
─────────────────────────────────────────
[Switch 1]     [Switch 2]     [Switch 3]
Control+Data   Control+Data   Control+Data
(แต่ละ Switch ตัดสินใจเอง — OSPF, BGP, STP แยกกัน)

SDN (Centralized Control):
───────────────────────────
┌──────────────────────────────────────┐
│           SDN Controller              │
│  (OpenDaylight, ONOS, Ryu)           │
│  Control Plane — รู้ Topology ทั้งหมด  │
└──────┬──────────┬──────────┬─────────┘
       │          │          │
       ▼          ▼          ▼
   [OpenFlow Switch] [OpenFlow Switch] [OpenFlow Switch]
   Data Plane       Data Plane        Data Plane
   (Forwarding according to Controller)
```

### 4.2 SDN Architecture

```
SDN Architecture (ONF Model):
─────────────────────────────
┌──────────────────────────────────────────────┐
│         APPLICATION LAYER                     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │Security │ │Network  │ │Load     │         │
│  │App      │ │Policy   │ │Balancer │         │
│  └────┬────┘ └────┬────┘ └────┬────┘        │
│       │           │           │              │
│  ┌────┴───────────┴───────────┴────┐         │
│  │   Northbound API (REST, JSON)    │         │
│  └──────────────────────────────────┘         │
├──────────────────────────────────────────────┤
│         CONTROL LAYER                         │
│  ┌──────────────────────────────────────┐    │
│  │        SDN Controller                │    │
│  │  (Network OS — Topology, Routing)    │    │
│  └──────────────────────────────────────┘    │
│  ┌──────────────────────────────────────┐    │
│  │   Southbound API (OpenFlow, NETCONF) │    │
│  └──────────────────────────────────────┘    │
├──────────────────────────────────────────────┤
│         INFRASTRUCTURE LAYER (Data Plane)     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │Switch 1 │ │Switch 2 │ │Switch 3 │        │
│  └─────────┘ └─────────┘ └─────────┘        │
└──────────────────────────────────────────────┘
```

### 4.3 ความท้าทายด้านความปลอดภัยของ SDN

| ความท้าทาย | คำอธิบาย | แนวทางป้องกัน |
|:------------|:---------|:--------------|
| **Single Point of Failure** | Controller ถูกโจมตี = ทั้ง Network พัง | Controller Cluster (High Availability) |
| **Controller Hijack** | ผู้โจมตีควบคุม Controller | Authentication + TLS ระหว่าง Controller-Switch |
| **Flow Table Flooding** | ส่ง Flow ใหม่จำนวนมาก — ทำให้ Switch ทำงานหนักเกินไป (Overload) | Rate Limiting, Flow Entry Timeout |
| **Man-in-the-Middle** | โจมตี Northbound/Southbound API | mTLS (Mutual TLS) |
| **Application Malicious** | App ที่รันบน Controller เป็น Malware | Application Sandbox, Code Review |
| **Data Plane Spoofing** | ปลอมแปลง Packet เพื่อหลบเลี่ยง Forwarding | OpenFlow Flow Table Validation |

### 4.4 แนวทางการรักษาความปลอดภัย SDN

```
SDN Security Best Practices:
─────────────────────────────
1. Controller Redundancy — Cluster อย่างน้อย 3 Nodes
2. mTLS — Southbound API (Controller ↔ Switch) ทั้งหมด
3. TLS — Northbound API (Controller ↔ App) ทั้งหมด
4. Authentication — ถ้า API ต้อง Authenticate ทุก Request
5. Flow Table Limits — จำกัดจำนวน Flow Entry ต่อ Switch
6. Logging — Log การเปลี่ยนแปลง Flow Table ทั้งหมด
7. Network Segmentation — แยก Management Network ของ SDN
8. SDN App Vetting — ตรวจสอบ App ก่อนติดตั้งบน Controller
9. Rate Limiting — จำกัด Packet-In Rate (ป้องกัน Controller DoS)
10. Update — อัปเดต SDN Controller Software สม่ำเสมอ
```

## 5. Network Function Virtualization (NFV)

### 5.1 หลักการของ NFV

NFV ย้าย Network Functions จาก Hardware Appliance ไปเป็น Virtual Machine หรือ Container:

```
Traditional vs NFV:
────────────────────
Traditional:
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│Router│ │FW    │ │IDS   │ │LB    │
│HW    │ │HW    │ │HW    │ │HW    │
└──────┘ └──────┘ └──────┘ └──────┘
(แต่ละ Function = Hardware แยกกัน)

NFV:
┌────────────────────────────────┐
│     Standard Server (x86)       │
│  ┌─────┐ ┌─────┐ ┌─────┐     │
│  │vRouter│ │vFW   │ │vIDS  │     │
│  │(VM)   │ │(VM)  │ │(VM)  │     │
│  └─────┘ └─────┘ └─────┘     │
│  ┌────────────────────────┐    │
│  │    Hypervisor (ESXi/KVM)│    │
│  └────────────────────────┘    │
│  ┌────────────────────────┐    │
│  │    Server Hardware      │    │
│  └────────────────────────┘    │
└────────────────────────────────┘
```

### 5.2 NFV Architecture (ETSI)

```
ETSI NFV Architectural Framework:
──────────────────────────────────
┌─────────────────────────────────────────────────┐
│         NFV Orchestrator (NFVO)                  │
│  ─ Lifecycle Management of NS + Resources        │
├─────────────────────────────────────────────────┤
│  ┌─────────────────┐  ┌──────────────────────┐  │
│  │ VNF Manager (VNFM)│  │ Virtual Infrastructure │
│  │ Manage VNF        │  │ Manager (VIM)         │
│  │ (FW, LB, Router)  │  │ (OpenStack, vCenter)  │
│  └─────────────────┘  └──────────────────────┘  │
├─────────────────────────────────────────────────┤
│         NFV Infrastructure (NFVI)                │
│  ┌────────┐ ┌────────┐ ┌────────┐              │
│  │ VNF 1  │ │ VNF 2  │ │ VNF 3  │              │
│  │(vFW)   │ │(vIDS)  │ │(vRouter)│              │
│  └────────┘ └────────┘ └────────┘              │
│  ┌──────────────────────────────────────────┐   │
│  │  Virtual Compute / Network / Storage      │   │
│  ├──────────────────────────────────────────┤   │
│  │  Hypervisor (KVM, ESXi)                   │   │
│  ├──────────────────────────────────────────┤   │
│  │  Hardware (Compute, Network, Storage)     │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 5.3 ความปลอดภัยของ NFV

| ความกังวล | ปัญหา | แนวทางป้องกัน |
|:-----------|:------|:--------------|
| **Hypervisor Escape** | Malware หนีจาก VM ไปยัง Hypervisor | Micro-segmentation, Hypervisor Hardening |
| **VNF Isolation** | VNF หนึ่งส่งผลกระทบ VNF อื่น | CPU Pinning, Memory Isolation |
| **VNF Image Tampering** | VNF Image ถูกแก้ไขก่อน Deploy | Image Signing + Integrity Check |
| **VNF Sprawl** | VNF จำนวนมากไม่ถูกจัดการ | VNF Lifecycle Management |
| **MANO Security** | NFV MANO Components ถูกโจมตี | RBAC, API Security |

## 6. Hybrid Cloud และ Multi-Cloud Security

### 6.1 Hybrid Cloud Architecture

```
Hybrid Cloud:
```

### 6.3 ความท้าทายของ Hybrid/Multi-Cloud

### 7.3 Security Group Rules

```
ALB SG:
├── Inbound: 80, 443 from 0.0.0.0/0 (ผ่าน WAF)
├── Inbound: 443 from CloudFront
└── Outbound: 8080 → App SG

App/API SG:
├── Inbound: 8080 from ALB SG
├── Inbound: 22 from Bastion SG (SSH)
├── Inbound: 8443 from Jenkins SG (CI/CD Webhook)
└── Outbound: 443 to Internet (API call ภายนอก)

DB SG:
├── Inbound: 5432 from App SG (PostgreSQL)
├── Inbound: 6379 from App SG (Redis)
└── Inbound: 22 from Bastion SG (SSH — Emergency)

S3 Bucket Policy:
├── Block Public Access: ✅ On
├── Server-Side Encryption: AES-256
├── Bucket Policy: อนุญาตเฉพาะ App Role เท่านั้น
└── Versioning: ✅ On (สำหรับ Ransomware Protection)
```

### 7.4 บทเรียนและข้อเสนอแนะ

1. **ใช้ Infrastructure as Code (IaC):** Terraform หรือ AWS CDK — ทุก Resource ต้องสร้างผ่าน IaC
2. **Least Privilege IAM:** ไม่ใช้ Root Account — ทุก Service ใช้ IAM Role
3. **Encryption ทุกชั้น:** At Rest (KMS) + In Transit (TLS)
4. **Logging:** CloudTrail + VPC Flow Logs + ALB Access Logs → ส่ง SIEM
5. **Backup:** Automated Backup + Cross-Region Replication
6. **Monitoring:** CloudWatch Alarm + GuardDuty + Security Hub
7. **Patch Management:** Systems Manager — Patch ทุกเดือน
8. **Incident Response:** CloudFormation Stack สำหรับ IR (Deploy Forensic Environment)

## สรุปท้ายบท (Chapter Summary)

1. **Shared Responsibility Model** — Cloud Provider รับผิดชอบ Security **ของ** Cloud, Customer รับผิดชอบ Security **ใน** Cloud

2. **Cloud Network Security** ใช้ Security Group (Stateful, Instance Level) + NACL (Stateless, Subnet Level) + WAF + Cloud Firewall

3. **SASE** รวม Network + Security Functions ใน Cloud Service เดียว — ใช้ SD-WAN + SWG + CASB + ZTNA

4. **ZTNA** เข้าถึง App โดยตรง — ไม่ต้อง VPN — ใช้หลัก "Never Trust, Always Verify" + Least Privilege

5. **SDN** แยก Control Plane และ Data Plane — มีความเสี่ยงเรื่อง Controller Hijack, Flow Table Flooding, API Security

6. **NFV** ย้าย Network Functions จาก Hardware → VM/Container — ต้องระวัง Hypervisor Escape, VNF Isolation

7. **Hybrid/Multi-Cloud** ต้องจัดการ Identity, Network, Policy, Visibility, และ Compliance — ใช้ CSPM Tools (Prisma, Wiz)

8. **กรณีศึกษา AWS** แสดงแนวทาง Defense in Depth: WAF → Shield → ALB → SG → NACL → IAM → Encryption → Logging

## คำถามทบทวน (Review Questions)

1. จงอธิบาย Shared Responsibility Model ในระบบคลาวด์ — เปรียบเทียบ IaaS, PaaS, SaaS ว่าคุณและ Provider รับผิดชอบอะไรบ้าง

2. ความแตกต่างระหว่าง Security Group และ Network ACL (NACL) ใน AWS คืออะไร? จงเปรียบเทียบ Stateful/Stateless, Rule Order, และ Use Case

3. SASE คืออะไร? จงอธิบาย components หลักของ SASE (SD-WAN, SWG, CASB, ZTNA, FWaaS, DLP)

4. ZTNA แตกต่างจาก VPN อย่างไร? จงอธิบายหลักการ "Never Trust, Always Verify" และ "Least Privilege"

5. SDN มีกี่ Layer (Application, Control, Infrastructure)? จงอธิบายความท้าทายด้านความปลอดภัยของแต่ละ Layer

6. NFV คืออะไร? จงอธิบายความแตกต่างระหว่าง Traditional Network Appliance และ NFV

7. Hybrid Cloud และ Multi-Cloud แตกต่างกันอย่างไร? จงอธิบายความท้าทายด้านความปลอดภัยอย่างน้อย 4 ข้อ

8. จงอธิบายหลักการออกแบบ Network Security สำหรับ Web Application บน Cloud — ครอบคลุม WAF, Shield, ALB, SG, NACL, IAM

9. จากกรณีศึกษาในบทนี้ จงอธิบายว่าเหตุใด RDS ต้องอยู่ใน Private Subnet และ SG ต้อง Allow Traffic จาก App SG เท่านั้น

10. จงเขียน Terraform snippet สำหรับ Security Group ที่อนุญาต HTTPS จาก ALB ไปยัง Web Server และ SSH จาก Bastion Host
