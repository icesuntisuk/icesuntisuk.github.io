# ส่วนที่ 2: Zero Trust Architecture (ZTA)

## 2.1 หลักการ Zero Trust

Zero Trust (NIST SP 800-207, 2020) — "Never Trust, Always Verify":

**หลักการ 7 ข้อของ Zero Trust (ตาม CISA):**

```
1. Continuous Verification — ตรวจสอบตลอดเวลา ไม่ใช่แค่ครั้งแรก
2. Limit Blast Radius — Micro-segmentation, Least Privilege
3. Automate Context Collection — Identity, Device, Location, Behavior
4. Assume Breach — ออกแบบระบบโดยคิดว่าถูกโจมตีแล้ว
5. Never Trust, Always Verify — ไม่เชื่อถือ Internal/Local โดยอัตโนมัติ
6. Least Privilege Access — ให้สิทธิ์เท่าที่จำเป็น
7. Data-Centric Security — ปกป้อง Data ไม่ใช่ Perimeter
```

```
ZT vs Traditional:
───────────────────
Traditional (Castle-and-Moat):
[Internet] ──▶ [Firewall ← Tight Security] ──▶ [Internal Network ← Loose]
                    │                                     │
                    │                                     │
               ถ้าเจาะ Firewall ได้ → เข้าถึงทุกอย่างได้ ❌

Zero Trust:
[Internet] ──▶ [ZTNA] ──▶ [App A]     (User → แค่ App A)
                           [App B]     (User → แค่ App B)
                           [Database]  (User → ไม่ได้เลย)
              │                               │
         ทุก Connection ต้อง Verify       Micro-segmentation
         + Authenticate                   + Least Privilege
```

## 2.2 BeyondCorp (Google)

Google BeyondCorp คือ Zero Trust Implementation ของ Google — เปลี่ยนจาก Network-Centric → User-Centric:

```
BeyondCorp Model:
──────────────────
Traditional Google (ก่อน 2010):
[Employee] ──▶ [VPN] ──▶ [Corporate Network] ──▶ [App]
                     │                          │
                เชื่อมต่อ VPN ก่อน         เข้าถึงทุก App

BeyondCorp (2011+):
[Employee] ──▶ [Internet] ──▶ [Access Proxy] ──▶ [App A]
                     │               │
                ไม่ต้อง VPN          ตรวจสอบ:
                                    ├── User Identity
                                    ├── Device Inventory
                                    ├── Device Certificate
                                    └── Context (Location, Time)
```

**บทเรียนจาก BeyondCorp:**
1. ไม่ต้อง VPN — พนักงานเข้าถึง App ได้จากทุกที่โดยตรง
2. Device Inventory ครบ — ทุกเครื่องต้องมี Certificate
3. Access Proxy — ทุก Request ผ่าน Proxy (ตรวจสอบทุกครั้ง)
4. App-level Access — พนักงานเข้าถึงเฉพาะ App ที่ได้รับอนุญาต
5. No Network-level Access — ไม่มีใครเห็น Network ทั้งหมด

## 2.3 ZTNA (Zero Trust Network Access)

ZTNA (SASE Component) — เข้าใกล้ทรัพยากรโดยไม่ต้องต่อ VPN:

```
ZTNA vs VPN:
─────────────
VPN:
├── User ได้ IP ใน Corporate Network
├── User เห็น Network ทั้งหมด (Lateral Movement ได้)
├── Once Connected → Full Access
└── เปิด Port ไปยัง Internal Network

ZTNA:
├── User เชื่อมต่อไปยัง App โดยตรง (ไม่ใช่ Network)
├── User ไม่เห็น Network — ไม่มี Lateral Movement
├── ทุก Request ถูกตรวจสอบ
└── App ถูกซ่อนจาก Internet (ไม่ต้องเปิด Port)
```

## 3.2 UEBA (User and Entity Behavior Analytics)

UEBA ใช้ ML เพื่อสร้าง Profile พฤติกรรมปกติของผู้ใช้และ Entity (Device, App):

```
UEBA Example — Insider Threat Detection:
──────────────────────────────────────────
User: john.smith (HR Department, Level 3)
Baseline (90 วัน):
├── Login Time: 08:00-18:00 (99.9%)
├── Login Location: Office (95%), Home (5%)
├── Data Access: HR Database (200 req/day), No DB access
├── File Download: < 50 MB/day
└── Email Recipients: Internal (90%), External (10%)

Anomaly:
├── Login Time: 03:15 AM (Sunday) 🔴
├── Login Location: Tor Exit Node (185.220.101.x) 🔴
├── Data Access: DB Server (first time!) 🔴
├── File Download: 5 GB (from DB Server) 🔴
└── Email: Forwarded 500 emails to external@gmail.com 🔴

Risk Score: 96/100 — 🔴 CRITICAL
Action: Disable account + Notify IR + Block IP
```

## 3.3 ข้อจำกัดของ AI/ML ด้านความปลอดภัย

| ข้อจำกัด | ปัญหา | แนวทางแก้ไข |
|:----------|:------|:------------|
| **False Positive** | AI แจ้งเตือนปกติเป็นผิด | Human-in-the-loop, Feedback Loop |

# ส่วนที่ 4: Post-Quantum Cryptography

## 4.1 Quantum Computing และผลกระทบ

เมื่อ Quantum Computer ที่มี Qubit เพียงพอ (หลายพัน Logical Qubit) มาถึง — การเข้ารหัสที่เป็นอยู่จะพัง:

| Algorithm | จุดประสงค์ | สถานะปัจจุบัน | ผลกระทบจาก Quantum |
|:----------|:-----------|:-------------|:-------------------|
| **RSA-2048** | Digital Signature, Key Exchange | ใช้ทั่วไป | ❌ Shor's Algorithm — แก้ได้ในไม่กี่นาที |
| **ECDSA / ECDH** | Key Exchange, Signature | ใช้ทั่วไป | ❌ Shor's Algorithm — แก้ได้ |
| **AES-256** | Symmetric Encryption | ใช้ทั่วไป | ✅ Grover's Algorithm — Reduction ครึ่งนึง (ยังปลอดภัยที่ 128-bit) |
| **SHA-256** | Hashing | ใช้ทั่วไป | ✅ Grover's Algorithm — Reduction ครึ่งนึง (ยังปลอดภัย) |

**Timeline ที่คาดการณ์:**
- **2030:** Quantum Computer ที่สามารถทำลาย RSA-2048 ได้? (ยังมีความไม่แน่นอนสูง)
- **NIST PQC Standardization:** 2024 — มาตรฐาน Post-Quantum Algorithms
- **Migration:** องค์กรควรเริ่มวางแผนตอนนี้ (Crypto Agility)

## 4.2 NIST PQC Standardization

NIST Post-Quantum Cryptography Standardization (2016-2024):

| Algorithm | ประเภท | วัตถุประสงค์ | Standard ปี |
|:----------|:-------|:------------|:-----------:|
| **CRYSTALS-Kyber** | KEM (Key Encapsulation) | Key Exchange | 2024 (FIPS 203) |
| **CRYSTALS-Dilithium** | Digital Signature | Signing | 2024 (FIPS 204) |
| **FALCON** | Digital Signature | Signing (smaller) | 2024 (FIPS 205) |
| **SPHINCS+** | Digital Signature | Signing (stateless) | 2024 (FIPS 205) |

```
Post-Quantum Migration:
────────────────────────
ก่อน Quantum:
TLS 1.3: ECDHE(X25519) + AES-256-GCM + RSA-2048 Signature

ระหว่าง Quantum (Hybrid):
TLS 1.3: ECDHE(X25519) + Kyber-768 + AES-256-GCM + Dilithium3 Signature

หลัง Quantum (Pure):
TLS 1.3: Kyber-1024 + AES-256-GCM + Dilithium5 Signature
```

## 4.3 Crypto Agility

Crypto Agility = ความสามารถในการเปลี่ยน Algorithm ได้โดยไม่ต้องเปลี่ยนระบบทั้งหมด:

```
Crypto Agility Requirements:
─────────────────────────────
1. Protocol Negotiation — รองรับ Algorithm หลายแบบ
2. Algorithm Independence — ไม่ Hardcode Algorithm
3. Hybrid Mode — รองรับ Classic + PQC พร้อมกัน
4. Versioning — Version Algorithm ได้
5. Migration Plan — เปลี่ยน Algorithm ได้โดย Downtime น้อย
```

# ส่วนที่ 5: 5G Security

## 5.1 สถาปัตยกรรมความปลอดภัยของ 5G

5G ปรับปรุงความปลอดภัยจาก 4G/LTE อย่างมีนัยสำคัญ:

| คุณสมบัติ | 4G (LTE) | 5G |
|:----------|:--------:|:--:|
| **Authentication** | EPS-AKA | 5G-AKA + EAP-AKA' |
| **SUCI** | IMSI (Plaintext) | SUCI (Encrypted SUPI) |
| **Network Slicing** | ❌ ไม่มี | ✅ |
| **Edge Computing** | ❌ จำกัด | ✅ MEC |
| **User Plane Integrity** | ❌ ไม่มี | ✅ (สำหรับ URLLC) |
| **Roaming Security** | ปานกลาง | SEAL (Security Edge Protection Proxy) |
| **Privacy** | IMSI ไม่เข้ารหัส | SUCI encrypts IMSI |

**SUCI (Subscription Concealed Identifier):**
- 4G: IMSI (International Mobile Subscriber Identity) ส่งแบบ Plaintext — ใครก็ดักจับได้
- 5G: SUCI = IMSI เข้ารหัสด้วย Public Key ของ Home Network — เฉพาะ Home Network เท่านั้นที่ถอดรหัสได้

## 5.2 Network Slicing

5G Network Slicing สร้าง Virtual Network แยกตามการใช้งาน:

```
5G Network Slicing:
────────────────────
┌─────────────────────────────────────────────────────────┐
│                  5G Core Network                         │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────┐ │
│  │  Slice 1       │  │  Slice 2       │  │  Slice 3    │ │
│  │  eMBB          │  │  URLLC         │  │  mMTC       │ │
│  │  Enhanced      │  │  Ultra-reliable│  │  Massive    │ │
│  │  Mobile        │  │  Low-latency   │  │  IoT        │ │
│  │  Broadband     │  │                │  │             │ │
│  ├────────────────┤  ├────────────────┤  ├────────────┤ │
│  │ Speed: 10 Gbps  │  │ Latency: 1 ms   │  │ Device: 1M   │ │
│  │ Use: Video     │  │ Use: Auto,     │  │ /km²        │ │
│  │ Streaming     │  │ Industry,      │  │ Use: Sensor │ │
│  │ Smartphone    │  │ Remote Surgery │  │ Smart Meter │ │
│  └────────────────┘  └────────────────┘  └────────────┘ │
└─────────────────────────────────────────────────────────┘
```

**Security Implications ของ Network Slicing:**
- แต่ละ Slice ต้อง Isolation แข็งแรง — บัคใน Slice หนึ่งไม่รั่วไปอีก Slice
- Slice Authentication — แต่ละ Slice ต้อง Authenticate แยก
- Slice-Specific Security Policy — Security Profile ต่างกันตาม Slice
- 5G Security Architecture กำหนดให้ Network Slicing ต้องมี Isolation ระดับ Network, Compute, Storage

## 5.3 Multi-access Edge Computing (MEC) Security

MEC นำ Compute มาไว้ใกล้ผู้ใช้ (Edge) — ลด Latency:

```
MEC Security:
─────────────
Traditional Cloud:
[Device] ────▶ [Cloud DC] — Latency 50-100ms
                      │
                 Security = ที่ Cloud

Edge (MEC):
[Device] ──▶ [Edge Node] — Latency 1-5ms
                 │            ─── ใกล้ Device
            Security = Edge + Cloud
                 │
            Edge Challenges:
            ├── Physical Security (Edge Node อยู่นอก Data Center)
            ├── Limited Resource (CPU/RAM ที่ Edge)
            ├── Update Management (Edge Node จำนวนมาก)
            └── Trust Model (Edge Node เชื่อถือได้?)

แนวทาง:
├── Trusted Execution Environment (TEE) — Intel SGX, ARM TrustZone
├── Attestation — ตรวจสอบ Integrity ของ Edge Node
├── Encryption — Data ทุกชั้น (Edge ↔ Cloud)
└── Zero Trust — Edge Node ต้อง Authentication ก่อนเชื่อมต่อ Cloud
```

# ส่วนที่ 6: SOAR และ Automation

## 6.1 หลักการของ SOAR

SOAR (Security Orchestration, Automation and Response) เชื่อมต่อเครื่องมือความปลอดภัยและ Automation การตอบสนอง:

```
SOAR Architecture:
───────────────────
┌─────────────────────────────────────────────────────┐
│                    SOAR Platform                      │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │  1. Orchestration — เชื่อมต่อ Tools              │ │
│  │  SIEM ↔ EDR ↔ Firewall ↔ Email ↔ IAM ↔ Ticketing│ │
│  └──────────────────────────────────────────────────┘ │
│  ┌──────────────────────────────────────────────────┐ │
│  │  2. Automation — ตอบสนองอัตโนมัติ               │ │
│  │  Playbook: Phishing → Block IP → Disable User    │ │
│  │            → Quarantine Email → Create Ticket    │ │
│  └──────────────────────────────────────────────────┘ │
│  ┌──────────────────────────────────────────────────┐ │
│  │  3. Response — ทำ Action บน Tools โดยอัตโนมัติ │ │
│  │  API Call: Firewall Block IP, EDR Isolate Host   │ │
│  │  API Call: Disable AD User, Create ServiceNow    │ │
│  └──────────────────────────────────────────────────┘ │
│  ┌──────────────────────────────────────────────────┐ │
│  │  4. Case Management — ติดตาม Incident           │ │
│  │  Timeline, Evidence, Communication, Report       │ │
│  └──────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

## 6.2 Security Playbook ตัวอย่าง

```
Playbook: Phishing Response (Automated)
─────────────────────────────────────────
Trigger: 🚨 User รายงานอีเมลต้องสงสัย / SIEM Alert

Step 1 — Analyze Email:
├── 🔍 Extract URLs, Attachments, Headers
├── 🔍 Sandbox URL + Attachment
├── 🔍 Check Threat Intelligence (VirusTotal, OTX)
└── 🧠 ML Classification: Phishing (Confidence: 98%)

Step 2 — Contain:
├── 🛑 Block Sender Domain on Email Gateway
├── 🛑 Block URLs on SWG (Secure Web Gateway)
├── 🛑 Quarantine email from all recipients
└── 📧 Notify: "You received a phishing email" + instructions

Step 3 — Investigate:
├── 🔍 SIEM Query: อีเมลนี้ถูกเปิดโดยกี่คน?
├── 🔍 EDR Query: เครื่องที่คลิก Link มีพฤติกรรมผิดปกติหรือไม่?
└── 📋 Create Incident Ticket (Priority: High)

Step 4 — Remediate:
├── 🛑 If machine infected → EDR Isolate Host
├── 🛑 If credential phished → Force Password Reset + MFA Re-enroll
├── 📧 Send Organization-wide Phishing Alert
└── 📊 Update Phishing Metrics Dashboard

Step 5 — Report:
├── 📊 MITRE ATT&CK Mapping
├── 📊 Timeline + Impact
└── 📊 Lessons Learned
```

## สรุปท้ายบท (Chapter Summary)

1. **Zero Trust (NIST SP 800-207)** — Never Trust, Always Verify — ใช้ Micro-segmentation, ZTNA, Least Privilege

2. **AI/ML ใน Network Security** — ใช้ UEBA (User Behavior Analytics) ตรวจจับ Insider Threat, Anomaly Detection — ต้องระวัง False Positive และ Adversarial ML

3. **Post-Quantum Cryptography** — Shor's Algorithm ทำลาย RSA/ECC — NIST เลือก CRYSTALS-Kyber (KEM), Dilithium/FALCON (Signature) — องค์กรควรเตรียม Crypto Agility

4. **5G Security** — 5G-AKA + SUCI (Encrypted IMI) + Network Slicing Isolation + MEC Edge Security

5. **SOAR** — Automation การตอบสนอง — Orchestration เชื่อม Tools — Playbook = กระบวนการอัตโนมัติ

6. **สายอาชีพ** — เริ่มจาก SOC Analyst, ต่อ Security Engineer, สู่ Security Architect / CISO — Certification: Security+ → CISSP → เฉพาะทาง

## คำถามทบทวน (Review Questions)

1. จงอธิบายหลักการ Zero Trust ตาม NIST SP 800-207 — "Never Trust, Always Verify" — และเปรียบเทียบกับ Traditional Castle-and-Moat Model

2. BeyondCorp ของ Google เปลี่ยนวิธีรักษาความปลอดภัยอย่างไร? จงอธิบายความแตกต่างระหว่าง BeyondCorp และ VPN

3. AI/ML ถูกนำมาใช้ใน Network Security อย่างไร? จงอธิบาย UEBA และยกตัวอย่างพฤติกรรมที่ UEBA สามารถตรวจจับได้

4. Quantum Computing มีผลกระทบต่อการเข้ารหัสแบบปัจจุบันอย่างไร? RSA-2048 และ AES-256 ต่างกันตรงไหน?

5. NIST PQC Standardization เลือก Algorithm ใดบ้าง? จงอธิบาย CRYSTALS-Kyber และ CRYSTALS-Dilithium

6. 5G Network Slicing คืออะไร? จงอธิบาย Security Implications ของ Network Slicing (Isolation, Authentication)

7. SUCI ใน 5G แตกต่างจาก IMSI ใน 4G อย่างไร? จงอธิบายว่าการเข้ารหัส IMSI ช่วยป้องกันการติดตามผู้ใช้อย่างไร

8. SOAR มีองค์ประกอบอะไร? จงเขียน Security Playbook สำหรับ Incident Response (ตั้งแต่ Trigger → Report)

9. Crypto Agility คืออะไร และเหตุใดจึงสำคัญต่อการเปลี่ยนผ่านสู่ Post-Quantum Cryptography?

10. จงวางแผนเส้นทางสายอาชีพ Network Security ของตนเอง — Certification ที่ควรสอบ, ตำแหน่งงาน, และทักษะที่ต้องพัฒนา
