# ส่วนที่ 1: มาตรฐานความปลอดภัยเครือข่ายไร้สาย

## 1. วิวัฒนาการของมาตรฐาน 802.11

### 1.1 มาตรฐาน IEEE 802.11

เครือข่ายไร้สายตามมาตรฐาน IEEE 802.11 มีวิวัฒนาการด้านความเร็วและความปลอดภัยดังนี้:

| มาตรฐาน | ปี | ความถี่ | ความเร็วสูงสุด | Security |
|:---------|:--:|:--------|:-------------:|:--------:|
| 802.11 | 1997 | 2.4 GHz | 2 Mbps | WEP (RC4) |
| 802.11b | 1999 | 2.4 GHz | 11 Mbps | WEP (RC4) |
| 802.11a | 1999 | 5 GHz | 54 Mbps | WEP (RC4) |
| 802.11g | 2003 | 2.4 GHz | 54 Mbps | WPA (TKIP) |
| 802.11n (Wi-Fi 4) | 2009 | 2.4 + 5 GHz | 600 Mbps | WPA2 (CCMP/AES) |
| 802.11ac (Wi-Fi 5) | 2013 | 5 GHz | 3.5 Gbps | WPA2/WPA3 |
| 802.11ax (Wi-Fi 6) | 2019 | 2.4 + 5 GHz | 9.6 Gbps | WPA3 (SAE) |
| 802.11be (Wi-Fi 7) | 2024 | 2.4 + 5 + 6 GHz | 46 Gbps | WPA3 (SAE) |

### 1.2 วิวัฒนาการด้านความปลอดภัย

```
WEP (1997) ─── RC4 + CRC-32 ❌
  │
  ├── WPA (2003) ─── TKIP + Michael ❌ (Deprecated)
  │     │
  │     └── WPA2 (2004) ─── CCMP + AES-CCMP ⚠️ (มี KRACK)
  │           │
  │           └── WPA2-Enterprise ─── 802.1X + EAP + RADIUS
  │
  └── WPA3 (2018) ─── SAE + AES-GCMP ✅ (แนะนำ)
        │
        ├── WPA3-Personal ─── SAE (Simultaneous Authentication of Equals)
        ├── WPA3-Enterprise ─── 802.1X + EAP-TLS + GCMP-256
        ├── OWE (Opportunistic Wireless Encryption) ─── ไม่ต้องใส่ Password
        └── Wi-Fi Enhanced Open ─── OWE + PMF
```

## 2. WEP (Wired Equivalent Privacy) ❌

### 2.1 หลักการทำงาน

WEP เป็นมาตรฐานรักษาความปลอดภัยแรกของ Wi-Fi (IEEE 802.11, 1997) ใช้ RC4 Stream Cipher ในการเข้ารหัส:

**ส่วนประกอบของ WEP:**
- **RC4:** Stream Cipher สำหรับ Encryption
- **CRC-32 (ICV):** Integrity Check Value — ตรวจสอบความถูกต้องของข้อมูล
- **IV (Initialization Vector):** ค่า 24-bit ที่เปลี่ยนทุก Packet

```
WEP Encrypted Packet Structure:
────────────────────────────────
┌──────────┬──────────┬──────────────────┬────────────┐
│ IV (24bit)│ Key ID   │   Data Payload    │ ICV (CRC32)│
│          │ (2bit)   │   (RC4 Encrypted) │ (Encrypted)│
└──────────┴──────────┴──────────────────┴────────────┘
```

### 2.2 จุดอ่อนของ WEP

| จุดอ่อน | รายละเอียด |
|:--------|:-----------|
| **IV ซ้ำ (IV Collision)** | IV มีขนาด 24-bit — เพียง 16,777,216 ค่า — ซ้ำแน่นอนในเครือข่ายที่ใช้งาน |
| **RC4 Weak Keys** | มี Weak Key ใน RC4 ที่ทำให้ Key Analysis เป็นไปได้ |
| **CRC-32 ไม่ปลอดภัย** | CRC-32 เป็น Linear Function — แก้ไขข้อมูลได้โดยไม่ต้องรู้ Key (Bit Flipping Attack) |
| **No Authentication** | WEP ไม่มีการตรวจสอบความถูกต้องของ Client — Spoofing ทำได้ง่าย |
| **Shared Key Authentication** | ส่ง Challenge ในรูปแบบ Plaintext + Encrypted Version — ทำให้รู้ RC4 Keystream |

### 2.3 การโจมตี WEP

**FMS Attack (Fluhrer, Mantin, Shamir — 2001):**
- ใช้จุดอ่อนของ RC4 IV (Weak IVs) เพื่อค้นหา WEP Key
- ต้องเก็บ Packet ประมาณ 250,000-500,000 Packet
- ใช้เวลาไม่กี่นาที

**KoreK Attack (2004):**
- ปรับปรุง FMS — ใช้ Weak IV ได้มากขึ้น
- ต้องใช้ Packet น้อยกว่า — ประมาณ 40,000-50,000 Packet

**PTW Attack (Pyshkin, Tews, Weinmann — 2007):**
- ไม่ต้องรอ Weak Key — ใช้ ARP Packet Replay + Statistical Analysis
- Crack WEP 64-bit ได้ใน 60 วินาที
- Crack WEP 128-bit ได้ใน 2-3 นาที

```bash
# การโจมตี WEP ด้วย aircrack-ng
airodump-ng wlan0mon                          # ดักจับ Packet
airodump-ng -c 6 --bssid AA:BB:CC:DD:EE:FF -w capture wlan0mon  # เก็บเฉพาะ Target
aireplay-ng -3 -b AA:BB:CC:DD:EE:FF wlan0mon  # ARP Replay — เร่งการเก็บ Packet
aircrack-ng -b AA:BB:CC:DD:EE:FF capture.cap  # Crack WEP Key
```

### 2.4 สรุป WEP

**WEP ควรถูกเลิกใช้โดยเด็ดขาด** — ไม่ว่าจะใช้ Key ขนาด 64-bit หรือ 128-bit — เครื่องมืออย่าง aircrack-ng สามารถ Crack ได้ในไม่กี่นาที

## 3. WPA (Wi-Fi Protected Access) ⚠️

### 3.1 หลักการทำงาน

WPA ถูกพัฒนาขึ้นในปี 2003 เพื่อแก้ปัญหาของ WEP โดยไม่ต้องเปลี่ยน Hardware (ใช้เฉพาะ Firmware Update) ใช้ **TKIP (Temporal Key Integrity Protocol)** แทน WEP เดิม:

**การปรับปรุงจาก WEP:**
1. **TKIP Key Mixing:** สร้าง Key เฉพาะแต่ละ Packet — ป้องกัน IV Collision
2. **Michael (MIC):** Message Integrity Code ที่ดีกว่า CRC-32
3. **Countermeasures:** ถ้าตรวจพบ Attack — Access Point หยุดทำงาน 60 วินาที
4. **IV Size เพิ่มเป็น 48-bit** — ลดโอกาส IV ซ้ำ

### 3.2 จุดอ่อนของ WPA/TKIP

| จุดอ่อน | รายละเอียด |
|:--------|:-----------|
| **Beck-Tews Attack (2008)** | ถอดรหัส Packet ขนาดเล็ก (เช่น ARP) ได้ด้วย QoS Channel |
| **Michael MIC Weakness** | MIC ของ TKIP ใช้สำหรับ Packet ขนาด < 18 bytes — ARP Packet |
| **Ohigashi-Morii Attack (2009)** | ปรับปรุง Beck-Tews — ถอดรหัส ARP ได้ใน 1 นาที |
| **Vanhoef-Piessens Attack (2014)** | ถอดรหัสและแทรก Packet ได้ — ใช้กับ WPA-TKIP 64-bit |
| **No Forward Secrecy** | ถ้าคนรู้ PSK — สามารถถอดรหัส Traffic ย้อนหลังได้ (ด้วย Nonce และ MIC) |

**Countermeasures ของ WPA:**
เมื่อตรวจพบ Attack (MIC Error มากกว่า 2 ครั้งใน 60 วินาที) — AP จะ:
- ปฏิเสธการเชื่อมต่อใหม่ทั้งหมดเป็นเวลา 60 วินาที
- ทำให้เกิด DoS (Denial of Service) ที่ผู้โจมตีสามารถโจมตี Countermeasure เองได้

### 3.3 สรุป WPA

WPA (TKIP) เป็นมาตรการชั่วคราว — ปัจจุบันถูก Deprecated — ควรย้ายไปใช้ WPA2 หรือ WPA3

## 4. WPA2 (IEEE 802.11i) ✅

### 4.1 หลักการทำงาน

WPA2 เปิดตัวในปี 2004 เป็นมาตรฐาน IEEE 802.11i ที่สมบูรณ์ ใช้ **CCMP (Counter Mode CBC-MAC Protocol)** กับ AES แทน TKIP

**CCMP Components:**
- **AES-CTR:** Encryption (AES ใน Counter Mode)
- **CBC-MAC:** Integrity (Cipher Block Chaining Message Authentication Code)
- **AES-CCMP:** รวม Encryption + Authentication ใน Algorithm เดียว (AEAD)

### 4.2 4-Way Handshake

หัวใจของ WPA2 คือ 4-Way Handshake — กระบวนการสร้าง Key สำหรับ Session:

```
4-Way Handshake:
─────────────────
Supplicant (Client)              Authenticator (AP)
      │                                   │
      │  PMK (Pairwise Master Key)        │  PMK
      │  ─── PSK [Personal] ───           │
      │  ─── From 802.1X [Enterprise] ──  │
      │                                   │
      │◀── (1) AP Nonce (ANonce) ────────│
      │                                   │
      │  PTK = PRF(PMK, ANonce, SNonce,   │
      │         MAC_AP, MAC_Client)        │
      │                                   │
      │── (2) SNonce + MIC ──────────────▶│
      │                                   │
      │  PTK = PRF(PMK, ANonce, SNonce,   │
      │         MAC_AP, MAC_Client)        │
      │                                   │
      │◀── (3) GTK + MIC ────────────────│
      │                                   │
      │── (4) ACK ───────────────────────▶│
      │                                   │
      │  DATA ENCRYPTION START            │
      │  (PTK = AES-CCMP)                 │
```

**Key Hierarchy ของ WPA2:**

```
PSK (256-bit) — Pre-Shared Key
    │
    ▼
PMK (256-bit) — Pairwise Master Key (PSK เท่ากับ PMK ใน WPA2-Personal)
    │
    ▼ (ผ่าน 4-Way Handshake)
PTK (512-bit) — Pairwise Transient Key
    ├── KCK (128-bit) — Key Confirmation Key (MIC ใน Handshake)
    ├── KEK (128-bit) — Key Encryption Key (ใช้ส่ง GTK)
    └── TK (128-bit) — Temporal Key (ใช้ AES-CCMP Encryption)

GTK (128-bit) — Group Temporal Key (ใช้ Broadcast/Multicast)
```

### 4.3 WPA2-Personal vs WPA2-Enterprise

| คุณสมบัติ | WPA2-Personal (PSK) | WPA2-Enterprise (802.1X) |
|:----------|:------------------:|:------------------------:|
| **Authentication** | Pre-Shared Key (Password) | RADIUS + 802.1X + EAP |
| **Key Uniqueness** | Key เดียวกันทุกคน | Key ต่างกันแต่ละ User |
| **Scalability** | ไม่เหมาะกับ >50 คน | รองรับผู้ใช้ไม่จำกัด |
| **Management** | เปลี่ยน Password = ทุกคนตั้งใหม่ | Management ผ่าน RADIUS |
| **Security** | PSK ถูก Crack ได้ (Wordlist) | สูง (Certificate-based) |
| **เหมาะกับ** | บ้าน, ร้านเล็ก | องค์กร |

### 4.4 การโจมตี WPA2-PSK — PMKID Attack

**PMKID (Pairwise Master Key Identifier):**

WPA2 AP ที่รองรับ 802.11r (Fast Roaming) จะส่ง PMKID ใน EAPOL-Key Frame แรกของ 4-Way Handshake:

```
PMKID = HMAC-SHA1(PMK, "PMK Name" || MAC_AP || MAC_Client)

ผู้โจมตีสามารถ:
1. ดักจับ PMKID จาก Beacon/eapol-start
2. โดยไม่ต้องรอ 4-Way Handshake เต็ม
3. Offline Dictionary Attack — หา PSK
```

**ข้อดีของการโจมตี PMKID เมื่อเทียบกับ 4-Way Handshake Capture:**
- ไม่ต้องรอ Client Connect (รอแค่ Beacon)
- ไม่ต้องทำ Deauthentication
- ไม่มี Client ก็ Attack ได้

### 4.5 การโจมตี WPA2 — KRACK (2017)

**KRACK (Key Reinstallation Attack)** โดย Mathy Vanhoef — ช่องโหว่ที่รุนแรงที่สุดของ WPA2:

**หลักการ:**
1. 4-Way Handshake ขั้นตอนที่ 3 — AP ส่ง GTK + MIC
2. Client ต้องส่ง ACK ขั้นตอนที่ 4 เพื่อยืนยัน
3. ถ้า AP ไม่ได้รับ ACK — AP ส่ง Frame 3 ซ้ำ
4. **KRACK:** ผู้โจมตีบังคับให้เกิดการส่ง Frame 3 ซ้ำ (โดย Block ACK)
5. Client Reinstall Key ที่ใช้แล้ว — ทำให้เกิด Nonce Reset
6. **Nonce Reuse** → AES-CTR Encryption ถูกทำลาย

```
KRACK Attack:
──────────────
AP                          Client              Attacker
 │                            │                    │
 │──── Msg1 (ANonce) ────────▶│                    │
 │◀─── Msg2 (SNonce+MIC) ────│                    │
 │──── Msg3 (GTK+MIC) ──────▶│                    │
 │                            │  PTK Installed     │
 │◀─── Msg4 (ACK) ───────────│                    │
 │                            │                    │
 │◀─── [Block ACK] ───────────────────────────────│ ← Attacker Blocks Msg4
 │──── Msg3 (GTK+MIC) ──────▶│                    │
 │                            │  PTK REINSTALLED!  │  ← Nonce Reset!
 │                            │  Nonce = 0         │
 │                            │  Encryption Broken │
```

**ผลกระทบของ KRACK:**
- ถอดรหัส Traffic ระหว่าง AP และ Client
- ปลอมแปลงและแทรก Packet (ARP Spoofing, DNS Spoofing)
- ส่งผลกระทบต่อ WPA2 ทุกแบบ — Personal และ Enterprise
- Windows, Android, iOS, Linux ทุก Version

**การแก้ไข KRACK:**
- Patch Firmware AP และ Client Driver
- WPA3 ไม่มีช่องโหว่นี้

## 5. WPA3 (2018) ✅

### 5.1 หลักการทำงาน

WPA3 เปิดตัวในปี 2018 โดย Wi-Fi Alliance เพื่อแก้ปัญหาความปลอดภัยของ WPA2:

**WPA3 มี 3 โหมด:**
1. **WPA3-Personal (SAE):** แทน PSK ด้วย SAE (Simultaneous Authentication of Equals)
2. **WPA3-Enterprise (Suite B):** ใช้ GCMP-256, ECDHE-384, SHA-384
3. **Wi-Fi Enhanced Open (OWE):** Opportunistic Wireless Encryption — เข้ารหัสแม้ไม่มี Password

### 5.2 SAE (Simultaneous Authentication of Equals)

SAE — หรือ **Dragonfly Key Exchange** — ใช้ **Dragonfly Handshake** (RFC 7664) แทน PSK:

**เปรียบเทียบ PSK vs SAE:**

| WPA2-PSK (PSK) | WPA3-SAE (SAE) |
|:---------------|:---------------|
| PSK ถูกใช้โดยตรงใน 4-Way Handshake | PSK ใช้สร้าง Commit เท่านั้น |
| Dictionary Attack แบบ Offline ได้ (Capture 4-Way แล้ว Crack) | Offline Dictionary Attack เป็นไปไม่ได้ |
| ไม่มี Forward Secrecy | มี Forward Secrecy (ECC Key Exchange) |
| PMKID Attack | ไม่มี PMKID ใน Beacon |
| KRACK Vulnerability | ไม่มี 4-Way Handshake แบบเดิม |

```
SAE Handshake (Dragonfly):
───────────────────────────
Station (Password)                    AP (Password)
      │                                      │
      │  Commit ──────────────────────────▶  │
      │  (Scalar + Element ของ ECC)          │
      │                                      │
      │  ◀── Commit ───────────────────────  │
      │  (Scalar + Element ของ ECC)          │
      │                                      │
      │  (Key Derivation: PMK = HKDF(         │
      │   shared_secret + counter)            │
      │                                      │
      │  Confirm ──────────────────────────▶  │
      │  ◀── Confirm ───────────────────────  │
      │                                      │
      │  (PMK Established — 4-Way Handshake   │
      │   สำหรับ Derive PTK)                  │
```

**ประโยชน์ของ SAE:**
- **Dictionary Attack Protection:** แม้ผู้โจมตีจะดักจับ Commit/Confirm ได้ — ไม่สามารถ Offline Crack Password ได้เพราะต้องคำนวณ Discrete Logarithm
- **Perfect Forward Secrecy:** ถ้า Password รั่วในอนาคต — Traffic ที่บันทึกไว้ก่อนหน้านี้ถอดรหัสไม่ได้
- **No KRACK:** SAE ไม่ใช้ 4-Way Handshake แบบเดิม — KRACK ไม่มีผล

### 5.3 WPA3-Enterprise

| WPA2-Enterprise | WPA3-Enterprise |
|:----------------|:----------------|
| AES-CCMP (128-bit) | AES-GCMP-256 |
| EAP-TLS, PEAP, EAP-TTLS | EAP-TLS (บังคับ — PEAP/EAP-TTLS Deprecated) |
| TLS 1.0/1.1 | TLS 1.2/1.3 |
| SHA-1 | SHA-384 |
| DH Group 1024-bit | ECDHE P-384 |

### 5.4 OWE (Opportunistic Wireless Encryption)

OWE — หรือ **Enhanced Open** — เป็นการเข้ารหัส Traffic บน Open Wi-Fi (ไม่มี Password) โดยใช้ Diffie-Hellman Key Exchange:

```
OWE: เปิดการเข้ารหัสโดยอัตโนมัติ
─────────────────────────────────
[Client] ──▶ [AP]  "ฉันอยากเชื่อมต่อ"
[AP]     ──▶ [Client] "DH Public Key ของฉัน"
[Client] ──▶ [AP]  "DH Public Key ของฉัน"
[AP]     ──▶ [Client] "PMK Established"
│
Traffic ถูกเข้ารหัส — แม้ไม่มี Password
ป้องกัน Sniffing บนเครือข่ายสาธารณะ
```

## 6. การโจมตีเครือข่ายไร้สาย

### 6.1 Evil Twin Attack

**หลักการ:** ผู้โจมตีตั้ง Access Point ปลอมที่ SSID เดียวกันกับ AP จริง — ทำให้ผู้ใช้เชื่อมต่อ (เข้าใจผิดว่า Signal ของ AP ปลอมแรงกว่า)

```
Evil Twin Attack:
──────────────────
                 [AP Real: CoffeeShop_Free]
                 SSID: CoffeeShop_Free
                 Signal: -65 dBm
                 
                 [Evil Twin: CoffeeShop_Free]
                 SSID: CoffeeShop_Free
                 Signal: -35 dBm ← แรงกว่า → Client เลือก AP ปลอม!
                 
Client ──▶ [Evil Twin AP] ──▶ [Internet]
                │
           อ่าน Traffic:
           ├── HTTP Request (Password, Cookie)
           ├── DNS Query → Redirect ไปยังหน้า Phishing
           └── MITM → แก้ไข Web Content
```

**การป้องกัน Evil Twin:**
- **802.11w (PMF — Protected Management Frames):** ป้องกัน Deauthentication Spoofing
- **Rogue AP Detection:** ใช้ WIPS หรือ Wireless Controller ตรวจสอบ AP ผิดปกติ
- **EAP-TLS (Certificate Validation):** ผู้ใช้ตรวจสอบ Certificate AP
- **VPN:** ใช้ VPN เสมอเมื่อใช้ Wi-Fi สาธารณะ

### 6.2 Deauthentication Attack (Deauth Attack)

**หลักการ:** ผู้โจมตีส่ง Deauthentication Frame ปลอม (ไม่ต้อง Authentication) ทำให้ Client ตัดการเชื่อมต่อจาก AP:

```bash
# Deauth Attack ด้วย aireplay-ng
aireplay-ng -0 5 -a AA:BB:CC:DD:EE:FF -c 11:22:33:44:55:66 wlan0mon
# -0: Deauth count, -a: BSSID AP, -c: Client MAC
```

**วัตถุประสงค์ของ Deauth Attack:**
- **Capture 4-Way Handshake:** บังคับให้ Client เชื่อมต่อใหม่ — จับ Handshake เพื่อ Crack PSK
- **ทำให้ Client เชื่อมต่อ Evil Twin:** Client หลุดจาก AP จริง → เชื่อมต่อ Evil Twin ที่แรงกว่า
- **DoS:** ทำให้ Client ไม่สามารถใช้งาน Wi-Fi ได้

**การป้องกัน Deauth Attack:**
- **802.11w (PMF):** เข้ารหัส Management Frame — ป้องกันปลอมแปลง Deauth Frame
- **WPA3 (SAE):** รองรับ PMF โดยค่าเริ่มต้น
- **Wireless IDS:** ตรวจจับ Deauth Flood Attack

| การโจมตี | คำอธิบาย | การป้องกัน |
|:----------|:---------|:-----------|
| **WPS Attack** | Brute-force WPS PIN (8 หลัก, ตรวจสอบแยก 2 ชุด — จริงๆ มี 11,000 ค่า) | ปิด WPS, ใช้ WPS Lockout |

## 7. 802.1X และ RADIUS

### 7.1 หลักการของ 802.1X

802.1X คือมาตรฐาน Network Access Control ที่ทำงานบน Layer 2 — ควบคุมการเข้าถึงเครือข่ายโดยต้อง Authentication ก่อน:

```
802.1X Architecture:
─────────────────────
┌────────────┐     ┌────────────┐     ┌────────────┐
│ Supplicant │────▶│ Authenticator│────▶│ Auth Server│
│ (Client)   │     │ (AP/Switch)  │     │ (RADIUS)   │
└────────────┘     └────────────┘     └────────────┘
    802.1X Client         │                 │
    wpa_supplicant     ถ่ายทอด EAP        FreeRADIUS
                                            
                  EAP over LAN (EAPOL):
                  ├── EAPOL-Start — Client ขอเชื่อมต่อ
                  ├── EAP-Request/Identity — AP ถามว่า Client คือใคร
                  ├── EAP-Response/Identity — Client บอกว่า ฉันคือ...
                  ├── EAP-Request (Method) — RADIUS ใช้ EAP Method
                  ├── EAP-Response — Client ตอบ
                  └── EAP-Success/Failure — ผ่าน/ไม่ผ่าน
```

### 7.2 EAP (Extensible Authentication Protocol)

EAP เป็น Framework สำหรับ Authentication — ไม่ใช่ Protocol เดียว:

| EAP Method | Security | Certificate | Password | เหมาะกับ |
|:-----------|:--------:|:-----------:|:--------:|:---------|
| **EAP-TLS** | ✅ สูงที่สุด | ✅ Client + Server | ❌ | Enterprise, อุปกรณ์ IT |
| **EAP-TTLS** | ✅ สูง | ✅ Server | ✅ | Legacy Clients |
| **PEAP** | ✅ สูง | ✅ Server | ✅ | Windows, Microsoft |
| **EAP-FAST** | ✅ ปานกลาง | ❌ (PAC) | ✅ | Cisco Environment |
| **EAP-MD5** | ❌ ต่ำ | ❌ | ✅ | ❌ ไม่ควรใช้ |
| **LEAP** | ❌ ต่ำ | ❌ | ✅ | ❌ ไม่ควรใช้ (Cisco Legacy) |

**EAP-TLS — ปลอดภัยที่สุด:**
- ต้องมี Certificate ทั้ง Client และ Server
- Mutual Authentication — ทั้งสองฝั่งตรวจสอบกัน
- ไม่มี Password — ใช้ Certificate
- รองรับ Smart Card, YubiKey

### 7.3 RADIUS Server

RADIUS (Remote Authentication Dial-In User Service) คือ Authentication Server ที่ใช้กับ 802.1X:

**FreeRADIUS — Open Source RADIUS:**

```bash
# /etc/freeradius/3.0/clients.conf — กำหนด AP/Switch
client 192.168.1.0/24 {
    secret = strong-radius-secret
    shortname = corporate-wireless
}

# /etc/freeradius/3.0/mods-config/files/authorize — ผู้ใช้
"john@example.com" Cleartext-Password := "password123"
"admin" Cleartext-Password := "securepass"
```

**RADIUS Proxy for Eduroam:**

Eduroam (Education Roaming) เป็นบริการที่ให้นักศึกษามหาวิทยาลัยต่างๆ ใช้ Wi-Fi ข้ามสถาบันได้ โดยใช้ RADIUS Proxy:

```
eduroam Architecture:
──────────────────────
[Student A] ──▶ [Univ B AP] ──▶ [Univ B RADIUS] ──▶ [Univ A RADIUS]
                  │                                     │
             802.1X + EAP-TLS/PEAP                   Verify: ใช่ นักศึกษาของเราหรือไม่
              Student A @ univ-a.edu                    └── Yes → Allow
```

### 7.4 RADIUS vs Active Directory

| คุณสมบัติ | RADIUS | Active Directory |
|:----------|:------:|:----------------:|
| บทบาท | Authentication Network | Authentication OS/App |
| Protocol | RADIUS (UDP 1812/1813) | LDAP, Kerberos |
| ใช้กับ | Network Device | Windows, Apps |
| Certificate | EAP-TLS | Domain Join |
| Integration | AD Backend | RADIUS Frontend |

## 8. การแบ่งแยกเครือข่ายไร้สาย

### 8.1 SSID Segmentation

แนวทางการแบ่งเครือข่ายไร้สายตามระดับความน่าเชื่อถือ:

```
SSID Segmentation:
───────────────────
┌────────────────────────────────────────────────────────┐
│                    Wireless Controller                   │
├────────────────────────────────────────────────────────┤
│  SSID: Corp-WiFi        │  VLAN 10 │  Internal Network   │
│  ───────────────────────┼──────────┼─────────────────────│
│  SSID: Guest-WiFi       │  VLAN 20 │  Internet Only       │
│  ───────────────────────┼──────────┼─────────────────────│
│  SSID: IoT-Network      │  VLAN 30 │  Isolated (IoT GW)   │
│  ───────────────────────┼──────────┼─────────────────────│
│  SSID: BYOD-Devices     │  VLAN 40 │  MDM Check → Limited │
│  ───────────────────────┼──────────┼─────────────────────│
│  SSID: Management       │  VLAN 99 │  Admin Only (Hidden) │
└────────────────────────────────────────────────────────┘
```

### 8.2 Guest Network Design

```
Guest Network — มาตรฐานความปลอดภัย:
──────────────────────────────────────
1. VLAN แยก — ไม่สามารถเข้าถึง Internal Network
2. Internet Only — อนุญาต HTTP/HTTPS, DNS
3. Bandwidth Limit — ไม่เกิน 5 Mbps/User
4. Captive Portal — Acceptable Use Policy
5. Session Timeout — ตัดการเชื่อมต่อทุก 4 ชม.
6. MAC Filtering — จำกัดจำนวน Device/MAC
7. Client Isolation — ผู้ใช้ Guest ไม่เห็นกันเอง
8. Rate Limiting — ป้องกัน Abuse

ตัวอย่าง ACL สำหรับ Guest:
──────────────────────────
permit udp any any eq 53       # DNS
permit tcp any any eq 80       # HTTP
permit tcp any any eq 443      # HTTPS
deny ip any 10.0.0.0/8        # Block Private IPs
deny ip any 172.16.0.0/12     # Block Private IPs
deny ip any 192.168.0.0/16    # Block Private IPs
permit ip any any              # Internet Only
```

### 8.3 IoT Network Design

```
IoT Network — ข้อควรพิจารณา:
──────────────────────────────
1. VLAN แยกเฉพาะ — IoT Devices ไม่ควรอยู่ในเครือข่ายหลัก
2. No Internet Access (ถ้าไม่จำเป็น) — IoT Devices ไม่ควรเข้าถึง Internet โดยไม่จำเป็น
3. IoT Gateway — Traffic IoT ผ่าน Gateway เพื่อควบคุม
4. Firmware Updates — ต้องมี Process อัปเดต Firmware
5. Device Authentication — 802.1X สำหรับ IoT (ถ้ารองรับ)
6. Traffic Inspection — ตรวจสอบ Traffic IoT ผ่าน IPS
7. MAC Whitelist — อนุญาตเฉพาะ MAC Address ที่รู้จัก
8. Network Segmentation — IoT แต่ละชนิด (Camera, Sensor, Printer) แยก VLAN กัน

ตัวอย่าง Device Segmentation:
├── Surveillance: VLAN 31 — Camera → NVR Only
├── HVAC/BMS:    VLAN 32 — Building Management
├── Printers:    VLAN 33 — Printer → Print Server
└── Medical:     VLAN 34 — Medical Device (HIPAA Compliant)
```

## 9. ความปลอดภัยของ Bluetooth และ IoT

### 9.1 Bluetooth Security

**Bluetooth Versions and Security:**

| Version | Security Features | จุดอ่อน |
|:--------|:-----------------|:--------|
| BR/EDR (Classic) | pairing, encryption | BlueBorne (2017), KNOB Attack |
| BLE 4.0-4.2 | LE Secure Connections | IV Reuse (บาง Implementation) |
| BLE 5.0+ | AES-CCM Encryption | Improved Key Management |
| BT 5.3+ | LE Secure Connections + CTKD | — |

**BlueBorne Attack (2017):**
- ช่องโหว่ใน Bluetooth Stack ของ Android, iOS, Windows, Linux
- ไม่ต้อง Pairing — ส่ง Packet ที่เป็นอันตรายไปยัง Bluetooth ที่เปิดไว้
- ผลกระทบ: RCE (Remote Code Execution), MITM
- อุปกรณ์กว่า 5.3 พันล้านเครื่องได้รับผลกระทบ

**การป้องกัน Bluetooth:**
- ปิด Bluetooth เมื่อไม่ใช้งาน
- อัปเดต Firmware และ Driver อย่างสม่ำเสมอ
- ใช้ Secure Simple Pairing (SSP) — เปรียบเทียบ PIN
- ใช้ Bluetooth เวอร์ชัน 5.0+ ที่มีการปรับปรุงความปลอดภัย

### 9.2 IoT Device Security Challenges

| ความท้าทาย | ปัญหา |
|:------------|:-------|
| **Default Credentials** | อุปกรณ์ IoT มักมี Username/Password มาตรฐาน (admin/admin) |
| **No Update Mechanism** | Firmware ไม่ได้รับการอัปเดต — ช่องโหว่ไม่ถูกแก้ |
| **Limited Resources** | CPU/RAM ไม่พอสำหรับ Encryption และ Protocol ที่ปลอดภัย |
| **Insecure Protocols** | ใช้ HTTP แทน HTTPS, Telnet แทน SSH, FTP แทน SFTP |
| **Data Privacy** | IoT Device ส่งข้อมูลไปยัง Cloud โดยไม่เข้ารหัส |
| **Physical Access** | อุปกรณ์ IoT มักถูกวางในพื้นที่ที่เข้าถึงได้ง่าย |
| **No Authentication** | Protocol IoT บางตัวไม่มีการ Authentication |

### 9.3 Mirai Botnet (2016)

**เหตุการณ์:** Mirai Botnet โจมตี DDoS ขนาด 1.2 Tbps ต่อ Dyn DNS — ทำให้ Twitter, Netflix, Reddit, GitHub ล่ม

**วิธีการ:**
1. สแกน IoT Devices ที่ใช้ Default Credentials (Telnet 23)
2. ติดตั้ง Malware — กลายเป็น Bot
3. Bot รอ Command จาก C2 Server
4. DDoS Attack — ส่ง Traffic ไปยัง Target

**บทเรียนจาก Mirai:**
- IoT Device มีความปลอดภัยต่ำมาก — Default Password, Telnet, No Update
- Firmware ของอุปกรณ์ IoT ไม่มี Secure Boot — ทำให้ติด Malware แบบถาวร
- IoT Devices ควรได้รับการแยก Network — ไม่ให้เข้าถึง Internal
- เปลี่ยน Default Credentials ทุกครั้ง

### 9.4 แนวทางการรักษาความปลอดภัย IoT

```
แนวทางปฏิบัติที่ดีสำหรับ IoT:
─────────────────────────────
1. เปลี่ยน Default Password ทันที
2. ปิดบริการที่ไม่จำเป็น (Telnet, FTP, UPnP)
3. ใช้ Network Segmentation — IoT VLAN แยก
4. Firmware Updates — อัปเดตสม่ำเสมอ
5. ใช้ Protocol ที่ปลอดภัย — HTTPS, SSH, MQTTS
6. Monitor Traffic IoT — ตรวจจับพฤติกรรมผิดปกติ
7. Device Authentication — 802.1X, Certificate
8. Disable Remote Access — ถ้าไม่จำเป็น
9. ใช้ IoT Gateway — ควบคุมและตรวจสอบ Traffic
10. ทำ Asset Inventory — รู้ว่ามี IoT อะไรบ้าง
```

## 10. Wireless IDS/IPS และแนวทางปฏิบัติที่ดี

### 10.1 WIDS/WIPS

Wireless Intrusion Detection/Prevention System:

| ความสามารถ | WIDS | WIPS |
|:------------|:----:|:----:|
| Rogue AP Detection | ✅ แจ้งเตือน | ✅ ปิดการเชื่อมต่อ (Deauth) |
| Evil Twin Detection | ✅ แจ้งเตือน | ✅ Block Client |
| Deauth Attack Detection | ✅ แจ้งเตือน | ✅ ส่ง Deauth ปลอมเพื่อป้องกัน |
| Client Misassociation | ✅ แจ้งเตือน | ✅ Disconnect |
| Honeypot Detection | ✅ แจ้งเตือน | ✅ Block |
| Spectrum Analysis | ✅ | ✅ |

**ตัวอย่าง WIDS Solution:**
- **Cisco ISE + CMX:** Enterprise WIDS
- **Aruba Wireless:** Integrated WIPS
- **Airtight Networks:** Dedicated WIPS
- **Kismet:** Open Source WIDS
- **Wifiphisher:** Rogue AP Framework (สำหรับ Penetration Test)

### 10.2 Best Practices สำหรับเครือข่ายไร้สายองค์กร

```
1. ใช้ WPA3-Enterprise (SAE + 802.1X + EAP-TLS)
   └── ถ้า WPA3 ไม่รองรับ → WPA2-Enterprise + 802.1X + PMF

2. 802.1X + RADIUS (FreeRADIUS, Cisco ISE, Aruba ClearPass)
   └── แทน PSK ทุกกรณี — PSK สำหรับบ้านเท่านั้น

3. 802.11w (Protected Management Frames)
   └── ป้องกัน Deauth Spoofing, Disconnect Attack

4. ใช้ Multiple SSID + VLAN Segmentation:
   ├── Corporate — 802.1X
   ├── Guest — Captive Portal + Internet Only
   ├── IoT — Isolated VLAN
   └── Management — Hidden SSID

5. WIDS/WIPS — ตรวจจับ Rogue AP, Evil Twin

6. Regular Security Audit:
   ├── War Driving — เดินตรวจสอบเครือข่ายไร้สาย
   ├── Spectrum Analysis — ตรวจสอบ Interference
   └── Penetration Test — ทดสอบการโจมตี

7. Firmware Update — อัปเดต Firmware AP และ Wireless Controller

8. Certificate Management — EAP-TLS Certificate ต้องมี Auto-renewal

9. Monitoring — Log การเชื่อมต่อทั้งหมดไปยัง SIEM

10. Incident Response — มี Plan เมื่อเจอ Wireless Attack
```

## สรุปท้ายบท (Chapter Summary)

1. **WEP ❌** — ใช้ RC4 + CRC-32 — Crack ได้ในไม่กี่นาทีด้วย aircrack-ng — ห้ามใช้เด็ดขาด

2. **WPA ⚠️** — ใช้ TKIP + Michael — ปรับปรุงจาก WEP แต่ยังมีจุดอ่อน — Deprecated

3. **WPA2 ✅** — ใช้ CCMP + AES — มาตรฐานปัจจุบัน — มี KRACK Vulnerability — ควร Patch

4. **WPA3 ✅** — ใช้ SAE (Dragonfly) + GCMP-256 — ปลอดภัยจาก Offline Dictionary Attack และ KRACK — แนะนำ

5. **Evil Twin + Deauth Attack** — ใช้ 802.11w (PMF) และ WIPS ในการป้องกัน

6. **802.1X + RADIUS + EAP-TLS** — มาตรฐานสำหรับ Enterprise — ใช้ Certificate ยืนยันตัวตน

7. **Network Segmentation** — แยก Guest, IoT, Corporate SSID+VLAN — จำกัดพื้นผิวการโจมตี

8. **IoT Security** — เปลี่ยน Default Password, แยก Network, ปิดบริการที่ไม่จำเป็น

9. **WIDS/WIPS** — ตรวจจับ Rogue AP, Evil Twin, และ Wireless Attack อัตโนมัติ

## คำถามทบทวน (Review Questions)

1. จงเปรียบเทียบ WEP, WPA, WPA2, และ WPA3 ในด้าน Algorithm, Authentication, และช่องโหว่ที่สำคัญ

2. จงอธิบาย 4-Way Handshake ของ WPA2 และ Key Hierarchy (PSK → PMK → PTK → KCK + KEK + TK)

3. KRACK คืออะไร และเหตุใดจึงสามารถถอดรหัส WPA2 Traffic ได้? จงอธิบายขั้นตอนการโจมตี

4. WPA3-SAE แตกต่างจาก WPA2-PSK อย่างไร? จงอธิบาย Dragonfly Handshake และความได้เปรียบด้านความปลอดภัย

5. 802.1X มีองค์ประกอบอะไรบ้าง (Supplicant, Authenticator, Auth Server)? จงอธิบายบทบาทของแต่ละองค์ประกอบ

6. จงเปรียบเทียบ EAP-TLS, PEAP, และ EAP-TTLS — และอธิบายว่าเหตุใด EAP-TLS จึงปลอดภัยที่สุด

7. Evil Twin Attack คืออะไร จงอธิบายวิธีการป้องกัน

8. Mirai Botnet ใช้ช่องโหว่ของ IoT Devices อย่างไร? จงอธิบายและเสนอแนวทางป้องกันองค์กรจาก IoT Botnet

9. จงออกแบบ SSID + VLAN Segmentation สำหรับมหาวิทยาลัย — โดยมีนักศึกษา, อาจารย์, บุคลากร, Guest, และ IoT Devices

10. จงอธิบายบทบาทของ WIDS/WIPS และยกตัวอย่างการตรวจจับที่ WIDS สามารถทำได้
