# CH-14: Network Access Control (NAC), Endpoint Security และ IoT Security

## วัตถุประสงค์การเรียนรู้ (Learning Objectives)

เมื่อจบบทนี้แล้ว ผู้เรียนสามารถ:
1. อธิบายหลักการ Network Access Control (NAC) — ทั้งก่อนการเชื่อมต่อและหลังการเชื่อมต่อได้
2. อธิบายการทำงานของ 802.1X, EAP, และ RADIUS ในการควบคุมการเข้าถึงเครือข่ายได้
3. อธิบายความแตกต่างระหว่าง EDR และ XDR และบทบาทในการป้องกัน Endpoint ได้
4. อธิบายแนวทางการรักษาความปลอดภัยสำหรับอุปกรณ์เคลื่อนที่ด้วย MDM และ BYOD Policy ได้
5. อธิบายความปลอดภัยของอุปกรณ์ IoT และ OT ในเครือข่ายองค์กรได้
6. ออกแบบ Network Segmentation สำหรับ IoT ด้วย VLAN segregation และ IoT Gateway ได้
7. วิเคราะห์กรณีศึกษาการโจมตีผ่าน IoT Botnet (Mirai, Mozi) และเสนอแนวทางการป้องกันได้

# ส่วนที่ 1: Network Access Control (NAC)

## 1. หลักการของ NAC

### 1.1 NAC คืออะไร

NAC (Network Access Control) คือระบบที่ **ตรวจสอบและควบคุม** อุปกรณ์ที่ขอเชื่อมต่อเข้ากับเครือข่าย — ก่อน (Pre-admission) และ หลัง (Post-admission) การเชื่อมต่อ:

```
NAC Workflow:
──────────────
[Endpoint] ──▶ [NAC] ──▶ [Network Access]
    │            │            │
    │        ตรวจสอบ:      │
    │        ├── OS Patch   │
    │        ├── Antivirus  │
    │        ├── Firewall   │
    │        ├── Disk Encrypt│
    │        └── Certificate│
    │            │            │
    │        ✗ Fail → Quarantine VLAN
    │        ✓ Pass → Corporate VLAN
```

**ประเภทของ NAC:**

| ประเภท | คำอธิบาย |
|:-------|:---------|
| **Pre-admission NAC** | ตรวจสอบก่อนเข้าเครือข่าย — ต้องผ่าน Policy → ถึงเข้าได้ |
| **Post-admission NAC** | ตรวจสอบต่อเนื่องหลังเข้าเครือข่าย — ถ้าผิดปกติ → ถูกถอด |
| **Agent-based NAC** | ติดตั้ง Agent บนเครื่อง — ตรวจสอบละเอียด |
| **Agentless NAC** | ตรวจสอบผ่าน Network Scan — ไม่ต้องติดตั้ง Agent |
| **Dissolvable Agent** | Agent ชั่วคราว — ทำงานผ่าน Browser แล้วหายไป |

### 1.2 NAC Policy Enforcement

```
นโยบาย NAC ทั่วไป:
────────────────────

Policy 1 — Corporate Devices (Managed):
├── Device: Company-issued Laptop
├── Domain-joined: ✅
├── Antivirus: ✅ (Defender updated)
├── OS Patch: ✅ (Within 7 days)
├── Disk Encryption: ✅ (BitLocker)
├── Firewall: ✅ (Enabled)
└── Result: → Corporate VLAN (Full Access)

Policy 2 — BYOD (Unmanaged):
├── Device: Personal Smartphone
├── Certificate: ✅ (MDM Enrolled)
├── Passcode: ✅ (6-digit+)
├── OS Version: ✅ (iOS 17+)
├── Jailbreak: ❌ (Not detected)
└── Result: → BYOD VLAN (Limited Access)

Policy 3 — Guest:
├── Device: Unknown Laptop
├── Authentication: ✅ (Captive Portal — Voucher Code)
├── Acceptable Use: ✅ (Accepted)
└── Result: → Guest VLAN (Internet Only)

Policy 4 — Non-Compliant:
├── Antivirus: ❌ (Disabled)
├── OS Patch: ❌ (Missing Critical Patch)
└── Result: → Quarantine VLAN (Fix → Re-scan)
```

### 1.3 Cisco ISE — NAC Solution ตัวอย่าง

Cisco Identity Services Engine (ISE) เป็น Enterprise NAC Solution:

```
Cisco ISE Architecture:
────────────────────────
┌──────────────────────────────────────────────┐
│           Cisco ISE (Policy Decision)         │
│  Rule:                                      │
│  User: AD Group = IT_Admin                   │
│  Device: Antivirus = Updated                 │
│  Location: Office                            │
│  → Allow: Full Access                        │
├──────────────────────────────────────────────┤
│           Policy Enforcement Points           │
│  ┌────────┐  ┌────────┐  ┌────────┐         │
│  │Switch  │  │Wireless │  │VPN     │         │
│  │(802.1X)│  │Controller│  │Gateway │         │

└──────────────────────────────────────────────┘
```

**Cisco ISE Features:**
- **802.1X Authentication:** ใช้กับ Wired + Wireless
- **Profiling:** ระบุ Device Type (Windows, macOS, iPhone, Printer, Camera)

- **Guest Management:** Captive Portal + Self-registration
- **BYOD Onboarding:** อุปกรณ์ส่วนตัวเข้าสู่ระบบ Enterprise
- **Endpoint Compliance:** ตรวจสอบ Antivirus, Patch, Firewall

## 2. 802.1X, EAP, และ RADIUS (NAC Context)

### 2.1 802.1X สำหรับ Wired Network

802.1X ไม่ใช่แค่สำหรับ Wi-Fi — ใช้กับ Switch Port (Wired) ได้ด้วย:

```
802.1X on Wired Switch:
────────────────────────
[User PC] ──▶ [Switch Port] ──▶ [RADIUS Server]
                │
           Port State:
           ├── Unauthorized — Allow only EAPOL (802.1X Traffic)
           ├── Authenticating — EAP Exchange
           └── Authorized — Allow Normal Traffic

เมื่อ PC ถอดสาย — Port กลับเป็น Unauthorized
เมื่อเสียบใหม่ — ต้อง Authenticate ใหม่
```

```cisco
! Cisco Switch Configuration for 802.1X
interface GigabitEthernet1/0/1
    switchport mode access
    authentication port-control auto          # เปิด 802.1X
    dot1x pae authenticator                    # ตั้ง Switch เป็น Authenticator
    dot1x timeout tx-period 30                 # Timeout
    spanning-tree portfast                     # Boot สั้นลง
    
! RADIUS Server
radius server ISE-SERVER
    address ipv4 192.168.1.100 auth-port 1812 acct-port 1813
    key radius-secret-key

! VLAN Assignment จาก RADIUS (Dynamic VLAN)
aaa new-model
aaa authentication dot1x default group radius
aaa authorization network default group radius
```

### 2.2 MAC Authentication Bypass (MAB)

สำหรับอุปกรณ์ที่ไม่รองรับ 802.1X (Printer, IP Phone, IoT) — ใช้ MAC Address เป็นตัว Authentication:

```
MAB Flow:
─────────
[Printer] ──▶ [Switch] ──▶ [RADIUS]
   |              |             |
   |   (No 802.1X)|             |
   |──────────────|             |
   |  MAC: AA:BB:CC:DD:EE:FF   |
   |              |───RADIUS Access-Request──▶|
   |              |    (MAC = Username+Password)│
   |              |◀──RADIUS Access-Accept────|
   |              |    (VLAN = IoT_VLAN)      |
   |   ✅ Access  |                            |
```

**ข้อควรระวังของ MAB:**
- MAC Address ปลอมแปลงได้ง่าย (MAC Spoofing)
- ใช้เป็น Secondary Method — 802.1X ก่อน → Fallback MAB
- ร่วมกับ Device Profiling เพื่อยืนยัน Device Type

### 2.3 RADIUS CoA (Change of Authorization)

RADIUS สามารถเปลี่ยน Authorization หลังเชื่อมต่อแล้ว — เช่น เมื่อตรวจพบว่าเครื่องติด Malware:

```
RADIUS CoA Flow:
─────────────────
[ISE] ──▶ RADIUS CoA Request ──▶ [Switch]
            "Change VLAN for user john"
            "From: Corp_VLAN → Quarantine_VLAN"

[Switch] ──▶ เครื่องของ John ถูกย้ายไป Quarantine โดยอัตโนมัติ
            โดยไม่ต้อง Deauth!
```

# ส่วนที่ 2: Endpoint Security

## 3. EDR (Endpoint Detection and Response)

### 3.1 หลักการของ EDR

EDR คือระบบที่ติดตั้ง Agent บน Endpoint (Laptop, Server) เพื่อ:
- เก็บข้อมูลกิจกรรมทุกระบบ (Process, File, Registry, Network, Memory)
- ตรวจจับพฤติกรรมต้องสงสัย (ไม่ใช่แค่ Signature)
- ตอบสนองอัตโนมัติ (Isolate, Kill Process, Quarantine File)

```
EDR Capabilities:
─────────────────
Continuous Recording:
├── Process Create/Terminate
├── File Create/Modify/Delete
├── Registry Modification
├── Network Connection
├── DLL/Load Library
├── Scheduled Task
└── PowerShell/Command Line

Detection:
├── MITRE ATT&CK Mapping — ตรงกับ Tactic/Technique
├── Behavioral Detection — ผิดปกติ
├── IOC Scan — ตรงกับ Known Malware
└── ML-based — Zero-day Detection

Response:
├── Isolate Host (ตัด Network)
├── Kill Malicious Process
├── Quarantine File
├── Block IP/Domain
├── Rollback Registry
└── Live Response (Remote Shell)
```

### 3.2 EDR vs Traditional Antivirus

| คุณสมบัติ | Antivirus (AV) | EDR |
|:----------|:--------------:|:---:|
| **Detection Method** | Signature-based | Behavioral + ML + IOC |
| **Zero-day Detection** | ❌ ไม่ได้ | ✅ Behavioral Analysis |
| **Forensic Record** | ❌ ไม่เก็บ | ✅ เก็บทุกกิจกรรม (90 วัน) |
| **Response** | Quarantine File | Isolate Host + Kill + Remediate |
| **MITRE Mapping** | ❌ | ✅ |
| **Threat Hunting** | ❌ | ✅ |
| **การทำงาน** | Scheduled Scan | Real-time 24/7 |
| **ย้อนกลับการโจมตี** | ❌ ไม่ได้ | ✅ Rollback |

### 3.3 EDR Solutions

| โซลูชัน | จุดเด่น |
|:--------|:--------|
| **CrowdStrike Falcon** | Cloud-native, AI Detection, อันดับ 1 Gartner |
| **Microsoft Defender for Endpoint** | ถ้าองค์กรใช้ Microsoft 365 — รวมกับ SIEM |
| **SentinelOne** | Autonomous AI — ตอบสนองโดยไม่ต้องมนุษย์ |
| **Carbon Black (VMware)** | Behavioral Analysis, Live Response |
| **Cybereason** | MalOp (Malicious Operation) Correlation |
| **Palo Alto Cortex XDR** | Network + Endpoint + Cloud รวมกัน |
| **Wazuh** | Open Source (ใช้ OSSEC Agent) |

```

│           └── ransomware.exe (PID 5400) ← Encrypt files
```

## 4. XDR (Extended Detection and Response)

### 4.1 หลักการของ XDR

XDR ขยายขอบเขตจาก EDR — รวม Data จากหลาย Sources:

```
XDR Coverage:
──────────────
┌─────────────────────────────────────────────────────┐
│                     XDR Platform                      │
│                                                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │ Endpoint  │  │ Network   │  │ Cloud     │           │
│  │ (EDR)     │  │ (NTA/IDS) │  │ (CSPM)    │           │
│  └──────────┘  └──────────┘  └──────────┘           │
│                                                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │ Email     │  │ Identity  │  │ SaaS      │           │
│  │ (Email GW)│  │ (AD/Azure)│  │ (M365)    │           │
│  └──────────┘  └──────────┘  └──────────┘           │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │  Correlation Engine — เชื่อมโยงทุกแหล่งข้อมูล     │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │  Automated Response (SOAR Playbook)               │ │
│  └──────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 4.2 EDR vs XDR

| คุณสมบัติ | EDR | XDR |
|:----------|:---:|:---:|
| **แหล่งข้อมูล** | Endpoint เท่านั้น | Endpoint + Network + Email + Cloud + Identity |
| **Correlation** | เฉพาะ Endpoint Events | ข้าม Data Source — รู้ภาพรวม |
| **Visibility** | เห็นเฉพาะเครื่อง | เห็นทั้ง Network, Cloud, User |
| **Detection** | Endpoint Attack | Multi-vector Attack |
| **Response** | Isolate Host | Isolate + Block IP + Reset Password + Disable User |
| **Complexity** | ต่ำ-กลาง | กลาง-สูง |

### 4.3 ตัวอย่าง XDR Detection (Multi-vector)

```
XDR Alert — Multi-vector Attack:
─────────────────────────────────
Alert: 🔴 XDR — Credential Theft + Lateral Movement

Data Sources Correlated:
├── Identity (Azure AD):
│   └── User: john.doe → Failed Login 10 ครั้ง จาก IP 185.130.5.253
├── Email (M365):
│   └── Phishing Email Sent to john.doe — "Password Reset"
├── Endpoint (MDE):
│   └── john.doe's laptop → PowerShell downloading mimikatz.exe
├── Network (NTA):
│   └── SMB Lateral Movement from 10.0.0.50 to 10.0.0.100
└── Cloud (Azure):
    └── New App Registration from john.doe's account (Abnormal)

Automated Response:
├── 🛑 Disable john.doe's Azure AD account
├── 🛑 Isolate both endpoints (10.0.0.50 and 10.0.0.100)
├── 🛑 Block IP 185.130.5.253 on Firewall
├── 🛑 Revoke OAuth tokens for new App Registration
└── 📧 Notify IR Team
```

## 5. MDM (Mobile Device Management)

### 5.1 หลักการของ MDM

MDM จัดการและรักษาความปลอดภัยอุปกรณ์เคลื่อนที่ในองค์กร:

```
MDM Capabilities:
──────────────────
Policy Enforcement:
├── Passcode Policy — ต้องมี PIN 6 หลัก+
├── Encryption — Device Encryption บังคับ
├── OS Update — บังคับอัปเดต Version ล่าสุด
├── App Management — อนุญาต/Block App
└── Remote Wipe — ลบข้อมูลเมื่ออุปกรณ์หาย

Compliance:
├── Jailbreak/Root Detection
├── Certificate Installation
├── Wi-Fi/VPN Configuration
└── Network Access — อุปกรณ์ที่ Compliant → เข้า Corporate Wi-Fi

Inventory:
├── Device Model, OS Version, Storage
├── Installed Apps
├── Location (GPS)
└── Security Status
```

**MDM Solutions:**

| โซลูชัน | จุดเด่น |
|:--------|:--------|
| **Microsoft Intune** | รวมกับ Azure AD/Office 365 |
| **Jamf** | Apple Device Management ที่ดีที่สุด |
| **VMware Workspace ONE** | Unified Endpoint Management (UEM) |
| **MobileIron** | Enterprise MDM |
| **Cisco Meraki SM** | ง่าย — เหมาะกับ SME |

### 5.2 BYOD (Bring Your Own Device) Policy

BYOD Policy คือแนวทางให้พนักงานนำอุปกรณ์ส่วนตัวมาใช้ทำงาน:

```
BYOD Architecture:
───────────────────
┌─────────────────────────────────────────────┐
│               BYOD Framework                  │
│                                                │
│  1. Containerization (Work Profile):          │
│     ├── Android Work Profile                   │
│     ├── iOS Managed Apple ID                   │
│     └── Data แยกจาก Personal                   │
│                                                │
│  2. Enrollment:                                │
│     ├── Self-service Portal                      │
│     ├── Certificate Installation               │
│     └── MDM Profile ติดตั้ง                      │
│                                                │
│  3. Security Policy:                           │
│     ├── PIN/Passcode (6 หลัก+)                 │
│     ├── Encryption (Device-level)              │
│     ├── No Jailbreak/Root                      │
│     └── OS Version ล่าสุด                       │
│                                                │
│  4. Data Protection:                           │
│     ├── Work Data Encrypted                     │
│     ├── Remote Wipe เฉพาะ Work Data             │
│     ├── No Copy/Paste ข้าม Profile              │
│     └── VPN สำหรับ Work Traffic                 │
└─────────────────────────────────────────────┘
```

**COPE (Corporate Owned, Personally Enabled):**
องค์กรซื้ออุปกรณ์ให้พนักงาน — พนักงานใช้ส่วนตัวได้บางส่วน:
- องค์กรควบคุม Device เต็มรูปแบบ
- Wipe ได้ทั้ง Device (ไม่ใช่แค่ Work Profile)
- เหมาะกับผู้บริหาร, พนักงานที่ต้องการ Security สูง

### 5.3 iOS vs Android Security

| คุณสมบัติ | iOS | Android (Enterprise) |
|:----------|:---:|:--------------------:|
| **App Store Control** | ✅ เฉพาะ App Store | ⚠️ Sideloading (ปิดได้) |
| **Sandboxing** | ✅ แข็งแรง | ✅ Android Work Profile |
| **Encryption** | ✅ Hardware-backed | ✅ File-based Encryption |
| **OS Update** | ✅ ทุกเครื่องพร้อมกัน | ⚠️ ขึ้นอยู่กับ OEM |
| **MDM Control** | ✅ Apple MDM Protocol | ✅ Android Enterprise |
| **Zero-day** | ⚠️ มีน้อย (Pegasus) | ⚠️ มีมากกว่า |
| **Jailbreak/Root** | ❌ ยาก (No JB) | ⚠️ Root ได้บ่อย |

### 5.4 Pegasus Spyware — กรณีศึกษา Mobile Threat

**Pegasus ของ NSO Group:**
- **Zero-click Exploit:** ไม่ต้องคลิก — แค่รับ iMessage ก็ติด
- **Target:** นักข่าว, นักเคลื่อนไหว, นักการเมือง
- **ความสามารถ:** อ่าน Message, เปิดกล้อง, บันทึกเสียง, GPS
- **การป้องกัน:** Lockdown Mode ใน iOS 16+ (ปิดฟังก์ชันที่ Pegasus ใช้)

**บทเรียน:**
1. Zero-day บน Mobile มีมูลค่าสูง — องค์กรที่มีความเสี่ยงสูงต้องใช้ Lockdown Mode
2. อัปเดต OS ทันทีที่มี Patch
3. ใช้ MDM ตรวจสอบ Device Health (Jailbreak Detection)
4. Mobile Threat Defense (MTD) — Lookout, Zimperium

# ส่วนที่ 3: IoT Security และ OT Security

## 6. IoT Security Challenges

### 6.1 ลักษณะของ IoT Devices

| ลักษณะ | คำอธิบาย | ผลกระทบความปลอดภัย |
|:--------|:---------|:-------------------|
| **Resource constrained** | CPU/RAM ต่ำ, Flash น้อย | เข้ารหัสไม่ได้, อัปเดตยาก |
| **Heterogeneous** | OS/Platform หลากหลาย | Patch Management ซับซ้อน |
| **Long lifecycle** | ใช้ 5-15 ปี | Firmware ไม่มี Patch |
| **Physical access** | วางในที่สาธารณะ | Physical Tampering |
| **Networked** | เชื่อมต่อ Internet โดยตรง | Exploit จากภายนอก |
| **Default credentials** | admin/admin, root/root | Brute-force ทันที |
| **No update mechanism** | อัปเดตต้อง Manual | ช่องโหว่ไม่ถูกแก้ |

### 6.3 OT Security (Operational Technology)

OT คือระบบที่ควบคุมกระบวนการทางกายภาพ (Industrial Control Systems — ICS, SCADA):

```
IT vs OT:
──────────
┌────────────────────────┬────────────────────────┐
│       IT Network        │       OT Network        │
│────────────────────────│────────────────────────│
│ Confidentiality ↑      │ Availability ↑          │
│ Integrity ↑            │ Safety ↑                │
│ Availability ↑         │ Integrity ↑             │
│────────────────────────│────────────────────────│
│ Patching = ปกติ         │ Patching = ต้อง Downtime │
│ Windows/Linux          │ RTOS, VxWorks, PLC      │
│ Change = ทุกวัน         │ Change = ต้อง Approval   │
│ อายุ Hardware = 3-5 ปี │ อายุ Hardware = 10-20 ปี │
└────────────────────────┴────────────────────────┘
```

**Purdue Model for ICS Security:**

```
Purdue Model (IEC 62443):
──────────────────────────
Level 5: Enterprise Network (ERP, Email)
Level 4: Site Business Planning (Historian, Asset Mgmt)
    ── DMZ ──
Level 3: Site Operations (SCADA, HMI)
Level 2: Control (PLC, DCS Controllers)
Level 1: Basic Control (Sensors, Actuators)
Level 0: Physical Process (Machines, Robots)
```

**การรักษาความปลอดภัย OT:**
- **Network Segmentation:** IT และ OT แยก Network เด็ดขาด — ใช้ One-way Data Diode
- **NIST SP 800-82:** Guide to ICS Security
- **IEC 62443:** มาตรฐานความปลอดภัยอุตสาหกรรม
- **OT-specific IDS:** Nozomi, Dragos, Claroty

### 7.3 IoT Gateway Architecture

IoT Gateway ทำหน้าที่เป็นตัวกลางระหว่าง IoT Devices และ Cloud:

```
IoT Gateway:
────────────
[IoT Device] ──[Local Protocol]──▶ [IoT Gateway] ──[MQTTS/HTTPS]──▶ [Cloud]
    |                   |                   |                  |
  Sensor/            Zigbee,           Protocol           AWS IoT/
  Camera             Z-Wave,           Translation         Azure IoT
                     BLE,              Local Processing    Data Lake
                     LoRaWAN           Edge Analytics      ML Model
                                       
                                       
ประโยชน์ของ IoT Gateway:
├── Protocol Translation — IoT Device ใช้ Protocol ต่างๆ → Gateway แปลงเป็น MQTT/HTTP
├── Local Processing — ประมวลผลใกล้ Device — ลด Latency
├── Security Hub — Authentication, Encryption, Firewall สำหรับ IoT
├── Firmware Management — อัปเดต Firmware IoT Device ผ่าน Gateway
└── Data Filtering — กรองข้อมูลก่อนส่ง Cloud — ลด Bandwidth
```

## 8. กรณีศึกษา: การโจมตี IoT Botnet

### 8.1 Mirai Botnet (2016)

**เหตุการณ์:** โจมตี DDoS สูงสุด 1.2 Tbps — Dyn DNS ล่ม — Twitter, Netflix, Reddit, GitHub ไม่สามารถใช้งานได้

**วิธีการทำงาน:**
```
1. Scanner — สแกนพอร์ต Telnet (23) ทั่วโลก
2. Credential Brute Force — ทดสอบ Default Password
   ├── root:root, admin:admin, support:support
   ├── 61 คู่ Credentials — ครอบคลุมอุปกรณ์ 90%
   └── (อุปกรณ์ IoT ส่วนใหญ่เปลี่ยน Password ไม่เป็น)
3. Infection — ติดตั้ง Malware บน Device
4. C2 Communication — Bot รอ Command จาก C2 Server
5. DDoS Attack:
   ├── SYN Flood
   ├── DNS Amplification
   ├── HTTP GET Flood
   └── GRE IP/ETH Flood
```

**อุปกรณ์ที่ถูกโจมตี:**
- IP Camera (Hikvision, Dahua)
- DVR (Digital Video Recorder)
- Router (Linksys, Netgear)
- CCTV Systems

**Mirai Source Code (เปิดเผยปี 2016):**
- หลังจาก Mirai Source Code ถูกปล่อย — มี Botnet หลายตัวที่ Fork จาก Mirai (Satori, Reaper, Mozi)

### 8.2 Mozi Botnet (2019-2022)

**ลักษณะของ Mozi:**
- Fork จาก Mirai — เพิ่ม P2P (Peer-to-Peer Architecture)
- ใช้ Protocol ของ eMule / Kad — กระจาย Command แบบ P2P
- ตรวจจับและกำจัดยากกว่า Mirai (ไม่มี C2 Server จุดเดียว)
- เป้าหมาย: Router IoT, Netgear, D-Link, Huawei

**P2P Architecture Advantage:**
```
Mirai (C2 Model):
[Bot] ◀───▶ [C2 Server] ← จุดอ่อน — ปิด C2 = Botnet ตาย

Mozi (P2P Model):
[Bot 1] ◀───▶ [Bot 2] ◀───▶ [Bot 3] ◀───▶ [Bot N]
ไม่มี C2 จุดเดียว — ปิด C2 ไม่ได้ (P2P Network)
```

### 8.3 แนวทางการป้องกัน IoT Botnet

```
มาตรการป้องกันองค์กร:
──────────────────────
1. Network Segmentation:
   ├── IoT VLAN แยกจาก Corporate Network
   ├── IoT Traffic ต้องผ่าน Gateway (ไม่ตรงถึง Internet)
   └── ACL: IoT → Internet = เฉพาะที่จำเป็น

2. Default Credentials:
   ├── เปลี่ยนทันทีที่ติดตั้ง
   ├── ใช้ Strong Password (12+ ตัว)
   └── เปลี่ยนจาก Factory Default

3. Firmware Updates:
   ├── ตรวจสอบ Firmware Version เป็นประจำ
   ├── อัปเดตเมื่อมี Patch
   └── ถ้า Vendor ไม่มี Patch → เปลี่ยนอุปกรณ์

4. Disable Unused Services:
   ├── ปิด Telnet (ใช้ SSH แทน)
   ├── ปิด UPnP ถ้าไม่จำเป็น
   ├── ปิด Remote Access ถ้าไม่ใช้
   └── ปิด WAN Administration บน Router

5. Monitoring:
   ├── ตรวจจับ IoT Device ที่ติดต่อ C2
   ├── ใช้ DNS Sinkhole สำหรับ Malware Domain
   ├── ตรวจสอบ Traffic ขาออก (Outbound) ที่ผิดปกติ
   └── SIEM Alert สำหรับ IoT Anomaly

6. Vendor Assessment:
   ├── เลือกอุปกรณ์ IoT ที่มี Security Feature
   ├── ตรวจสอบว่า Vendor มีกระบวนการ Patch
   └── หลีกเลี่ยงอุปกรณ์ที่ไม่เคยอัปเดต Firmware
```

## สรุปท้ายบท (Chapter Summary)

1. **NAC** ควบคุมอุปกรณ์ที่เข้าเครือข่าย — ตรวจสอบ Compliance (Patch, Antivirus, Encryption) — ก่อน (Pre-admission) และ หลัง (Post-admission) การเชื่อมต่อ

2. **802.1X + RADIUS** — ใช้กับ Wired (Switch Port) และ Wireless — รองรับ Dynamic VLAN Assignment และ CoA (Change of Authorization)

3. **EDR** ตรวจจับและตอบสนองต่อภัยคุกคามบน Endpoint — Behavioral Detection, MITRE ATT&CK Mapping, Live Response — ดีกว่า Antivirus แบบเดิม

4. **XDR** ขยายจาก EDR — รวม Data จาก Endpoint + Network + Email + Cloud + Identity — Correlate Multi-vector Attack

5. **MDM + BYOD** จัดการอุปกรณ์เคลื่อนที่ — Enforce Passcode, Encryption, Remote Wipe, Work Profile — BYOD ใช้ Containerization แยกข้อมูลส่วนตัวและงาน

6. **IoT Security** — อุปกรณ์ IoT มีข้อจำกัด (Resource, Update, Credential) — ต้องแยก VLAN, ใช้ IoT Gateway, เปลี่ยน Default Password, ตรวจสอบ Firmware

7. **OT Security** — ระบบควบคุมอุตสาหกรรม — Availability และ Safety สำคัญที่สุด — ใช้ Purdue Model + Network Isolation + OT-specific IDS

8. **Mirai/Mozi Botnet** — โจมตี IoT Device ด้วย Default Credentials — ป้องกันด้วย Network Segmentation, Firmware Update, และ Monitoring

## คำถามทบทวน (Review Questions)

1. NAC คืออะไร? จงอธิบายความแตกต่างระหว่าง Pre-admission NAC และ Post-admission NAC

2. 802.1X ทำงานอย่างไรบน Switch Port? จงอธิบายขั้นตอนการ Authentication — ตั้งแต่เสียบสายจนถึงได้รับ VLAN

3. MAB (MAC Authentication Bypass) คืออะไร และข้อควรระวังในการใช้งานคืออะไร?

4. EDR แตกต่างจาก Antivirus แบบดั้งเดิมอย่างไร? จงอธิบาย Detection, Forensic, และ Response ที่ EDR ทำได้

5. XDR คืออะไร? จงอธิบายความแตกต่างระหว่าง EDR และ XDR พร้อมยกตัวอย่างการตรวจจับ Multi-vector Attack

6. BYOD Policy ควรมีองค์ประกอบอะไรบ้าง? จงอธิบาย Containerization (Work Profile) และความสำคัญ

7. MDM สามารถป้องกันภัยคุกคามบนอุปกรณ์เคลื่อนที่ (เช่น Pegasus) ได้อย่างไร?

8. IoT Security มีความท้าทายอะไรบ้าง? จงอธิบาย OWASP IoT Top 10 อย่างน้อย 5 ข้อ

9. Mirai Botnet ทำงานอย่างไร? จงอธิบายตั้งแต่ Scanning ถึง DDoS Attack และเสนอแนวทางป้องกัน

10. ให้ออกแบบ Network Segmentation สำหรับ Smart Building ที่มี: IP Cameras 50 ตัว, Access Control 20 จุด, HVAC 10 จุด, และ Corporate Network — ระบุ VLAN, ACL, และ IoT Gateway
