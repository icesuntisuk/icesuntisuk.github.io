# CH-10: DNS Security และ Email Security

## วัตถุประสงค์การเรียนรู้ (Learning Objectives)

เมื่อจบบทนี้แล้ว ผู้เรียนสามารถ:
1. อธิบายความสำคัญของ DNS ต่อความปลอดภัยของเครือข่ายและประเภทของการโจมตีทาง DNS ได้
2. อธิบายหลักการทำงานของ DNSSEC (DNS Security Extensions) และการกำหนดค่าเบื้องต้นได้
3. อธิบายแนวคิด DNS Filtering และ DNS Sinkhole พร้อมการประยุกต์ใช้ในองค์กรได้
4. อธิบายกลไก SPF, DKIM, DMARC และการทำงานร่วมกันในการป้องกันการปลอมแปลงอีเมลได้
5. วิเคราะห์การโจมตีแบบ Phishing และ Spear-Phishing และเสนอแนวทางป้องกันได้
6. อธิบายบทบาทของ Email Gateway Security — Anti-spam, Anti-malware, DLP — ได้
7. วิเคราะห์กรณีศึกษาการโจมตีผ่าน DNS และอีเมลในองค์กรได้

# ส่วนที่ 1: DNS Security

## 1. ความสำคัญของ DNS ต่อความปลอดภัยของเครือข่าย

### 1.1 DNS คืออะไร

DNS (Domain Name System) คือระบบที่แปลงชื่อโดเมนที่มนุษย์อ่านเข้าใจได้ (เช่น `www.example.com`) ไปเป็น IP Address (เช่น `93.184.216.34`) ที่เครื่องคอมพิวเตอร์ใช้ในการสื่อสาร

```
DNS Resolution Process:
────────────────────────
[Client] ──▶ [Local DNS Resolver] ──▶ [Root Server]
              (ISP / 8.8.8.8)           ↓
              (Cache)                 [.com TLD Server]
                                        ↓
                                      [Authoritative DNS]
                                      (example.com)
                                        ↓
                                   93.184.216.34
                                        ↓
[Client] ◀── 93.184.216.34 ◀────────────
```

### 1.2 ทำไม DNS จึงสำคัญต่อความปลอดภัย

- **DNS เป็นพื้นฐานของการสื่อสารทุกอย่างบนอินเทอร์เน็ต** — ถ้า DNS ถูกโจมตี ทุกอย่างพัง
- **DNS ถูกออกแบบในยุคที่ความปลอดภัยไม่ใช่ปัจจัยหลัก** — DNS Protocol (RFC 1035, 1987) ไม่มี Authentication และ Encryption
- **DNS Traffic มักถูกมองข้าม** — Firewall หลายตัวอนุญาต DNS Outbound โดยไม่ตรวจสอบ
- **DNS ถูกใช้เป็น C2 Channel** — ผู้โจมตีใช้ DNS tunneling เพื่อติดต่อ C2 Server
- **DNS สามารถถูกใช้เป็น Vector ในการโจมตี DDoS** — DNS Amplification

### 1.3 สถิติการโจมตีทาง DNS

- 79% ขององค์กรเคยถูกโจมตีทาง DNS ในปี 2023 (IDC, 2023)
- ค่าเสียหายเฉลี่ยต่อครั้งจากการโจมตีทาง DNS: $924,000 (IDC, 2023)
- DNS Tunneling ถูกใช้ในมากกว่า 30% ของ APT Attacks (Mandiant, 2023)
- DNS Amplification DDoS คิดเป็น 15% ของ DDoS Attacks ทั้งหมด (Cloudflare, 2023)

## 2. การโจมตีทาง DNS

### 2.1 DNS Spoofing (DNS Cache Poisoning)

**หลักการ:** ผู้โจมตีแทรก DNS Record ปลอมใน Cache ของ DNS Resolver เพื่อให้ผู้ใช้ถูกส่งไปยังเว็บปลอม

```
Normal DNS:
[Client] ──▶ [DNS Resolver] ──▶ [Authoritative: 93.184.216.34]
              ↑                                      ↑
            "bank.com=93.184.216.34"              (ร้านค้าจริง)

DNS Spoofing:
[Client] ──▶ [DNS Resolver] ──▶ [Attacker's DNS]
              ↑                      ↑
            "bank.com=5.5.5.5"    (เว็บปลอม)
              ↑
          ถูกส่งไปยัง 5.5.5.5 — หน้าเว็บปลอมที่เหมือนจริง
          → กรอก Username/Password → ข้อมูลรั่วไหล
```

**เทคนิคการโจมตี:**
- **Man-in-the-Middle:** Intercept DNS Query และตอบกลับด้วย IP ปลอม
- **DNS Cache Poisoning:** ส่ง DNS Response ปลอมไปยัง Resolver (ใช้ Transaction ID Prediction)
- **Rogue DNS Server:** ตั้ง DNS Server ของตนเองให้เหยื่อใช้

**การป้องกัน:**
- DNSSEC (ป้องกันการปลอมแปลง DNS Response)
- DNS over HTTPS (DoH) / DNS over TLS (DoT)
- ใช้ Trusted DNS Resolver (8.8.8.8, 1.1.1.1, 9.9.9.9)
- ติดตาม Transaction ID แบบสุ่ม

### 2.2 DNS Tunneling

**หลักการ:** ผู้โจมตีใช้ DNS Query/Response เป็นช่องทางในการส่งข้อมูล — ใช้ซ่อน C2 Communication หรือ Data Exfiltration

```
DNS Tunneling:
────────────────
[Infected Machine]       [DNS Server]        [Attacker C2]
      │                       │                    │
      │── DNS Query ─────────▶│                    │
      │   c2secretdata.evil.com│                    │
      │                       │── Forward ────────▶│
      │                       │◀─ TXT Record ──────│
      │◀── DNS Response ─────│   "command=cat /etc/passwd"
      │   TXT "cm93IHNlY3JldA=="│                    │
```

**การประยุกต์ใช้ของ Hacker:**
- **C2 Communication:** ส่ง Command ไปยัง Malware ผ่าน DNS Response
- **Data Exfiltration:** แปลงข้อมูลเป็น Subdomain และส่งออก (Base64 + DNS Query)
- **Tunnel TCP over DNS:** ใช้ DNS เป็น Proxy (iodine, dnscat2)

**เครื่องมือ DNS Tunneling ที่รู้จัก:**
| เครื่องมือ | รายละเอียด |
|:-----------|:-----------|
| **iodine** | TCP/IP over DNS — รองรับการทำงานคล้าย VPN |
| **dnscat2** | C2 Channel over DNS — สร้าง Command Session |
| **DNS2TCP** | TCP Tunnel over DNS |
| **Heyoka** | DNS Exfiltration Tool — ใช้ SPF Record ช่วยซ่อน Traffic |

**การป้องกัน DNS Tunneling:**
- **DNS Traffic Analysis:** ตรวจสอบ DNS Query ที่มีขนาดใหญ่ผิดปกติ
- **Whitelist Domains:** อนุญาต DNS Query เฉพาะ Domain ที่จำเป็น
- **DNS Sinkhole:** เปลี่ยน DNS Response สำหรับ Malicious Domains
- **Limit DNS Query Rate:** ตรวจจับ DNS Query ปริมาณมากผิดปกติ
- **Inspect TXT Record:** ตรวจสอบ TXT Record Response ขนาดใหญ่

### 2.3 DNS Amplification DDoS

**หลักการ:** ผู้โจมตีใช้ DNS Servers ที่เปิด Recursive Query เพื่อ Amplify Traffic โจมตีเหยื่อ

```
DNS Amplification DDoS:
────────────────────────
[Attacker] ──▶ [Open DNS Resolver] ──▶ [Victim]
   |                | (x50-70)             |
   |── ANY? ──────▶│                       |
   |(Spoofed Source│                       |
   | = Victim IP)  │── Response (3,500B) ─▶│
                   │                       │
[Bot 1] ── ANY ──▶│                       │
[Bot 2] ── ANY ──▶│                       │
[Bot 3] ── ANY ──▶│                       │
        ...        │  Bandwidth Amplified  │
     50,000 Bots   │  30,000 queries/sec   │
                   │  × 3,500 bytes = 105MB/s
                   │  → 840 Mbps to Victim │
```

**ปัจจัยที่ทำให้ DNS Amplification มีความรุนแรงสูง:**
- **Amplification Factor:** DNS Response มีขนาดใหญ่กว่า Query 50-70 เท่า (โดยเฉพาะ `ANY` query)
- **Source IP Spoofing:** ผู้โจมตีปลอม Source IP เป็นของเหยื่อ
- **Open DNS Resolvers:** มี DNS Servers หลายล้านเครื่องที่เปิด Recursive Query

**การป้องกัน:**
- **ปิด Recursive Query บน Authoritative DNS Servers**
- **Response Rate Limiting (RRL):** จำกัดการตอบกลับ DNS Query ที่ผิดปกติ
- **Network Ingress Filtering (BCP 38):** ป้องกัน Source IP Spoofing
- **ใช้ DDoS Protection Service** (Cloudflare, Akamai, AWS Shield)

### 2.4 การโจมตีทาง DNS อื่นๆ

| การโจมตี | คำอธิบาย | ผลกระทบ |
|:----------|:---------|:--------|
| **DNS Hijacking** | เปลี่ยน Nameserver ของ Domain โดยไม่ได้รับอนุญาต | ผู้ใช้ถูกส่งไปยังเว็บปลอม |
| **NXDOMAIN Attack** | ส่ง DNS Query ไปยัง Domain ที่ไม่มีอยู่ — Resolver ใช้ทรัพยากรในการค้นหา | CPU Saturation |
| **Random Subdomain Attack** | ส่ง DNS Query ไปยัง Subdomain แบบสุ่มจำนวนมาก | ทำ Cache Poisoning, DDoS |
| **Phantom Domain Attack** | DNS Resolver รอ Response จาก Domain ที่ตอบช้า | CPU Saturation |
| **Domain Generation Algorithm (DGA)** | Malware สร้าง Domain ใหม่ทุกวันเพื่อหลบเลี่ยง Sinkhole | C2 Communication |

## 3. DNSSEC (DNS Security Extensions)

### 3.1 หลักการของ DNSSEC

DNSSEC เพิ่มความปลอดภัยให้กับ DNS โดยใช้ **Digital Signature** เพื่อยืนยันว่า DNS Response มาจาก Authoritative Source จริงและไม่ถูกแก้ไขระหว่างทาง

**DNSSEC ไม่ได้เข้ารหัส DNS Data — แค่เซ็นรับรองความถูกต้อง (Data Integrity + Origin Authentication)**

```
DNSSEC Chain of Trust:
───────────────────────
                       [Root Zone]
                      KSK (Key Signing Key)
                           │
                   ZSK (Zone Signing Key)
                    เซ็น DS Record
                            │
                     [.com TLD Zone]
                       DS Record ↕ Root
                       KSK + ZSK
                            │
                     เซ็น DS Record
                           │
                 [example.com Zone]
                   DS Record ↕ .com
                   KSK + ZSK
                      DNSKEY
                      RRSIG (สำหรับทุก Record)
                      NSEC/NSEC3 (สำหรับ Non-existence)
```

### 3.2 DNSSEC Record Types

| Record Type | คำอธิบาย |
|:------------|:---------|
| **DNSKEY** | Public Key ที่ใช้ในการตรวจสอบ Signature |
| **RRSIG** | Digital Signature ของ DNS Record (RRSIG A, RRSIG MX, RRSIG NS, etc.) |
| **DS (Delegation Signer)** | Hash ของ DNSKEY ของ Zone ลูก — ใช้สร้าง Chain of Trust |
| **NSEC / NSEC3** | ยืนยันว่า Domain Name ที่ถามไม่มีอยู่ (Authenticated Denial of Existence) |
| **CDS / CDNSKEY** | Child-to-parent — แจ้ง Parent Zone เกี่ยวกับ Key Change |

### 3.3 กระบวนการตรวจสอบ DNSSEC

```
Client → Resolver (Validating Resolver):
─────────────────────────────────────────
1. Client ถาม "what is www.example.com?"
2. Resolver ถาม Root: "DS of .com?" → Root ส่ง DS(.com) + RRSIG
3. Resolver ถาม .com TLD: "DS of example.com?" → .com ส่ง DS(example.com) + RRSIG
4. Resolver ถาม example.com: "A record of www.example.com?" → example.com ส่ง A + RRSIG + DNSKEY
5. Resolver ตรวจสอบ:
   - DNSKEY(example.com) → verify RRSIG(A record) ✅
   - DS(example.com) → verify DNSKEY(example.com) ✅
   - DS(.com) → verify DNSKEY(.com) ✅
   - Root Key → verify DS(.com) ✅ (Trust Anchor)
6. Resolver ส่ง A record ให้ Client ✅ (Authenticated)
```

### 3.4 การติดตั้ง DNSSEC (ตัวอย่าง — BIND9)

```bind
# /etc/bind/named.conf.options — เปิด DNSSEC Validation
options {
    dnssec-enable yes;
    dnssec-validation yes;
    dnssec-lookaside auto;
    managed-keys-directory "/var/cache/bind";
};

# /etc/bind/named.conf — เปิด DNSSEC สำหรับ Zone
zone "example.com" {
    type master;
    file "/etc/bind/db.example.com";
    dnssec-policy default;
    inline-signing yes;
};
```

```bash
# สร้าง Key สำหรับ DNSSEC
dnssec-keygen -a ECDSAP256SHA256 -n ZONE example.com
dnssec-keygen -a ECDSAP256SHA256 -n ZONE -f KSK example.com

# เซ็น Zone
dnssec-signzone -A -3 $(head -c 1000 /dev/random | sha1sum | cut -b 1-16) \
    -N INCREMENT -o example.com -t db.example.com
```

### 3.5 ความท้าทายของ DNSSEC

| ความท้าทาย | รายละเอียด |
|:------------|:-----------|
| **Complexity** | การจัดการ Key Rotation ซับซ้อน (KSK, ZSK, Key Rollover) |
| **Zone Walking** | NSEC Record ทำให้ผู้โจมตีสามารถเดิน (Enumerate) ทุก Record ใน Zone ได้ (NSEC3 แก้ปัญหานี้) |
| **Performance** | DNS Response มีขนาดใหญ่ขึ้น (RRSIG, DNSKEY) — อาจเกิน 512 bytes (ต้องใช้ EDNS0) |
| **Adoption** | เฉพาะ ~30% ของ .com Zone ที่ใช้ DNSSEC (2024) |
| **Key Compromise** | หาก KSK รั่ว — ทั้ง Zone ต้อง Resign และแจ้ง Parent Zone |

## 4. DNS Filtering และ DNS Sinkhole

### 4.1 DNS Filtering

DNS Filtering คือการกรอง DNS Query ตาม Policy — ป้องกันผู้ใช้เข้าถึงเว็บที่เป็นอันตรายหรือไม่เหมาะสม:

```
DNS Filtering:
───────────────
[User] ──▶ [DNS Filter] ──▶ [Internet]
              │
         ตรวจสอบ Domain:
         ├── example.com (safe) → Forward → นำไปสู่ IP จริง
         ├── malware-evil.com (malicious) → Block → แสดง Block Page
         └── social-media.com (policy) → Block → เฉพาะเวลางาน
```

**บริการ DNS Filtering:**
- **Cisco Umbrella (OpenDNS):** Block Malicious Domains + Content Filtering
- **Cloudflare Gateway:** DNS Filtering + Zero Trust
- **Quad9 (9.9.9.9):** Block Malicious Domains (ฟรี)
- **AdGuard DNS:** Block โฆษณา + Malware + Tracking
- **Self-hosted:** Pi-hole (Open Source), BIND + RPZ

### 4.2 Pi-hole — DNS Sinkhole ตัวอย่าง

Pi-hole เป็น DNS Sinkhole ที่ Block Domain ที่ไม่พึงประสงค์ (Ad, Tracker, Malware):

```
Pi-hole Architecture:
─────────────────────
[Client] ──▶ [Pi-hole (DNS Sinkhole)]
               │
          ตรวจสอบ Domain กับ Blocklist:
          ├── safe.com → Forward to Upstream DNS (8.8.8.8)
          └── ads.com  → Return 0.0.0.0 (Null Route)
```

```bash
# Block Malicious Domains บน Pi-hole
pihole -b malware-domain.com    # เพิ่ม Domain ใน Blacklist
pihole -g                        # Update Gravity (Blocklist)
pihole status                    # ดูสถิติ

# Sample Pi-hole Statistics:
# Total queries: 245,892
# Blocked: 34,781 (14.1%)
# Top blocked: doubleclick.net, googleadservices.com, facebook.com/tr
```

### 4.3 DNS Sinkhole ในการตอบสนองต่อเหตุการณ์

ในการตอบสนองต่อเหตุการณ์ (Incident Response) — DNS Sinkhole ใช้เปลี่ยน DNS Resolution สำหรับ Malicious Domain เป็น Internal Server เพื่อ:

1. **ป้องกัน Malware ติดต่อ C2** — โดยไม่ต้อง Block ที่ Firewall
2. **ตรวจจับเครื่องที่ติด Malware** — DNS Query ที่ไปยัง Sinkhole บ่งบอกว่าเครื่องนั้นติด Malware
3. **วิเคราะห์ Malware Behavior** — ดูว่า Malware พยายามติดต่อ Domain ใดบ้าง

```
Response Policy Zone (RPZ) — BIND:
─────────────────────────────────────
zone "rpz.local" {
    type master;
    file "/etc/bind/db.rpz";
    allow-query { localhost; };
};

# /etc/bind/db.rpz — แก้ไข DNS Response สำหรับ Malicious Domains
$TTL 86400
@   IN  SOA localhost. admin.localhost. (1 3h 1h 1w 1h)
    IN  NS  localhost.

malware-c2.evil.com       CNAME .       # NXDOMAIN — ทำให้ DNS Resolution ล้มเหลว
malware-c2.evil.com       A    10.0.0.2 # หรือ Redirect ไปยัง Sinkhole Server (วิเคราะห์ Traffic)
*.dynamic-dns.org         CNAME .       # Block Dynamic DNS ทั้งหมด
```

## 5. DNS Security Best Practices

```
องค์กรควรปฏิบัติ:
──────────────────
1. ใช้ DNSSEC Validation — ทุก DNS Resolver ในองค์กร
2. ใช้ DNS Filtering — Block Malicious Domains
3. เปิด Response Rate Limiting (RRL) — ป้องกัน DNS Amplification
4. ปิด Recursive Query บน Authoritative DNS Server
5. ใช้ DNS over HTTPS/TLS (DoH/DoT) — ป้องกัน DNS Spoofing
6. ตรวจสอบ DNS Log — ตรวจจับ DNS Tunneling และ Beaconing
7. ใช้ Split-horizon DNS — Internal และ External DNS แยกกัน
8. ใช้ RPZ (Response Policy Zone) — สำหรับ Incident Response
9. อัปเดต BIND/Unbound/PowerDNS อย่างสม่ำเสมอ
10. Monitor Query Rate — ตรวจจับ DNS Anomaly
```

# ส่วนที่ 2: Email Security

## 6. พื้นฐานความปลอดภัยของอีเมล

### 6.1 ความสำคัญของ Email Security

- อีเมลเป็นช่องทางหลักในการโจมตี — มากกว่า 90% ของ Cyber Attack เริ่มต้นจากอีเมล (Verizon DBIR 2023)
- **Business Email Compromise (BEC)** ทำให้เกิดความเสียหายมากกว่า $50 พันล้านตั้งแต่ปี 2013 (FBI IC3)
- อีเมลเป็นเป้าหมายของการโจมตีแบบ Phishing, Spear-Phishing, Whaling, Ransomware, และ Malware Distribution

### 6.2 โพรโทคอลอีเมลพื้นฐาน

```
Email Delivery Path:
─────────────────────
[Sender] ──▶ [MUA] ──▶ [MTA (Outbound)] ──▶ [Internet]
              │                                  │
              │ SMTP (Submission)                 │ SMTP (Relay)
              │ Port 587 (STARTTLS)               │
              │ Port 465 (SMTPS)                  │
              ▼                                   ▼
[Recipient] ◀── [MUA] ◀── [MTA (Inbound)] ◀──────┘
              │            │
              │ POP3/IMAP   │
              │ Port 995    │ Spam Filter
              │ Port 993    │ Virus Scan
              │             │ DLP
```

**โพรโทคอลที่เกี่ยวข้อง:**
- **SMTP (Simple Mail Transfer Protocol):** ส่งอีเมล (Port 25, 465, 587)
- **POP3 (Post Office Protocol 3):** ดาวน์โหลดอีเมล (Port 110, 995)
- **IMAP (Internet Message Access Protocol):** จัดการอีเมลบน Server (Port 143, 993)

## 7. SPF, DKIM, DMARC

### 7.1 SPF (Sender Policy Framework)

SPF คือกลไกตรวจสอบว่าเซิร์ฟเวอร์ที่ส่งอีเมลนั้น **ได้รับอนุญาตให้ส่งอีเมลในนามของโดเมนนั้น** หรือไม่

**การทำงาน:**
1. เจ้าของโดเมนประกาศรายชื่อ IP Address ที่ได้รับอนุญาตให้ส่งอีเมล ในรูปแบบ TXT Record
2. เซิร์ฟเวอร์ผู้รับตรวจสอบ SPF Record ของโดเมนผู้ส่ง — ว่า Sending IP อยู่ในรายการที่อนุญาตหรือไม่

```
SPF Record:
───────────
example.com  TXT  "v=spf1 ip4:203.0.113.0/24 include:_spf.google.com ~all"

v=spf1                  → SPF Version 1
ip4:203.0.113.0/24      → IP 203.0.113.0-255 ได้รับอนุญาต
include:_spf.google.com → Google Workspace IPs ได้รับอนุญาต
~all                    → SoftFail (ถือว่าสงสัย) / -all = HardFail (ปฏิเสธ)
```

**SPF Mechanisms:**
| Mechanism | คำอธิบาย |
|:----------|:---------|
| `all` | ตรงกับ Source IP ทุกตัว |
| `ip4:192.0.2.0/24` | ตรงกับ IP Range |
| `ip6:2001:db8::/32` | ตรงกับ IPv6 Range |
| `a` | IP ของ A Record ของโดเมน |
| `mx` | IP ของ MX Record ของโดเมน |
| `include:domain.com` | รวม SPF Record จากอีกโดเมน |
| `exists:%{i}._spf.domain.com` | Macro — Query DNS แบบ Dynamic |

**SPF Qualifiers:**
| Qualifier | ผลลัพธ์ | ความหมาย |
|:----------|:-------|:---------|
| `+` (default) | Pass | อนุญาต |
| `-` | Fail | ปฏิเสธ |
| `~` | SoftFail | ยอมรับแต่อาจ Mark เป็น Spam |
| `?` | Neutral | ไม่มีการตัดสิน (Neutral) |

### 7.2 DKIM (DomainKeys Identified Mail)

DKIM ใช้ **Digital Signature** เพื่อยืนยันว่าอีเมลไม่ได้ถูกแก้ไขระหว่างทาง (Integrity) และมาจากโดเมนที่อ้างจริง (Authentication)

**การทำงาน:**
1. เซิร์ฟเวอร์ผู้ส่งเซ็น Header (From, Subject, Date, Message-ID) และ Body ด้วย Private Key
2. Signature ถูกเพิ่มใน DKIM-Signature Header
3. เซิร์ฟเวอร์ผู้รับดึง Public Key จาก DNS (selector._domainkey.example.com) และตรวจสอบ Signature

```
DKIM Signature ใน Email Header:
────────────────────────────────
DKIM-Signature: v=1; a=rsa-sha256; c=relaxed/relaxed;
                d=example.com; s=google;
                h=from:to:subject:date:message-id;
                bh=47DEQpj8HBSa+/TImW+5JCeuQeRkm5NMpJWZG3hSuFU=;
                b=Au3wQkLmN8p8Hfl9aXo7Yz3Rx... (Digital Signature)

v=1             → DKIM Version
a=rsa-sha256    → อัลกอริทึม (RSA + SHA-256)
c=relaxed/relaxed → Canonicalization (Header/Body)
d=example.com   → Domain
s=google        → Selector (Public Key Location)
h=...           → Headers ที่เซ็น
bh=...          → Body Hash
b=...           → Digital Signature
```

**DKIM Public Key ใน DNS:**
```
google._domainkey.example.com  TXT  "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQC..."

v=DKIM1   → DKIM Version
k=rsa     → Key Type
p=...     → Public Key (Base64)
```

### 7.3 DMARC (Domain-based Message Authentication, Reporting & Conformance)

DMARC บอกผู้รับอีเมลว่าควรทำอย่างไร **เมื่ออีเมลไม่ผ่าน SPF และ/หรือ DKIM**:

**การทำงาน:**
1. DMARC Policy ถูกประกาศใน DNS TXT Record ของโดเมน
2. ผู้รับตรวจสอบว่า From Header (RFC5322.From) ตรงกับ Domain ที่ตรวจสอบ SPF/DKIM หรือไม่ (Alignment)
3. ถ้าไม่ผ่าน — ทำตาม Policy (none / quarantine / reject)
4. ส่งรายงาน (Aggregate Report, Forensic Report) กลับไปยังเจ้าของโดเมน

```
DMARC Record:
─────────────
_dmarc.example.com  TXT  "v=DMARC1; p=reject; rua=mailto:dmarc@example.com; ruf=mailto:dmarc-forensic@example.com; pct=100; sp=quarantine; adkim=s; aspf=s"

v=DMARC1                     → DMARC Version
p=reject                     → นโยบายหลัก (reject/quarantine/none)
pct=100                      → ใช้กับ 100% ของอีเมล
rua=mailto:dmarc@example.com → ส่ง Aggregate Report ทุกวัน (XML)
ruf=mailto:dmarc-forensic@example.com → ส่ง Forensic Report รายฉบับ
sp=quarantine                → นโยบายสำหรับ Subdomain
adkim=s                      → DKIM Alignment Mode (s=strict, r=relaxed)
aspf=s                       → SPF Alignment Mode (s=strict, r=relaxed)
```

**DMARC Policy:**
| Policy | คำอธิบาย |
|:-------|:---------|
| `p=none` | ตรวจสอบแต่ไม่ทำอะไร — ใช้เก็บรายงาน (Monitoring Mode) |
| `p=quarantine` | Mark อีเมลเป็น Spam (ส่งไปยัง Spam Folder) |
| `p=reject` | ปฏิเสธอีเมล — ไม่ส่งถึงผู้รับ (แนะนำ) |

### 7.4 BIMAI (Brand Indicators for Message Identification)

BIMAI แสดง Logo ขององค์กรใน Email Client (Gmail, Yahoo, Fastmail) — เพิ่มความเชื่อมั่นและลด Phishing:

```
BIMAI Record:
─────────────
default._bimi.example.com  TXT  "v=BIMI1; l=https://example.com/logo.svg; a=https://example.com/logo-vmc.pem"

v=BIMI1               → BIMI Version
l=https://...logo.svg → Logo URL (SVG)
a=https://...vmc.pem  → VMC (Verified Mark Certificate) — รับรอง Logo
```

**ข้อกำหนดของ BIMAI:**
1. ต้องมี **DMARC p=quarantine หรือ p=reject** อย่างน้อย
2. Logo ต้องเป็น SVG ที่ผ่านการตรวจ (VMC)
3. Email Client ต้องรองรับ BIMAI

### 7.5 MTA-STS (SMTP MTA Strict Transport Security)

MTA-STS บังคับให้ใช้ TLS สำหรับ SMTP Connection — ป้องกันการโจมตี Downgrade Attack:

```
_mta-sts.example.com  TXT  "v=STSv1; id=20240501;"

https://mta-sts.example.com/.well-known/mta-sts.txt
─────────────────────────────────────────────────────
version: STSv1
mode: enforce
mx: mail.example.com
mx: mail2.example.com
max_age: 86400
```

### 7.6 ตัวอย่างการตั้งค่าครบชุด

```
# DNS Records สำหรับ Email Security (example.com)

# SPF
example.com          TXT  "v=spf1 ip4:203.0.113.0/24 include:_spf.google.com -all"

# DKIM
google._domainkey    TXT  "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQC..."

# DMARC
_dmarc               TXT  "v=DMARC1; p=reject; rua=mailto:dmarc@example.com; ruf=mailto:dmarc-forensic@example.com; pct=100"

# BIMI

# MTA-STS
```

## 8. การป้องกัน Phishing และ Spear-Phishing ทางอีเมล

### 8.1 ประเภทของ Phishing

| ประเภท | คำอธิบาย | ตัวอย่าง |
|:-------|:---------|:---------|
| **Mass Phishing** | ส่งอีเมลหลอกจำนวนมาก — ไม่ระบุเป้าหมาย | "ธนาคารของคุณถูกระงับ — กรุณาคลิกที่นี่" |
| **Spear Phishing** | ระบุเป้าหมายเป็นรายบุคคล — มีข้อมูลส่วนตัว | "สวัสดีคุณสมชาย, เราได้รับแจ้ง..." |
| **Whaling** | เจาะจงผู้บริหารระดับสูง (CEO, CFO) | "CEO ต้องการโอนเงินด่วน — ทำรายการตอนนี้" |
| **CEO Fraud (BEC)** | ปลอมตัวเป็นผู้บริหาร | ปลอมอีเมล CEO สั่ง CFO โอนเงิน |
| **Clone Phishing** | คัดลอกอีเมลจริงแล้วแก้ Link | แก้ Invoice ที่ส่งมาก่อนหน้านี้ |
| **Smishing** | Phishing ผ่าน SMS | "คุณได้รับพัสดุ — คลิก https://bit.ly/..." |
| **Vishing** | Phishing ผ่านโทรศัพท์ | "โทรจากธนาคาร — ขอยืนยัน OTP" |

### 8.2 Business Email Compromise (BEC) / CEO Fraud

BEC เป็นการโจมตีที่สร้างความเสียหายสูงที่สุดในโลกของ Cyber Crime — FBI รายงานความเสียหายมากกว่า $50 พันล้าน (2013-2023)

**รูปแบบการโจมตี BEC:**
```
Scenario: CEO Fraud
────────────────────
1. ผู้โจมตีปลอมอีเมล CEO (ใช้ Display Name เหมือน — ceo@examp1e.com vs ceo@example.com)
2. ส่งถึง CFO — "กรุณาโอนเงิน $50,000 ไปยังบัญชีนี้สำหรับการชำระเงิน Supplier ด่วน"
3. CFO เชื่อว่าเป็น CEO (ชื่อเหมือน, Signature เหมือน)
4. CFO โอนเงิน — เงินไปยังบัญชีปลายทาง

การป้องกัน:
├── DMARC p=reject — ป้องกัน Domain Spoofing
├── Sender Policy — ระบุ IP ที่ได้รับอนุญาต
├── Multi-factor Authentication สำหรับการโอนเงิน
├── Dual Approval — การโอนเงินทุกครั้งต้องมี 2 คนอนุมัติ
└── Call-back Verification — โทรยืนยันก่อนโอนเงิน
```

### 8.3 เทคนิคที่ Phishing ใช้

| เทคนิค | คำอธิบาย |
|:--------|:---------|
| **Display Name Spoofing** | ตั้งชื่อผู้ส่งให้เหมือน CEO (แต่ Email จริงคือ attacker@evil.com) |
| **Domain TypoSquatting** | ใช้ Domain ที่สะกดคล้าย: examp1e.com, g00gle.com, rnicrosoft.com |
| **Homograph Attack** | ใช้ตัวอักษรที่เหมือนกันจากภาษาต่างๆ: example.com (ใช้ Cyrillic 'е') |
| **URL Obfuscation** | ซ่อน URL จริงด้วย Link Shortener หรือ Anchor Text |
| **Social Engineering** | สร้างความเร่งด่วน, ความกลัว, หรือความโลภ |
| **Attachment** | แนบไฟล์ Malicious (PDF, Word, Excel, ZIP) |
| **Brand Impersonation** | ปลอมหน้า Login ของธนาคาร, Microsoft, Google, PayPal |

### 8.4 แนวทางการป้องกัน Phishing

```
แนวทางป้องกัน Phishing:
────────────────────────-

เชิงเทคนิค:
├── SPF/DKIM/DMARC (p=reject) — ป้องกัน Domain Spoofing
├── Email Gateway Security — Anti-spam, Anti-malware, URL Protection
├── DMARC Reporting — ตรวจสอบอีเมลที่ปลอมแปลง
├── URL Sandboxing — คลิก URL ในอีเมล ต้องเข้า Sandbox ก่อน
├── Attachment Sandboxing — ไฟล์แนบถูกเปิดใน Sandbox
├── Sender Reputation — ตรวจสอบ Sender Reputation (Talos, Spamhaus)
├── Banner Warning — เพิ่ม Banner ในอีเมลจากภายนอก
└── MFA — Multi-factor Authentication ทุกบัญชี

เชิงนโยบาย:
├── Security Awareness Training — ฝึกพนักงานทุกคน
├── Phishing Simulation — ทดสอบพนักงานด้วยอีเมลทดสอบ
├── Reporting Button — ให้พนักงานรายงานอีเมลต้องสงสัย
├── Incident Response — Process การรับมือเมื่อพนักงานคลิก
├── Dual Approval — การโอนเงินต้อง 2 คน
└── Call-back Verification — โทรยืนยันการขอโอนเงิน
```

## 9. Email Gateway Security

### 9.1 บทบาทของ Email Gateway

Email Gateway คือระบบที่ทำหน้าที่ตรวจสอบอีเมลทุกฉบับก่อนถึงผู้รับ:

```
Email Gateway:
───────────────
[Internet] ──▶ [Email Gateway] ──▶ [Mail Server] ──▶ [User]
                    │
               การตรวจสอบ:
               ├── Spam Filtering (Bayesian, ML)
               ├── Virus/Malware Scan (ClamAV, Sophos)
               ├── URL Reputation Check
               ├── Attachment Sandboxing
               ├── DLP (Data Loss Prevention)
               ├── SPF/DKIM/DMARC Validation
               └── Policy Enforcement
```

### 9.2 ฟังก์ชันของ Email Gateway

**1. Anti-spam:**
- Bayesian Filtering — วิเคราะห์ Content
- ML-based Classification — TensorFlow, Neural Network
- Sender Reputation — DNSBL (Spamhaus, Barracuda)
- Greylisting — ปฏิเสธอีเมลครั้งแรก ให้ลองส่งใหม่
- Rate Limiting — จำกัดจำนวนอีเมลจาก Sender เดียว

**2. Anti-malware:**
- Signature-based (ClamAV, Sophos)
- Heuristic Analysis — ตรวจจับพฤติกรรมน่าสงสัย
- Attachment Analysis — ตรวจสอบ Macro, JavaScript, Exploit
- Sandbox Execution — เปิดไฟล์ในสภาพแวดล้อมจำลอง

**3. Anti-phishing:**
- SPF/DKIM/DMARC Validation
- URL Analysis — ตรวจสอบ URL กับ Threat Intelligence Feed
- Display Name Analysis — ตรวจจับ Display Name Spoofing
- Lookalike Domain Detection — examp1e.com, micros0ft.com

**4. DLP (Data Loss Prevention):**
- ตรวจจับข้อมูลอ่อนไหว (บัตรเครดิต, เลขบัตรประชาชน, PDPA Data)
- Encryption — เข้ารหัสอีเมลที่มีข้อมูลอ่อนไหว
- Policy-based Routing — ส่งอีเมลสำคัญผ่าน TLS เสมอ
- Attachment Control — Block การส่งไฟล์บางประเภท (.exe, .zip, .scr)

### 9.3 การตั้งค่า Email Gateway (ตัวอย่าง — Postfix + SpamAssassin + ClamAV)

```bash
# /etc/postfix/main.cf — Postfix Configuration
smtpd_recipient_restrictions =
    permit_mynetworks
    reject_unauth_destination
    check_policy_service unix:private/policy-spf  # SPF Check
    reject_rbl_client zen.spamhaus.org             # DNSBL
    reject_rbl_client bl.spamcop.net

# SpamAssassin Integration
smtpd_data_restrictions =
    check_policy_service inet:127.0.0.1:10023

# Content Filter — Amavis (SpamAssassin + ClamAV)
content_filter = smtp-amavis:[127.0.0.1]:10024
```

```bash
# /etc/spamassassin/local.cf — SpamAssassin Configuration
required_score 5.0      # คะแนนที่ถือว่าเป็น Spam
report_safe 0           # ส่ง Original Message + Headers
rewrite_header Subject [SPAM]

# RBL Checks
use_razor2 1
use_pyzor 1
use_dcc 1

# Custom Rules
header LOCAL_SPF_FAIL Authentication-Results: =~ /spf=fail/
score LOCAL_SPF_FAIL 3.0
```

## 10. กรณีศึกษา: การโจมตีผ่าน DNS และอีเมล

### 10.1 กรณีศึกษา 1: DNS Spoofing — การโจมตีระบบ DNS ขององค์กร (2019)

**เหตุการณ์:** DNS Cache Poisoning โจมตี Crypto Wallet Provider
- ผู้โจมตีทำ DNS Spoofing — ทำให้ DNS Resolver ของผู้ใช้คืน IP ปลอม
- ผู้ใช้เปิด URL ของ Crypto Wallet → ถูกส่งไปยังเว็บปลอม
- กรอก Wallet Seed Phrase → เงินถูกขโมยไป $150,000

**บทเรียน:**
1. DNSSEC สามารถป้องกัน DNS Spoofing ได้
2. ควรใช้ DNSSEC-validating Resolver
3. ผู้ใช้ควรตรวจสอบ Certificate (HTTPS) ก่อนกรอกข้อมูลสำคัญ

### 10.2 กรณีศึกษา 2: DNS Tunneling — APT29 (Cozy Bear) — SolarWinds

**เหตุการณ์:** APT29 ใช้ DNS Tunneling สำหรับ C2 Communication ในการโจมตี SolarWinds
- Malware สร้าง DNS Query ไปยัง Subdomain แบบสุ่ม (DGA)
- DNS Query ไปยัง Domain ที่ attacker ควบคุม
- Response ถูกซ่อนใน TXT Record (Command + Data Exfiltration)
- ตรวจพบยากเพราะ DNS Traffic ปริมาณไม่มาก

**บทเรียน:**
1. DNS Traffic Analysis สำคัญ — ตรวจสอบ DNS Query ที่ผิดปกติ
2. DNS Query Rate Monitoring — ตรวจจับ Beaconing
3. DNS Sinkhole + TI (Threat Intelligence) Feed
4. ใช้ DNS over HTTPS (DoH) เพื่อเข้ารหัส DNS Traffic

### 10.3 กรณีศึกษา 3: BEC CEO Fraud — $2.9 Billion Loss

**เหตุการณ์:** การโจมตี BEC ที่สร้างความเสียหายทั่วโลกมากกว่า $2.9 พันล้าน (FBI IC3 2023)
- ผู้โจมตีปลอมเป็น CEO ส่งอีเมลไปยัง CFO
- ใช้ Display Name Spoofing — ชื่อเหมือน CEO ทุกประการ
- สร้างความเร่งด่วน — "ต้องโอนเงิน Supplier ภายใน 1 ชม."
- CFO โอนเงิน $50,000 ไปยังบัญชีปลายทาง (ในไทย, จีน, รัสเซีย)

**บทเรียน:**
1. DMARC p=reject — ป้องกัน Domain Spoofing
2. MFA สำหรับทุกการทำธุรกรรม
3. Dual Approval — การโอนเงินต้องมี 2 คน
4. Security Awareness — CEO ต้องไม่ขอให้โอนเงินผ่านอีเมล
5. Impersonation Protection — บริการอย่าง Proofpoint, Mimecast

### 10.4 กรณีศึกษา 4: การโจมตีแบบ Supply Chain — SolarWinds Orion

**เหตุการณ์:** SolarWinds Orion (2020) — Nation-State Attack
- ผู้โจมตีแทรก Malicious Code ในอัปเดตซอฟต์แวร์ Orion
- Malware ใช้ DNS สำหรับ C2 — DNS Tunneling + DGA
- ส่งผลกระทบต่อองค์กร 18,000+ แห่งทั่วโลก
- ใช้เวลา 7-9 เดือนกว่าจะตรวจพบ

**บทเรียน:**
1. Supply Chain Risk — ตรวจสอบซอฟต์แวร์ Third-party
2. DNS Monitoring — NSRecord, DNS Query Analysis
3. Zero Trust Architecture — ตรวจสอบทุก Traffic ทุกระบบ

## สรุปท้ายบท (Chapter Summary)

1. **DNS Security:** DNS เป็นพื้นฐานของโครงสร้างพื้นฐานอินเทอร์เน็ต — ถูกโจมตีได้หลายวิธี เช่น Spoofing, Tunneling, Amplification DDoS

2. **DNSSEC** เพิ่ม Digital Signature ใน DNS Response เพื่อยืนยันความถูกต้องและป้องกันการปลอมแปลง — ใช้ Chain of Trust จาก Root Zone ถึง Domain

3. **DNS Filtering และ DNS Sinkhole** กรองและ Block DNS Query ไปยัง Domain ที่เป็นอันตราย — ใช้ในองค์กรเพื่อป้องกัน Malware C2 และ Content Filtering

4. **Email Authentication สามชั้น (SPF, DKIM, DMARC):**
   - **SPF:** ตรวจสอบ IP ที่ได้รับอนุญาตให้ส่งอีเมล
   - **DKIM:** ใช้ Digital Signature ยืนยันความถูกต้องของอีเมล
   - **DMARC:** กำหนดนโยบายเมื่อ SPF/DKIM ไม่ผ่าน

5. **Phishing** เป็นภัยคุกคามอันดับหนึ่งทางอีเมล — ต้องป้องกันทั้งเชิงเทคนิค (Email Gateway, DMARC) และเชิงนโยบาย (Training, Simulation, MFA)

6. **Email Gateway Security** ประกอบด้วย Anti-spam, Anti-malware, DLP, URL Analysis, และ Sandboxing

7. **BEC (CEO Fraud)** สร้างความเสียหายสูงที่สุด — ป้องกันด้วย DMARC, Dual Approval, Call-back Verification, และ Security Awareness

## คำถามทบทวน (Review Questions)

1. จงอธิบายการโจมตี DNS Spoofing (DNS Cache Poisoning) และ DNSSEC สามารถป้องกันได้อย่างไร

2. DNS Tunneling คืออะไร และเหตุใดจึงเป็นอันตรายต่อองค์กร? จงอธิบายวิธีการตรวจจับและป้องกัน

3. DNS Amplification DDoS ทำงานอย่างไร? จงอธิบาย Amplification Factor และแนวทางการป้องกัน

4. จงอธิบาย Chain of Trust ของ DNSSEC — จาก Root Zone ถึง example.com — พร้อมอธิบายบทบาทของ RRSIG, DNSKEY, และ DS Record

5. SPF, DKIM, และ DMARC ทำงานร่วมกันอย่างไร? จงยกตัวอย่างการโจมตีที่ป้องกันได้ด้วย DMARC p=reject

6. DMARC Alignment คืออะไร? จงอธิบายความแตกต่างระหว่าง Strict และ Relaxed Alignment สำหรับ SPF และ DKIM

7. Business Email Compromise (BEC) / CEO Fraud คืออะไร? จงอธิบายวิธีป้องกัน 3 วิธีที่ต่างกัน

8. Email Gateway มีฟังก์ชันอะไรบ้างในการป้องกัน Phishing? จงอธิบาย Anti-spam, Anti-malware, Sandboxing, และ DLP

9. จากกรณีศึกษา SolarWinds (DNS Tunneling + Supply Chain Attack) จงวิเคราะห์ว่ามาตรการป้องกันใดที่องค์กรทั่วไปสามารถนำไปใช้ได้

10. จงเขียน DNS Records ครบชุด (SPF, DKIM, DMARC, BIMI, MTA-STS) สำหรับโดเมน mycompany.com ที่ใช้ Google Workspace และมี Mail Server อยู่ที่ IP 198.51.100.0/24
