4.1 ภาพรวมของ Threat Modeling และ Risk Analysis

Threat Modeling เป็นกิจกรรมสำคัญของ Secure SDLC ที่ควรเกิดตั้งแต่ขั้นตอน Requirements และ Design ช่วยให้ทีมมองเห็นปัญหาเชิงสถาปัตยกรรมตั้งแต่ต้น เช่น การไม่ตรวจสิทธิ์ระหว่างบริการ การส่งข้อมูลสำคัญผ่านช่องทางที่ไม่เข้ารหัส หรือการออกแบบที่ทำให้ผู้ใช้ทั่วไปเข้าถึงฟังก์ชันของผู้ดูแลระบบได้

OWASP อธิบาย Threat Modeling ผ่านคำถามหลัก 4 ข้อ:

Interactive: OWASP 4 คำถามของ Threat Modeling

คลิกแต่ละคำถามเพื่อดูคำอธิบาย

1. เรากำลังสร้างอะไร (What are we building?)

การเข้าใจระบบ ขอบเขต สินทรัพย์ สถาปัตยกรรม และผู้เกี่ยวข้อง ก่อนวิเคราะห์ภัยคุกคาม ผลลัพธ์ที่ได้: System Scope, DFD, Asset List

2. อะไรอาจผิดพลาดได้ (What can go wrong?)

การค้นหาภัยคุกคาม Abuse Case และ Attack Vector ผ่านกรอบเช่น STRIDE, Attack Tree, Kill Chain ผลลัพธ์ที่ได้: Threat List, STRIDE Analysis, Attack Tree

3. เราจะจัดการกับสิ่งที่ผิดพลาดอย่างไร (What are we going to do about it?)

การเลือก Risk Treatment และ Security Controls เพื่อลดความเสี่ยง เช่น MFA, RBAC, Prepared Statement ผลลัพธ์ที่ได้: Risk Register, Security Requirements

4. เราตรวจสอบได้อย่างไรว่าทำงานถูกต้อง (Did we do a good job?)

การตรวจสอบว่ามาตรการที่เลือกใช้ได้ผลจริง ผ่าน Verification Plan, Test Cases, Security Review ผลลัพธ์ที่ได้: Test Cases, Verification Plan, Monitoring

คำถาม 4 ข้อนี้ทำให้ Threat Modeling เป็นกระบวนการคิดอย่างเป็นระบบ ตั้งแต่เข้าใจระบบ ค้นหาภัยคุกคาม ตัดสินใจลดความเสี่ยง และตรวจสอบว่ามาตรการได้ผล

4.1.1 ทำไม Threat Modeling จึงสำคัญต่อ Software Security

ซอฟต์แวร์จำนวนมากล้มเหลวเพราะออกแบบผิดตั้งแต่แรก เช่น API ให้ Client ส่ง user_id มาหรือไม่ตรวจสอบ, Microservices เชื่อใจกันเองภายใน, Payment Callback ไม่ตรวจ Digital Signature ปัญหาเหล่านี้เป็น design flaw ที่ unit test ไม่พบ

ปัญหาเชิงออกแบบสิ่งที่อาจเกิดขึ้นThreat Modeling ช่วยถาม
Client ส่งราคาเองผู้ใช้แก้ราคาก่อนชำระเงินข้อมูลใดไม่ควรเชื่อจาก client
Service ภายในไม่ยืนยันตัวตนผู้โจมตี pivot ในระบบจุดข้าม trust boundary
Admin API ซ่อนด้วย UIผู้ใช้เรียก API โดยตรงAuthorization อยู่ backend หรือไม่
Webhook ไม่ตรวจ signatureปลอมสถานะชำระเงินสำเร็จเชื่อ third-party อย่างไร
Q

คำถาม: หากระบบผ่าน test ทั้งหมดแต่ไม่ตรวจ object-level authorization ระบบ "ทำงานถูกต้อง" หรือ "ปลอดภัย" เพราะเหตุใด

4.1.2 ผลลัพธ์ที่ควรได้จาก Threat Modeling

ผลลัพธ์ใช้ทำอะไรต่อ

Asset Listจัดลำดับสิ่งที่ต้องปกป้อง
DFDค้นหา attack surface และ trust boundary
Threat Listสร้าง security requirement และ risk register

ข้อควรจำ: Threat Model ที่ดีต้องนำไปปฏิบัติได้ ถ้าไม่ทำให้เกิด Security Requirements, Design Changes, Test Cases แสดงว่ายังไม่เกิดคุณค่าจริง

4.2.1 ตัวอย่างการแยกคำศัพท์

ระบบ /api/bookings/{booking_id} แบบ ID เรียงลำดับ
  • Asset: ข้อมูลการจองและข้อมูลผู้ใช้
  • Threat: ผู้ใช้ทั่วไปเดา booking_id ของผู้อื่น
  • Vulnerability: ไม่มี Object-level Authorization Check
  • Impact: ข้อมูลส่วนบุคคลและประวัติการจองรั่วไหล
  • Likelihood: สูง เพราะ ID เดาง่ายและ endpoint เปิดให้เรียก
  • Control: ตรวจ owner, ใช้ UUID, เพิ่ม Audit Log

4.3 ขั้นตอนการทำ Threat Modeling

กระบวนการ 7 ขั้นที่เหมาะกับทีมพัฒนาและห้องเรียน:

Interactive: 7 ขั้นตอน

คลิกแต่ละขั้นเพื่อดูรายละเอียด

1. กำหนด Scope

ขอบเขตการวิเคราะห์ ครอบคลุมอะไร ไม่ครอบคลุมอะไร ระบุผู้ใช้ ข้อมูลสำคัญ และข้อจำกัด

2. ระบุ Asset

ข้อมูล บริการ กระบวนการที่ต้องปกป้อง แต่ละ Asset มี Security Objective เช่น Confidentiality, Integrity, Availability

3. สร้าง DFD

แผนภาพการไหลของข้อมูล เริ่มจาก Context Diagram แล้วแตก Level 1

4. Trust Boundary

จุดข้ามเขตความไว้วางใจ ต้องมี Authentication, Authorization, Validation, Encryption

5. STRIDE

ตั้งคำถามครบ 6 มุม: Spoofing, Tampering, Repudiation, Info Disclosure, DoS, EoP

6. จัดลำดับ Risk

ใช้ Likelihood + Impact ใน Risk Matrix บันทึก Risk Register พร้อมเจ้าของ

7. Control

เลือก Treatment (Mitigate/Transfer/Accept/Avoid) ระบุ Owner Due Date และ Residual Risk

4.3.1 กำหนด Scope และ Business Context

Scope ระบบ E-Commerce
  • ครอบคลุม: สมัครสมาชิก, เข้าสู่ระบบ, ตะกร้าสินค้า, ชำระเงิน, ประวัติคำสั่งซื้อ
  • ไม่ครอบคลุม: ระบบคลังสินค้าภายใน, ระบบบัญชีหลังบ้าน, ระบบ CRM
  • ผู้ใช้หลัก: ลูกค้า, ผู้ดูแลร้าน, Payment Gateway, Email Service
  • ข้อมูลสำคัญ: ข้อมูลส่วนบุคคล, ที่อยู่จัดส่ง, Order History, Payment Token
  • ข้อจำกัด: ไม่เก็บเลขบัตรเครดิตเต็ม ใช้ Token จาก Payment Gateway

4.3.3 ระบุ Assumption และ Constraint

Assumption: สิ่งที่ทีมเชื่อว่าเป็นจริง เช่น Database ไม่เปิดสู่อินเทอร์เน็ต, Admin ใช้ MFA, HTTPS เสมอ

Constraint: ข้อจำกัดที่มีผลต่อการเลือก Control เช่น เวลา 2 Sprint, รองรับ 10,000 คน, ปฏิบัติตาม PDPA

A

ตัวอย่าง Assumption ที่อันตราย: ทีมเชื่อว่า API ถูกเรียกผ่าน Frontend เท่านั้น จึงไม่ตรวจ Authorization ที่ Backend แต่ผู้โจมตีเรียก API โดยตรงได้

DFD ช่วยให้ทีมเห็นว่าใครส่งข้อมูลอะไร ไปที่ไหน ผ่านกระบวนการใด และข้อมูลถูกเก็บตรงไหน

องค์ประกอบตัวอย่าง

Trust Boundary ที่ควรระบุ:

  1. Customer Browser ↔ Web Frontend: จากอุปกรณ์ผู้ใช้เข้าสู่ระบบองค์กร
  2. Web Frontend ↔ Backend API: Presentation Layer ไป Application Layer
  3. Backend API ↔ Database: Application Layer ไป Data Layer
  4. Backend API ↔ Payment Gateway: องค์กรไป Third-party

วิธีอ่าน DFD เพื่อหาภัยคุกคาม: ถามซ้ำทุก data flow: ผู้ส่งเป็นใครและพิสูจน์ตัวตนอย่างไร มีสิทธิ์หรือไม่ ข้อมูลถูกแก้ไขระหว่างทางได้หรือไม่ ถูกเปิดเผยใน response/log หรือไม่ ปลายทางตรวจ input หรือเชื่อทันที มีหลักฐานตรวจสอบย้อนหลังหรือไม่

ข้อผิดพลาด DFDผลกระทบวิธีแก้
วาดเฉพาะ UIมองไม่เห็น API/DBวาด Process และ Data Store
ไม่ใส่ Third-partyมองข้าม Supply Chainใส่ Payment, Email, Analytics, CDN
ไม่ระบุ Trust Boundaryไม่รู้จุดตรวจสิทธิ์วาดเส้นแบ่งความไว้วางใจ
ไม่ระบุทิศทางข้อมูลวิเคราะห์ยากใส่ลูกศรทุกเส้น

4.5 STRIDE Framework

STRIDE แบ่งภัยคุกคามเป็น 6 หมวด:

ตัวภัยคุกคามกระทบคำถามสำคัญ
SSpoofingAuthenticationใครปลอมตัวเป็นผู้ใช้หรือบริการอื่นได้
TTamperingIntegrityข้อมูลถูกแก้ไขโดยไม่ได้รับอนุญาตได้หรือไม่
RRepudiationNon-repudiationผู้ใช้ปฏิเสธการกระทำเพราะไม่มีหลักฐานได้หรือไม่
IInformation DisclosureConfidentialityข้อมูลรั่วสู่ผู้ไม่มีสิทธิ์ได้หรือไม่
DDenial of ServiceAvailabilityระบบถูกทำให้ใช้การไม่ได้หรือไม่
EElevation of PrivilegeAuthorizationผู้ใช้ได้สิทธิ์สูงกว่าที่ควรหรือไม่

Interactive: STRIDE Explorer

คลิกแต่ละหมวดเพื่อดูตัวอย่างและ Control

SSpoofing — การปลอมตัวตน

ตัวอย่าง: ขโมย Session Token, ปลอม API Key, Phishing, ไม่มี mTLS

Control: MFA, Mutual TLS, Signed JWT, Credential Rotation

TTampering — การแก้ไขข้อมูล

ตัวอย่าง: แก้ราคาใน request, ปลอม Payment Callback, แก้ Log

Control: Server-side Validation, HMAC, Immutable Log

RRepudiation — ปฏิเสธความรับผิดชอบ

ตัวอย่าง: ปฏิเสธว่าไม่ได้ยกเลิก booking, Admin ปฏิเสธว่าไม่ได้เปลี่ยนสิทธิ์

Control: Audit Log (User ID, Timestamp, Action), Log Integrity, NTP Sync

IInformation Disclosure — เปิดเผยข้อมูล

ตัวอย่าง: API ส่ง password_hash, Error แสดง Stack Trace, Object Storage Public

Control: Data Minimization, Encryption, Secret Redaction, Secure Error Handling

DDenial of Service — ปฏิเสธบริการ

ตัวอย่าง: Request จำนวนมากไป CPU-heavy endpoint, อัปโหลดไฟล์ใหญ่ล้นพื้นที่

Control: Rate Limit, Quota, Timeout, Circuit Breaker, Autoscaling

EElevation of Privilege — ยกระดับสิทธิ์

ตัวอย่าง: ผู้ใช้เรียก Admin API, JWT role=admin ไม่ตรวจ, Container root

Control: Least Privilege, RBAC, Backend Authorization, Container Hardening

4.5.1 Spoofing

ตัวอย่าง: ขโมย Session Token, ปลอม API Key, Phishing, Service ไม่มี mTLS Control: MFA, Session Token ที่ปลอดภัย, Mutual TLS, Signed JWT, Credential Rotation

4.5.2 Tampering

ตัวอย่าง: แก้ราคาใน request, ปลอม Payment Callback, แก้ Log Control: Server-side Validation, HMAC, Immutable Log, Database Constraint

4.5.3 Repudiation

ตัวอย่าง: ปฏิเสธว่าไม่ได้ยกเลิก booking, Admin เปลี่ยนสิทธิ์ไม่มีหลักฐาน Control: Audit Log (User ID, Timestamp, Action), Log Integrity, NTP Sync

4.5.4 Information Disclosure

ตัวอย่าง: API ส่ง password_hash, Error แสดง SQL/Stack Trace, Log เก็บ Token Control: Data Minimization, Field-level Access Control, Encryption, Secret Redaction

4.5.5 Denial of Service

ตัวอย่าง: Request ท่วม CPU-heavy endpoint, อัปโหลดใหญ่ล้นพื้นที่ Control: Rate Limit, Request Size Limit, Timeout, Circuit Breaker, Queue, DDoS Protection

4.5.6 Elevation of Privilege

ตัวอย่าง: ผู้ใช้เรียก Admin API, JWT role=admin ไม่ตรวจลายเซ็น Control: Least Privilege, RBAC/ABAC, Backend Authorization, Container Hardening

  1. ใช้ DFD จาก Lab 4.1 เลือก 8 จุดขึ้นไป
  2. วิเคราะห์ STRIDE ที่เกี่ยวข้องกับแต่ละจุด
  3. เขียน Threat Statement อย่างน้อย 12 รายการตามรูปแบบที่แนะนำ
  4. เสนอ Control อย่างน้อย 1 รายการต่อ Threat
Lab 4.3: Risk Register และ Risk Treatment (60 นาที)
  1. เลือกภัยคุกคาม 8 รายการจาก Lab 4.2
  2. ประเมิน Likelihood และ Impact กำหนด Risk Level ด้วย Matrix
  3. เลือก Treatment (Mitigate/Transfer/Accept/Avoid)
  4. ระบุ Control, Owner, Due Date, Status และ Residual Risk
Lab 4.4: CVSS เทียบ Business Risk (45-60 นาที)
  1. เลือกช่องโหว่ 3 รายการ (IDOR, SQL Injection, Missing Rate Limit)
  2. จัดลำดับด้วย CVSS อย่างเดียว
  3. จัดลำดับใหม่ด้วย Business Risk (รวมบริบท Asset, Exposure)
  4. อธิบายว่าลำดับเปลี่ยนหรือไม่ เพราะเหตุใด ส่งสรุป 1 หน้า

คำถามท้ายบท

  1. Threat Modeling แตกต่างจาก Risk Assessment และ Vulnerability Scoring อย่างไร
  2. เหตุใด DFD และ Trust Boundary จึงเป็นพื้นฐานสำคัญของ Threat Modeling
  3. อธิบาย STRIDE ทั้ง 6 หมวด พร้อมตัวอย่างภัยคุกคามและ Control
  4. เขียน Threat Statement สำหรับ "ผู้ใช้ดูข้อมูลคำสั่งซื้อของผู้อื่นได้"
  5. Risk Matrix มีข้อดีและข้อจำกัดอย่างไร
  6. CVSS v4.0 มี Metric Group ใดบ้าง และเหตุใด CVSS จึงไม่เท่ากับ Risk
  7. DREAD มีประโยชน์และข้อจำกัดอะไรในการเรียน Threat Modeling
  8. เปรียบเทียบ STRIDE กับ PASTA
  9. จากกรณี Equifax 2017 การทำ TM และ Risk Register ช่วยลดความเสี่ยงใดได้บ้าง
  10. จากกรณี SolarWinds 2020 Attack Surface ของซอฟต์แวร์รวมอะไรบ้างนอกเหนือจากตัวแอป

สรุปท้ายบท

สาระสำคัญของบทที่ 4

Threat Modeling ช่วยให้ทีมเห็นภัยคุกคามตั้งแต่ช่วงออกแบบ ไม่ใช่รอให้พบช่องโหว่หลังระบบเสร็จ DFD และ Trust Boundary ช่วยให้เห็นจุดที่ต้องตรวจตัวตน ตรวจสิทธิ์ เข้ารหัส และบันทึกหลักฐาน

STRIDE เป็นกรอบถามคำถามครบ 6 มุม ได้แก่ Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service และ Elevation of Privilege เมื่อจับคู่กับ DFD จะลดโอกาสตกหล่น

Risk Assessment จัดลำดับด้วย Likelihood และ Impact บันทึกใน Risk Register ที่มีเจ้าของ Due Date Treatment และ Residual Risk CVSS v4.0 ช่วยสื่อสารความรุนแรงของช่องโหว่ แต่ไม่ใช่คะแนน Risk ทางธุรกิจทั้งหมด

กรณีศึกษา Equifax, Capital One, SolarWinds, MOVEit, Log4Shell แสดงว่า TM และ Risk Analysis ป้องกันความเสียหายจริง Threat Modeling ไม่ใช่งาน Security Team เท่านั้น แต่เป็นกิจกรรมร่วมของ Developer, Architect, QA, DevOps, Product Owner และผู้บริหาร
