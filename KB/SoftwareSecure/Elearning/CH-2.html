CIA Triad, Authentication, Authorization, Accountability

จำนวนชั่วโมง: บรรยาย 2 ชม. + ปฏิบัติการ 2 ชม. | รวม 4 ชม.

วัตถุประสงค์การเรียนรู้
  1. อธิบาย CIA Triad (Confidentiality, Integrity, Availability) และยกตัวอย่างการประยุกต์ใช้ในบริบทซอฟต์แวร์ พร้อมวิเคราะห์ความขัดแย้งระหว่างหลักการทั้งสาม
  2. เข้าใจและแยกแยะความแตกต่างระหว่าง Identification, Authentication, Authorization, และ Accountability (IAAA)
  3. อธิบายหลักการ Non-repudiation และ Privacy พร้อมยกตัวอย่างการประยุกต์ใช้
  4. เข้าใจหลักการออกแบบความปลอดภัยขั้นพื้นฐานของ Saltzer & Schroeder และสามารถประยุกต์ใช้ในการออกแบบซอฟต์แวร์
  5. ประยุกต์ใช้หลักการ Least Privilege, Defense in Depth, Fail Safe, และ Privacy by Design ในการวิเคราะห์และออกแบบระบบ

2.1 CIA Triad ในบริบทของซอฟต์แวร์

CIA Triad เป็นหลักการพื้นฐานด้านความมั่นคงปลอดภัยของสารสนเทศที่ประกอบด้วย 3 เสาหลัก ได้แก่ Confidentiality (การรักษาความลับ), Integrity (ความถูกต้องครบถ้วน), และ Availability (ความพร้อมใช้งาน) แม้จะเป็นแนวคิดที่มีมาก่อนยุคดิจิทัล แต่การประยุกต์ใช้ในบริบทของซอฟต์แวร์มีรายละเอียดที่เฉพาะเจาะจง

Interactive: CIA Triad — คลิกเพื่อดูรายละเอียด

คลิกที่การ์ดแต่ละใบเพื่อดูคำอธิบาย มาตรการ และตัวอย่างการละเมิดที่มีชื่อเสียง

🔒
Confidentiality
การรักษาความลับ

คำอธิบาย: การปกป้องข้อมูลไม่ให้ถูกเปิดเผยแก่บุคคลหรือระบบที่ไม่ได้รับอนุญาต

มาตรการ:

  • Encryption at Rest (AES-256)
  • Encryption in Transit (TLS 1.3, SSH)
  • Access Control (RBAC, ACL)
  • Data Masking
  • Memory Protection (ASLR, DEP)

การละเมิดที่มีชื่อเสียง:

  • Equifax (2017): 147 ล้านราย รั่วไหลจากช่องโหว่ที่ไม่ได้อัปเดต
  • Facebook/Cambridge Analytica (2018): 87 ล้านคน
  • Meltdown/Spectre: CPU vulnerability
✅
Integrity
ความถูกต้องครบถ้วน

คำอธิบาย: การทำให้แน่ใจว่าข้อมูลและระบบไม่ถูกแก้ไขเปลี่ยนแปลงโดยไม่ได้รับอนุญาต

มาตรการ:

  • Hashing (SHA-256 Checksum)
  • Digital Signatures
  • Version Control (Git, SVN)
  • Audit Logs
  • Integrity Monitoring (Tripwire, OSSEC)
  • Checksum / MAC (HMAC, CRC)

ตัวอย่าง:

  • Software Update: ตรวจสอบ Signature และ Checksum
  • Database Transactions: ACID
  • Blockchain: Hash chain
  • Git Commit Signing: GPG หรือ SSH key
⚡
Availability
ความพร้อมใช้งาน

คำอธิบาย: การทำให้แน่ใจว่าระบบและข้อมูลสามารถเข้าถึงได้เมื่อต้องการ — ระบบที่ไม่สามารถใช้งานได้ย่อมไร้ค่า

มาตรการ:

  • Redundancy (Active-Passive, Active-Active)
  • Load Balancing (NGINX, AWS ALB)
  • Fault Tolerance (Clustering, Failover)
  • DDoS Protection (Cloudflare, AWS Shield)
  • Auto Scaling (Kubernetes HPA)
  • Backup & Recovery
  • Circuit Breaker (Hystrix, Resilience4j)

ตัวอย่าง:

  • Netflix: Chaos Engineering (Chaos Monkey)
  • Cloud Services: SLA 99.99% (downtime ≤ 52.56 นาที/ปี)
  • ระบบจองตั๋วเครื่องบิน: High Availability
หมายเหตุสำคัญ: ความสัมพันธ์พึ่งพากันของ CIA Triad: แม้ CIA Triad จะถูกนำเสนอเป็นสามเสาหลักที่แยกจากกัน แต่ในระบบจริง ทั้งสามมีความสัมพันธ์พึ่งพากัน (Interdependence) — ความล้มเหลวในด้านหนึ่งมักส่งผลกระทบต่ออีกด้านหนึ่ง ตัวอย่างเช่น กรณี Barclays IT Glitch ปี 2025 ที่ข้อผิดพลาดในระบบสำรองข้อมูล (Availability Failure) ทำให้ข้อมูลสถานะบัญชีของลูกค้าหายไป ส่งผลให้ข้อมูลขาดความถูกต้อง (Integrity Failure) ตามมา และทำให้ผู้ใช้ไม่สามารถทำธุรกรรมได้อย่างถูกต้อง (Confidentiality Impact ต่อข้อมูลการเงิน) นักศึกษาควรมอง CIA Triad เป็นระบบความสัมพันธ์ที่เชื่อมโยงกัน ไม่ใช่สามส่วนที่แยกขาดจากกัน

2.1.1 Confidentiality (การรักษาความลับ)

Confidentiality หมายถึง การปกป้องข้อมูลไม่ให้ถูกเปิดเผยแก่บุคคลหรือระบบที่ไม่ได้รับอนุญาต ในบริบทของซอฟต์แวร์ หมายถึงการทำให้แน่ใจว่าข้อมูลจะถูกอ่านได้โดยผู้ที่มีสิทธิ์เท่านั้น

มาตรการคำอธิบายตัวอย่าง
Encryption at Restเข้ารหัสข้อมูลขณะจัดเก็บเข้ารหัสฐานข้อมูลด้วย AES-256
Encryption in Transitเข้ารหัสข้อมูลขณะส่งผ่านเครือข่ายHTTPS (TLS 1.3), SSH, VPN
Access Controlควบคุมการเข้าถึงข้อมูลRBAC, ACL
Data Maskingซ่อนข้อมูลบางส่วนแสดงเลขบัตรเครดิตเฉพาะ 4 หลักท้าย
Memory Protectionป้องกันการอ่านหน่วยความจำASLR, DEP, Secure Enclave
  • ระบบธนาคารออนไลน์: ข้อมูลบัญชีและประวัติการเงินต้องถูกเข้ารหัสทั้งขณะจัดเก็บและขณะส่ง
  • ระบบโรงพยาบาล: เวชระเบียนผู้ป่วยต้องถูกปกปิดตามกฎหมาย (PDPA/HIPAA)
  • แอปพลิเคชันส่งข้อความ: WhatsApp ใช้ End-to-End Encryption
📌
ข้อควรจำ: Confidentiality ไม่ใช่แค่การเข้ารหัส — ครอบคลุม Access Control, การจัดการ Key, และการป้องกัน Data Leakage ภัยคุกคามต่อ Confidentiality ไม่ได้มาแต่แฮกเกอร์ภายนอก — Insider Threat มีสัดส่วนสูง การเข้ารหัสที่ผิดวิธี (เช่น ใช้ ECB mode, ใช้คีย์ที่ Hardcode) แย่กว่าการไม่เข้ารหัส

2.1.2 Integrity (ความถูกต้องครบถ้วน)

Integrity หมายถึง การทำให้แน่ใจว่าข้อมูลและระบบไม่ถูกแก้ไขเปลี่ยนแปลงโดยไม่ได้รับอนุญาต รวมถึงการทำให้แน่ใจว่าข้อมูลที่ได้รับเป็นข้อมูลที่ถูกต้องและสมบูรณ์

มาตรการคำอธิบายตัวอย่าง
Hashingสร้างค่า Hash เพื่อตรวจสอบความถูกต้องSHA-256 Checksum สำหรับไฟล์ดาวน์โหลด
Digital Signaturesเซ็นชื่อดิจิทัลเพื่อยืนยันแหล่งที่มาเซ็น DLL, APK, Git Commit
Version Controlระบบควบคุมเวอร์ชันGit, SVN — ดูประวัติการแก้ไข
Audit Logsบันทึกการเปลี่ยนแปลงทั้งหมดใคร แก้ไขอะไร เมื่อไหร่
Integrity Monitoringตรวจจับการเปลี่ยนแปลงโดยไม่ได้รับอนุญาตTripwire, OSSEC
Checksum / MACตรวจสอบความถูกต้องของข้อมูลที่รับส่งHMAC, CRC
  • Software Update: ตรวจสอบ Signature และ Checksum ก่อนติดตั้ง (ป้องกัน Supply Chain Attack)
  • Database Transactions: ACID (Atomicity, Consistency, Isolation, Durability)
  • Blockchain: ทุกบล็อกมี Hash ของบล็อกก่อนหน้า
  • Git Commit Signing: เซ็น Git commit ด้วย GPG หรือ SSH key
📌
ข้อควรจำ: Integrity ครอบคลุมทั้ง Data Integrity และ System Integrity การใช้ Hashing เพียงอย่างเดียวไม่เพียงพอ — ต้องมี Mechanisms เพื่อป้องกัน Man-in-the-Middle Message Authentication Code (MAC) และ Digital Signatures ช่วยแก้ปัญหานี้โดยใช้ Secret Key หรือ Private Key

2.1.3 Availability (ความพร้อมใช้งาน)

Availability หมายถึง การทำให้แน่ใจว่าระบบซอฟต์แวร์และข้อมูลสามารถเข้าถึงได้เมื่อต้องการ โดยผู้มีสิทธิ์ — ระบบที่ไม่สามารถใช้งานได้ย่อมไร้ค่าไม่ว่าจะปลอดภัยแค่ไหน

มาตรการคำอธิบายตัวอย่าง
Redundancyมีระบบสำรองเซิร์ฟเวอร์สำรอง (Active-Passive, Active-Active)
Load BalancingกระจายโหลดNGINX, AWS ALB, HAProxy
Fault Toleranceทนต่อความผิดพลาดClustering, Failover, Replication
DDoS Protectionป้องกันการโจมตี DoSCloudflare, AWS Shield, Rate Limiting
Auto ScalingปรับขนาดตามโหลดAWS Auto Scaling, Kubernetes HPA
Backup & Recoveryสำรองและกู้คืนข้อมูลDatabase Snapshot, Disaster Recovery Plan
Circuit Breakerป้องกัน Cascading FailureNetflix Hystrix, Resilience4j
  • Netflix: Chaos Engineering (Chaos Monkey) — จงใจทำให้บริการล่มเพื่อทดสอบ
  • Cloud Services: SLA 99.99% — downtime ไม่เกิน 52.56 นาทีต่อปี
  • ระบบจองตั๋วเครื่องบิน: ต้องมี High Availability ช่วง Peak Season
📌
ข้อควรจำ: Availability มีต้นทุนสูง — ยิ่งต้องการสูง (99.999% = Five Nines) ยิ่งต้องลงทุนมาก DDoS Attack เป็นภัยคุกคามสำคัญต่อ Availability — ทุกองค์กรควรมีแผนรับมือ Availability ไม่ใช่แค่ Infrastructure — รวมถึง Code Quality, การจัดการ Dependency, และ Incident Response

2.1.4 ความขัดแย้งระหว่าง CIA Triad (Trade-offs)

หลักการ CIA Triad ทั้งสามมักขัดแย้งกัน การเพิ่มความแข็งแกร่งด้านหนึ่งอาจลดทอนอีกด้านหนึ่ง

ความขัดแย้งคำอธิบายตัวอย่าง
Confidentiality ↔ Availabilityยิ่งป้องกันแน่นหนา ยิ่งเข้าถึงยากMFA หลายขั้นตอนอาจล็อกผู้ใช้ออกจากระบบ
Integrity ↔ Availabilityตรวจสอบทุกครั้งอาจทำให้ระบบช้าChecksum ทุก request เพิ่ม latency
Confidentiality ↔ Integrityเข้ารหัสอย่างเดียวไม่รับประกันความถูกต้องข้อมูลที่เข้ารหัสอาจถูกแก้ไข (ถ้าไม่มี AEAD)
Security ↔ Usabilityปลอดภัยสูงมักใช้งานยากเปลี่ยนรหัสผ่านทุก 30 วัน → จดใส่ sticky note

การจัดการ Trade-offs

  1. Risk-Based Approach: ประเมินความเสี่ยงของแต่ละสินทรัพย์ — ข้อมูลธนาคาร → Confidentiality สูง, E-Commerce → Availability สูง, Blockchain → Integrity สูง
  2. Layered Controls: ใช้ Defense in Depth — แต่ละชั้นเน้นคนละด้านของ CIA
  3. Context-Aware Decisions: ปรับระดับความปลอดภัยตามบริบท — โอนเงินมาก → MFA, ดูรายการ → Password only
  4. Continuous Monitoring: เฝ้าระวังและปรับ Trade-offs ตามสถานการณ์
ตัวอย่างการตัดสินใจ Trade-off ในโลกจริง

ธนาคารออนไลน์ต้องตัดสินใจระหว่าง Security และ Availability: ถ้าใส่ MFA ทุกครั้งที่ login → ปลอดภัย แต่ผู้ใช้ไม่สะดวก — อาจเสียลูกค้า ถ้าใส่ MFA เฉพาะการโอนเงินต่างประเทศ → สมดุล ถ้าไม่ใส่ MFA เลย → ง่ายต่อการถูกโจมตี

📌
ข้อควรจำ: ไม่มีระบบใดปลอดภัย 100% และ CIA Triad ที่สมบูรณ์แบบไม่มีอยู่จริง Trade-offs คือการตัดสินใจทางธุรกิจ (Business Decision) ไม่ใช่แค่ทางเทคนิค การทำ Risk Assessment และการสื่อสาร Trade-offs กับผู้มีส่วนได้ส่วนเสีย (Stakeholders) เป็นทักษะสำคัญ
💬 คำถามสำหรับการอภิปรายในชั้นเรียน: จงยกตัวอย่างสถานการณ์ที่ Confidentiality และ Availability ขัดแย้งกันในชีวิตจริง และเสนอแนวทางการจัดการกับความขัดแย้งนั้น

2.1.5 Parkerian Hexad — การขยายแนวคิด CIA สู่มิติความปลอดภัยที่สมบูรณ์ยิ่งขึ้น

Donn B. Parker นักวิจัยด้านความปลอดภัยสารสนเทศ ได้เสนอแนวคิด Parkerian Hexad ในปี 1998 โดยเพิ่มเติมอีก 3 มิติที่ CIA Triad แบบดั้งเดิมครอบคลุมไม่ถึง

มิติคำอธิบายความสำคัญต่อซอฟต์แวร์
Confidentialityการรักษาความลับข้อมูลไม่ถูกเปิดเผยแก่ผู้ไม่ได้รับอนุญาต
Integrityความถูกต้องครบถ้วนข้อมูลและระบบไม่ถูกแก้ไขโดยไม่ได้รับอนุญาต
Availabilityความพร้อมใช้งานระบบและข้อมูลเข้าถึงได้เมื่อต้องการ
Possession/Controlการครอบครองหรือควบคุมข้อมูลข้อมูลอาจถูกคัดลอกโดยไม่สูญเสียการครอบครอง — DLP, DRM, Cloud Data Residency
Authenticityความแท้จริงของข้อมูลและแหล่งที่มาDigital Signature, Code Signing, Certificate Validation
Utilityประโยชน์ใช้สอยของข้อมูลข้อมูลที่เข้ารหัสแต่ถอดรหัสไม่ได้ย่อมไร้ค่า — การจัดการ Key, Data Recovery

📌
ข้อควรจำ: Parkerian Hexad มีประโยชน์ในการวิเคราะห์ความเสี่ยงที่ละเอียดขึ้น โดยเฉพาะในยุคที่ CIA Triad แบบดั้งเดิมอาจไม่ครอบคลุมทุกมิติ Possession/Control และ Utility มีความสำคัญเพิ่มขึ้นในยุค Cloud และ Ransomware Authenticity เป็นรากฐานของ Zero Trust

2.2 IAAA Framework (Identification, Authentication, Authorization, Accountability)

IAAA Framework เป็นแนวคิดที่อธิบายกระบวนการพิสูจน์และควบคุมตัวตนในระบบซอฟต์แวร์ ประกอบด้วย 4 ขั้นตอนที่ทำงานร่วมกัน

1. Identificationการระบุตัวตน
2. Authenticationการพิสูจน์ตัวตน
3. Authorizationการกำหนดสิทธิ์
4. Accountabilityตรวจสอบย้อนหลัง

2.2.1 Identification (การระบุตัวตน)

Identification คือขั้นตอนที่ผู้ใช้ "บอก" ระบบว่าตนคือใคร — เป็นการอ้างตัวตน (Claim of Identity) โดยยังไม่มีการพิสูจน์

รูปแบบตัวอย่างข้อควรระวัง
Usernamejohn.smith@company.comต้องไม่ซ้ำ, ไม่ควรใช้ข้อมูลส่วนบุคคลที่เดาง่าย
User IDuser_12345ควรเป็นค่า Internal ไม่เปิดเผย
Email Addressuser@example.comนิยมใช้เป็น Identifier แต่ก็ทำให้เดาตัวตนได้
Phone Number+66 81 234 5678เปลี่ยนเบอร์ได้ ทำให้ติดตามยาก
Biometricใบหน้า, ลายนิ้วมือไม่เปลี่ยนตาม, มี Privacy Concerns
  • Enumeration Attack: ถ้าระบบตอบกลับแตกต่างเมื่อ username มีอยู่ vs ไม่มีอยู่ — ควรตอบกลับเหมือนกัน (เช่น "Invalid username or password")
  • Predictable IDs: การใช้ ID แบบ Sequential (/user/1, /user/2) — ควรใช้ UUID หรือ Random ID
  • Username Policy: ควรมีนโยบาย username ที่ชัดเจน
📌
ข้อควรจำ: Identification เป็นเพียงการ "บอก" ว่าเป็นใคร ยังไม่มีการพิสูจน์ ควรแยก Identifier ภายใน (Internal ID) ออกจาก Identifier ภายนอก (Display Name, Email) ใช้ Identifier ที่ไม่เปลี่ยน (Immutable) เท่านั้นเป็น Primary Key

2.2.2 Authentication (การพิสูจน์ตัวตน)

Authentication คือกระบวนการ "พิสูจน์" ว่าผู้ใช้เป็นบุคคลที่อ้างตัวจริง — เป็นการยืนยัน Claim of Identity ด้วยหลักฐาน (Credential)

ปัจจัยคำอธิบายตัวอย่าง
Something You Knowความลับที่ผู้ใช้จำได้Password, PIN, Security Question
Something You Haveอุปกรณ์ที่ผู้ใช้มีOTP จากมือถือ, Hardware Token, Smart Card
Something You Areลักษณะทางกายภาพลายนิ้วมือ, ใบหน้า, เส้นเลือดม่านตา
Somewhere You Areตำแหน่งที่ตั้งIP Address, GPS Location
Something You Doพฤติกรรมลายเซ็น, จังหวะการพิมพ์, Gait Analysis

Multi-Factor Authentication (MFA)

การใช้ปัจจัยมากกว่า 1 ประเภท เช่น รหัสผ่าน (สิ่งที่รู้) + OTP จากมือถือ (สิ่งที่ครอบครอง), รหัสผ่าน (สิ่งที่รู้) + ลายนิ้วมือ (สิ่งที่คุณเป็น)

แนวปฏิบัติที่ดีสำหรับ Authentication

  1. Password Hashing: ใช้ Argon2id, bcrypt, หรือ PBKDF2
  2. Rate Limiting: จำกัดจำนวนครั้งในการ login
  3. Account Lockout: หลังล้มเหลวหลายครั้ง (ระวัง DoS)
  4. Password Policy: เน้นความยาว (12+ chars) มากกว่าความซับซ้อน
  5. MFA โดย Default: โดยเฉพาะ Admin Accounts
  6. Secure Credential Recovery
📌
ข้อควรจำ: Authentication ≠ Authorization Password ยังใช้มากที่สุดแต่อ่อนแอที่สุด — ควรเสริมด้วย MFA Never Roll Your Own Authentication

โมเดล Authorization ที่สำคัญ

1

จุดอ่อน: ผู้ใช้มักให้สิทธิ์มากเกินไป, มัลแวร์ใช้สิทธิ์ของเจ้าของ

2
MAC (Mandatory Access Control)

ระบบกำหนดสิทธิ์ตามนโยบายกลาง — SELinux, AppArmor, Windows MIC

จุดแข็ง: ปลอดภัยสูง | จุดอ่อน: จัดการซับซ้อน

3
RBAC (Role-Based Access Control)

สิทธิ์ตามบทบาท — User → Role → Permission

Admin > Editor > Viewer

4
ABAC (Attribute-Based Access Control)

สิทธิ์จากคุณลักษณะ (User, Resource, Environment)

5. ReBAC (Relationship-Based Access Control)

สิทธิ์จากความสัมพันธ์ (Relationship) — (alice, editor, doc:report)

  • ระบบที่ใช้: Google Zanzibar, OpenFGA, SpiceDB
  • จุดแข็ง: จัดการ Permission ซับซ้อนได้ดี — Collaborative Products
  • จุดอ่อน: ซับซ้อนในการ Implement

MACต่ำสูงยากMilitary, Government
RBACปานกลางปานกลางปานกลางEnterprise Applications
ABACสูงสูงซับซ้อนCloud, Microservices
ReBACสูงมากสูงมากซับซ้อนGoogle Drive, GitHub, Figma

หลักการสำคัญของ Authorization

  1. Least Privilege: ให้สิทธิ์เท่าที่จำเป็น
  2. Default Deny: ปฏิเสธทุกอย่าง ยกเว้นที่ได้รับอนุญาต
  3. Separation of Duties: แยกคนรับผิดชอบหน้าที่สำคัญ
  4. Authorization ที่ทุกระดับ: API Gateway, Service Layer, Data Layer
📌
ข้อควรจำ: Broken Access Control เป็นอันดับ 1 ของ OWASP Top 10 การตรวจสอบ Authorization ต้องทำที่ Server-side เท่านั้น อย่าเชื่อข้อมูล Authorization ที่ส่งจาก Client

2.4.2 Audit Logs สำหรับการตรวจสอบ

Audit Logs เป็นรากฐานสำคัญของทั้ง Accountability และ Non-repudiation — เป็นการบันทึกเหตุการณ์ทั้งหมดในระบบอย่างเป็นระบบ

หลักการสำคัญของ Audit Log

  4. Correlatable: เชื่อมโยงเหตุการณ์ที่เกี่ยวข้องกัน (Correlation ID)

สาระสำคัญของบทที่ 2

CIA Triad เป็นรากฐานของความมั่นคงปลอดภัยสารสนเทศ ประกอบด้วย Confidentiality (การรักษาความลับ), Integrity (ความถูกต้องครบถ้วน), และ Availability (ความพร้อมใช้งาน) ทั้งสามมีความสัมพันธ์พึ่งพากันและมักขัดแย้งกัน การจัดการ Trade-offs ต้องใช้ Risk-Based Approach และ Context-Aware Decisions

IAAA Framework อธิบายกระบวนการ Identification, Authentication, Authorization, และ Accountability ที่ทำงานร่วมกันในการควบคุมการเข้าถึงระบบ หลักการออกแบบความปลอดภัยของ Saltzer & Schroeder ทั้ง 11 หลักการเป็นแนวทางที่ใช้ได้กับทุกระบบ

Non-repudiation และ Privacy เป็นแนวคิดที่เชื่อมโยงกับ CIA Triad และ IAAA — การออกแบบระบบที่ปลอดภัยต้องคำนึงถึงทั้ง Security และ Privacy ตั้งแต่ต้น (Privacy by Design)
