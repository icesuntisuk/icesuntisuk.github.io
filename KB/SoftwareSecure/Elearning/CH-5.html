5.1 ภาพรวม Input Validation และ Injection Attacks

Input Validation คือการตรวจสอบข้อมูลที่เข้าระบบว่ามีชนิด รูปแบบ ความยาว ช่วงค่า โครงสร้าง และความหมายตรงตามที่ระบบคาดหวังหรือไม่ ส่วน Injection เกิดเมื่อระบบนำข้อมูลที่ไม่น่าเชื่อถือไปปะปนกับคำสั่งที่มีตัวแปลความหมาย เช่น SQL, Shell, HTML, JavaScript, LDAP, XPath หรือ Template Expression

รับข้อมูลForm, API, Header, File, Webhook
ตรวจสอบชนิด รูปแบบ ความยาว ช่วงค่า
แยกคำสั่งParameterization หรือ Safe API
แสดงผลContext-aware Encoding
ติดตามLog, Test, Review

5.1.1 ข้อมูลนำเข้าที่ไม่น่าเชื่อถือ

ข้อมูลที่ต้องตรวจสอบไม่ได้มีเฉพาะฟอร์มหน้าเว็บ แต่รวมถึง query string, JSON body, HTTP header, cookie, file metadata, URL parameter, webhook, message queue, cache, log และข้อมูลจากฐานข้อมูลเดิมที่อาจถูกป้อนเข้ามาในอดีต

เลือกกฎและใส่ค่าทดลอง

5.1.2 Injection ใน OWASP Top 10 2021

OWASP Top 10 2021 จัด Injection อยู่ในหมวด A03 และรวม Cross-Site Scripting ไว้ในหมวดนี้ด้วย เพราะเป็นการฉีด script ให้ browser ของเหยื่อรันคำสั่งที่ผู้โจมตีควบคุม

ประเภทบริบทที่ถูกฉีดผลกระทบทั่วไป
SQL Injectionคำสั่ง SQLอ่านหรือแก้ข้อมูล, bypass login
NoSQL InjectionQuery object หรือ operatorอ่านข้อมูลเกินสิทธิ์หรือ bypass query
XSSHTML/JavaScript ใน browserสวมรอยผู้ใช้หรือแก้ DOM
OS Command InjectionShell commandรันคำสั่งบน server
SSTITemplate engineอ่านข้อมูลหรือรันโค้ดฝั่ง server

5.2 Validation, Sanitization, Encoding และ Parameterization

สี่คำนี้มีหน้าที่ต่างกัน Validation ใช้ตรวจข้อมูลตอนรับเข้า Sanitization ใช้ลบหรือปรับข้อมูลบางส่วน Encoding ใช้ก่อนแสดงผล และ Parameterization ใช้แยกข้อมูลออกจากคำสั่ง

เทคนิคใช้เมื่อใดเป้าหมายข้อจำกัด
Validationตอนรับข้อมูลตรวจชนิด รูปแบบ ความยาว ช่วงค่าไม่พอถ้านำข้อมูลไปใช้ผิดบริบท
Sanitizationเมื่อต้องรับข้อมูลบางส่วนลบหรือปรับส่วนอันตรายเสี่ยงหากกฎไม่ครบ
Encodingก่อนแสดงผลทำให้ข้อมูลไม่ถูกตีความเป็นโค้ดต้องตรงบริบท
Parameterizationก่อนส่งคำสั่งไป interpreterแยกข้อมูลออกจากคำสั่งใช้ไม่ได้กับทุกตำแหน่ง เช่น ชื่อตาราง

5.2.2 Canonicalization และ Encoding Issues

ควรแปลงข้อมูลให้อยู่ในรูปมาตรฐานก่อน validation เช่น URL decoding, unicode normalization หรือ path normalization แต่ต้องระวังการ decode ซ้ำจนเกิดความหมายใหม่ที่ไม่คาดคิด

5.3 SQL Injection

SQL Injection เกิดเมื่อระบบสร้าง SQL statement ด้วยการนำข้อมูลผู้ใช้ไปต่อเป็น string แล้วส่งให้ฐานข้อมูลตีความ วิธีป้องกันหลักคือ prepared statement หรือ parameterized query

# ตัวอย่างเพื่อการสอน: ไม่ควรใช้ในระบบจริง
sql = "SELECT * FROM users WHERE username = '" + username + "'"
cursor.execute(sql)
# แยกข้อมูลออกจากคำสั่ง
sql = "SELECT * FROM users WHERE username = ?"
cursor.execute(sql, (username,))
SQL

Prepared statement ทำให้ฐานข้อมูลรู้โครงสร้างคำสั่งก่อน แล้วรับค่าพารามิเตอร์เป็นข้อมูล ไม่ใช่ส่วนหนึ่งของ syntax เพิ่มเติม

5.3.1 ประเภท SQL Injection

ประเภทลักษณะสิ่งที่ควรสอน
In-bandผลลัพธ์กลับทาง response เดียวกันเห็นผลชัดใน lab
Blind Boolean-basedอนุมานจากผล true/falseแม้ไม่เห็น error ก็ยังมีช่องโหว่ได้
Blind Time-basedอนุมานจากเวลาตอบสนองใช้สอนแนวคิด ไม่ต้องทำกับระบบจริง
Second-orderpayload ถูกบันทึกก่อนแล้วเกิดผลภายหลังข้อมูลจากฐานข้อมูลเดิมก็ต้องระวัง

5.3.2 Stored Procedure, ORM และ Least Privilege

Stored procedure และ ORM ช่วยลดความเสี่ยงได้เมื่อใช้ถูกต้อง แต่ยังเสี่ยงหากสร้าง dynamic SQL จาก string หรือใช้ raw query จากข้อมูลผู้ใช้ ควรเสริมด้วย least privilege สำหรับบัญชีฐานข้อมูล

5.4 NoSQL Injection

NoSQL Injection เกิดเมื่อระบบให้ผู้ใช้ควบคุมโครงสร้าง query หรือ operator โดยไม่ตรวจ schema และชนิดข้อมูลอย่างเข้มงวด แนวทางหลักคือ schema validation, type enforcement และไม่ส่ง object จากผู้ใช้เข้า query โดยตรง

5.5 Cross-Site Scripting (XSS) และ DOM XSS

XSS ทำให้ browser ของผู้ใช้รัน script ที่ผู้โจมตีควบคุมในบริบทของเว็บไซต์ ผลกระทบรวมถึงสวมรอยผู้ใช้ อ่านข้อมูลในหน้าเว็บ และทำลายกลไกป้องกัน CSRF

ประเภทลักษณะตัวอย่าง
Reflected XSSpayload มาจาก request แล้วสะท้อนกลับทันทีช่องค้นหาแสดงคำค้นโดยไม่ encode
Stored XSSpayload ถูกบันทึกแล้วแสดงให้ผู้ใช้อื่นcomment, profile, review
DOM-based XSSเกิดจาก JavaScript ฝั่ง browser ใช้ข้อมูลกับ dangerous sinkนำค่าจาก URL ไปใส่ innerHTML

5.5.1 Context-aware Output Encoding

บริบทความเสี่ยงแนวป้องกัน
HTML Contentแทรก tag หรือ scriptHTML entity encoding
HTML Attributeหลุดจากค่า attributeAttribute encoding และใส่ quote
JavaScriptกลายเป็น codeหลีกเลี่ยงฝังข้อมูลดิบใน script
URLใช้ scheme อันตรายตรวจ allowlist ของ scheme/domain

Content Security Policy จำกัดแหล่งโหลด script, style, image และ resource อื่นๆ เพื่อลดผลกระทบจาก XSS แต่ไม่ควรใช้แทน output encoding

5.6 Command Injection และ OS Injection

Command Injection เกิดเมื่อ application ส่งข้อมูลผู้ใช้เข้า shell หรือ system command โดยไม่แยก command กับ argument อย่างปลอดภัย แนวป้องกันคือหลีกเลี่ยง shell ใช้ API ของภาษา ส่ง argument เป็น list และตรวจ allowlist

# ตัวอย่างเพื่อการสอน: ไม่ควรใช้ในระบบจริง
os.system("ping " + host)
# แยก argument และไม่ใช้ shell
subprocess.run(["ping", "-c", "1", host], shell=False)
  • ตรวจชนิดข้อมูล เช่น IP address
  • ใช้ allowlist
  • จำกัด timeout และสิทธิ์ process
  • ไม่เปิดเผย output ภายในมากเกินไป

5.7 LDAP Injection, XPath Injection และ Template Injection

ประเภทเกิดจากแนวป้องกัน
LDAP Injectionต่อ string สร้าง LDAP filterใช้ API/escaping เฉพาะทางและ allowlist
XPath Injectionต่อ string สร้าง XPath queryใช้ variable binding และ validation
SSTIให้ผู้ใช้ควบคุม template expressionไม่ให้ผู้ใช้ควบคุม template, sandbox, low privilege

5.8 File Upload Vulnerabilities

File Upload เสี่ยงเพราะไฟล์มีทั้ง content, metadata, filename, MIME type, extension และอาจถูกนำไปเปิด แสดง ประมวลผล หรือรันภายหลัง

ความเสี่ยงตัวอย่างผลกระทบ
Executable uploadไฟล์ script ใน web rootรันโค้ดบน server
MIME spoofingไฟล์อันตรายปลอมเป็น imagebypass validation
ไฟล์ใหญ่เกินอัปโหลดจำนวนมากDoS หรือ storage exhaustion
Filename injectionชื่อไฟล์มี path พิเศษoverwrite หรือ path traversal

5.9 Cross-Site Request Forgery (CSRF)

CSRF หลอกให้ browser ของผู้ใช้ที่ล็อกอินอยู่ส่ง request ไปยังเว็บไซต์เป้าหมายโดยอาศัย cookie session ที่ browser แนบให้อัตโนมัติ การป้องกันหลักคือ CSRF token, SameSite cookie, Origin check และ Fetch Metadata

ประเด็นCSRFXSS
กลไกหลอก browser ส่ง requestทำให้ browser รัน script
แนวป้องกันToken, SameSite, Origin checkOutput encoding, safe DOM API, CSP
ความสัมพันธ์XSS อาจขโมยหรือใช้ token ได้มักร้ายแรงกว่าในบริบท browser

  2. ทดลองเฉพาะ input ที่ lab กำหนด
  3. แก้ toy app ด้วย prepared statement
  4. ส่งภาพก่อน/หลังแก้และอธิบายเหตุผล

Lab 5.2: Reflected/Stored XSS และ Encoding

  1. ใช้ Juice Shop หรือ WebGoat
  2. จำแนกประเภท XSS
  3. แก้ toy app ด้วย context-aware encoding
  4. เพิ่ม CSP เป็น defense in depth

Lab 5.3: DOM XSS และ Safe DOM API

  1. ระบุ source และ sink
  2. เปลี่ยน innerHTML เป็น textContent เมื่อไม่ต้องการ HTML
  3. อธิบายผลต่างก่อนและหลังแก้

Lab 5.4: Command Injection ในสภาพแวดล้อมจำลอง

  1. ใช้ DVWA หรือ toy app ภายในเครื่อง
  2. วิเคราะห์ความเสี่ยงจาก shell metacharacter
  3. แก้ด้วยการแยก argument และไม่ใช้ shell

Lab 5.5: File Upload และ CSRF Defense

  1. ตรวจ file upload ด้วย allowlist, size limit และ safe storage
  2. เพิ่ม CSRF token, SameSite cookie และ Origin check
  3. อธิบายว่า XSS ทำให้ CSRF protection ล้มเหลวได้อย่างไร

คำถามท้ายบท

  1. Validation, sanitization, encoding และ parameterization ต่างกันอย่างไร
  2. เหตุใด allowlist จึงเหมาะกว่า blocklist ในหลายกรณี
  3. Prepared statement ป้องกัน SQL Injection ได้อย่างไร
  4. Stored procedure และ ORM ยังเสี่ยงได้เมื่อใด
  5. Reflected, stored และ DOM-based XSS ต่างกันอย่างไร
  6. CSP ช่วยอะไรและมีข้อจำกัดอะไร
  7. Command Injection ต่างจาก Argument Injection อย่างไร
  8. File Upload ที่ปลอดภัยต้องตรวจอะไรบ้าง
  9. CSRF ต่างจาก XSS อย่างไร
  10. จาก Log4Shell ทำไมข้อมูลที่ถูก log จึงยังเป็น input ที่เสี่ยงได้

สรุปท้ายบท

สาระสำคัญของบทที่ 5

Injection เกิดจากการปะปนระหว่างข้อมูลกับคำสั่ง การป้องกันที่แข็งแรงต้องแยกข้อมูลออกจากคำสั่งด้วย parameterization, safe API และ context-aware encoding

Input validation เป็นด่านแรก แต่ไม่พอ ต้องใช้ output encoding, least privilege, CSP, CSRF protection, safe file handling, monitoring และ security test ร่วมกัน

กิจกรรมทั้งหมดในบทนี้ต้องทำเฉพาะในสภาพแวดล้อมจำลองที่ได้รับอนุญาต และต้องนำผลลัพธ์ไปสู่ secure requirement, test case และ code review checklist
