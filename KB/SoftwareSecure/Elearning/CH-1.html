บทนำเกี่ยวกับความมั่นคงปลอดภัยของซอฟต์แวร์

บรรยาย 2 ชม. + ปฏิบัติการ 1 ชม. | รวม 3 ชม.

วัตถุประสงค์การเรียนรู้
  1. อธิบายความหมาย ขอบเขต และความสำคัญของความมั่นคงปลอดภัยของซอฟต์แวร์ในยุคดิจิทัล พร้อมยกตัวอย่างเหตุการณ์จริง
  2. จำแนกความแตกต่างระหว่าง Software Security, Application Security, Network Security และ Information Security ได้
  3. อธิบายระบบการจำแนกและให้คะแนนช่องโหว่มาตรฐานสากล ได้แก่ CVE, CWE, NVD และ CVSS
  4. ประยุกต์ใช้ Security Mindset และหลักการสำคัญในการคิดด้านความปลอดภัย เช่น Defense in Depth, Zero Trust, Assume Breach
  5. ระบุประเภทของผู้โจมตี ภัยคุกคาม และพื้นผิวการถูกโจมตี (Attack Surface) พร้อมแนวทางป้องกัน
  6. อธิบายจริยธรรมและกฎหมายที่เกี่ยวข้องกับการทดสอบความมั่นคงปลอดภัยของซอฟต์แวร์

1.1 ภาพรวมและนิยามความมั่นคงปลอดภัยของซอฟต์แวร์

1.1.1 ความหมายและขอบเขตของ Software Security

ความมั่นคงปลอดภัยของซอฟต์แวร์ (Software Security) หมายถึง กระบวนการ แนวปฏิบัติ และเทคโนโลยีที่นำมาใช้เพื่อให้แน่ใจว่าซอฟต์แวร์สามารถทำงานได้ตามที่ออกแบบไว้ โดยไม่มีช่องโหว่ที่ทำให้เกิดความเสียหายต่อข้อมูล ระบบ หรือผู้ใช้ ซอฟต์แวร์ที่ "ปลอดภัย" ไม่ได้หมายถึงซอฟต์แวร์ที่ไม่มีช่องโหว่เลย แต่หมายถึงซอฟต์แวร์ที่มีการบริหารจัดการความเสี่ยงอย่างเหมาะสม และมีมาตรการป้องกันที่เพียงพอต่อภัยคุกคามที่คาดการณ์ได้

Software Security ครอบคลุมตั้งแต่ขั้นตอนการออกแบบสถาปัตยกรรม การเขียนโค้ด การทดสอบ การปรับใช้ ไปจนถึงการบำรุงรักษาและตอบสนองต่อเหตุการณ์ ซึ่งแตกต่างจากแนวคิดด้านความปลอดภัยอื่นๆ ที่มุ่งเน้นเฉพาะในส่วนใดส่วนหนึ่ง

Software Security มีความสำคัญมากขึ้นเรื่อยๆ เนื่องจากซอฟต์แวร์กลายเป็นส่วนสำคัญของทุกองค์กรและทุกชีวิตประจำวันของผู้คน รายงานจาก IBM Cost of a Data Breach Report ปี 2025 ระบุว่าต้นทุนเฉลี่ยของการถูกละเมิดข้อมูลทั่วโลกอยู่ที่ 4.44 ล้านเหรียญสหรัฐต่อครั้ง สำหรับสหรัฐอเมริกาอยู่ที่ 10.22 ล้านเหรียญสหรัฐ ซึ่งเป็นค่าสูงสุดเป็นประวัติการณ์ ค่าใช้จ่ายต่อหนึ่งระเบียนข้อมูลที่ถูกขโมยเฉลี่ย 173 เหรียญสหรัฐ

📌
ข้อควรจำ:
  • Software Security ครอบคลุมตั้งแต่การออกแบบ การพัฒนา การทดสอบ ไปจนถึงการบำรุงรักษา
  • ช่องโหว่ในซอฟต์แวร์เป็นสาเหตุหลักของการถูกละเมิดข้อมูลในปัจจุบัน
  • ต้นทุนเฉลี่ยของการถูกละเมิดข้อมูลทั่วโลกในปี 2025 อยู่ที่ 4.44 ล้านเหรียญสหรัฐต่อเหตุการณ์
💬
คำถามสำหรับการอภิปรายในชั้นเรียน:

ในชีวิตประจำวันของนักศึกษา มีซอฟต์แวร์ใดบ้างที่นักศึกษาใช้งานและคิดว่าอาจมีช่องโหว่ทางความปลอดภัย ท่านคิดว่าซอฟต์แวร์เหล่านั้นควรมีมาตรการป้องกันใดบ้าง

1.1.2 ความแตกต่างระหว่าง Software Security, Application Security, Network Security และ Information Security

แม้ว่าคำศัพท์เหล่านี้มักถูกใช้แทนที่กัน แต่มีความหมายและขอบเขตที่ต่างกันดังนี้

Software Security เป็นแนวคิดระดับสูงที่มุ่งเน้นความปลอดภัยของซอฟต์แวร์ในทุกขั้นตอนของวงจรการพัฒนาซอฟต์แวร์ (Secure SDLC) รวมถึงการออกแบบสถาปัตยกรรม การเขียนโค้ดที่ปลอดภัย การทดสอบความปลอดภัย และการบำรุงรักษา เป็นหน้าที่ของนักพัฒนา สถาปนิกซอฟต์แวร์ และวิศวกรความปลอดภัย

Application Security (AppSec) เป็นสาขาย่อยของ Software Security ที่มุ่งเน้นเฉพาะความปลอดภัยของแอปพลิเคชันที่ทำงานอยู่ โดยเฉพาะเว็บแอปพลิเคชันและ API ครอบคลุมการป้องกันช่องโหว่เช่น SQL Injection, XSS, Broken Access Control

Network Security เป็นการรักษาความปลอดภัยของโครงสร้างพื้นฐานเครือข่าย โดยใช้ Firewall, IDS/IPS, VPN, Network Segmentation ป้องกันการโจมตีที่อาศัยช่องโหว่ในโปรโตคอลเครือข่าย

Information Security (InfoSec) เป็นแนวคิดที่กว้างที่สุด ครอบคลุมการป้องกันข้อมูลทุกรูปแบบทั้งดิจิทัลและกายภาพ โดยมีเป้าหมายเพื่อรักษา CIA Triad (Confidentiality, Integrity, Availability)

ข้อควรจำ:
  • Software Security ครอบคลุมกว้างกว่า Application Security และเป็นส่วนหนึ่งของ Information Security
  • การรักษาความปลอดภัยแบบแยกส่วนไม่เพียงพอ ต้องอาศัยการรวมกำลังกันทุกระดับ
  • ซอฟต์แวร์ที่ปลอดภัยต้องเริ่มต้นจากการออกแบบ ไม่ใช่การเพิ่มมาตรการป้องกันทีหลัง

1.1.3 เหตุผลที่ซอฟต์แวร์มีช่องโหว่

ช่องโหว่ในซอฟต์แวร์เกิดจากหลายปัจจัยหลัก ดังนี้

1. ความซับซ้อนของระบบ (Complexity) — Chrome มีโค้ดมากกว่า 30 ล้านบรรทัด Windows 11 มีโค้ดประมาณ 50-60 ล้านบรรทัด แอปพลิเคชันมือถือมี third-party libs มากกว่า 100 ตัว ความซับซ้อนหมายถึงจำนวนจุดที่อาจผิดพลาดมีมากตามไปด้วย

2. ข้อจำกัดด้านทรัพยากร (Resource Constraints) — ต้นทุนและเวลา: ผู้บริหารกดดันให้ส่งมอบเร็ว ขาดบุคลากร: Dev ไม่ได้อบรมด้านความปลอดภัย งบประมาณ: Security มักถูกมองว่าเป็นต้นทุน

3. ความผิดพลาดของมนุษย์ (Human Error) — การเขียนโค้ดโดยไม่ตรวจสอบข้อมูลนำเข้า Misconfiguration ลืมป้องกันฟังก์ชันสำคัญ ใช้ API หรือ lib ที่ไม่ปลอดภัย

4. การใช้ซอฟต์แวร์คอมโพเนนต์จากภายนอก (Third-party Dependencies) — โครงการโดยเฉลี่ยมี dependencies กว่า 2,000 ตัว (Synopsys OSSRA) ช่องโหว่ใน dependency ตัวเดียว (Log4j) ส่งผลกระทบต่อซอฟต์แวร์นับแสนตัว

5. ข้อผิดพลาดในการออกแบบ (Design Flaws) — สถาปัตยกรรมไม่คำนึงถึงความปลอดภัย Authentication/Authorization ผิดพลาด Session Management ไม่ปลอดภัย การเข้ารหัสไม่ถูกต้อง

6. ภัยคุกคามที่เปลี่ยนแปลงตลอดเวลา (Evolving Threat Landscape) — เทคนิคโจมตีพัฒนาขึ้นทุกวัน Zero-day ใหม่ถูกค้นพบเสมอ AI, IoT, Cloud สร้างพื้นผิวการโจมตีใหม่

ตัวอย่างจากโลกแห่งความเป็นจริง

ในปี 2017 Equifax ถูกโจมตีผ่าน CVE-2017-5638 ใน Apache Struts ทั้งที่ Apache ออกแพตชตั้งแต่เดือนมีนาคม 2017 แต่ Equifax ไม่ได้อัปเดต ส่งผลให้ข้อมูลส่วนบุคคล 143 ล้านรายถูกขโมย ค่าใช้จ่ายสูงกว่า 1.4 พันล้านเหรียญสหรัฐ

📌
ข้อควรจำ:
  • ซอฟต์แวร์มีช่องโหว่เพราะซับซ้อน ถูกจำกัดด้านทรัพยากร และมนุษย์ผิดพลาด
  • การใช้ไลบรารีภายนอกเพิ่มความเสี่ยงต่อ Supply Chain Attack
  • การอัปเดตและรักษาความปลอดภัยอย่างต่อเนื่องเป็นสิ่งจำเป็น
💬
คำถามสำหรับการอภิปรายในชั้นเรียน:

หากนักศึกษาเป็นผู้บริหารบริษัทซอฟต์แวร์แห่งหนึ่ง ท่านจะจัดสรรงบประมาณและเวลาให้กับความปลอดภัยอย่างไรในขณะที่ต้องส่งมอบซอฟต์แวร์ตามกำหนดการ

1.2 ระบบการจำแนกและให้คะแนนช่องโหว่มาตรฐานสากล

ในวงการความมั่นคงปลอดภัยทางไซเบอร์ มีมาตรฐานสากลที่สำคัญ 4 ระบบ ได้แก่ CVE, CWE, NVD และ CVSS

1.2.1 CVE (Common Vulnerabilities and Exposures)

CVE เป็นระบบบัญชีรายชื่อช่องโหว่ที่รู้จักต่อสาธารณะ ดำเนินการโดย MITRE สนับสนุนโดย CISA สหรัฐอเมริกา CVE-ID มีรูปแบบ CVE-ปี-เลขที่ เช่น CVE-2017-5638 (Equifax), CVE-2021-44228 (Log4Shell), CVE-2023-34362 (MOVEit) แต่ละรายการประกอบด้วย CVE ID, คำอธิบาย, References และ Date

⚠️
สิ่งที่ CVE ไม่ใช่:
  • CVE ไม่ได้บอกความรุนแรงของช่องโหว่
  • CVE ไม่ได้ประเมินผลกระทบต่อระบบของท่าน
  • CVE เป็นแค่ "ป้าย" ระบุตัวตนของช่องโหว่ ไม่ใช่รายงานฉบับสมบูรณ์

1.2.2 CWE (Common Weakness Enumeration)

CWE เป็นระบบจำแนกประเภทของจุดอ่อนในซอฟต์แวร์ CVE ตอบ "ช่องโหว่นี้คืออะไร (What)" ส่วน CWE ตอบ "จุดอ่อนประเภทไหนที่ทำให้เกิด (Why)"

ตัวอย่าง CWE-ID ที่พบบ่อย: CWE-79 (XSS), CWE-89 (SQL Injection), CWE-287 (Improper Authentication), CWE-200 (Exposure of Sensitive Information), CWE-502 (Deserialization of Untrusted Data), CWE-20 (Improper Input Validation), CWE-798 (Hard-coded Credentials), CWE-269 (Improper Privilege Management)

1.2.3 NVD (National Vulnerability Database)

NVD คือคลังข้อมูลช่องโหว่แห่งชาติสหรัฐฯ โดย NIST ทำหน้าที่ "เติมเต็ม" CVE ด้วยคะแนน CVSS, CWE Mapping, CPE และ Reference Tags ดังนั้น CVE บอกว่าช่องโหว่คืออะไร NVD บอกว่ามีความรุนแรงแค่ไหน

1.2.4 CVSS (Common Vulnerability Scoring System)

CVSS เป็นมาตรฐานเปิดสำหรับให้คะแนนความรุนแรงของช่องโหว่ พัฒนาโดย FIRST.org ปัจจุบันคือ CVSS v4.0 (ในอุตสาหกรรมยังนิยม v3.1 เป็นหลัก)

Base Metrics: Attack Vector (AV), Attack Complexity (AC), Privileges Required (PR), User Interaction (UI), Scope (S), Confidentiality (C), Integrity (I), Availability (A)

1.4 Security Mindset และหลักคิดด้านความปลอดภัย

1.4.1 Attack vs Defense Thinking

Security Mindset คือวิธีคิดที่คำนึงถึงความปลอดภัย โดยเฉพาะการคิดในมุมของผู้โจมตี (Adversarial Thinking)

Defense Thinking: มุ่งเน้นการป้องกัน ถามว่า "ฉันจะทำให้ระบบนี้ปลอดภัยได้อย่างไร"

Attack Thinking: มุ่งเน้นหาจุดอ่อน ถามว่า "ฉันจะโจมตีระบบนี้ด้วยวิธีใดได้บ้าง"

ตัวอย่าง Security Mindset
  • เมื่อรับไฟล์อัปโหลด คิดว่า "จะเกิดอะไรขึ้นถ้าส่งไฟล์ 100 GB หรือ .exe แทน .jpg"
  • เมื่อออกแบบ Authentication คิดว่า "จะเกิดอะไรขึ้นถ้า login 1000 ครั้งต่อวินาที"
  • เมื่อใช้ lib ภายนอก คิดว่า "จะเกิดอะไรขึ้นถ้า lib นี้มีช่องโหว่"

1.4.2 หลักการ Assume Breach และ Zero Trust

Assume Breach: สมมติว่าถูกเจาะแล้ว ออกแบบให้จำกัดความเสียหาย ใช้ Network Segmentation, Continuous Monitoring, Encryption, Least Privilege, Incident Response Plan

Zero Trust: "อย่าไว้ใจใคร ตรวจสอบเสมอ" 3 หลักการ: 1. Verify Explicitly, 2. Least Privilege Access, 3. Assume Breach ประกอบด้วย Micro-segmentation, MFA, Continuous Monitoring, Automation

1.4.3 การตัดสินใจภายใต้ Trade-offs

ไม่มีระบบใดปลอดภัย 100% ต้องประนีประนอมระหว่าง:

  • Security vs Usability: MFA ปลอดภัยขึ้นแต่ใช้งานยากขึ้น
  • Security vs Cost: งบประมาณจำกัด ROI วัดยาก
  • Security vs Performance: การเข้ารหัสและ logging ทำให้ระบบช้าลง

ต้องใช้ Risk Management: ประเมิน Asset Value, Likelihood, Impact, เลือก Risk Treatment

1.4.4 หลักการ Defense in Depth

"การป้องกันแบบหลายชั้น" — ไม่พึ่งพามาตรการเดียว แต่สร้างกำแพงหลายชั้น

Interactive: Defense in Depth Layers

คลิกแต่ละชั้นเพื่อดูรายละเอียด

ชั้นที่ 1 — Security Requirements
Security Requirements: กำหนดตั้งแต่ตอนออกแบบ เช่น ต้องมี MFA, เข้ารหัสข้อมูลผู้ใช้, Audit Log
ชั้นที่ 2 — Secure Coding
Secure Coding: Parameterized Query, Input Validation, Output Encoding, หลีกเลี่ยง eval()
ชั้นที่ 3 — SAST (Static Testing)
SAST: วิเคราะห์โค้ดต้นฉบับก่อนคอมไพลด้วย SonarQube, Semgrep, CodeQL
ชั้นที่ 4 — Dependency Scanning (SCA)
SCA: สแกน Third-party Components ด้วย Trivy, Snyk, OWASP Dependency-Check
ชั้นที่ 5 — WAF
WAF: กรอง request อันตรายด้วย ModSecurity, AWS WAF, Cloudflare WAF
ชั้นที่ 6 — DAST (Dynamic Testing)
DAST: ทดสอบขณะทำงานจริงด้วย OWASP ZAP, Burp Suite
ชั้นที่ 7 — Security Monitoring
Security Monitoring: SIEM, IDS/IPS ตรวจจับพฤติกรรมผิดปกติ
ชั้นที่ 8 — Incident Response
Incident Response: แผนและทีมรับมือเหตุการณ์, Disaster Recovery, Forensics
📌
ข้อควรจำ:
  • ไม่มีมาตรการใดป้องกันทุกชนิดได้
  • Defense in Depth คือการสร้างกำแพงหลายชั้น
  • ยิ่งชั้นมาก ยิ่งปลอดภัย แต่ก็ยิ่งใช้ทรัพยากรมาก
💬
คำถาม:

ในฐานะนักพัฒนา ท่านคิดว่าชั้นใดสำคัญที่สุด เพราะเหตุใด

1.5 ประเภทของภัยคุกคามและผู้โจมตี

Interactive: ประเภทผู้โจมตี

Black Hat Hacker (หมวกดำ) — บุกรกระบบโดยไม่ได้รับอนุญาตเพื่อผลประโยชน์ส่วนตัว ผิดกฎหมายอาญา ตัวอย่าง: Kevin Mitnick, LockBit, Clop

White Hat Hacker (หมวกขาว) — ได้รับอนุญาตให้ทดสอบระบบ ถูกต้องตามกฎหมาย รายงานช่องโหว่ ตัวอย่าง: Penetration Tester, Bug Bounty Hunter

Grey Hat Hacker (หมวกเทา) — อยู่กลางระหว่างดำและขาว เจาะโดยไม่ได้รับอนุญาตแต่อาจไม่มีเจตนาร้าย การกระทำอาจผิดกฎหมายแม้มีเจตนาดี

Script Kiddie (Skiddie) — ใช้เครื่องมือสำเร็จรูปโดยไม่เข้าใจลึกซึ้ง มักโจมตีแบบสุ่ม มีทักษะจำกัดแต่เครื่องมือมีประสิทธิภาพสูง

APT (Advanced Persistent Threat) — มีความสามารถสูง สนับสนุนโดยรัฐ เป้าหมายเฉพาะเจาะจง ต่อเนื่องยาวนาน มีทรัพยากรมหาศาล ใช้ Zero-day เป็นอาวุธหลัก

กลุ่ม APT: APT29 (Cozy Bear/รัสเซีย), APT41 (Winnti/จีน), Lazarus Group (เกาหลีเหนือ), APT28 (Fancy Bear/รัสเซีย)

วิธีการ: Reconnaissance → Initial Compromise → Establish Foothold → Lateral Movement → Data Exfiltration

1.5.4 Insider Threat (ภัยคุกคามจากภายใน)

เกิดจากบุคคลภายในองค์กร: พนักงาน, อดีตพนักงาน, ผู้รับเหมา, คู่ค้า

  • Malicious Insider (เจตนาร้าย) 23% — พนักงานไม่พอใจ, รับสินบน, อดีตพนักงาน
  • Accidental Insider (ไม่ได้ตั้งใจ) 63% — คลิก Phishing, ส่งข้อมูลผิด, ตั้งค่าผิด
  • Compromised Insider (ถูกประนีประนอม) 14% — บัญชีถูกขโมย, อุปกรณ์ถูกมัลแวร์

สถิติ Ponemon Institute: ต้นทุนเฉลี่ยต่อเหตุการณ์ 16.2 ล้านเหรียญสหรัฐ (2023)

การป้องกัน: Least Privilege, UBA, อบรมพนักงาน, Onboarding/Offboarding รัดกุม

💬
คำถาม:

องค์กรควรมีมาตรการตรวจสอบพนักงานมากน้อยแค่ไหนโดยไม่ละเมิดสิทธิส่วนบุคคล

1.6 Attack Surface (พื้นผิวการถูกโจมตี)

Attack Surface คือพื้นที่หรือช่องทางทั้งหมดที่ผู้โจมตีใช้โจมตีระบบ

Network Attack Surface: พอร์ตเปิด (SSH 22, HTTP 80, HTTPS 443, DB 3306/5432), API, บริการ

Software Attack Surface: Input Points, Third-party Components, Error Messages, Config Files

Human Attack Surface: พนักงาน, ผู้ดูแล, ผู้ใช้ปลายทาง

Physical Attack Surface: อุปกรณ์, USB, เอกสาร

เลือกพอร์ตแล้วกดสแกน

การลด Attack Surface: ปิดบริการ/พอร์ตไม่จำเป็น, Network Segmentation, Firewall, ACL, จำกัด Input, API Gateway

📌
ข้อควรจำ:
  • ยิ่งซอฟต์แวร์มีฟังก์ชันมาก Attack Surface ยิ่งใหญ่
  • Attack Surface เพิ่มขึ้นตามความซับซ้อน
  • การลด Attack Surface เป็นวิธีป้องกันที่มีประสิทธิภาพ

1.7 ทักษะและเครื่องมือพื้นฐาน

1. ความรู้ด้านโปรแกรมมิ่ง: Python, JavaScript, Java, C#, Go, Input Validation, Type Safety

2. ความรู้ด้าน OS: Linux (CLI, Permissions), Windows (AD, Group Policy)

3. ความรู้ด้านเครือข่าย: OSI Model, TCP/IP, HTTP/HTTPS, DNS, TLS, Wireshark, nmap

4. ความรู้ด้าน Web: HTTP Methods, Headers, Cookies, Session, REST API, CORS

เครื่องมือ: SAST (SonarQube, Semgrep, CodeQL), DAST (OWASP ZAP, Burp Suite), SCA (Trivy, Snyk), Network (Nmap, Wireshark), Exploitation (Metasploit, SQLMap), Crypto (OpenSSL, Hashcat), Cloud (ScoutSuite, Prowler), OSINT (Shodan, Recon-ng)

1.8 จริยธรรมและกฎหมายที่เกี่ยวข้อง

1.8.1 จริยธรรม

  4. รักษาความลับ
  5. ไม่นำวิชาความรู้ไปใช้ในทางที่ผิด
