# CH-13: Security Testing (SAST, DAST, Penetration Testing)

---

## วัตถุประสงค์การเรียนรู้

เมื่อจบบทนี้นักศึกษาสามารถ:

1. อธิบายภาพรวมของ Security Testing ประเภทต่าง ๆ ได้แก่ SAST, DAST, IAST, RASP, SCA และ Fuzz Testing รวมถึงความแตกต่างด้านการทำงาน จุดแข็ง และข้อจำกัดของแต่ละประเภท
2. ใช้เครื่องมือ SAST เช่น Semgrep และ SonarQube ในการสแกน Source Code เพื่อค้นหาช่องโหว่ และวิเคราะห์ผลลัพธ์เพื่อแยกแยะ True Positive ออกจาก False Positive
3. ใช้เครื่องมือ DAST เช่น OWASP ZAP และ Burp Suite ในการสแกน Web Application แบบ Active Scan และ Passive Scan พร้อมวิเคราะห์ผลลัพธ์ที่ได้
4. อธิบายมาตรฐานการทดสอบเจาะระบบ (Penetration Testing) ได้แก่ PTES, OWASP WSTG, NIST SP 800-115 และ OSSTMM และดำเนินการทดสอบตามขั้นตอนของ PTES
5. ใช้เครื่องมือ SCA เช่น Trivy, Snyk และ OWASP Dependency-Check ในการสแกน Dependencies และ Container Images รวมถึงการสร้างและใช้งาน SBOM
6. บูรณาการเครื่องมือ Security Testing ใน CI/CD Pipeline ตามหลักการ DevSecOps และ Shift Left
7. เขียนรายงานผล Security Testing ที่ครอบคลุม Executive Summary, Technical Findings, Risk Rating ด้วย CVSS และ Remediation Recommendations

## ขอบเขตและข้อกำหนดด้านจริยธรรมของบทนี้

บทนี้ครอบคลุมหลักการและเครื่องมือสำหรับการทดสอบความปลอดภัยของซอฟต์แวร์ในทุกขั้นตอนของการพัฒนา ตั้งแต่การวิเคราะห์ Source Code (SAST) การทดสอบขณะรันไทม์ (DAST/IAST) ไปจนถึงการทดสอบเจาะระบบ (Penetration Testing) เนื้อหามีตัวอย่างเกี่ยวกับเทคนิคการโจมตีเพื่อการเรียนรู้เชิงป้องกันเท่านั้น กิจกรรมปฏิบัติการทั้งหมดต้องทำในสภาพแวดล้อมที่ได้รับอนุญาต เช่น เครื่องเสมือน (Virtual Machine), สภาพแวดล้อม Docker ที่แยกออกจากระบบจริง หรือแอปพลิเคชันตัวอย่างที่อาจารย์จัดให้เท่านั้น

**ข้อควรจำ:** ห้ามนำเทคนิคและเครื่องมือในบทนี้ไปทดสอบกับระบบของบุคคลอื่นโดยไม่ได้รับอนุญาตเป็นลายลักษณ์อักษร การใช้เครื่องมือเช่น Semgrep, OWASP ZAP, Burp Suite, Trivy หรือเทคนิคการทดสอบเจาะระบบกับระบบที่ไม่ใช่ของตนเองหรือไม่ได้รับอนุญาตอาจมีความผิดทั้งทางอาญาตาม พ.ร.บ. ว่าด้วยการกระทำผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2560 (รวมถึงฉบับแก้ไขเพิ่มเติม) และทางแพ่งตามประมวลกฎหมายแพ่งและพาณิชย์ มาตรา 420 (ละเมิด) หากพบข้อมูลส่วนบุคคลระหว่างการทดสอบ ต้องปฏิบัติตาม พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA) ด้วย — ห้ามนำข้อมูลออกจากระบบ ต้องทำลายหลังการใช้ และแจ้งเจ้าของข้อมูล การทดสอบระบบต้องได้รับอนุญาตเป็นลายลักษณ์อักษรก่อนดำเนินการทุกครั้ง รวมถึงต้องมี Rules of Engagement ที่ชัดเจนระหว่างผู้ทดสอบและเจ้าของระบบ

## เนื้อหา

### 13.1 ภาพรวมของ Security Testing

Security Testing เป็นกระบวนการประเมินความปลอดภัยของซอฟต์แวร์เพื่อค้นหาช่องโหว่ จุดอ่อน และการกำหนดค่าที่ไม่ปลอดภัยก่อนที่ผู้ไม่หวังดีจะนำไปใช้ในการโจมตี การทดสอบความปลอดภัยที่มีประสิทธิภาพต้องครอบคลุมหลายมิติ ตั้งแต่การวิเคราะห์ Source Code ไปจนถึงการทดสอบในสภาพแวดล้อมจริง

#### 13.1.1 ความสำคัญของการทดสอบความปลอดภัย

การค้นพบช่องโหว่ในช่วงท้ายของวงจรการพัฒนาซอฟต์แวร์มีค่าใช้จ่ายที่สูงกว่าการค้นพบในช่วงต้นอย่างมีนัยสำคัญ แบบจำลองค่าใช้จ่ายสัมพัทธ์ในการแก้ไขช่องโหว่ที่ยอมรับในอุตสาหกรรม (industry-accepted model) มีดังนี้:

| ช่วงที่ค้นพบช่องโหว่ | ค่าใช้จ่ายสัมพัทธ์ในการแก้ไข |
|---------------------|------------------------------|
| Requirements / Design | 1 เท่า (ฐานอ้างอิง) |
| Implementation / Coding | 6.5 เท่า |
| Integration / Testing | 15 เท่า |
| Staging / Pre-Production | 40 เท่า |
| Production | 60–100 เท่า |

การทดสอบความปลอดภัยในวงจรการพัฒนาซอฟต์แวร์จึงมุ่งเน้นแนวคิด **Shift Left** — การเลื่อนการทดสอบไปยังช่วงต้นของ SDLC ให้มากที่สุด เพื่อลดค่าใช้จ่ายและเวลาในการแก้ไขช่องโหว่

#### 13.1.2 ประเภทของ Security Testing ตามแนวทาง NIST SP 800-115

NIST SP 800-115 (Technical Guide to Information Security Testing and Assessment) แบ่งการทดสอบความปลอดภัยออกเป็น 3 ประเภทหลักตามลักษณะการดำเนินการ:

| ประเภท | คำอธิบาย | ตัวอย่าง |
|--------|----------|---------|
| **Review Techniques** | การตรวจสอบโดยไม่ต้องรันระบบ | Architecture Review, Code Review, Policy Review |
| **Target Identification and Analysis Techniques** | การระบุและวิเคราะห์เป้าหมาย | Network Discovery, Vulnerability Scanning, SAST |
| **Target Vulnerability Validation Techniques** | การยืนยันช่องโหว่ของเป้าหมาย | Penetration Testing, DAST, IAST |

นอกจากนี้ OWASP ASVS (Application Security Verification Standard) ยังเป็นกรอบที่ใช้ในการตรวจสอบความปลอดภัยของแอปพลิเคชันในเชิงลึก ครอบคลุมทั้งการตรวจสอบโค้ด, การทดสอบเจาะระบบ, และการตรวจสอบการออกแบบ

#### 13.1.3 ประเภทของ Security Testing Tools

| ประเภท | แนวทางการทดสอบ | ต้องใช้ Source Code? | แอปพลิเคชันต้องรัน? | ช่วง SDLC | พบ Runtime Issues? | อัตรา False Positive |
|--------|----------------|---------------------|--------------------|-----------|-------------------|---------------------|
| **SAST** | White-box (วิเคราะห์ Source Code) | ใช่ | ไม่ | Dev, Build | ไม่ | ปานกลาง–สูง |
| **DAST** | Black-box (ทดสอบจากภายนอก) | ไม่ | ใช่ | Test, Staging, Prod | ใช่ | ต่ำ–ปานกลาง |
| **IAST** | Grey-box (Agent ใน Runtime) | ไม่ (แต่ต้องติดตั้ง Agent) | ใช่ | Test, Staging | ใช่ | ต่ำมาก |
| **RASP** | Protection (ป้องกันขณะรันไทม์) | ไม่ (แต่ติดตั้งเป็น Agent) | ใช่ | Production | ใช่ (ป้องกัน) | ไม่มี (เป็น Protection) |
| **SCA** | Dependency Analysis | ใช่ (manifest files) | ไม่ | Dev, Build | ไม่ (แต่ระบุ known CVEs) | ต่ำ |

> **ข้อควรจำ:** ไม่มีเครื่องมือใดที่สามารถค้นหาช่องโหว่ได้ครบถ้วนทุกประเภท การใช้เครื่องมือหลายประเภทประกอบกันให้ครอบคลุมทั้ง Static, Dynamic และ Dependency Analysis จึงเป็นแนวทางที่แนะนำ

### 13.2 SAST (Static Application Security Testing)

SAST เป็นวิธีการทดสอบความปลอดภัยแบบ White-box ที่วิเคราะห์ Source Code, Bytecode หรือ Binary โดยไม่ต้องรันโปรแกรม เครื่องมือ SAST จะตรวจสอบโค้ดในเชิงสถิติเพื่อหารูปแบบที่อาจเป็นช่องโหว่ เช่น Buffer Overflow, SQL Injection, Cross-Site Scripting, Insecure Cryptographic Storage และอื่น ๆ

#### 13.2.1 หลักการทำงานของ SAST

เครื่องมือ SAST ใช้เทคนิคการวิเคราะห์โค้ดแบบต่าง ๆ ดังนี้:

| เทคนิค | คำอธิบาย | จุดแข็ง | จุดอ่อน |
|--------|----------|---------|---------|
| **Pattern Matching** | เปรียบเทียบโค้ดกับรูปแบบที่กำหนดไว้ล่วงหน้า (Rule-based) | ง่าย เร็ว ตรวจจับรูปแบบที่รู้จักได้ดี | False Positive สูง, ไม่พบช่องโหว่ที่ไม่ตรงกับรูปแบบ |
| **Data Flow Analysis** | วิเคราะห์เส้นทางการไหลของข้อมูลจาก Source (ผู้ใช้) ไปยัง Sink (ฟังก์ชันอันตราย) | พบช่องโหว่แบบ Taint Analysis เช่น SQLi, XSS | ใช้ทรัพยากรมาก, อาจพลาดช่องโหว่ที่ซับซ้อน |
| **Control Flow Analysis** | วิเคราะห์ลำดับการทำงานของโปรแกรม | ตรวจจับ Resource Leak, Unreachable Code | ไม่ครอบคลุม Data Flow |
| **Abstract Syntax Tree (AST)** | วิเคราะห์โครงสร้างไวยากรณ์ของโค้ดแบบต้นไม้ | ตรวจจับ Syntax Error และโครงสร้างที่ไม่ปลอดภัย | ไม่ครอบคลุมความหมายในเชิงลึก |

#### 13.2.2 เครื่องมือ SAST ที่นิยม

| เครื่องมือ | ประเภท | ภาษา/แพลตฟอร์มที่รองรับ | จุดเด่น |
|-----------|--------|------------------------|---------|
| **Semgrep** | Open Source | Python, Java, Go, JS/TS, Ruby, C, OCaml | กฎที่ปรับแต่งได้ง่าย, รองรับ CI/CD, ชุมชนขนาดใหญ่ |
| **SonarQube** | Open Source / Commercial | 30+ ภาษา | Quality Gates, Technical Debt, CI/CD Integration |
| **CodeQL** (GitHub) | Commercial (ฟรีสำหรับ OSS) | C/C++, C#, Go, Java, JS/TS, Python | Query-based, GitHub Integration, Code Scanning |
| **Bandit** | Open Source | Python | เฉพาะ Python, ติดตั้งและใช้งานง่าย |
| **Checkmarx** | Commercial | 25+ ภาษา | CxSAST, CxFlow สำหรับ CI/CD |
| **Fortify (Micro Focus)** | Commercial | 25+ ภาษา | Audit Workbench, CloudScan |

**ตัวอย่างการใช้ Semgrep:**

```yaml
# semgrep-rules/sql-injection.yaml
# กฎ Semgrep สำหรับตรวจจับ SQL Injection ใน Python
rules:
  - id: python-sqli-detected
    patterns:
      - pattern: |
          cursor.execute("..." + $QUERY + "...")
      - pattern-not: |
          cursor.execute("... %s ..." % ($SAFE,))
    message: >
      Detected potential SQL injection. Use parameterized queries
      instead of string concatenation.
    languages: [python]
    severity: WARNING
    metadata:
      cwe: "CWE-89: SQL Injection"
      owasp: "A05:2025 - Injection"
      category: security
```

```bash
# รัน Semgrep เพื่อสแกนโปรเจกต์
semgrep --config=auto /path/to/project

# รัน Semgrep ด้วยกฎที่กำหนดเอง
semgrep --config=/path/to/semgrep-rules/ /path/to/project

# รัน Semgrep และส่งออกผลลัพธ์ในรูปแบบ JSON
semgrep --config=auto --json /path/to/project

# รัน Semgrep แบบไม่ระบุ severity ต่ำ
semgrep --config=auto --severity ERROR,WARNING /path/to/project
```

**ตัวอย่างการใช้ Bandit สำหรับ Python:**

```bash
# ติดตั้ง Bandit
pip install bandit

# สแกนโปรเจกต์ Python
bandit -r /path/to/project

# ส่งออกผลลัพธ์ในรูปแบบ JSON
bandit -r /path/to/project -f json -o bandit-report.json

# ละเว้นเส้นทางที่ไม่ต้องการสแกน
bandit -r /path/to/project --exclude /path/to/project/tests
```

#### 13.2.3 การจัดการ False Positives

ปัญหาหลักของ SAST คือ False Positive — ผลลัพธ์ที่แจ้งเตือนช่องโหว่ที่ไม่มีอยู่จริง แนวทางในการจัดการ:

1. **Triage Findings:** จัดลำดับความสำคัญของผลลัพธ์ตามความรุนแรง (CVSS Score), ความน่าเชื่อถือของกฎ (Rule Confidence), และบริบทของโค้ด
2. **Suppression with Justification:** ระงับ False Positive ที่ชัดเจนพร้อมระบุเหตุผล (Audit Trail)
3. **Custom Rules Tuning:** ปรับแต่งกฎให้เหมาะสมกับบริบทของโปรเจกต์
4. **Baseline Comparison:** ใช้ Baselines เพื่อระบุเฉพาะ Finding ใหม่ที่เพิ่มเข้ามาในโค้ดที่เปลี่ยนแปลง
5. **Verify with DAST:** ยืนยันช่องโหว่ที่ SAST พบด้วย DAST หรือการตรวจสอบด้วยตนเอง

```python
# ตัวอย่างการ Suppress False Positive ใน Bandit
import hashlib  # nosec  # แจ้ง Bandit ว่าไม่ต้องตรวจสอบบรรทัดนี้

# หรือใช้ # nosec เพื่อระงับการแจ้งเตือนทุกรายการในบรรทัด
password = "supersecret"  # nosec
```

#### 13.2.4 SAST Workflow ใน IDE และ CI/CD

**IDE Integration:**
- **VS Code:** ใช้ Semgrep Extension, SonarLint
- **IntelliJ IDEA:** ใช้ SonarLint, Qodana
- **GitHub Code Scanning:** ใช้ CodeQL ใน GitHub Actions

**CI/CD Integration:**

```yaml
# .github/workflows/semgrep.yml
name: Semgrep SAST Scan
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  semgrep:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Semgrep
        uses: semgrep/semgrep-action@v1
        with:
          config: p/default
          audit_on: push
          severity: ERROR,WARNING
```

#### 13.2.5 ข้อจำกัดของ SAST

1. **ไม่สามารถตรวจสอบ Runtime Behavior:** SAST ไม่ทราบว่าโปรแกรมทำงานอย่างไรในสภาพแวดล้อมจริง
2. **ไม่พบช่องโหว่ที่เกิดจาก Configuration:** ปัญหาเช่น Misconfigured Cloud Services หรือ Weak TLS Configuration ไม่สามารถตรวจพบได้จาก Source Code
3. **False Positive สูง:** โดยเฉพาะกับภาษาที่มี Dynamic Typing หรือ Reflection
4. **การวิเคราะห์ Dependency Chains:** การติดตาม Data Flow ผ่านหลาย Layer อาจไม่สมบูรณ์
5. **ภาษาและ Framework ที่ไม่รองรับ:** เครื่องมือบางตัวไม่ครอบคลุมภาษาเฉพาะหรือ Framework ใหม่

> **ข้อควรจำ:** SAST ควรใช้เป็นส่วนหนึ่งของการทดสอบความปลอดภัย ไม่ใช่เครื่องมือเดียวที่พึ่งพา การใช้ SAST ร่วมกับ DAST และการตรวจสอบด้วยมนุษย์ให้ผลลัพธ์ที่ดีที่สุด

### 13.3 DAST (Dynamic Application Security Testing)

DAST เป็นวิธีการทดสอบความปลอดภัยแบบ Black-box ที่ทดสอบแอปพลิเคชันขณะทำงานจริง (Runtime) โดยไม่ต้องเข้าถึง Source Code เครื่องมือ DAST จะส่งคำขอที่ออกแบบมาเพื่อโจมตีไปยังแอปพลิเคชันและวิเคราะห์การตอบสนองเพื่อหาช่องโหว่

#### 13.3.1 หลักการทำงานของ DAST

DAST ทำงานโดยการจำลองการโจมตีจากมุมมองของผู้ไม่หวังดีที่อยู่ภายนอกระบบ:

1. **Crawling/Spidering:** สำรวจโครงสร้างของแอปพลิเคชันเพื่อค้นหา URL, Parameters, Forms
2. **Passive Scanning:** วิเคราะห์ HTTP Request/Response โดยไม่ส่งคำขอที่เปลี่ยนแปลง
3. **Active Scanning:** ส่ง Payload ที่ออกแบบมาเพื่อกระตุ้นช่องโหว่
4. **Analysis:** วิเคราะห์การตอบสนองเพื่อตรวจสอบว่าช่องโหว่ถูกกระตุ้นสำเร็จหรือไม่

#### 13.3.2 Active Scan vs Passive Scan

| มิติ | Passive Scan | Active Scan |
|-----|-------------|-------------|
| ลักษณะ | วิเคราะห์การรับส่งข้อมูลปกติ ไม่เปลี่ยนแปลง Request | ส่งคำขอที่ดัดแปลง (Payload) เพื่อทดสอบช่องโหว่ |
| ผลกระทบ | ไม่กระทบต่อระบบ (ปลอดภัย) | อาจทำให้ระบบเสียหายหรือทำงานผิดปกติได้ |
| ความเร็ว | รวดเร็ว | ช้า เนื่องจากต้องส่งหลาย Payload ต่อจุด |
| การครอบคลุม | จำกัดเฉพาะสิ่งที่สังเกตเห็นในการรับส่งข้อมูล | ครอบคลุมหลายรูปแบบการโจมตี |
| ความเสี่ยง | ต่ำมาก | ปานกลางถึงสูง (ต้องใช้ในสภาพแวดล้อมที่ไม่ใช่ Production) |

#### 13.3.3 เครื่องมือ DAST ที่นิยม

| เครื่องมือ | ประเภท | จุดเด่น |
|-----------|--------|---------|
| **OWASP ZAP** (Zed Attack Proxy) | Open Source | ฟรี, รองรับ Passive/Active Scan, API Testing, Automation Framework, มี Community Add-ons |
| **Burp Suite** (PortSwigger) | Commercial (มี Community Edition ฟรี) | Professional Edition มีความสามารถขั้นสูง, Scanner, Repeater, Intruder, Extender |
| **StackHawk** | Commercial | CI/CD-native, รองรับ GraphQL, gRPC, การทดสอบ API |
| **Netsparker / Invicti** | Commercial | Proof-Based Scanning — ยืนยันช่องโหว่อัตโนมัติโดยไม่ต้องตรวจสอบด้วยคน |

**ตัวอย่างการใช้ OWASP ZAP ผ่าน Command Line:**

```bash
# 1. รัน ZAP ในโหมด Headless (ไม่มี GUI)
zap.sh -daemon -host 0.0.0.0 -port 8080

# 2. ใช้ ZAP API เพื่อเริ่ม Spider (Quick Scan)
curl "http://localhost:8080/JSON/spider/action/scan/?url=http://target-site.com&maxChildren=10"

# 3. ตรวจสอบสถานะ Spider
curl "http://localhost:8080/JSON/spider/view/status/"

# 4. เริ่ม Active Scan
curl "http://localhost:8080/JSON/ascan/action/scan/?url=http://target-site.com&recurse=true"

# 5. ดึงผลลัพธ์การสแกน
curl "http://localhost:8080/JSON/ascan/view/scanProgress/?scanId=0"

# 6. ตรวจสอบ Alerts ที่พบ
curl "http://localhost:8080/JSON/core/view/alerts/?baseurl=http://target-site.com"
```

**ตัวอย่างการใช้ OWASP ZAP API ผ่าน Python (zapv2):**

```python
from zapv2 import ZAPv2

# เชื่อมต่อกับ ZAP Daemon
zap = ZAPv2(apikey='', proxies={'http': 'http://127.0.0.1:8080', 'https': 'http://127.0.0.1:8080'})

target = 'http://test-site.com'

# เริ่ม Spider
print(f'Spidering target {target}')
scan_id = zap.spider.scan(target)
while int(zap.spider.status(scan_id)) < 100:
    print(f'Spider progress: {zap.spider.status(scan_id)}%')
zap.spider.stop(scan_id)

# เริ่ม Active Scan
print(f'Active Scanning target {target}')
scan_id = zap.ascan.scan(target)
while int(zap.ascan.status(scan_id)) < 100:
    print(f'Active Scan progress: {zap.ascan.status(scan_id)}%')

# แสดงผลลัพธ์
alerts = zap.core.alerts()
for alert in alerts:
    print(f'Alert: {alert["alert"]} | Risk: {alert["risk"]} | URL: {alert["url"]}')
```

#### 13.3.4 API Testing ด้วย DAST

DAST สมัยใหม่รองรับการทดสอบ API โดยตรง ไม่ใช่แค่การโจมตีผ่านหน้าเว็บ:

```yaml
# ตัวอย่าง ZAP API Scan Configuration (openapi-scan.yaml)
env:
  contexts:
    - name: "API Context"
      urls:
        - "http://api.example.com"
  parameters:
    failOnError: true
    progressStats: true

jobs:
  - type: openapi
    parameters:
      apiFile: "/path/to/openapi-spec.yaml"
      targetUrl: "http://api.example.com"
  
  - type: activeScan
    parameters:
      context: "API Context"
      
  - type: alertFilter
    parameters:
      - ruleId: 10003  # Retire.js findings
        newRisk: FalsePositive
```

#### 13.3.5 Authentication Handling ใน DAST

การทดสอบแอปพลิเคชันที่ต้องมีการยืนยันตัวตนเป็นความท้าทายของ DAST:

| วิธีการ | คำอธิบาย | ข้อดี | ข้อเสีย |
|--------|----------|------|--------|
| **Session Management** | กำหนด Session Token ให้ DAST ใช้ | ง่าย เร็ว | Token อาจหมดอายุระหว่างการสแกน |
| **Form-Based Auth** | DAST กรอกฟอร์ม Login อัตโนมัติ | ครอบคลุมมากขึ้น | ต้องกำหนดค่าฟอร์ม, CAPTCHA อาจเป็นอุปสรรค |
| **API Key/Bearer Token** | ใช้ API Key หรือ JWT ในการยืนยันตัวตน | เหมาะสำหรับ API Testing | ต้องเตรียม Token ที่ถูกต้อง |
| **OAuth 2.0** | DAST รองรับ OAuth 2.0 Flow | ครอบคลุมการทดสอบ Flow จริง | ตั้งค่าซับซ้อน |

> **ข้อควรจำ:** การทดสอบ DAST กับสภาพแวดล้อม Production ต้องได้รับการอนุญาตเป็นลายลักษณ์อักษรและควรใช้เฉพาะ Passive Scan หรือกำหนดขอบเขตให้ชัดเจน Active Scan ควรดำเนินการในสภาพแวดล้อม Staging หรือ Test เท่านั้น

### 13.4 IAST (Interactive Application Security Testing)

IAST เป็นวิธีการทดสอบความปลอดภัยแบบ Grey-box ที่ทำงานโดยการติดตั้ง Agent ภายในแอปพลิเคชัน (Instrumentation Agent) เพื่อวิเคราะห์การทำงานของแอปพลิเคชันในระหว่างการทดสอบ (ทั้ง Manual และ Automated Test) แบบ Real-time

#### 13.4.1 หลักการทำงานของ IAST

IAST Agent จะถูกติดตั้งใน Application Server (หรือ Sidecar) และเฝ้าสังเกต:

1. **Data Flow (Taint Analysis):** ติดตามเส้นทางการไหลของข้อมูลจากอินพุตของผู้ใช้ไปยังฟังก์ชันอันตราย
2. **Control Flow:** วิเคราะห์ว่าข้อมูลที่ป้อนเข้าไปถูกนำไปใช้ในบริบทที่ปลอดภัยหรือไม่
3. **Code Coverage:** แจ้งเตือนเมื่อมีโค้ดที่ไม่ได้ถูกทดสอบ
4. **Configuration Analysis:** ตรวจสอบการตั้งค่าความปลอดภัยของเฟรมเวิร์ก

#### 13.4.2 Taint Analysis ใน IAST

Taint Analysis เป็นเทคนิคหลักของ IAST ที่ติดตามข้อมูลจาก **Source** (แหล่งที่มาที่ไม่น่าเชื่อถือ) ไปยัง **Sink** (ฟังก์ชันที่อาจเป็นอันตราย):

```python
# ตัวอย่าง Taint Analysis ที่ IAST ตรวจจับ
from flask import request, render_template_string

@app.route('/greet')
def greet():
    name = request.args.get('name', '')  # Source: Input จากผู้ใช้
    # Sink: IAST ตรวจจับว่า name ถูกส่งไปยัง render_template_string
    # ซึ่งเป็น Template Engine ที่อาจเกิด SSTI (Server-Side Template Injection)
    return render_template_string(f"Hello {{name}}")  # Safe — IAST ตรวจพบว่ามี Escaping

    # ถ้าเป็น:
    # return render_template_string(f"Hello {name}")  # ไม่ปลอดภัย — SSTI
    # IAST จะ Alert ทันทีว่า name (Tainted) ถูกใช้ใน Sink ที่อันตราย
```

#### 13.4.3 ข้อดีของ IAST เมื่อเทียบกับ SAST และ DAST

| มิติ | SAST | DAST | IAST |
|-----|------|------|------|
| อัตรา False Positive | สูง | ปานกลาง | **ต่ำที่สุด** (ตรวจสอบยืนยันที่ Runtime) |
| การครอบคลุม Code | ทั้งหมด (Static) | เฉพาะที่เข้าถึงได้ | **เฉพาะที่ถูกทดสอบจริง** |
| การระบุตำแหน่งที่แน่นอน | ระดับบรรทัดโค้ด | ระดับ URL/Parameter | **ระดับบรรทัดโค้ด** (จาก Stack Trace ที่ Runtime) |
| การทำงานใน CI/CD | ดี | ปานกลาง | **ดี** (Agent-based) |
| Resource Usage | ต่ำ (แค่สแกนไฟล์) | ปานกลาง (ต้องรอสแกนเสร็จ) | **ต่ำ** (วิเคราะห์ ณ. Runtime) |

#### 13.4.4 เครื่องมือ IAST

| เครื่องมือ | ประเภท | แพลตฟอร์ม | จุดเด่น |
|-----------|--------|-----------|---------|
| **Contrast Assess** | Commercial | Java, .NET, Node.js, Python, Go | IAST ชั้นนำ, False Positive Rate ต่ำมาก, CI/CD Integration |
| **HCL AppScan** (IAST Option) | Commercial | Java, .NET | รองรับทั้ง SAST+DAST+IAST |
| **Seeker (Synopsys)** | Commercial | Java, .NET, Node.js, Python | รองรับการวิเคราะห์หลายภาษา |
| **Veracode Interactive Analysis** | Commercial | Java, .NET | แพลตฟอร์ม Cloud-based |

### 13.5 RASP (Runtime Application Self-Protection)

RASP เป็นเทคโนโลยีที่แตกต่างจากเครื่องมือทดสอบความปลอดภัยอื่น ๆ ตรงที่ RASP ไม่ใช่เครื่องมือสำหรับค้นหาช่องโหว่ แต่เป็นเครื่องมือสำหรับ **ป้องกัน** การโจมตีในขณะที่แอปพลิเคชันทำงานจริง (Production)

#### 13.5.1 หลักการทำงานของ RASP

RASP ถูกติดตั้งเป็น Agent หรือ Library ใน Application Server และทำงานใน 3 โหมดหลัก:

| โหมด | คำอธิบาย | การใช้งาน |
|------|----------|-----------|
| **Monitor Mode** | บันทึกและแจ้งเตือนเมื่อตรวจพบการโจมตี แต่ไม่บล็อก | ใช้ในช่วงเริ่มต้นเพื่อทำความเข้าใจพฤติกรรม |
| **Block Mode** | บล็อกการโจมตีทันทีที่ตรวจพบ | ใช้ใน Production เมื่อมั่นใจในความถูกต้อง |
| **Protect Mode** | ป้องกันโดยการ Cleanse Input หรือ Redirect | ใช้เมื่อต้องการความยืดหยุ่น |

#### 13.5.2 ความแตกต่างระหว่าง RASP และ Testing Tools

| มิติ | SAST/DAST/IAST | RASP |
|-----|---------------|------|
| วัตถุประสงค์ | ค้นหาช่องโหว่ (Testing) | ป้องกันการโจมตี (Protection) |
| ช่วง SDLC | Dev, Test, Staging | **Production** |
| การตอบสนอง | รายงาน (Report) | **แทรกแซงและบล็อก** (Block) |
| ผลกระทบต่อประสิทธิภาพ | ไม่มี (วิเคราะห์ภายนอก) | มี (แทรกแทรกซึมใน Runtime) — ต้องปรับแต่ง |

#### 13.5.3 RASP vs WAF

| มิติ | WAF (Web Application Firewall) | RASP |
|-----|-------------------------------|------|
| ตำแหน่ง | หน้าแอปพลิเคชัน (Network Edge) | **ใน**แอปพลิเคชัน (ได้ Context เต็ม) |
| การมองเห็น | เห็น HTTP Request/Response | เห็น Code, Data Flow, Runtime Context |
| การตรวจจับ | Signature-based + Behavioral | **Context-aware** — เข้าใจ Business Logic |
| Bypass | สามารถ Bypass ได้ | ยากที่จะ Bypass (อยู่ใน Application) |
| การจัดการ | ตั้งค่ากฎที่ WAF | ตั้งค่าใน Agent |

#### 13.5.4 เครื่องมือ RASP

| เครื่องมือ | ประเภท | แพลตฟอร์ม |
|-----------|--------|-----------|
| **Contrast Protect** | Commercial | Java, .NET, Node.js, Python, Go |
| **HCL AppScan** (RASP Option) | Commercial | Java, .NET |
| **Imperva RASP** | Commercial | Java, .NET |
| **Signal Sciences (Fastly)** | Commercial | ทำงานเป็น Agent ใน Web Server |

> **ข้อควรจำ:** RASP ไม่ใช่ Test Tool — ไม่ควรใช้ RASP แทนการทดสอบความปลอดภัย แต่ควรใช้เป็นชั้นป้องกันเสริมหลังจากที่ได้ทดสอบและแก้ไขช่องโหว่ด้วย SAST/DAST แล้ว

### 13.6 Penetration Testing

Penetration Testing (การทดสอบเจาะระบบ) เป็นกระบวนการจำลองการโจมตีจริงต่อระบบเป้าหมายเพื่อประเมินความปลอดภัยในมุมมองของผู้โจมตี แตกต่างจาก SAST/DAST ที่เป็น Automated Scanning ตรงที่ Penetration Testing ต้องอาศัยทักษะ ความคิดสร้างสรรค์ และการตีความของมนุษย์ร่วมกับเครื่องมือ

#### 13.6.1 มาตรฐานและระเบียบวิธี

| มาตรฐาน | หน่วยงาน | รายละเอียด |
|---------|---------|-------------|
| **PTES** (Penetration Testing Execution Standard) | ชุมชนความปลอดภัย | 7 ขั้นตอน: Pre-engagement → Intelligence Gathering → Threat Modeling → Vulnerability Analysis → Exploitation → Post-Exploitation → Reporting |
| **OWASP WSTG** (Web Security Testing Guide) v4.2 | OWASP | 12 หมวดหมู่, 91 Test Cases (ครอบคลุม Information Gathering, Configuration, Identity, Auth, AuthZ, Session, Input Validation, Error Handling, Cryptography, Business Logic, Client-Side, API) |
| **NIST SP 800-115** | NIST | Technical Guide to Information Security Testing and Assessment, ครอบคลุมทั้ง Discovery, Vulnerability Identification และ Exploitation |
| **OSSTMM** (Open Source Security Testing Methodology Manual) | ISECOM | ครอบคลุม Human Security, Physical Security, Wireless, Telecom, Data Networks และ Compliance |

#### 13.6.2 ขั้นตอนของ Penetration Testing ตาม PTES

**1. Pre-engagement Interactions:**
- การกำหนดขอบเขต (Scope) — ระบบใดบ้างที่ได้รับอนุญาตให้ทดสอบ
- Rules of Engagement (RoE) — เวลาทดสอบ, วิธีการ, การแจ้งเตือน
- การเซ็นสัญญาอนุญาต (Authorization Letter)
- ช่องทางการติดต่อกรณีฉุกเฉิน (Emergency Contact)

**2. Intelligence Gathering / Reconnaissance:**
- Passive Recon: OSINT (Shodan, Censys, Google Dork, WHOIS, DNS Records)
- Active Recon: nmap, dnsrecon, subdomain enumeration
- เทคโนโลยีเป้าหมาย (Wappalyzer, WhatWeb, BuiltWith)

```bash
# Passive Recon — ใช้ Shodan API
curl -s "https://api.shodan.io/shodan/host/203.0.113.50?key=SHODAN_API_KEY"

# Active Recon — Scan พอร์ตด้วย nmap
nmap -sV -sC -p- -T4 target.com

# Subdomain Enumeration
sublist3r -d target.com
```

**3. Threat Modeling:**
- ระบุทรัพย์สินที่มีค่า (Crown Jewels)
- วิเคราะห์เส้นทางการโจมตี (Attack Path)
- ใช้ Framework เช่น STRIDE (Spoofing, Tampering, Repudiation, Info Disclosure, DoS, Elevation of Privilege)

**4. Vulnerability Analysis:**
- สแกนช่องโหว่อัตโนมัติด้วย Nessus, OpenVAS, OWASP ZAP
- Manual Verification — ยืนยันผลลัพธ์และลด False Positive
- การค้นหาช่องโหว่ที่ไม่ครอบคลุมโดย Automatic Scanner (Business Logic Flaws)

**5. Exploitation:**
- ยืนยันว่าช่องโหว่สามารถถูกโจมตีได้จริง
- ใช้เครื่องมือเช่น Metasploit, Burp Suite, SQLMap, หรือ Custom Exploit
- **ข้อจำกัด:** ดำเนินการตาม RoE เท่านั้น — ห้ามทำให้ระบบเสียหายหากไม่ได้รับอนุญาต

**6. Post-Exploitation:**
- การยกระดับสิทธิ์ (Privilege Escalation)
- การเคลื่อนที่ในแนวราบ (Lateral Movement) — ถ้าอยู่ในขอบเขต
- การรักษาการเข้าถึง (Persistence)
- การรวบรวมข้อมูล (Data Exfiltration Simulation)

**7. Reporting:**
- Executive Summary — สำหรับผู้บริหาร
- Technical Findings — สำหรับทีมพัฒนา
- Risk Rating — CVSS Scoring
- Remediation Recommendations
- Screenshots และ PoC Code

#### 13.6.3 แนวทางการทดสอบ: Black-box, White-box, Grey-box

| แนวทาง | คำอธิบาย | ข้อมูลที่ให้ผู้ทดสอบ | ข้อดี | ข้อเสีย |
|--------|----------|-------------------|------|--------|
| **Black-box** | ไม่ทราบข้อมูลภายในระบบเลย | เฉพาะ URL หรือ IP | จำลองการโจมตีจากภายนอกได้สมจริง | ใช้เวลานาน, อาจพลาดช่องโหว่ภายใน |
| **White-box** | ทราบข้อมูลทั้งหมด (Source Code, Architecture, Credentials) | Full Access | ครอบคลุมมากที่สุด, พบช่องโหว่ลึก | ไม่สมจริง, ใช้ทรัพยากรมาก |
| **Grey-box** | ทราบข้อมูลบางส่วน (Credentials พื้นฐาน, API Docs) | Limited Access | สมดุลระหว่างความสมจริงและความครอบคลุม | ต้องกำหนดระดับข้อมูลที่ให้ |

#### 13.6.4 Rules of Engagement (RoE)

RoE เป็นเอกสารที่กำหนดกติกาในการทดสอบเจาะระบบ:

| หัวข้อ | รายละเอียด |
|-------|------------|
| **ขอบเขต (Scope)** | IP Address, Domain Name, API Endpoint, Application ที่ได้รับอนุญาต |
| **นอกขอบเขต (Out of Scope)** | ระบบที่ไม่ได้รับอนุญาต, ระบบของบุคคลที่สาม, ข้อมูลส่วนบุคคล |
| **เวลาทดสอบ** | ช่วงเวลาที่อนุญาตให้ทดสอบ (นอกเวลาทำการ, วันหยุด) |
| **วิธีการที่อนุญาต** | Social Engineering, Physical Access, DoS (ถ้าได้รับอนุญาต) |
| **การแจ้งเตือน** | ต้องแจ้งทีม SOC/Security ก่อนและระหว่างการทดสอบ |
| **ข้อมูลติดต่อฉุกเฉิน** | ผู้ประสานงานที่ติดต่อได้ 24/7 หากเกิดเหตุไม่คาดฝัน |
| **การจัดการข้อมูล** | ข้อมูลที่พบระหว่างการทดสอบต้องถูกทำลายหลังส่งรายงาน |

> **ข้อควรจำ:** การทดสอบเจาะระบบโดยไม่มี RoE ที่ชัดเจนหรือไม่ได้รับอนุญาตเป็นลายลักษณ์อักษรอาจเข้าข่ายความผิดทางอาญาตาม พ.ร.บ. ว่าด้วยการกระทำผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2560 ผู้ทดสอบควรตรวจสอบให้แน่ใจว่ามีเอกสารอนุญาตที่ลงนามโดยผู้มีอำนาจก่อนเริ่มดำเนินการเสมอ

#### 13.6.5 OWASP Web Security Testing Guide (WSTG) v4.2

WSTG v4.2 เป็นแนวทางการทดสอบความปลอดภัยของเว็บแอปพลิเคชันที่ละเอียดและได้รับการยอมรับในวงกว้าง ประกอบด้วย 12 หมวดหมู่หลัก:

| หมวดหมู่ | รหัส | ขอบเขต |
|---------|------|---------|
| 1. Information Gathering | WSTG-INFO | การรวบรวมข้อมูลเบื้องต้น, Fingerprint เทคโนโลยี, การค้นหา Hidden Content |
| 2. Configuration and Deployment Management | WSTG-CONF | การกำหนดค่าเซิร์ฟเวอร์, TLS, HTTP Methods, Cloud Storage Exposure |
| 3. Identity Management | WSTG-IDNT | Account Enumeration, Username Policy, Registration Flow |
| 4. Authentication | WSTG-ATHN | Default Credentials, Brute Force, MFA Bypass, Password Reset |
| 5. Authorization | WSTG-AUTHZ | IDOR, Path Traversal, Privilege Escalation, RBAC Testing |
| 6. Session Management | WSTG-SESS | Cookie Attributes, CSRF, Session Fixation, Logout Behavior |
| 7. Input Validation | WSTG-INPV | SQLi, XSS, SSTI, Command Injection, XXE, HTTP Splitting |
| 8. Error Handling | WSTG-ERRH | Stack Trace Disclosure, Error Message Verbosity |
| 9. Cryptography | WSTG-CRYP | Weak Ciphers, Insecure TLS, Padding Oracle, Hash Strength |
| 10. Business Logic | WSTG-BUSL | Race Conditions, Price Manipulation, Workflow Bypass |
| 11. Client-Side | WSTG-CLNT | DOM XSS, Clickjacking, HTML Injection, WebSocket Misuse |
| 12. API Testing | WSTG-APIT | REST/GraphQL/gRPC, Mass Assignment, API Key Exposure |

### 13.7 Software Composition Analysis (SCA)

SCA เป็นกระบวนการวิเคราะห์ความปลอดภัยของ Component และ Dependency ของบุคคลที่สามที่ถูกนำมาใช้ในซอฟต์แวร์ โดยเปรียบเทียบ Version ของ Library กับฐานข้อมูล CVE (Common Vulnerabilities and Exposures) ที่รู้จัก

#### 13.7.1 ความสำคัญของ SCA

ตามรายงาน OWASP Top 10:2025, A03 — Software Supply Chain Failures เป็นหมวดหมู่ใหม่ที่สะท้อนถึงความเสี่ยงที่เพิ่มขึ้นจาก Dependency ที่ไม่ปลอดภัย แนวคิดหลักคือ "คุณไม่สามารถรับผิดชอบความปลอดภัยของโค้ดที่คุณไม่ได้เขียน"

สถิติที่น่าสนใจ:
- มากกว่า 90% ของแอปพลิเคชันสมัยใหม่ใช้ Open Source Components
- โดยเฉลี่ยแอปพลิเคชันหนึ่งมี Dependency มากกว่า 500 ตัว (รวม Transitive)
- ช่องโหว่ใน Transitive Dependency (Dependency ของ Dependency) คิดเป็น 40–60% ของช่องโหว่ทั้งหมด

#### 13.7.2 เครื่องมือ SCA

| เครื่องมือ | ประเภท | ภาษา/แพลตฟอร์ม | จุดเด่น |
|-----------|--------|----------------|---------|
| **Snyk** | Commercial (มี Free Tier) | 30+ ภาษา | Fix PRs อัตโนมัติ, Container Scanning, IaC Scanning, CI/CD Integration |
| **Dependabot** (GitHub) | ฟรี (สำหรับ GitHub) | GitHub Ecosystem | Security Updates อัตโนมัติ, Version Updates, Integrated with GitHub |
| **Trivy** (Aqua Security) | Open Source | Container, OS Packages, ภาษา | สแกน Container Image, Filesystem, Git Repo, SBOM, ใช้งานง่าย |
| **OWASP Dependency-Check** | Open Source | Java, .NET, Python, Ruby, Node.js | ฟรี, OWASP Project, Jenkins/CLI/Maven Plugin |

#### 13.7.3 ตัวอย่างการใช้ Trivy

```bash
# ติดตั้ง Trivy บน macOS
brew install trivy

# สแกน Container Image
trivy image nginx:1.25

# สแกน Container Image ด้วยรูปแบบผลลัพธ์ที่กำหนด
trivy image --severity CRITICAL,HIGH nginx:1.25

# สแกน Filesystem
trivy fs /path/to/project

# สแกน Git Repository
trivy repo https://github.com/aquasecurity/trivy

# สแกนเฉพาะ OS Packages
trivy image --pkg-types os nginx:1.25

# สร้าง SBOM (Software Bill of Materials) ในรูปแบบ CycloneDX
trivy image --format cyclonedx --output sbom.json nginx:1.25

# สแกน SBOM ที่มีอยู่
trivy sbom sbom.json

# ส่งออกผลลัพธ์ในรูปแบบ HTML
trivy image --format template --template "html" nginx:1.25
```

#### 13.7.5 SBOM (Software Bill of Materials)

SBOM เป็นรายการของ Component ทั้งหมดที่ประกอบขึ้นเป็นซอฟต์แวร์ รวมถึง Version, License, และ Dependency Relationship มาตรฐาน SBOM ที่นิยม:

| มาตรฐาน | รายละเอียด | จุดเด่น |
|---------|-----------|---------|
| **CycloneDX** (OWASP) | มาตรฐานแบบ XML/JSON ที่เน้นความปลอดภัย | รองรับ Component Metadata, License, Service, Vulnerability, Cryptography |
| **SPDX** (Linux Foundation) | มาตรฐาน ISO/IEC 5962:2021 | เน้น License Compliance, ISO มาตรฐานสากล |
| **SWID** (ISO/IEC 19770-2) | มาตรฐาน ISO สำหรับ Tagging ซอฟต์แวร์ | เน้น Identification และ Inventory |

**ความสำคัญของ SBOM ต่อ Security Testing:**

1. **Visibility:** ทราบว่า Dependency อะไรบ้างที่ถูกใช้งาน รวมถึง Transitive Dependency
2. **Vulnerability Management:** เมื่อมี CVE ใหม่ — ตรวจสอบได้ทันทีว่าได้รับผลกระทบหรือไม่
3. **License Compliance:** ตรวจสอบ License ของทุก Component ให้สอดคล้องกับนโยบายองค์กร
4. **Supply Chain Security:** ปฏิบัติตาม Executive Order (USA) 14028 และแนวทาง NIST SSDF

**ตัวอย่าง SBOM ในรูปแบบ CycloneDX:**

```json
{
  "$schema": "http://cyclonedx.org/schema/bom-1.5.schema.json",
  "bomFormat": "CycloneDX",
  "specVersion": "1.5",
  "version": 1,
  "metadata": {
    "component": {
      "name": "my-app",
      "version": "1.0.0",
      "type": "application"
    }
  },
  "components": [
    {
      "type": "library",
      "name": "express",
      "version": "4.18.2",
      "licenses": [
        { "license": { "id": "MIT" } }
      ],
      "purl": "pkg:npm/express@4.18.2"
    },
    {
      "type": "library",
      "name": "lodash",
      "version": "4.17.21",
      "licenses": [
        { "license": { "id": "MIT" } }
      ],
      "purl": "pkg:npm/lodash@4.17.21"
    }
  ],
  "dependencies": [
    {
      "ref": "pkg:npm/my-app@1.0.0",
      "dependsOn": [
        "pkg:npm/express@4.18.2",
        "pkg:npm/lodash@4.17.21"
      ]
    }
  ]
}
```

```bash
# สร้าง SBOM ด้วย CycloneDX (Node.js)
npx @cyclonedx/cyclonedx-npm --output-file sbom.json

# สร้าง SBOM ด้วย Trivy
trivy image --format cyclonedx --output sbom.json my-app:latest

# สร้าง SBOM ในรูปแบบ SPDX
trivy image --format spdx-json --output spdx.json my-app:latest
```

#### 13.7.6 License Compliance

| หมวด License | ตัวอย่าง | ข้อควรระวัง |
|-------------|---------|------------|
| **Permissive** | MIT, Apache 2.0, BSD | ใช้งานได้อิสระ, ต้องระบุ Copyright |
| **Weak Copyleft** | LGPL, MPL | ใช้ใน Library ได้ถ้า Dynamic Link |
| **Strong Copyleft** | GPL, AGPL | ห้ามรวมใน Commercial Code ถ้าไม่เปิด Source ทั้งหมด |
| **Proprietary** | Commercial Licenses | ต้องซื้อสิทธิ์การใช้งาน |

### 13.8 Fuzz Testing

Fuzz Testing หรือ Fuzzing เป็นเทคนิคการทดสอบความปลอดภัยที่ป้อนข้อมูลที่ผิดปกติ ไม่คาดคิด หรือสุ่ม (Fuzz) เข้าไปในโปรแกรมเพื่อกระตุ้นให้เกิดพฤติกรรมที่ไม่ปลอดภัย เช่น Crash, Memory Corruption, หรือ Assertion Failure

#### 13.8.1 ประเภทของ Fuzzing

| ประเภท | คำอธิบาย | ข้อดี | ข้อเสีย |
|--------|----------|------|--------|
| **Dumb Fuzzing** | ป้อนข้อมูลสุ่มโดยไม่รู้โครงสร้าง | ง่าย, เร็ว | ครอบคลุมน้อย |
| **Smart/Generational Fuzzing** | ป้อนข้อมูลตามรูปแบบที่รู้จัก (Protocol, File Format) | ครอบคลุมมากกว่า | ต้องมีความรู้เรื่องรูปแบบข้อมูล |
| **Mutation-based Fuzzing** | เริ่มจากข้อมูลที่ถูกต้องแล้วเปลี่ยนแปลงเล็กน้อย | ได้ผลดีกับ Input ที่ซับซ้อน | อาจพลาดกรณีที่ต้องเปลี่ยนโครงสร้าง |
| **Coverage-guided Fuzzing** | ป้อนข้อมูลและติดตาม Code Coverage เพื่อเพิ่มการครอบคลุม | ครอบคลุมมากที่สุด | ซับซ้อน, ใช้ทรัพยากรมาก |

#### 13.8.2 ช่องโหว่ที่ Fuzzing ช่วยค้นหา

| ประเภทช่องโหว่ | คำอธิบาย | ตัวอย่าง CVE |
|----------------|----------|-------------|
| **Buffer Overflow** | การเขียนข้อมูลเกินขนาด Buffer | CVE-2024-2961 (glibc) |
| **Integer Overflow** | การคำนวณเลขเกินขอบเขตที่ตัวแปรรองรับ | CVE-2025-31496 (Linux Kernel) |
| **Format String** | การใช้ User Input เป็น Format String | CVE-2025-1394 (GNU Tar) |
| **Use-After-Free** | การใช้หน่วยความจำที่ถูกปล่อยไปแล้ว | CVE-2025-24816 (Linux Kernel) |
| **Null Pointer Dereference** | การเข้าถึง Null Pointer | CVE-2025-11571 (PHP) |

#### 13.8.3 เครื่องมือ Fuzzing

| เครื่องมือ | ประเภท | เหมาะกับ |
|-----------|--------|---------|
| **AFL** (American Fuzzy Lop) | Coverage-guided | C/C++ Binary |
| **libFuzzer** | Coverage-guided (In-process) | C/C++ Library |
| **Jazzer** | Coverage-guided | Java/Kotlin |
| **zzuf** | Mutation-based | File Input, Network Input |
| **Burp Suite Intruder** | Mutation-based | Web Application, API |

#### 13.9.2 Quality Gates

Quality Gates เป็นเกณฑ์ที่ใช้ตัดสินว่า Build ผ่านหรือไม่ผ่านตามผลการทดสอบความปลอดภัย:

| Quality Gate | ตัวอย่างค่า | การกระทำ |
|-------------|------------|----------|
| **Critical Severity** | 0 พบ | Blocker — ห้าม Deploy |
| **High Severity** | ≤ 3 รายการ | Blocker — ห้าม Deploy |
| **Medium Severity** | ≤ 10 รายการ | Warning — ต้องแก้ไขภายใน 7 วัน |
| **Low Severity** | ≤ 20 รายการ | Info — บันทึกใน Backlog |
| **Code Coverage** | ≥ 80% | Warning หากต่ำกว่า |
| **Dependency Check** | 0 Known Critical CVE | Blocker — ห้าม Deploy |

**ตัวอย่างการตั้งค่า Quality Gate ใน SonarQube:**

```yaml
# sonar-project.properties
sonar.qualitygate=true
sonar.qualitygate.wait=true
sonar.qualitygate.timeout=300
sonar.exclusions=**/test/**,**/vendor/**
sonar.coverage.exclusions=**/test/**
```

#### 13.9.3 NIST SSDF v1.2 (Secure Software Development Framework)

NIST SP 800-218 (SSDF v1.2) เป็นกรอบมาตรฐานสำหรับการพัฒนาซอฟต์แวร์อย่างปลอดภัยที่美国政府กำหนดให้ผู้ขายซอฟต์แวร์ต้องปฏิบัติตาม (Executive Order 14028) มี 4 กลุ่มหลัก:

| กลุ่ม | รหัส | แนวทางปฏิบัติ | เครื่องมือที่เกี่ยวข้อง |
|------|------|--------------|----------------------|
| **Prepare** (PO) | PO.1–PO.5 | กำหนด Security Requirements, Training, Risk Management | Threat Modeling Tools |
| **Protect** (PS) | PS.1–PS.3 | ปกป้อง Source Code, Integrity, Artifacts | Code Signing, SBOM |
| **Produce** (PW) | PW.1–PW.9 | SAST, DAST, SCA, Fuzz Testing, Hardening | SAST/DAST/SCA Tools |
| **Respond** (RV) | RV.1–RV.3 | Vulnerability Disclosure, Incident Response, Patch | SCA Continuous Monitoring |

#### 13.9.4 Policy-as-Code และ Security Gates อัตโนมัติ

Policy-as-Code คือการกำหนดนโยบายความปลอดภัยในรูปแบบของโค้ดที่สามารถตรวจสอบได้โดยอัตโนมัติ:

```yaml
# rego-policy/deny_high_critical.rego
package security

# ปฏิเสธการ Deploy หากมี Critical Severity Finding
deny[reason] {
    finding := data.findings[_]
    finding.severity == "CRITICAL"
    reason := sprintf(
        "Cannot deploy: %s found in %s (CVE: %s)",
        [finding.severity, finding.file, finding.id])
}
```

**ตัวอย่างการกำหนด Security Gates ด้วย GitHub Actions:**

```yaml
# .github/workflows/security-gate.yml
name: Security Gate
on:
  pull_request:
    branches: [main]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Semgrep SAST
        uses: semgrep/semgrep-action@v1
        with:
          config: p/ci
          severity: ERROR,WARNING
  
  sca:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy FS Scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          scan-ref: .
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH
  
  security-gate:
    runs-on: ubuntu-latest
    needs: [sast, sca]
    if: failure()
    steps:
      - name: Block PR
        run: |
          echo "❌ Security Gate FAILED — Critical/High findings detected"
          echo "Please review and fix the findings before merging."
          exit 1
```

#### 13.9.5 Shift Left — การเลื่อนการทดสอบความปลอดภัยไปทางซ้าย

Shift Left เป็นแนวคิดในการทำ Security Testing ให้เร็วที่สุดใน SDLC เพื่อลดค่าใช้จ่ายและเวลาในการแก้ไข:

| ขั้นตอน | Shift Left (ทำเร็ว) | Shift Right (ทำช้า) |
|---------|-------------------|--------------------|
| Pre-commit | SAST (เฉพาะไฟล์ที่เปลี่ยน), Pre-commit Hooks | — |
| PR Review | SAST (Full), SCA, Linting | — |
| Build | SAST (Full), Container Scan | — |
| Test/Staging | DAST (Active + Passive), IAST | SAST (Deep Scan) |
| Staging / Pre-Prod | DAST (Full), Fuzz Testing | DAST, Pentest |
| Production | — | RASP, WAF, DAST (Passive) |

> **ข้อควรจำ:** Shift Left ≠ ทำทุกอย่างทางซ้าย การทดสอบบางประเภท (เช่น DAST Active Scan, Penetration Testing) ควรทำในสภาพแวดล้อมที่เหมือน Production เพื่อให้ได้ผลลัพธ์ที่แม่นยำที่สุด

### 13.10 การเขียนรายงานผล Security Testing

รายงานผล Security Testing ที่มีประสิทธิภาพต้องสามารถสื่อสารกับผู้มีส่วนได้ส่วนเสียทุกระดับ ตั้งแต่ผู้บริหารไปจนถึงทีมพัฒนา

#### 13.10.1 โครงสร้างรายงาน

**1. Executive Summary (สรุปสำหรับผู้บริหาร):**
- ภาพรวมของขอบเขตและวัตถุประสงค์การทดสอบ
- จำนวนช่องโหว่ที่พบแยกตามระดับความรุนแรง
- ความเสี่ยงโดยรวมของระบบ
- คำแนะนำเชิงกลยุทธ์

**2. Scope และ Methodology:**
- ระบบที่ได้รับการทดสอบ (IP, URL, Application Name)
- วันที่และระยะเวลาที่ทดสอบ
- วิธีการทดสอบ (SAST, DAST, Manual Pentest)
- ข้อจำกัด (Limitations)

**3. Technical Findings (รายละเอียดทางเทคนิค):**
- ชื่อและประเภทของช่องโหว่ (CWE, OWASP Category)
- ตำแหน่งที่พบ (File, Line, URL, Parameter)
- ความรุนแรง (CVSS 3.1 Score)
- รายละเอียดของช่องโหว่ (Root Cause, Impact)
- หลักฐาน (Screenshot, Request/Response, PoC Code)
- คำแนะนำในการแก้ไข (Remediation)

**4. Risk Rating Matrix:**
- จำนวน Findings แยกตามความรุนแรง

| ระดับความรุนแรง | คะแนน CVSS | จำนวน | สถานะ |
|----------------|------------|-------|--------|
| Critical | 9.0–10.0 | 2 | ต้องแก้ไขทันที |
| High | 7.0–8.9 | 5 | ต้องแก้ไขภายใน 7 วัน |
| Medium | 4.0–6.9 | 8 | ต้องแก้ไขภายใน 30 วัน |
| Low | 0.1–3.9 | 12 | ต้องแก้ไขภายใน 90 วัน |
| Info | 0.0 | 15 | บันทึกใน Backlog |

**5. Remediation Recommendations:**
- คำแนะนำเฉพาะสำหรับแต่ละ Finding
- ลำดับความสำคัญในการแก้ไข
- แนวทางแก้ไขระยะสั้น (Workaround) และระยะยาว (Fix)

**6. Screenshots / Evidence:**
- ภาพหน้าจอของช่องโหว่ (Screenshot)
- Request/Response ที่พิสูจน์ช่องโหว่
- Proof-of-Concept (PoC) Code (ถ้ามี)

**7. Appendix:**
- รายการเครื่องมือที่ใช้และ Version
- CVSS Vector Strings
- Glossary

#### 13.10.2 ตัวอย่างการรายงาน Findings

**Example Finding: SQL Injection ใน Login Endpoint**

```
หัวข้อ: SQL Injection ใน `/api/login`
ประเภท: A05:2025 — Injection (CWE-89: SQL Injection)
ความรุนแรง: CVSS 3.1: 9.1 (CRITICAL) — AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:N
ตำแหน่ง: POST /api/login (Parameter: username)

รายละเอียด:
Endpoint `/api/login` ของ API Server ไม่ได้ใช้ Parameterized Query
ในการตรวจสอบ Username และ Password ส่งผลให้ผู้โจมตีที่ไม่ผ่านการยืนยันตัวตน
สามารถส่ง Payload SQL Injection เพื่อ bypass การตรวจสอบสิทธิ์ได้

คำขอที่ใช้ทดสอบ:
POST /api/login HTTP/1.1
Host: api.example.com
Content-Type: application/json

{"username": "admin' OR '1'='1", "password": "test"}

การตอบสนอง:
HTTP/1.1 200 OK
{"token": "eyJhbGciOiJIUzI1NiIs...", "role": "admin"}

Root Cause:
โค้ดใน auth_service.py บรรทัดที่ 45–48:
  query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
  cursor.execute(query)

คำแนะนำ:
1. เปลี่ยนเป็น Parameterized Query ทันที:
   query = "SELECT * FROM users WHERE username=%s AND password=%s"
   cursor.execute(query, (username, password))
2. เพิ่ม Input Validation — จำกัด character ที่อนุญาตใน username
3. ใช้ ORM (SQLAlchemy) ที่ป้องกัน SQL Injection โดยค่าเริ่มต้น
4. เพิ่ม Rate Limiting บน Login Endpoint

หลักฐาน: [รูปภาพ Screenshot, Request/Response Log]
```

---

**คำถามสำหรับการอภิปรายในชั้นเรียน:**

1. ในสถานการณ์ที่ทีมพัฒนามีงบประมาณจำกัดและสามารถเลือกใช้เครื่องมือ Security Testing ได้เพียง 2 ประเภท นักศึกษาจะเลือกประเภทใด เพราะเหตุใด จงอธิบายเหตุผลโดยอ้างอิงถึง Cost of Late Discovery และ Shift Left

2. SAST มีอัตรา False Positive ที่สูงกว่า DAST และ IAST นักศึกษาคิดว่าการมี False Positive จำนวนมากเป็นปัญหาที่ร้ายแรงหรือไม่ เพราะเหตุใด องค์กรควรมีกระบวนการจัดการ False Positive อย่างไร

3. ในมุมมองของนักพัฒนา การต้องรอให้ SAST/SCA Scan เสร็จก่อน Merge Pull Request อาจทำให้工作效率ลดลง นักศึกษามีข้อเสนอแนะในการปรับสมดุลระหว่างความปลอดภัยและความเร็วในการพัฒนาอย่างไร

4. CVE-2026-33634 เป็น Supply Chain Attack ที่เกิดกับ Trivy ซึ่งเป็นเครื่องมือ SCA เสียเอง นักศึกษาคิดว่าบทเรียนอะไรบ้างที่สามารถนำไปปรับใช้ในการรักษาความปลอดภัยของเครื่องมือ Security Testing ในองค์กร

5. การทดสอบเจาะระบบ (Penetration Testing) แบบ White-box ให้ความครอบคลุมมากที่สุด แต่ใช้ทรัพยากรมากและไม่สมจริง ในทางกลับกัน Black-box มีความสมจริงแต่ครอบคลุมน้อยกว่า นักศึกษาคิดว่าองค์กรควรเลือกแนวทางใดในการทดสอบประจำปี เพราะเหตุใด

### Lab 13.1: การใช้ Semgrep สแกน Source Code ตัวอย่าง — วิเคราะห์ True Positive vs False Positive

**วัตถุประสงค์:** เรียนรู้การใช้ Semgrep สแกน Source Code เพื่อค้นหาช่องโหว่ และแยกแยะระหว่าง True Positive กับ False Positive

**เวลาที่ใช้:** 45–60 นาที

**เครื่องมือ:** Semgrep, Source Code ตัวอย่าง (Python/JavaScript)

**ขั้นตอน:**

1. ติดตั้ง Semgrep:
   ```bash
   # ติดตั้งด้วย pip
   pip install semgrep
   
   # ตรวจสอบการติดตั้ง
   semgrep --version
   ```

2. สร้างไฟล์ Source Code ตัวอย่างที่มีช่องโหว่ (`vulnerable_app.py`):
   ```python
   import sqlite3
   import hashlib
   from flask import Flask, request, render_template_string
   
   app = Flask(__name__)
   
   # ช่องโหว่ 1: SQL Injection
   @app.route('/login')
   def login():
       username = request.args.get('username')
       password = request.args.get('password')
       
       conn = sqlite3.connect('users.db')
       cursor = conn.cursor()
       # SQL Injection — ต่อ string โดยตรง
       query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
       cursor.execute(query)
       user = cursor.fetchone()
       
       if user:
           return "Login successful"
       return "Login failed"
   
   # ช่องโหว่ 2: Hardcoded Password
   API_SECRET = "sk-abcdefghijklmnopqrstuvwxyz123456"  # Hardcoded Secret
   
   # ช่องโหว่ 3: MD5 Hash (deprecated)
   def hash_password(password):
       return hashlib.md5(password.encode()).hexdigest()  # MD5 ไม่ปลอดภัย
   
   # ช่องโหว่ 4: Command Injection
   @app.route('/ping')
   def ping():
       import os
       host = request.args.get('host')
       # Command Injection
       result = os.system(f"ping -c 1 {host}")
       return f"Ping result: {result}"
   
   # ช่องโหว่ 5: Template Injection (SSTI)
   @app.route('/greet')
   def greet():
       name = request.args.get('name')
       # SSTI — ใช้ render_template_string โดยตรง
       return render_template_string(f"Hello {name}")
   
   if __name__ == '__main__':
       app.run(debug=True)  # Debug Mode ใน Production
   ```

3. รัน Semgrep ด้วยกฎเริ่มต้น:
   ```bash
   semgrep --config=auto vulnerable_app.py
   ```

4. รัน Semgrep ด้วยกฎเฉพาะสำหรับ Python:
   ```bash
   semgrep --config=p/python vulnerable_app.py
   ```

5. รัน Semgrep ด้วยกฎเฉพาะ (SQL Injection):
   ```bash
   semgrep --config=https://semgrep.dev/c/p/python
   ```

6. วิเคราะห์ผลลัพธ์:
   ```bash
   semgrep --config=auto --json vulnerable_app.py
   ```

7. ปรับปรุงโค้ดเพื่อแก้ไขช่องโหว่และรัน Semgrep อีกครั้ง

**สิ่งที่ต้องส่ง:**

1. ภาพ Screenshot ของ Semgrep Output ที่แสดง Findings ทั้งหมด
2. ตารางวิเคราะห์ Findings ว่าเป็น True Positive หรือ False Positive พร้อมเหตุผล

| Finding | ประเภท (ช่องโหว่) | True/False Positive | เหตุผล |
|---------|-------------------|--------------------|--------|
| SQL Injection | SQL Injection | True Positive | การต่อ String โดยตรงใน SQL Query |
| ... | ... | ... | ... |

3. โค้ดที่ได้รับการแก้ไขแล้ว โดยแสดงบรรทัดที่เปลี่ยนแปลง
4. ภาพ Screenshot ของ Semgrep Output หลังแก้ไข — ต้องไม่มี Findings ที่มีความรุนแรงสูง

### Lab 13.2: การใช้ OWASP ZAP สแกน Web Application (Juice Shop) — Active vs Passive Scan

**วัตถุประสงค์:** เรียนรู้การใช้ OWASP ZAP ในการสแกน Web Application และเปรียบเทียบความแตกต่างของผลลัพธ์ระหว่าง Active Scan และ Passive Scan

**เวลาที่ใช้:** 45–60 นาที

**เครื่องมือ:** OWASP ZAP, Docker, OWASP Juice Shop

**ขั้นตอน:**

1. รัน OWASP Juice Shop ด้วย Docker:
   ```bash
   docker run --rm -d -p 3000:3000 --name juice-shop bkimminich/juice-shop
   ```

2. ติดตั้งและรัน OWASP ZAP:
   ```bash
   # macOS
   brew install --cask zap
   
   # Linux
   # ดาวน์โหลดจาก https://www.zaproxy.org/download/
   ```

3. เปิด ZAP และตั้งค่า:
   - เปิด ZAP Desktop
   - ไปที่ Tools → Options → Local Proxies
   - ตั้งค่า Address: localhost, Port: 8080
   - ตั้งค่า Browser Proxy ให้ใช้ localhost:8080

4. **Passive Scan:**
   - เปิด Browser และไปที่ `http://localhost:3000`
   - นำทางไปยังหน้าต่าง ๆ ของ Juice Shop (Login, Search, Product, Basket, etc.)
   - กลับไปที่ ZAP → ดู Alerts Tab
   - สังเกต Findings ที่ ZAP ตรวจพบผ่าน Passive Scan

5. **Active Scan:**
   - ใน ZAP → คลิกขวาที่ `http://localhost:3000` ใน Sites Tree
   - เลือก Attack → Active Scan
   - ตั้งค่า: Policy = Default, Scope = In Scope Only
   - คลิก Start Scan
   - รอให้ Active Scan เสร็จสมบูรณ์ (อาจใช้เวลา 10–15 นาที)

6. **เปรียบเทียบผลลัพธ์:**
   ```bash
   # ใช้ ZAP API เพื่อรับผลลัพธ์
   curl "http://localhost:8080/JSON/core/view/alerts/?baseurl=http://localhost:3000&start=0&count=100"
   ```

7. ส่งออกรายงาน:
   - ใน ZAP → Report → Generate Report
   - เลือก Traditional HTML Report
   - บันทึกไฟล์เป็น `zap-report.html`

8. ทดสอบ API Scan:
   ```bash
   # ใช้ ZAP API เพื่อเริ่ม Spider
   curl "http://localhost:8080/JSON/spider/action/scan/?url=http://localhost:3000"
   
   # ตรวจสอบสถานะ
   curl "http://localhost:8080/JSON/spider/view/status/"
   
   # เริ่ม Active Scan หลัง Spider เสร็จ
   curl "http://localhost:8080/JSON/ascan/action/scan/?url=http://localhost:3000"
   ```

**สิ่งที่ต้องส่ง:**

1. จำนวน Findings จาก Passive Scan (แยกตาม Risk: High/Medium/Low/Info)
2. จำนวน Findings จาก Active Scan (แยกตาม Risk)
3. เปรียบเทียบความแตกต่าง — Active Scan พบช่องโหว่ประเภทใดเพิ่มเติมจาก Passive Scan บ้าง
4. ตัวอย่างช่องโหว่ High Risk อย่างน้อย 1 รายการ พร้อมรายละเอียด:
   - URL และ Parameter
   - คำอธิบายช่องโหว่
   - แนวทางแก้ไข
5. ภาพ Screenshot ของ ZAP Report

**การล้างข้อมูล:**
```bash
docker stop juice-shop
```

### Lab 13.3: การใช้ Trivy สแกน Container Image และ Dependencies

**ขั้นตอน:**

### Lab 13.4: ดำเนินการ Penetration Testing ขนาดเล็กตาม PTES — เขียนรายงาน

**วัตถุประสงค์:** ดำเนินการทดสอบเจาะระบบขนาดเล็กตามขั้นตอนของ PTES และเขียนรายงานผลการทดสอบ

**เวลาที่ใช้:** 60–75 นาที

**เครื่องมือ:** OWASP ZAP, Burp Suite (Community Edition), nmap, curl, เว็บเบราว์เซอร์

**ขอบเขตการทดสอบ:**
- เป้าหมาย: OWASP Juice Shop (`http://localhost:3000`)
- การทดสอบ: เฉพาะ Passive Recon + Active Recon ที่ไม่ทำให้ระบบเสียหาย
- ข้อจำกัด: ไม่ทดสอบ Denial of Service, ไม่ทดสอบ Brute Force ที่รุนแรง

**ขั้นตอน:**

**Phase 1: Pre-engagement**
- กำหนดขอบเขตและ Rules of Engagement (RoE)
- ระบุข้อมูลติดต่อฉุกเฉิน

**Phase 2: Intelligence Gathering**

```bash
# ตรวจสอบ HTTP Headers
curl -sI http://localhost:3000 | head -20

# ตรวจสอบ Robots.txt
curl -s http://localhost:3000/robots.txt

# ตรวจสอบ Technology Stack
curl -s http://localhost:3000 | grep -i "express\|angular\|react\|vue"
```

**Phase 3: Threat Modeling**
- ระบุทรัพย์สินที่มีค่า: User Data, Auth Tokens, Admin Functions
- ระบุ Attack Surface: Login, Search, API Endpoints, File Upload

**Phase 4: Vulnerability Analysis**

```bash
# ใช้ ZAP Passive Scan (Lab 13.2)
# ใช้ ZAP Active Scan (Lab 13.2)

# ทดสอบ SQL Injection ด้วย curl
curl "http://localhost:3000/rest/products/search?q='"
curl "http://localhost:3000/rest/products/search?q=1' OR '1'='1"

# ทดสอบ XSS
curl "http://localhost:3000/#/search?q=<script>alert(1)</script>"

# ทดสอบ IDOR
curl "http://localhost:3000/rest/user/profile"

# ทดสอบ Directory Traversal
curl "http://localhost:3000/ftp/../../../etc/passwd"
```

**Phase 5: Exploitation (จำกัดเฉพาะที่ไม่ก่อให้เกิดความเสียหาย)**

```bash
# ทดสอบการ Bypass Authentication
curl -X POST http://localhost:3000/rest/user/login \
  -H "Content-Type: application/json" \
  -d '{"email": "' OR 1=1--", "password": "test"}'

# ทดสอบ JWT Weakness — ใช้ HS256 Algorithm
# (ต้องติดตั้ง Burp Suite Extension: JSON Web Token Attacker)
```

**Phase 6: Post-Exploitation (จำลอง)**
- วิเคราะห์ว่าหากเจาะระบบสำเร็จจะสามารถทำอะไรได้บ้าง
- ระบุข้อมูลที่อาจถูกรั่วไหล

**Phase 7: Reporting**

เขียนรายงานตามรูปแบบต่อไปนี้:


## สรุปท้ายบท

Security Testing เป็นกระบวนการที่จำเป็นในการพัฒนาซอฟต์แวร์อย่างปลอดภัย โดยครอบคลุมหลายมิติตั้งแต่การวิเคราะห์ Source Code (SAST) การทดสอบขณะรันไทม์ (DAST, IAST) ไปจนถึงการป้องกันใน Production (RASP) แต่ละประเภทมีจุดแข็งและข้อจำกัดที่แตกต่างกัน — SAST วิเคราะห์โค้ดได้ครอบคลุมทุกบรรทัดแต่มี False Positive สูงและไม่พบ Runtime Issues, DAST ทดสอบจากมุมมองของผู้โจมตีได้สมจริงแต่ครอบคลุมเฉพาะฟังก์ชันที่เข้าถึงได้, IAST มี False Positives ต่ำที่สุดเนื่องจากตรวจสอบยืนยันที่ Runtime, RASP ป้องกันการโจมตีแบบ Real-time ใน Production แต่ไม่ใช่เครื่องมือทดสอบ และ SCA ตรวจสอบ Dependency และ License Compliance ที่จำเป็นในยุคที่ Supply Chain Attacks เพิ่มสูงขึ้น

Penetration Testing ตามมาตรฐาน PTES, OWASP WSTG, NIST SP 800-115 และ OSSTMM เป็นกระบวนการจำลองการโจมตีที่ต้องอาศัยทักษะของมนุษย์ร่วมกับเครื่องมืออัตโนมัติ โดยมี Rules of Engagement (RoE) ที่ชัดเจนเพื่อให้การทดสอบอยู่ในขอบเขตที่ได้รับอนุญาต

DevSecOps และแนวคิด Shift Left ช่วยให้ Security Testing เป็นส่วนหนึ่งของ CI/CD Pipeline ตั้งแต่ Pre-commit ไปจนถึง Production โดยใช้ Quality Gates, Policy-as-Code และเครื่องมืออัตโนมัติเพื่อรักษาสมดุลระหว่างความปลอดภัยและความเร็วในการพัฒนา รายงานผล Security Testing ที่มีคุณภาพต้องสามารถสื่อสารกับผู้มีส่วนได้ส่วนเสียทุกระดับ ตั้งแต่ Executive Summary สำหรับผู้บริหารไปจนถึง Technical Findings และ Remediation Recommendations สำหรับทีมพัฒนา
