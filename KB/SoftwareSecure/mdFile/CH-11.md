# CH-11: Mobile Application Security

## วัตถุประสงค์การเรียนรู้

เมื่อจบบทนี้นักศึกษาสามารถ:

1. อธิบายองค์ประกอบของ OWASP Mobile Security Ecosystem ได้แก่ MASVS v2.1.0 (8 กลุ่มควบคุม), MASTG v1.7.0, MASWE และ OWASP Mobile Top 10 2024 ได้อย่างถูกต้อง
2. วิเคราะห์ความแตกต่างด้านความปลอดภัยระหว่างแพลตฟอร์ม Android และ iOS ในด้าน App Sandbox, Permission Model, Entitlements และการเข้ารหัสข้อมูล
3. ระบุและอธิบายช่องโหว่ที่พบบ่อยในแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ เช่น Intent Spoofing, WebView Injection, Insecure Data Storage และ Insecure Communication
4. เลือกใช้แนวทางป้องกัน Mobile API Communication ที่ปลอดภัย ได้แก่ Certificate Pinning, OAuth 2.0 with PKCE และ Device Fingerprinting
5. ออกแบบกลไก Secure Data Storage บนอุปกรณ์เคลื่อนที่โดยใช้ Android Keystore, iOS Keychain, EncryptedSharedPreferences และ SQLCipher
6. ใช้เครื่องมือทดสอบความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ ได้แก่ MobSF, Burp Suite, Drozer และ Objection (Frida) เพื่อวิเคราะห์ช่องโหว่
7. วิเคราะห์กรณีเหตุการณ์จริงด้านความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่และระบุบทเรียนที่นำไปปรับใช้ได้

## ขอบเขตและข้อกำหนดด้านจริยธรรมของบทนี้

บทนี้ครอบคลุมหลักการรักษาความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ทั้งในระบบปฏิบัติการ Android และ iOS เนื้อหามีตัวอย่างเกี่ยวกับช่องโหว่และการโจมตีเพื่อการเรียนรู้เชิงป้องกันเท่านั้น กิจกรรมปฏิบัติการทั้งหมดต้องทำในสภาพแวดล้อมที่ได้รับอนุญาต เช่น อุปกรณ์ทดสอบในห้องปฏิบัติการ เครื่องเสมือน (Virtual Machine) หรือแอปพลิเคชันตัวอย่างที่อาจารย์จัดให้เท่านั้น

**ข้อควรจำ:** ห้ามนำเทคนิคในบทนี้ไปทดสอบกับแอปพลิเคชันหรือระบบของบุคคลอื่นโดยไม่ได้รับอนุญาตเป็นลายลักษณ์อักษร การใช้เครื่องมือเช่น MobSF, Burp Suite, Drozer และ Frida กับแอปพลิเคชันที่ไม่ใช่ของตนเองหรือไม่ได้รับอนุญาตอาจมีความผิดทางกฎหมาย พ.ร.บ. ว่าด้วยการกระทำผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2560 และแก้ไขเพิ่มเติม การทดสอบแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ต้องทำบนอุปกรณ์ที่นักศึกษาเป็นเจ้าของและอยู่ในการควบคุมของนักศึกษาเท่านั้น

## เนื้อหา

### 11.1 OWASP Mobile Security Ecosystem

OWASP (Open Web Application Security Project) ได้พัฒนาเฟรมเวิร์กด้านความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่อย่างครอบคลุม ประกอบด้วยสามองค์ประกอบหลักที่ทำงานร่วมกัน ได้แก่ MASVS (Mobile Application Security Verification Standard), MASTG (Mobile Application Security Testing Guide) และ MASWE (Mobile Application Security Weakness Enumeration)

#### 11.1.1 OWASP MASVS v2.1.0 (มกราคม 2024)

MASVS เป็นมาตรฐานสำหรับการตรวจสอบความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ เวอร์ชันล่าสุด v2.1.0 แบ่งกลุ่มควบคุมออกเป็น 8 กลุ่ม ดังนี้:

| กลุ่มควบคุม | รหัส | คำอธิบาย |
|------------|------|----------|
| MASVS-STORAGE | การจัดเก็บข้อมูลอย่างปลอดภัย | การเข้ารหัสข้อมูลที่จัดเก็บในอุปกรณ์, การจัดการ Keychain/Keystore, การป้องกันข้อมูลรั่วไหลผ่านไฟล์ขยะ |
| MASVS-CRYPTO | การเข้ารหัส | การใช้ Algorithm ที่ปลอดภัย, การจัดการคีย์อย่างถูกต้อง, การไม่ใช้การเข้ารหัสที่พัฒนาขึ้นเอง |
| MASVS-AUTH | การยืนยันตัวตนและอนุญาต | การยืนยันตัวตนของผู้ใช้, การจัดการ Session, Biometric Authentication, Local Authentication |
| MASVS-NETWORK | การสื่อสารผ่านเครือข่าย | TLS, Certificate Pinning, การป้องกัน MITM, การสื่อสารที่เข้ารหัส |
| MASVS-PLATFORM | ปฏิสัมพันธ์กับแพลตฟอร์ม | Intent, IPC, URL Scheme, WebView, การเรียกใช้ API ของระบบปฏิบัติการอย่างปลอดภัย |
| MASVS-CODE | ความปลอดภัยของโค้ด | Input Validation, Error Handling, การป้องกัน Code Tampering, การคอมไพล์อย่างปลอดภัย |
| MASVS-RESILIENCE | ความทนทานต่อการถูกดัดแปลง | การป้องกันการถูก Reverse Engineering, การตรวจจับการถูกดัดแปลง, การป้องกันการถูกดีบัก |
| MASVS-PRIVACY | ความเป็นส่วนตัว | การจัดการข้อมูลส่วนบุคคล, การแจ้งให้ผู้ใช้ทราบ, การปฏิบัติตามกฎหมายคุ้มครองข้อมูลส่วนบุคคล |

#### 11.1.2 OWASP MASTG v1.7.0

MASTG เป็นคู่มือการทดสอบความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ที่ครอบคลุมทั้งการทดสอบแบบ Static Analysis และ Dynamic Analysis ปัจจุบันมี 12,846 stars บน GitHub MASTG แบ่งการทดสอบออกเป็นหมวดหมู่ตามกลุ่มควบคุมของ MASVS โดยแต่ละหมวดหมู่มี Test Case ที่ระบุขั้นตอนการทดสอบ เครื่องมือที่ต้องใช้ และเกณฑ์การประเมินผล

MASTG ครอบคลุมการทดสอบในด้านต่อไปนี้:
- การทดสอบการจัดเก็บข้อมูล (MASVS-STORAGE)
- การทดสอบการเข้ารหัส (MASVS-CRYPTO)
- การทดสอบการยืนยันตัวตน (MASVS-AUTH)
- การทดสอบการสื่อสารผ่านเครือข่าย (MASVS-NETWORK)
- การทดสอบปฏิสัมพันธ์กับแพลตฟอร์ม (MASVS-PLATFORM)
- การทดสอบความปลอดภัยของโค้ด (MASVS-CODE)
- การทดสอบความทนทานต่อการถูกดัดแปลง (MASVS-RESILIENCE)
- การทดสอบความเป็นส่วนตัว (MASVS-PRIVACY)

#### 11.1.3 OWASP MASWE

MASWE (Mobile Application Security Weakness Enumeration) เป็นโครงการที่เชื่อมโยงระหว่าง MASVS และ MASTG โดย MASWE จะระบุช่องโหว่หรือจุดอ่อน (Weakness) ที่สอดคล้องกับแต่ละกลุ่มควบคุมใน MASVS และเชื่อมต่อไปยัง Test Case ที่เกี่ยวข้องใน MASTG ทำให้ผู้พัฒนาสามารถติดตามได้ว่าช่องโหว่ใดควรแก้ไขตามกลุ่มควบคุมใด และจะทดสอบอย่างไร

#### 11.1.4 OWASP Mobile Top 10 2024

OWASP Mobile Top 10 2024 (Final v1.0) เป็นรายการความเสี่ยงด้านความปลอดภัยที่สำคัญที่สุด 10 อันดับสำหรับแอปพลิเคชันบนอุปกรณ์เคลื่อนที่:

| อันดับ | ความเสี่ยง | คำอธิบาย |
|-------|-----------|----------|
| M1 | Improper Credential Usage | การใช้ข้อมูลประจำตัวอย่างไม่เหมาะสม เช่น การเก็บรหัสผ่านในรูปแบบ Plaintext หรือการใช้ Credential ที่รั่วไหล |
| M2 | Inadequate Supply Chain Security | ความปลอดภัยของห่วงโซ่อุปทานที่ไม่เพียงพอ เช่น การใช้ Third-Party SDK ที่ไม่ปลอดภัย |
| M3 | Insecure Authentication/Authorization | การยืนยันตัวตนและอนุญาตที่ไม่ปลอดภัย |
| M4 | Insufficient Input/Output Validation | การตรวจสอบ Input และ Output ที่ไม่เพียงพอ |
| M5 | Insecure Communication | การสื่อสารที่ไม่ปลอดภัย เช่น การไม่ใช้ TLS หรือการไม่ตรวจสอบ Certificate |
| M6 | Inadequate Privacy Controls | การควบคุมความเป็นส่วนตัวที่ไม่เพียงพอ |
| M7 | Insufficient Binary Protections | การป้องกัน Binary ที่ไม่เพียงพอ เช่น การไม่ป้องกัน Reverse Engineering |
| M8 | Security Misconfiguration | การกำหนดค่าที่ผิดพลาดด้านความปลอดภัย |
| M9 | Insecure Data Storage | การจัดเก็บข้อมูลที่ไม่ปลอดภัย |
| M10 | Insufficient Cryptography | การใช้การเข้ารหัสที่ไม่เพียงพอ |

### 11.2 ความปลอดภัยบน Android

#### 11.2.1 Application Sandbox

Android ใช้กลไก Sandbox ในระดับ Linux Kernel โดยแต่ละแอปพลิเคชันจะถูกกำหนด Linux User ID (UID) ที่ไม่ซ้ำกัน แอปพลิเคชันแต่ละตัวจะทำงานใน Process ของตนเอง และมีไดเรกทอรีส่วนตัวที่ `/data/data/<package_name>` ซึ่งแอปพลิเคชันอื่นไม่สามารถเข้าถึงได้โดยตรง

กลไก Sandbox นี้ทำงานดังนี้:
- แต่ละแอปพลิเคชันได้รับ UID ที่ไม่ซ้ำกันเมื่อติดตั้ง
- ไฟล์และทรัพยากรของแอปพลิเคชันถูกจำกัดการเข้าถึงด้วยสิทธิ์ระดับ Linux
- การสื่อสารระหว่างแอปพลิเคชันต้องผ่าน IPC (Inter-Process Communication) เช่น Intent, Binder, Content Provider
- แอปพลิเคชันไม่สามารถอ่านไฟล์ของแอปพลิเคชันอื่นได้โดยตรง

#### 11.2.2 Permission Model

Android มีโมเดลการขอสิทธิ์ที่พัฒนามาอย่างต่อเนื่อง:

| Android เวอร์ชัน | การเปลี่ยนแปลงสำคัญ |
|-----------------|---------------------|
| Android 6.0 (API 23) | เริ่มใช้ Runtime Permissions — ผู้ใช้สามารถปฏิเสธสิทธิ์ขณะรันได้ |
| Android 10 (API 29) | เริ่มใช้ Scoped Storage — จำกัดการเข้าถึงไฟล์ภายนอก |
| Android 11 (API 30) | Auto-Reset Permissions — สิทธิ์ที่ไม่ได้ใช้จะถูกรีเซ็ตอัตโนมัติ |

**ประเภทของ Permissions ใน Android:**

| ประเภท | คำอธิบาย | ตัวอย่าง |
|--------|----------|----------|
| Normal | อนุญาตอัตโนมัติ ไม่ต้องขอผู้ใช้ | `INTERNET`, `ACCESS_NETWORK_STATE` |
| Dangerous | ต้องขอ Runtime Permission | `CAMERA`, `ACCESS_FINE_LOCATION`, `READ_CONTACTS` |
| Signature | ได้รับก็ต่อเมื่อแอปเซ็นด้วย Certificate เดียวกัน | `BIND_ACCESSIBILITY_SERVICE` |
| Special | ต้องผ่านการตั้งค่าใน Settings พิเศษ | `SYSTEM_ALERT_WINDOW`, `REQUEST_INSTALL_PACKAGES` |

**ตัวอย่างการขอ Runtime Permission ใน Kotlin:**

```kotlin
// Android — ขอ Runtime Permission ด้วย Activity Result API
class MainActivity : AppCompatActivity() {
    private val requestPermissionLauncher = registerForActivityResult(
        ActivityResultContracts.RequestPermission()
    ) { isGranted: Boolean ->
        if (isGranted) {
            // สิทธิ์ได้รับแล้ว — ดำเนินการต่อ
        } else {
            // สิทธิ์ถูกปฏิเสธ — แจ้งผู้ใช้
        }
    }

    private fun requestCameraPermission() {
        when {
            ContextCompat.checkSelfPermission(
                this, Manifest.permission.CAMERA
            ) == PackageManager.PERMISSION_GRANTED -> {
                // มีสิทธิ์แล้ว
            }
            shouldShowRequestPermissionRationale(Manifest.permission.CAMERA) -> {
                // อธิบายเหตุผลที่ต้องใช้สิทธิ์
            }
            else -> {
                requestPermissionLauncher.launch(Manifest.permission.CAMERA)
            }
        }
    }
}
```

#### 11.2.3 APK Structure และ Reverse Engineering

ไฟล์ APK (Android Package Kit) เป็นไฟล์ที่ถูกบีบอัด (ZIP Archive) ซึ่งประกอบด้วยส่วนประกอบต่าง ๆ ดังนี้:

| องค์ประกอบ | คำอธิบาย |
|------------|----------|
| `AndroidManifest.xml` | ไฟล์ Manifest ที่ประกาศ Component, Permission, Features (ถูก Compile เป็น Binary XML) |
| `classes.dex` | โค้ดที่ถูก Compile เป็น DEX (Dalvik Executable) |
| `resources.arsc` | ไฟล์ Resource ที่ถูก Compile |
| `res/` | โฟลเดอร์เก็บ Resource ดิบ (รูปภาพ, Layout XML) |
| `lib/` | โฟลเดอร์เก็บ Native Library (.so) สำหรับแต่ละสถาปัตยกรรม |
| `META-INF/` | โฟลเดอร์เก็บข้อมูลลายเซ็นและ Manifest |

**เครื่องมือสำหรับ Reverse Engineering Android:**

| เครื่องมือ | หน้าที่ | รายละเอียด |
|-----------|--------|------------|
| JADX | Decompile DEX เป็น Java Code | แสดงโค้ด Java ที่อ่านเข้าใจได้จากไฟล์ DEX |
| APKTool | Decode/Rebuild ทรัพยากร | ถอดรหัส AndroidManifest.xml และ Resource File |
| Jadx-GUI | ส่วนติดต่อผู้ใช้แบบกราฟิกของ JADX | แสดงโครงสร้างคลาสและโค้ดแบบ Interactive |
| Ghidra | Reverse Engineering Native Library | วิเคราะห์ไฟล์ .so ที่เขียนด้วย C/C++ |

**คำสั่งสำหรับการวิเคราะห์ APK ด้วย APKTool และ JADX:**

```bash
# ถอด APK ด้วย APKTool
apktool d target.apk -o output_dir/

# ตรวจสอบ AndroidManifest ที่ถอดแล้ว
cat output_dir/AndroidManifest.xml

# Decompile DEX ด้วย JADX
jadx -d output_java/ target.apk

# ใช้ JADX แบบ GUI
jadx-gui target.apk
```

#### 11.2.4 Intent Spoofing และ Intent Injection

Intent เป็นกลไก IPC (Inter-Process Communication) หลักของ Android ที่ใช้สื่อสารระหว่าง Component (Activity, Service, Broadcast Receiver) ทั้งภายในและระหว่างแอปพลิเคชัน

**Intent Spoofing:** เกิดขึ้นเมื่อ Component ที่ถูก Export (มี `<exported=true>`) ไม่ได้ตรวจสอบ Intent ที่ได้รับ ทำให้แอปพลิเคชันอื่นสามารถส่ง Intent มาปลอมแปลงคำสั่งได้

**Intent Injection:** เกิดขึ้นเมื่อแอปพลิเคชันสร้าง Intent จากข้อมูลที่ผู้ใช้ควบคุม โดยไม่ได้ตรวจสอบความถูกต้อง ทำให้ผู้โจมตีสามารถแทรกคำสั่งที่เป็นอันตรายได้

**ตัวอย่าง CVE ที่เกี่ยวข้องกับ Intent:**

| CVE | ปี | รายละเอียด | ผลกระทบ |
|-----|-----|-----------|----------|
| CVE-2025-48564 | 2025 | Intent filter bypass via race condition ใน Android 13-16 | ผู้โจมตีสามารถข้าม Intent Filter ได้ผ่าน Race Condition |
| CVE-2025-48566 | 2025 | User profile boundary bypass ใน Android | การข้ามขอบเขตระหว่าง User Profile |
| CVE-2025-22433 | 2025 | Cross-profile intent filter bypass (CVSS 7.8) | การข้าม Intent Filter ระหว่าง Profile ส่งผลให้เกิด Privilege Escalation |
| CVE-2025-32326 | 2025 | Confused deputy attack | แอปพลิเคชันที่ได้รับสิทธิ์สูงถูกหลอกให้ดำเนินการแทนแอปพลิเคชันที่ไม่ได้รับสิทธิ์ |
| CVE-2025-22441 | 2025 | Confused deputy via RemoteViews | การใช้ RemoteViews เพื่อโจมตีแบบ Confused Deputy |

**กรณีศึกษา: EngageSDK Intent Redirection (Microsoft, เมษายน 2026)**

Microsoft รายงานช่องโหว่ Intent Redirection ใน Third-Party SDK ที่พบในแอปพลิเคชัน Android มากกว่า 50 ล้านครั้งที่ติดตั้ง ช่องโหว่เกิดจากการใช้ `parseUri()` ด้วย Flag `URI_ALLOW_UNSAFE` ซึ่งทำให้ผู้โจมตีสามารถส่ง Intent ที่ไม่ปลอดภัยไปยัง Component ภายในแอปพลิเคชันได้

**แนวทางป้องกัน Intent Spoofing/Injection:**

```kotlin
// ไม่ปลอดภัย: ไม่ตรวจสอบ Intent ที่เข้ามา
class VulnerableActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val data = intent.getStringExtra("data")
        // ใช้ data โดยตรงโดยไม่ตรวจสอบแหล่งที่มา
    }
}

// ปลอดภัย: ตรวจสอบ Caller และ Input
class SecureActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        // ตรวจสอบว่า Caller เป็นแอปพลิเคชันที่เชื่อถือได้
        val callingUid = callingUid
        val packageName = packageManager.getNameForUid(callingUid)
        
        if (trustedPackages.contains(packageName)) {
            val data = intent.getStringExtra("data")
            // ตรวจสอบและ Sanitize ก่อนใช้งาน
            if (isValidInput(data)) {
                processData(data)
            }
        } else {
            finish()  // ปิด Activity หาก Caller ไม่ได้รับอนุญาต
        }
    }
}
```

#### 11.2.5 Content Provider Security

Content Provider เป็น Component ที่ให้แอปพลิเคชันอื่นเข้าถึงข้อมูลผ่าน URI interface หาก Content Provider ถูก Export โดยไม่มีการป้องกันข้อมูล ผู้โจมตีสามารถเข้าถึงข้อมูลที่ละเอียดอ่อนได้

**CVE-2025-48619 (Android 14-16):** ช่องโหว่ใน `ContentProvider.java` ที่ทำให้แอปพลิเคชันแบบ Read-Only สามารถตัดทอน (Truncate) ไฟล์ได้เนื่องจากตรรกะการตรวจสอบสิทธิ์ที่ผิดพลาด

**แนวทางป้องกัน Content Provider:**

```xml
<!-- ปลอดภัย: ไม่ Export Content Provider โดยไม่จำเป็น -->
<provider
    android:name=".SecureProvider"
    android:authorities="com.example.app.provider"
    android:exported="false" />

<!-- หากจำเป็นต้อง Export ต้องใช้ Permission -->
<provider
    android:name=".SharedProvider"
    android:authorities="com.example.app.shared"
    android:exported="true"
    android:readPermission="com.example.READ_DATA"
    android:writePermission="com.example.WRITE_DATA" />
```

#### 11.2.6 WebView Security

WebView เป็น Component ที่ใช้แสดงเนื้อหาเว็บภายในแอปพลิเคชัน Android ซึ่งมีความเสี่ยงด้านความปลอดภัยที่สำคัญ:

| ความเสี่ยง | คำอธิบาย | แนวทางป้องกัน |
|-----------|----------|--------------|
| JavaScript Bridge Injection | การใช้ `addJavascriptInterface()` สามารถทำให้เกิด Remote Code Execution (RCE) บน API < 17 | ใช้เฉพาะบน API 17+ และตรวจสอบ Input ทุกครั้ง |
| Cleartext Traffic | WebView ที่อนุญาตการรับส่งข้อมูลแบบไม่เข้ารหัส (HTTP) | ตั้งค่า `android:usesCleartextTraffic="false"` ใน Manifest |
| File Access | WebView ที่เปิดให้เข้าถึงไฟล์ภายในเครื่องผ่าน `file://` | ตั้งค่า `setAllowFileAccess(false)` |
| Cross-Site Scripting (XSS) | การโจมตีผ่าน JavaScript ที่ถูก Inject ใน WebView | ใช้ Input Validation และ Output Encoding |

**ตัวอย่างการตั้งค่า WebView ที่ปลอดภัย:**

```kotlin
// การตั้งค่า WebView ที่ปลอดภัย
val webView = findViewById<WebView>(R.id.webview)

// ปิด JavaScript ถ้าไม่จำเป็น
webView.settings.javaScriptEnabled = false

// ถ้าจำเป็นต้องใช้ JavaScript ให้ใช้เฉพาะ API 17+
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.JELLY_BEAN_MR1) {
    webView.addJavascriptInterface(object {
        @JavascriptInterface
        fun processData(data: String) {
            // ตรวจสอบและ Sanitize ข้อมูล
        }
    }, "AndroidBridge")
}

// ปิด File Access
webView.settings.allowFileAccess = false

// ปิด File Access จาก URL
webView.settings.allowFileAccessFromFileURLs = false
webView.settings.allowUniversalAccessFromFileURLs = false

// ใช้ HTTPS เท่านั้น
webView.loadUrl("https://trusted-site.com")
```

#### 11.2.7 การจัดเก็บข้อมูลที่ไม่ปลอดภัย (Insecure Storage)

ข้อมูลที่ละเอียดอ่อนที่ถูกจัดเก็บในรูปแบบที่ไม่ปลอดภัยเป็นหนึ่งในช่องโหว่ที่พบบ่อยที่สุดในการทดสอบความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่

| วิธีการจัดเก็บ | ความเสี่ยง | ระดับความปลอดภัย |
|---------------|-----------|-------------------|
| SharedPreferences (Plaintext) | ข้อมูลถูกจัดเก็บเป็น XML ที่อ่านได้ง่าย | ไม่ปลอดภัย |
| SQLite Database (ไม่เข้ารหัส) | ฐานข้อมูลสามารถอ่านได้ด้วยเครื่องมือเช่น SQLite Browser | ไม่ปลอดภัย |
| Internal Storage (ไฟล์ Plaintext) | แอปพลิเคชันอื่นไม่สามารถเข้าถึงได้ แต่อ่านได้บนอุปกรณ์ที่ Root แล้ว | ปลอดภัยระดับพื้นฐาน |
| External Storage (SD Card) | แอปพลิเคชันใดก็ได้อ่านได้ หากมีสิทธิ์ `READ_EXTERNAL_STORAGE` | ไม่ปลอดภัย |

### 11.3 ความปลอดภัยบน iOS

#### 11.3.1 iOS App Sandbox และ Entitlements

iOS ใช้กลไก Sandbox ที่เข้มงวดกว่า Android โดยแอปพลิเคชันของบุคคลที่สามทุกรุ่นทำงานภายใต้ผู้ใช้ `mobile` ซึ่งเป็น Non-Privileged User พร้อมกับการบังคับใช้ Mandatory Access Control (MAC) ผ่าน TrustedBSD

**Entitlements:** เป็น Key-Value Pairs ที่ถูกเซ็นไว้ใน Code Signature ของแอปพลิเคชัน ไม่สามารถเปลี่ยนแปลงได้หลังจากเซ็นแล้ว Entitlements กำหนดสิทธิ์พิเศษที่แอปพลิเคชันได้รับ เช่น:
- `com.apple.security.personal-information.addressbook` — การเข้าถึง Address Book
- `com.apple.developer.networking.wifi-info` — การเข้าถึงข้อมูล WiFi
- `com.apple.developer.nfc.readersession.formats` — การเข้าถึง NFC

#### 11.3.2 Keychain Services

Keychain เป็นระบบจัดเก็บข้อมูลที่ละเอียดอ่อนใน iOS ที่จัดการโดย Daemon ชื่อ `securityd` และใช้ฐานข้อมูล SQLite ในการจัดเก็บ Keychain เป็นกลไกหลักสำหรับการจัดเก็บข้อมูลที่เข้ารหัส เช่น รหัสผ่าน, คีย์เข้ารหัส, Token การยืนยันตัวตน

**Accessibility Levels ของ Keychain:**

| ค่า | คำอธิบาย |
|-----|----------|
| `kSecAttrAccessibleWhenUnlocked` | สามารถเข้าถึงได้เมื่ออุปกรณ์ปลดล็อกแล้วเท่านั้น |
| `kSecAttrAccessibleAfterFirstUnlock` | สามารถเข้าถึงได้หลังจากปลดล็อกครั้งแรก จนกว่าจะปิดเครื่อง |
| `kSecAttrAccessibleAlways` | สามารถเข้าถึงได้ตลอดเวลา (ไม่แนะนำ) |
| `kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly` | สามารถเข้าถึงได้เมื่อตั้งรหัสผ่านและปลดล็อกเท่านั้น ไม่สำรองข้อมูล |

ACL (Access Control Lists) ของ Keychain ถูกประเมินภายใน Secure Enclave ทำให้ผู้โจมตีไม่สามารถ bypass การตรวจสอบสิทธิ์ได้

**ตัวอย่างการใช้งาน Keychain ใน Swift:**

```swift
import Security

// การจัดเก็บข้อมูลใน Keychain
func saveToKeychain(key: String, data: Data) -> Bool {
    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: key,
        kSecValueData as String: data,
        kSecAttrAccessible as String: kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        kSecUseDataProtectionKeychain as String: true
    ]
    
    // ลบข้อมูลเก่าก่อน (ถ้ามี)
    SecItemDelete(query as CFDictionary)
    
    // เพิ่มข้อมูลใหม่
    let status = SecItemAdd(query as CFDictionary, nil)
    return status == errSecSuccess
}

// การอ่านข้อมูลจาก Keychain
func readFromKeychain(key: String) -> Data? {
    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: key,
        kSecReturnData as String: true,
        kSecMatchLimit as String: kSecMatchLimitOne
    ]
    
    var item: CFTypeRef?
    let status = SecItemCopyMatching(query as CFDictionary, &item)
    
    guard status == errSecSuccess, let data = item as? Data else {
        return nil
    }
    return data
}
```

**Biometric Authentication กับ Keychain:**

```swift
import LocalAuthentication

// การกำหนดให้ Keychain Item ต้องยืนยัน Biometric
func saveWithBiometric(key: String, data: Data) {
    // กำหนด Access Control ด้วย Biometric
    let access = SecAccessControlCreateWithFlags(
        nil,
        kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        .biometryCurrentUser,  // Face ID หรือ Touch ID
        nil
    )
    
    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrAccount as String: key,
        kSecValueData as String: data,
        kSecAttrAccessControl as String: access as Any
    ]
    
    SecItemDelete(query as CFDictionary)
    SecItemAdd(query as CFDictionary, nil)
}
```

#### 11.3.3 Data Protection API

iOS Data Protection API ให้ความสามารถในการเข้ารหัสไฟล์ในระดับระบบปฏิบัติการโดยใช้ฮาร์ดแวร์เข้ารหัสของอุปกรณ์:

| ระดับการป้องกัน | คำอธิบาย |
|----------------|----------|
| `NSFileProtectionComplete` | เข้ารหัสไฟล์และไม่สามารถเข้าถึงได้เมื่ออุปกรณ์ถูกล็อก |
| `NSFileProtectionCompleteUnlessOpen` | เข้ารหัสไฟล์ แต่ไฟล์ที่เปิดอยู่แล้วสามารถเข้าถึงได้ต่อเมื่อปิดไฟล์ |
| `NSFileProtectionCompleteUntilFirstUserAuthentication` | เข้ารหัสไฟล์หลังจากผู้ใช้ปลดล็อกครั้งแรก — ระดับค่าเริ่มต้น |
| `NSFileProtectionNone` | ไม่มีการเข้ารหัส — ไม่แนะนำ |

**ตัวอย่างการใช้งาน Data Protection API ใน Swift:**

```swift
import Foundation

// การบันทึกไฟล์ด้วย Data Protection
let data = "Sensitive content".data(using: .utf8)!
let fileURL = FileManager.default.urls(
    for: .documentDirectory, in: .userDomainMask
).first!.appendingPathComponent("secret.txt")

// เขียนไฟล์และกำหนด Protection Level
try data.write(
    to: fileURL,
    options: [.completeFileProtection] // NSFileProtectionComplete
)

// หรือกำหนดภายหลัง
try (fileURL as NSURL).setResourceValue(
    URLFileProtection.complete,
    forKey: .fileProtectionKey
)
```

#### 11.3.4 App Transport Security (ATS)

ATS เป็นกลไกที่บังคับใช้โดย iOS ตั้งแต่เวอร์ชัน 9.0 เป็นต้นไป โดยกำหนดให้การเชื่อมต่อทั้งหมดต้องใช้ TLS 1.2 ขึ้นไป พร้อมกับ Forward Secrecy การเชื่อมต่อที่ไม่เป็นไปตามข้อกำหนดจะถูกบล็อกโดยระบบปฏิบัติการ

**ข้อยกเว้น ATS (ไม่แนะนำให้ใช้):**

```xml
<!-- Info.plist — ATS Exception (ไม่แนะนำสำหรับ Production) -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSAllowsArbitraryLoads</key>
    <false/>
    
    <!-- หากจำเป็นต้องยกเว้นสำหรับบางโดเมน -->
    <key>NSExceptionDomains</key>
    <dict>
        <key>legacy-api.example.com</key>
        <dict>
            <key>NSExceptionMinimumTLSVersion</key>
            <string>TLSv1.0</string>
        </dict>
    </dict>
</dict>
```

#### 11.3.5 การตรวจจับการถูก Jailbreak

Jailbreak Detection เป็นเทคนิคที่แอปพลิเคชันใช้ตรวจสอบว่าอุปกรณ์ถูก Jailbreak หรือไม่ อย่างไรก็ตาม เทคนิคนี้สามารถเลี่ยงได้ด้วยเครื่องมือเช่น Frida

**ตัวอย่างการตรวจจับการถูก Jailbreak ใน Swift:**

```swift
func isJailbroken() -> Bool {
    // ตรวจสอบการมีอยู่ของไฟล์ที่เกี่ยวข้องกับการ Jailbreak
    let jailbreakPaths = [
        "/Applications/Cydia.app",
        "/Applications/Sileo.app",
        "/usr/sbin/sshd",
        "/etc/apt",
        "/var/lib/cydia"
    ]
    
    for path in jailbreakPaths {
        if FileManager.default.fileExists(atPath: path) {
            return true
        }
    }
    
    // ตรวจสอบความสามารถในการเขียนนอก Sandbox
    do {
        try "test".write(
            toFile: "/private/jailbreak-test.txt",
            atomically: true,
            encoding: .utf8
        )
        // ถ้าเขียนสำเร็จ — แสดงว่าอุปกรณ์ถูก Jailbreak
        try FileManager.default.removeItem(atPath: "/private/jailbreak-test.txt")
        return true
    } catch {
        return false
    }
}
```

#### 11.3.6 Secure Enclave

Secure Enclave เป็น Coprocessor ด้านความปลอดภัยในฮาร์ดแวร์ที่เริ่มใช้ตั้งแต่ iPhone 5s เป็นต้นไป Secure Enclave ถูกแยกออกจาก Processor หลักอย่างสิ้นเชิง และคีย์เข้ารหัสจะไม่มีทางออกจากฮาร์ดแวร์นี้

**ความสามารถของ Secure Enclave:**
- จัดเก็บและประมวลผลข้อมูล Biometric (Face ID, Touch ID)
- สร้างและจัดเก็บคีย์เข้ารหัสที่ไม่สามารถถูกอ่านโดยระบบปฏิบัติการ
- ยืนยันรหัสผ่าน (Passcode) โดยไม่เปิดเผยข้อมูล
- ยืนยันธุรกรรม Apple Pay
- ประเมิน ACL ของ Keychain

### 11.4 Mobile API Security

#### 11.4.1 Certificate Pinning และ Public Key Pinning

Certificate Pinning เป็นเทคนิคที่แอปพลิเคชันยืนยันว่า Certificate ของเซิร์ฟเวอร์ตรงกับ Certificate ที่คาดไว้ ป้องกันการโจมตีแบบ MITM แม้ว่า Certificate Authority (CA) จะถูกบุกรุก

| วิธีการ | คำอธิบาย | ข้อดี | ข้อเสีย |
|--------|----------|------|--------|
| Certificate Pinning | ตรึงค่า Certificate จริง (Public Key + Metadata) | ตรวจจับการเปลี่ยนแปลง Certificate ได้ | ต้องอัปเดตแอปเมื่อ Certificate หมดอายุ |
| Public Key Pinning | ตรึงเฉพาะ Public Key ของ Certificate | ทนต่อการ Renew Certificate (ถ้าใช้ Key เดิม) | ต้องจัดการ Key Rotation |

**แนวทางที่แนะนำ:** Public Key Pinning เป็นวิธีที่แนะนำเนื่องจากสามารถ Renew Certificate ได้โดยไม่ต้องอัปเดตแอป (ถ้าใช้ Key คู่เดิม)

#### 11.4.2 การกำหนดค่า Network Security Configuration บน Android

Android แนะนำให้ใช้ Network Security Configuration (NSC) สำหรับการ Pinning ตั้งแต่ API 24 (Android 7.0):

```xml
<!-- res/xml/network_security_config.xml -->
<?xml version="1.0" encoding="utf-8"?>
<network-security-config>
    <!-- การกำหนดค่า Pinning สำหรับ Production Domain -->
```

#### 11.4.3 Certificate Pinning บน iOS ด้วย URLSessionDelegate

```swift
import Foundation

class PinningURLSessionDelegate: NSObject, URLSessionDelegate {
    let pinnedPublicKeyHash = "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="
    
    func urlSession(
        _ session: URLSession,
        didReceive challenge: URLAuthenticationChallenge,
        completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void
    ) {
        // ตรวจสอบว่าเป็นการท้าทายแบบ Server Trust
        guard challenge.protectionSpace.authenticationMethod ==
              NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust else {
            completionHandler(.performDefaultHandling, nil)
            return
        }
        
        // รับ Certificate จาก Server
        if let serverCertificate = SecTrustGetCertificateAtIndex(serverTrust, 0) {
            let serverPublicKey = SecCertificateCopyKey(serverCertificate)
            let serverPublicKeyData = SecKeyCopyExternalRepresentation(serverPublicKey!, nil)!
            let serverPublicKeyHash = sha256(data: serverPublicKeyData as Data)
            
            // เปรียบเทียบ Public Key Hash
            if serverPublicKeyHash == pinnedPublicKeyHash {
                completionHandler(.useCredential, URLCredential(trust: serverTrust))
            } else {
                completionHandler(.cancelAuthenticationChallenge, nil)
            }
        }
    }
    
    private func sha256(data: Data) -> String {
        // สร้าง SHA-256 hash และเข้ารหัส Base64
        var hash = [UInt8](repeating: 0, count: Int(CC_SHA256_DIGEST_LENGTH))
        data.withUnsafeBytes { buffer in
            _ = CC_SHA256(buffer.baseAddress, CC_LONG(data.count), &hash)
        }
        return Data(hash).base64EncodedString()
    }
}
```

#### 11.4.4 OAuth 2.0 สำหรับอุปกรณ์เคลื่อนที่ด้วย PKCE

OAuth 2.0 Authorization Code Flow with PKCE (Proof Key for Code Exchange) ตาม RFC 7636 เป็นมาตรฐานที่จำเป็นสำหรับแอปพลิเคชันบนอุปกรณ์เคลื่อนที่เนื่องจากไม่สามารถเก็บ Client Secret อย่างปลอดภัยได้

**ขั้นตอนของ PKCE Flow:**

| ขั้นตอน | คำอธิบาย |
|--------|----------|
| 1. สร้าง Code Verifier | สุ่มค่า String ที่มีความยาว 43-128 ตัวอักษร |
| 2. สร้าง Code Challenge | นำ Code Verifier มาทำ SHA-256 hash (S256) |
| 3. ส่ง Authorization Request | ส่ง Code Challenge ไปยัง Authorization Server |
| 4. รับ Authorization Code | รับ Authorization Code หลังจากผู้ใช้อนุญาต |
| 5. ส่ง Token Request | ส่ง Authorization Code + Code Verifier ไปยัง Token Endpoint |
| 6. รับ Access Token | เซิร์ฟเวอร์ตรวจสอบ Code Verifier ว่าตรงกับ Code Challenge หรือไม่ |

**ข้อกำหนด:** OAuth 2.1 กำหนดให้ PKCE เป็นข้อบังคับสำหรับทุกประเภทของ Client โดยใช้ S256 (SHA-256 based) เป็นค่าเริ่มต้น

**ตัวอย่างการใช้งาน PKCE ใน Kotlin:**

```kotlin
import java.security.MessageDigest
import java.util.Base64

class PKCEGenerator {
    companion object {
        // สร้าง Code Verifier แบบสุ่ม
        fun generateCodeVerifier(): String {
            val random = java.security.SecureRandom()
            val bytes = ByteArray(32)
            random.nextBytes(bytes)
            return Base64.getUrlEncoder().withoutPadding()
                .encodeToString(bytes)
        }
        
        // สร้าง Code Challenge ด้วย S256
        fun generateCodeChallenge(verifier: String): String {
            val digest = MessageDigest.getInstance("SHA-256")
            val hash = digest.digest(verifier.toByteArray())
            return Base64.getUrlEncoder().withoutPadding()
                .encodeToString(hash)
        }
    }
}
```

| วิธีการ | ระดับความปลอดภัย | คำอธิบาย |

| Proxy Server (แนะนำมากที่สุด) | ปลอดภัยที่สุด | คีย์อยู่บนเซิร์ฟเวอร์ ไม่เคยถูกส่งไปยังอุปกรณ์ |

### 11.5 Secure Data Storage บนอุปกรณ์เคลื่อนที่

#### 11.5.1 Android Keystore

Android Keystore เป็นระบบที่ใช้ฮาร์ดแวร์ (TEE — Trusted Execution Environment) ในการจัดการคีย์เข้ารหัส โดยคีย์ Material จะไม่มีทางเข้าไปใน Process ของแอปพลิเคชัน:

```kotlin
import java.security.KeyStore
import javax.crypto.Cipher
import javax.crypto.KeyGenerator
import javax.crypto.SecretKey
import android.security.keystore.KeyGenParameterSpec
import android.security.keystore.KeyProperties

class AndroidKeystoreManager {
    private val keyStore = KeyStore.getInstance("AndroidKeyStore").apply {
        load(null)
    }
    
    fun generateKey(keyAlias: String) {
        val keyGenerator = KeyGenerator.getInstance(
            KeyProperties.KEY_ALGORITHM_AES,
            "AndroidKeyStore"
        )
        
        val spec = KeyGenParameterSpec.Builder(
            keyAlias,
            KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
        )
            .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
            .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
            .setKeySize(256)
            .setUserAuthenticationRequired(true)  // ต้องยืนยันตัวตนผู้ใช้
            .build()
        
        keyGenerator.init(spec)
        keyGenerator.generateKey()
    }
    
    fun encrypt(keyAlias: String, plaintext: String): ByteArray {
        val secretKey = keyStore.getEntry(keyAlias, null) as KeyStore.SecretKeyEntry
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.ENCRYPT_MODE, secretKey.secretKey)
        
        // IV (Initialization Vector) ต้องถูกเก็บไว้เพื่อใช้ในการ Decrypt
        val iv = cipher.iv
        val ciphertext = cipher.doFinal(plaintext.toByteArray())
        
        return iv + ciphertext  // รวม IV + ข้อมูลที่เข้ารหัส
    }
}
```

#### 11.5.2 iOS Keychain (Secure Enclave Backed)

iOS Keychain ที่ใช้ Secure Enclave เป็นการจัดเก็บข้อมูลที่ปลอดภัยที่สุดบนอุปกรณ์ iOS โดยคีย์จะถูกจัดเก็บและประมวลผลภายใน Secure Enclave เท่านั้น:

```swift
// การใช้ Secure Enclave ในการสร้างคีย์
func generateSecureEnclaveKey() -> SecKey? {
    let attributes: [String: Any] = [
        kSecAttrKeyType as String: kSecAttrKeyTypeECSECPrimeRandom,
        kSecAttrKeySizeInBits as String: 256,
        kSecAttrTokenID as String: kSecAttrTokenIDSecureEnclave,

    // คีย์ถูกสร้างภายใน Secure Enclave — private key ไม่มีทางออกมา
```

#### 11.5.3 EncryptedSharedPreferences (AndroidX Security)

EncryptedSharedPreferences เป็นไลบรารีจาก AndroidX Security ที่ใช้ AES-256 GCM ในการเข้ารหัสข้อมูลที่จัดเก็บใน SharedPreferences โดยใช้คีย์จาก Android Keystore:

```kotlin
import androidx.security.crypto.EncryptedSharedPreferences
import androidx.security.crypto.MasterKey

class SecurePreferences(context: Context) {
    private val masterKey = MasterKey.Builder(context)
        .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
        .build()
    
    private val securePrefs = EncryptedSharedPreferences.create(
        context,
        "secure_prefs",
        masterKey,
        EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
        EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
    )
    
    fun saveToken(token: String) {
        securePrefs.edit().putString("auth_token", token).apply()
    }
    
    fun getToken(): String? {
        return securePrefs.getString("auth_token", null)
    }
}
```

SQLCipher เป็นส่วนขยายของ SQLite ที่เข้ารหัสทั้งฐานข้อมูลด้วย AES-256-CBC และ HMAC-SHA512 พร้อม PBKDF2 สำหรับการ Derive Key:

```kotlin
}
```

### Lab 11.1: การวิเคราะห์ APK ด้วย Mobile Security Framework (MobSF)

**วัตถุประสงค์:** เรียนรู้การใช้ MobSF ในการวิเคราะห์ความปลอดภัยของ APK แบบ Static Analysis

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** MobSF v4.4.6, ไฟล์ APK ตัวอย่าง (ที่อาจารย์จัดให้)

**ขั้นตอน:**

1. รัน MobSF ด้วย Docker:
   ```bash
   docker run --rm -it -p 8000:8000 opensecurity/mobile-security-framework-mobsf:v4.4.6
   ```

2. เปิดเบราว์เซอร์ไปที่ `http://localhost:8000`

3. อัปโหลดไฟล์ APK ตัวอย่างที่อาจารย์จัดให้

4. รอให้ MobSF วิเคราะห์เสร็จ (ประมาณ 2-5 นาที)

5. ตรวจสอบผลลัพธ์ในแต่ละหมวดหมู่:
   - **File Analysis:** ตรวจสอบไฟล์ภายใน APK
   - **Permissions:** ดู Permissions ที่แอปพลิเคชันขอ
   - **Manifest Analysis:** ตรวจสอบ Exported Components
   - **Code Analysis:** ดูช่องโหว่ที่พบในโค้ด
   - **Network Analysis:** ดู URL และโดเมนที่แอปพลิเคชันเชื่อมต่อ

6. บันทึกรายงานในรูปแบบ PDF หรือ HTML

**สิ่งที่ต้องส่ง:**

1. รายชื่อ Permissions ที่แอปพลิเคชันขอ แบ่งเป็น Normal และ Dangerous
2. Exported Components ที่พบ (Activity, Service, Broadcast Receiver, Content Provider)
3. ช่องโหว่ที่มีความเสี่ยงสูงที่สุด 3 รายการ พร้อมคำอธิบาย
4. รายชื่อโดเมนที่แอปพลิเคชันเชื่อมต่อ

### Lab 11.2: การดักจับ Mobile App Traffic ด้วย Burp Suite

**วัตถุประสงค์:** เรียนรู้การตั้งค่า Proxy และดักจับ Traffic ของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** Burp Suite Community Edition, อุปกรณ์ Android/Emulator

**ขั้นตอน:**

1. เปิด Burp Suite และตั้งค่า Proxy เป็น `127.0.0.1:8080`

2. ตั้งค่า Proxy บนอุปกรณ์ Android:
   ```bash
   # เชื่อมต่ออุปกรณ์กับเครือข่ายเดียวกัน
   # ตั้งค่า WiFi Proxy → IP ของคอมพิวเตอร์ + Port 8080
   ```

3. ติดตั้ง CA Certificate ของ Burp Suite บนอุปกรณ์:
   - เปิดเบราว์เซอร์ไปที่ `http://burpsuite`
   - ดาวน์โหลด `cacert.der`
   - ติดตั้งใน Settings → Security → Install from Storage

4. ทดสอบเปิดแอปพลิเคชันตัวอย่าง และสังเกต Traffic ใน Burp Suite

5. ตรวจสอบว่า HTTPS Traffic ถูกถอดรหัสหรือไม่:
   - ถ้า Certificate Pinning ถูกใช้งาน → การเชื่อมต่อจะถูกบล็อก
   - ถ้าไม่มีการ Pinning → สามารถดู Request/Response ได้

6. วิเคราะห์ API Endpoint ที่แอปพลิเคชันเรียกใช้:
   - บันทึก URL Endpoint ที่พบ
   - ตรวจสอบ Request Method และ Headers
   - ตรวจสอบ Request Body และ Response

**สิ่งที่ต้องส่ง:**

1. ภาพ Screenshot แสดง Burp Suite ที่ดักจับ Traffic ได้
2. รายการ API Endpoint ที่แอปพลิเคชันเรียกใช้
3. การตรวจสอบว่ามี Certificate Pinning หรือไม่
4. ข้อมูลที่เปิดเผย (ถ้ามี) เช่น API Key, Token, ข้อมูลส่วนบุคคล

### Lab 11.3: การตรวจสอบ Android Manifest และ iOS Info.plist

**วัตถุประสงค์:** เรียนรู้การตรวจสอบการตั้งค่าความปลอดภัยใน Manifest และ Info.plist

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** APKTool, MobSF, ไฟล์ APK/IPA ตัวอย่าง

**ขั้นตอน:**

1. ถอด APK ด้วย APKTool:
   ```bash
   apktool d target.apk -o output_dir/
   ```

2. ตรวจสอบ `AndroidManifest.xml` ในประเด็นต่อไปนี้:
   - `android:exported="true"` ใน Activity, Service, Receiver
   - `android:allowBackup="true"`
   - `android:usesCleartextTraffic="true"`
   - `android:debuggable="true"`
   - Intent Filter ที่ไม่มีการตรวจสอบ
   - `<uses-permission>` ที่เป็น Dangerous Permission

3. สำหรับ iOS ใช้ MobSF หรือเครื่องมืออื่นตรวจสอบ `Info.plist`:
   - `NSAppTransportSecurity` — มี `NSAllowsArbitraryLoads` หรือไม่
   - `NSCameraUsageDescription`, `NSLocationWhenInUseUsageDescription` — คำอธิบายที่เหมาะสม
   - `CFBundleURLTypes` — URL Scheme ที่เปิดรับ
   - `UIBackgroundModes` — การทำงานเบื้องหลัง

4. สร้างตารางเปรียบเทียบความแตกต่างระหว่าง Android Manifest และ iOS Info.plist

**สิ่งที่ต้องส่ง:**

1. รายการ Component ที่ถูก Export ใน Android Manifest
2. การตั้งค่าที่ไม่ปลอดภัยที่พบ (เช่น Cleartext Traffic, Allow Backup, Debuggable)
3. การตั้งค่า ATS Exception ใน iOS Info.plist (ถ้ามี)
4. ตารางเปรียบเทียบ Android Manifest vs iOS Info.plist

### Lab 11.4: Android Reverse Engineering ด้วย JADX และ APKTool

**วัตถุประสงค์:** เรียนรู้การ Decompile และวิเคราะห์โค้ดของแอปพลิเคชัน Android

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** JADX, APKTool, ไฟล์ APK ตัวอย่าง

**ขั้นตอน:**

1. ติดตั้ง JADX:
   ```bash
   # macOS
   brew install jadx
   ```

2. Decompile APK ด้วย JADX:
   ```bash
   jadx-gui target.apk
   # หรือ
   jadx -d output_java/ target.apk
   ```

3. วิเคราะห์ Source Code ที่ Decompile ได้:
   - ค้นหา Hardcoded API Key: ค้นหาคำว่า `api_key`, `apikey`, `secret`, `token`
   - ค้นหา SharedPreferences: ค้นหาการใช้ `getSharedPreferences` และ `putString`
   - ค้นหา WebView: ค้นหาการใช้ `WebView`, `setJavaScriptEnabled`, `addJavascriptInterface`
   - ค้นหา Intent: ค้นหาการใช้ `getIntent`, `startActivity`, `Intent`

4. ตรวจสอบคลาสที่มีความสำคัญด้านความปลอดภัย:
   ```bash
   # ค้นหา String ที่น่าสนใจในโค้ด
   grep -r "https://" output_java/
   grep -r "password" output_java/
   grep -r "secret" output_java/
   ```

5. วิเคราะห์ ProGuard/R8 Obfuscation:
   - ตรวจสอบว่าโค้ดถูก Obfuscate หรือไม่
   - ถ้าถูก Obfuscate — คลาสและเมธอดจะมีชื่อที่อ่านยาก (a.a(), b.c())
   - ถ้าไม่ถูก Obfuscate — โค้ดจะอ่านได้ชัดเจน

**สิ่งที่ต้องส่ง:**

1. ภาพ Screenshot ของ JADX ที่แสดง Source Code ที่ Decompile ได้
2. Hardcoded Credential หรือ API Key ที่พบ (รายงานผ่าน MobSF ก่อน)
3. การใช้ WebView ที่พบและการประเมินความปลอดภัย
4. การใช้ Intent และการตรวจสอบ Caller
5. การประเมิน Obfuscation — มีหรือไม่ และมีผลต่อการวิเคราะห์อย่างไร

## สรุปท้ายบท

ความปลอดภัยของแอปพลิเคชันบนอุปกรณ์เคลื่อนที่เป็นเรื่องที่ซับซ้อนและต้องพิจารณาทั้งในด้านความปลอดภัยของแพลตฟอร์ม (Android และ iOS), การสื่อสารผ่านเครือข่าย, การจัดเก็บข้อมูล, และการทำงานร่วมกับ API OWASP MASVS v2.1.0 เป็นมาตรฐานที่ครอบคลุม 8 กลุ่มควบคุม ตั้งแต่การจัดเก็บข้อมูลอย่างปลอดภัยไปจนถึงความเป็นส่วนตัว ในขณะที่ OWASP Mobile Top 10 2024 ระบุความเสี่ยงที่สำคัญที่สุด 10 อันดับ

Android และ iOS มีแนวทางด้านความปลอดภัยที่แตกต่างกันอย่างมีนัยสำคัญ Android ใช้ Linux UID-based Sandbox และ Runtime Permissions ในขณะที่ iOS ใช้ MAC-based Sandbox ผ่าน TrustedBSD และ Entitlements ที่ถูกเซ็นใน Code Signature การทำความเข้าใจความแตกต่างนี้สำคัญต่อการพัฒนาแอปพลิเคชันที่ปลอดภัยบนทั้งสองแพลตฟอร์ม

Mobile API Security โดยเฉพาะ Certificate Pinning, OAuth 2.0 with PKCE, และการป้องกัน MITM เป็นสิ่งจำเป็นสำหรับการสื่อสารที่ปลอดภัยระหว่างแอปพลิเคชันกับเซิร์ฟเวอร์ ในด้านการจัดเก็บข้อมูล Android Keystore และ iOS Keychain (Secure Enclave) เป็นกลไกหลักที่ควรใช้ร่วมกับ EncryptedSharedPreferences และ SQLCipher

เครื่องมือทดสอบความปลอดภัยเช่น MobSF, Burp Suite, Drozer และ Objection ช่วยให้นักพัฒนาสามารถตรวจสอบและปรับปรุงความปลอดภัยของแอปพลิเคชันได้อย่างเป็นระบบ ในขณะที่เหตุการณ์จริงในปี 2025-2026 (EngageSDK, Google Gemini API Key Leak, AppsFlyer Supply Chain Attack) แสดงให้เห็นว่าภัยคุกคามต่อแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ยังคงเพิ่มขึ้นอย่างต่อเนื่อง
