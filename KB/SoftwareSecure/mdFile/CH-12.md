# CH-12: Network Security สำหรับซอฟต์แวร์


---

## วัตถุประสงค์การเรียนรู้

เมื่อจบบทนี้นักศึกษาสามารถ:

1. อธิบายพื้นฐานด้าน Network Security ที่เกี่ยวข้องกับนักพัฒนา ได้แก่ โมเดล OSI ในแต่ละชั้น ปัญหาด้านความปลอดภัยของ TCP/IP (IP Spoofing, TCP SYN Flood, Session Hijacking) และแนวทางป้องกัน DNS Security (DNSSEC, DNS over HTTPS/TLS)
2. วิเคราะห์ความแตกต่างระหว่าง TLS 1.2 และ TLS 1.3 ในด้าน Handshake, Cipher Suites, Forward Secrecy และเลือกใช้การตั้งค่าที่ปลอดภัยสำหรับซอฟต์แวร์
3. ติดตั้งและจัดการ Certificate สำหรับ HTTPS โดยใช้ Let's Encrypt, ACME Protocol และ Certificate Transparency รวมถึงการใช้งาน mTLS สำหรับ Service-to-Service Authentication
4. จำแนกความแตกต่างระหว่าง Network Firewall, WAF, Stateful vs Stateless Firewall และ Security Group/Network ACL ในระบบคลาวด์ พร้อมอธิบายแนวคิด Zero Trust Network Access (ZTNA)
5. ระบุประเภทของ DDoS Attack และเลือกใช้มาตรการป้องกันที่เหมาะสม ได้แก่ CDN, Rate Limiting, Auto Scaling และ Load Balancing
6. ใช้เครื่องมือ Network Monitoring และ Threat Detection เช่น Suricata, Wireshark, Zeek และ SIEM เพื่อตรวจจับและวิเคราะห์ภัยคุกคามทางเครือข่าย
7. บูรณาการแนวทาง Network Security ในกระบวนการพัฒนาซอฟต์แวร์ ได้แก่ Structured Logging, Distributed Tracing และการตั้งค่า CAA DNS Records

## ขอบเขตและข้อกำหนดด้านจริยธรรมของบทนี้

บทนี้ครอบคลุมหลักการรักษาความปลอดภัยทางเครือข่ายในมุมมองของนักพัฒนาซอฟต์แวร์ เนื้อหามีตัวอย่างเกี่ยวกับเทคนิคการโจมตีเพื่อการเรียนรู้เชิงป้องกันเท่านั้น กิจกรรมปฏิบัติการทั้งหมดต้องทำในสภาพแวดล้อมที่ได้รับอนุญาต เช่น ห้องปฏิบัติการทดสอบ, เครื่องเสมือน (Virtual Machine) หรือระบบที่อาจารย์จัดให้เท่านั้น

**ข้อควรจำ:** ห้ามนำเทคนิคในบทนี้ไปทดสอบกับระบบของบุคคลอื่นโดยไม่ได้รับอนุญาตเป็นลายลักษณ์อักษร การใช้เครื่องมือเช่น nmap, Metasploit, Burp Suite หรือเทคนิคการโจมตีทางเครือข่ายกับระบบที่ไม่ใช่ของตนเองหรือไม่ได้รับอนุญาตอาจมีความผิดทางกฎหมาย พ.ร.บ. ว่าด้วยการกระทำผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2560 และที่แก้ไขเพิ่มเติม การทดสอบระบบต้องได้รับอนุญาตเป็นลายลักษณ์อักษรก่อนดำเนินการทุกครั้ง

## เนื้อหา

### 12.1 พื้นฐาน Network Security สำหรับนักพัฒนา

Network Security เป็นองค์ประกอบที่สำคัญของความปลอดภัยของซอฟต์แวร์โดยรวม แม้ว่านักพัฒนาซอฟต์แวร์จะไม่ได้ดูแลโครงสร้างพื้นฐานเครือข่ายโดยตรง แต่ความเข้าใจในหลักการด้าน Network Security ช่วยให้นักพัฒนาสามารถเขียนโค้ดที่ปลอดภัยและทำงานร่วมกับทีมโครงสร้างพื้นฐานได้อย่างมีประสิทธิภาพ

#### 12.1.1 โมเดล OSI กับความปลอดภัยในแต่ละชั้น

โมเดล OSI (Open Systems Interconnection) แบ่งการสื่อสารออกเป็น 7 ชั้น สำหรับนักพัฒนาซอฟต์แวร์ ชั้นที่เกี่ยวข้องมากที่สุดคือ Layer 7 (Application), Layer 4 (Transport) และ Layer 3 (Network):

| ชั้น OSI | โปรโตคอล | ความเสี่ยงด้านความปลอดภัย | แนวทางป้องกันที่นักพัฒนาควรทราบ |
|----------|-----------|--------------------------|--------------------------------|
| L7 Application | HTTP/HTTPS, DNS, SSH, FTP | Injection, XSS, Broken Auth, API Abuse | Input Validation, WAF, Authentication ที่ปลอดภัย, Security Headers |
| L6 Presentation | TLS/SSL, MIME | Protocol Downgrade, Weak Cipher | ใช้ TLS 1.3 เท่านั้น, ปิด Cipher ที่ไม่ปลอดภัย |
| L5 Session | Sockets, NetBIOS | Session Hijacking, Session Fixation | Session ID ที่ปลอดภัย (CSPRNG), HttpOnly/Secure Cookies |
| L4 Transport | TCP, UDP | SYN Flood, Port Scanning, Connection Hijacking | SYN Cookies, Rate Limiting, Firewall Rules |
| L3 Network | IP, ICMP | IP Spoofing, Smurf Attack, DDoS | Egress Filtering, Ingress Filtering (BCP 38) |

**สำหรับนักพัฒนา:** จุดที่นักพัฒนาควรให้ความสำคัญมากที่สุดคือ Layer 7 (Application) และ Layer 4 (Transport) เนื่องจากเป็นชั้นที่ซอฟต์แวร์สามารถควบคุมได้โดยตรง การเข้ารหัสด้วย TLS อยู่ที่ Layer 6 แต่ถูกเรียกใช้งานจาก Layer 7 ผ่านไลบรารีเช่น OpenSSL, BoringSSL หรือ crypto/tls

#### 12.1.2 TCP/IP Security Issues

**IP Spoofing:**
IP Spoofing เป็นเทคนิคที่ผู้โจมตีปลอมแปลง Source IP Address ในหัวข้อแพ็กเกต IP เพื่อทำให้ดูเหมือนว่าแพ็กเกตมาจากแหล่งที่เชื่อถือได้ การโจมตีแบบนี้มักใช้ในการโจมตีแบบ Reflection DDoS (เช่น DNS Amplification, NTP Amplification) หรือการโจมตีที่ต้องการหลบเลี่ยง Access Control Lists (ACL)

แนวทางป้องกัน IP Spoofing:
- **Egress Filtering:** กรองแพ็กเกตขาออกที่ Source IP ไม่ตรงกับเครือข่ายภายในตามแนวทาง BCP 38 (RFC 2827)
- **Ingress Filtering:** กรองแพ็กเกตขาเข้าที่ Source IP มาจากเครือข่ายภายในทางอินเทอร์เฟซภายนอก
- **uRPF (Unicast Reverse Path Forwarding):** ตรวจสอบว่า Source IP ของแพ็กเกตสามารถเข้าถึงได้ผ่านอินเทอร์เฟซที่แพ็กเกตเข้ามาจริง

**TCP SYN Flood:**
TCP SYN Flood เป็นการโจมตีแบบ Denial of Service (DoS) ที่อาศัยช่องโหว่ในกลไก Three-Way Handshake ของ TCP ผู้โจมตีจะส่งแพ็กเกต SYN จำนวนมากโดยไม่ตอบกลับ ACK ทำให้เซิร์ฟเวอร์ต้องจัดสรรทรัพยากรเพื่อรอการเชื่อมต่อที่ไม่มีวันสมบูรณ์ ซึ่งทำให้ Backlog Queue เต็มและไม่สามารถรับการเชื่อมต่อใหม่ได้

แนวทางป้องกัน TCP SYN Flood:
- **SYN Cookies:** เทคนิคที่เข้ารหัสข้อมูลสถานะการเชื่อมต่อไว้ใน Sequence Number ของ SYN-ACK แทนการจัดสรรทรัพยากรล่วงหน้า ทำให้เซิร์ฟเวอร์ไม่ต้องจองทรัพยากรจนกว่า Three-Way Handshake จะสมบูรณ์ SYN Cookies ถูกเปิดใช้งานเป็นค่าเริ่มต้นใน Linux ตั้งแต่ Kernel 2.6.x
- **SYN Proxy:** ให้ Firewall หรือ Load Balancer รับมือ Three-Way Handshake แทนเซิร์ฟเวอร์จริง
- **เพิ่ม TCP Backlog Queue:** การเพิ่มค่า `tcp_max_syn_backlog` และ `somaxconn`

**Session Hijacking:**
Session Hijacking คือการที่ผู้โจมตีสามารถขโมยหรือคาดเดา Session ID ของผู้ใช้ที่ถูกต้องแล้วนำไปใช้ในการเข้าถึงระบบเสมือนเป็นผู้ใช้นั้น กลไกการโจมตีรวมถึงการดักจับ Cookie ที่ไม่มีการเข้ารหัส, การคาดเดา Session ID ที่อ่อนแอ, หรือ Cross-Site Scripting (XSS) ที่ขโมย Cookie

แนวทางป้องกัน Session Hijacking:
- **CSPRNG (Cryptographically Secure Pseudo-Random Number Generator):** ใช้สำหรับสร้าง Session ID ที่ไม่สามารถคาดเดาได้ การใช้ฟังก์ชันเช่น `random_bytes()` ใน PHP หรือ `crypto/rand` ใน Go
- **HttpOnly และ Secure Flags:** ตั้งค่า Cookie Flag `HttpOnly` เพื่อป้องกันการเข้าถึงผ่าน JavaScript และ `Secure` เพื่อให้ส่งผ่าน HTTPS เท่านั้น
- **SameSite Flag:** ใช้ `SameSite=Lax` หรือ `SameSite=Strict` เพื่อป้องกัน CSRF-based Session Hijacking
- **Session Timeout และ Rotation:** กำหนดอายุ Session และสร้าง Session ID ใหม่หลังจากการยืนยันตัวตนสำเร็จ

**CVE-2025-6515 (เผยแพร่: 20 ตุลาคม 2025):** ช่องโหว่ใน oatpp-mcp (MCP SSE endpoint) ที่ส่งคืน Instance Pointer เป็น Session ID ซึ่งไม่ใช่ค่าที่ไม่ซ้ำกันและไม่ปลอดภัยทางด้านการเข้ารหัส ผู้โจมตีที่สามารถเข้าถึงเครือข่ายของเซิร์ฟเวอร์ oatpp-mcp สามารถคาดเดา Session ID ในอนาคตและทำการโจมตีแบบ Session Hijacking เพื่อควบคุม Session ของ Client ที่ถูกต้อง คะแนน CVSS 3.1 อยู่ที่ 6.8 (Medium) จัดอยู่ใน CWE-330 (Use of Insufficiently Random Values)

#### 12.1.3 DNS Security

DNS (Domain Name System) เป็นโปรโตคอลพื้นฐานที่ไม่มีกลไกความปลอดภัยในตัว การโจมตี DNS ที่พบบ่อยรวมถึง DNS Spoofing/Cache Poisoning, DNS Tunneling, DNS Amplification DDoS และ DNS Hijacking

**DNSSEC (DNS Security Extensions):**
DNSSEC เป็นชุดของส่วนขยายมาตรฐานที่เพิ่มความสมบูรณ์ของข้อมูล (Data Integrity) และการยืนยันแหล่งที่มา (Source Authentication) ให้กับ DNS โดยใช้ Digital Signature ข้อมูล DNS ที่ได้รับการเซ็นชื่อด้วย DNSSEC จะมี Resource Record (RR) พิเศษเช่น RRSIG, DNSKEY, DS และ NSEC

NIST SP 800-81r3 (Secure Domain Name System Deployment Guide ฉบับปรับปรุงครั้งที่ 3) เป็นเอกสารมาตรฐานที่เผยแพร่ในเดือนมีนาคม ค.ศ. 2026 โดย NIST ให้คำแนะนำเกี่ยวกับการปรับใช้ DNS อย่างปลอดภัยในองค์กร ครอบคลุม 3 ด้านหลัก:
1. **การปกป้องข้อมูล DNS ด้วย DNSSEC:** ปกป้องความสมบูรณ์ของข้อมูล DNS จากการถูกแก้ไขระหว่างทาง
2. **การใช้ Encrypted DNS:** ปกป้องความเป็นส่วนตัวของคำถาม DNS ด้วย DoT, DoH, DoQ
3. **บทบาทของ DNS ใน Zero Trust Architecture:** DNS ทำหน้าที่เป็น Policy Enforcement Point (PEP) และแหล่งข้อมูลสำหรับการประเมินคำขอเข้าถึง

**ข้อควรทราบ:** DNSSEC ให้เฉพาะการปกป้องความสมบูรณ์ของข้อมูล (Integrity) และการยืนยันตัวตน (Authentication) เท่านั้น ไม่ได้ให้การเข้ารหัส (Encryption) หรือการปกป้องความเป็นส่วนตัว (Privacy) คุณสมบัติเหล่านั้นมาจาก DNS over TLS (DoT) และ DNS over HTTPS (DoH)

**DNS over HTTPS (DoH) และ DNS over TLS (DoT):**

| คุณสมบัติ | DNS over TLS (DoT) | DNS over HTTPS (DoH) |
|-----------|-------------------|---------------------|
| พอร์ต | 853 (พอร์ตเฉพาะ) | 443 (รวมกับ HTTPS) |
| โปรโตคอลพื้นฐาน | TLS | HTTPS (HTTP/2 หรือ HTTP/3) |
| การระบุตัวตน | ง่าย — พอร์ตเฉพาะที่ใช้สำหรับ DoT | ยาก — รวมอยู่ในการรับส่ง HTTPS ปกติ |
| การถูกบล็อก | ง่าย — บล็อกพอร์ต 853 | ยาก — แยกไม่ออกจาก HTTPS ทั่วไป |
| มาตรฐาน | RFC 7858 | RFC 8484 |
| การใช้งาน | Enterprise, Stub Resolver | Browser, Application |

**การทำงานร่วมกันของ DNSSEC และ DoH/DoT:**
การใช้งาน DNSSEC ร่วมกับ DoH/DoT ให้การปกป้องใน 2 มิติที่แตกต่างกัน:
- DNSSEC ปกป้องความสมบูรณ์ของข้อมูล DNS (ตอบคำถามว่า "ข้อมูลนี้ถูกต้องหรือไม่")
- DoH/DoT ปกป้องความเป็นส่วนตัวของคำถาม DNS (ตอบคำถามว่า "ใครถามอะไร")

**CVE-2025-66414 (เผยแพร่: 2 ธันวาคม 2025):** ช่องโหว่ใน MCP TypeScript SDK ที่ไม่เปิดใช้งาน DNS Rebinding Protection เป็นค่าเริ่มต้นสำหรับเซิร์ฟเวอร์ที่ใช้ HTTP เมื่อเซิร์ฟเวอร์ MCP ที่ใช้ HTTP ทำงานบน localhost โดยไม่มีการยืนยันตัวตน เว็บไซต์ที่ไม่หวังดีสามารถใช้ DNS Rebinding เพื่อเลื่อนนโยบาย Same-Origin และส่งคำขอไปยังเซิร์ฟเวอร์ MCP ในเครื่อง ส่งผลให้ผู้โจมตีสามารถเรียกใช้เครื่องมือหรือเข้าถึงทรัพยากรที่เปิดเผยโดยเซิร์ฟเวอร์ MCP ได้ คะแนน CVSS 3.1 อยู่ที่ 8.1 (High) CWE-1188 (Insecure Default Initialization of Resource) แก้ไขในเวอร์ชัน 1.24.0

#### 12.1.4 การแบ่ง Network Segments

การแบ่งเครือข่ายออกเป็นส่วน (Network Segmentation) เป็นแนวทางป้องกันความปลอดภัยที่จำกัดการแพร่กระจายของการโจมตีและลด Attack Surface:

| Segment | คำอธิบาย | บริการที่อยู่ | แนวทางการเข้าถึง |
|---------|----------|-------------|------------------|
| **DMZ (Demilitarized Zone)** | เครือข่ายที่เปิดให้เข้าถึงจากอินเทอร์เน็ตได้ | Web Server, Reverse Proxy, API Gateway, DNS Server | เปิดเฉพาะพอร์ตที่จำเป็น (80, 443, 53) |
| **Internal Network** | เครือข่ายภายในที่ไม่เปิดสู่สาธารณะ | Application Server, Database, Internal API | เข้าถึงได้จาก DMZ เท่านั้น ไม่เปิดสู่อินเทอร์เน็ต |
| **Management Network** | เครือข่ายสำหรับผู้ดูแลระบบ | SSH/Bastion Host, Monitoring, Logging, Backup | จำกัดเฉพาะผู้ดูแลที่ได้รับอนุญาต, ใช้ VPN หรือ Jump Box |
| **Development/Test Network** | เครือข่ายสำหรับพัฒนาและทดสอบ | Dev/Staging Servers, CI/CD | แยกจาก Production อย่างเด็ดขาด ไม่มีข้อมูลจริง |

**หลักการสำคัญ:** เซิร์ฟเวอร์ใน DMZ ไม่ควรสามารถเชื่อมต่อไปยัง Internal Network ได้โดยตรง (ต้องผ่าน Reverse Proxy หรือ API Gateway ที่มีการตรวจสอบสิทธิ์) และ Management Network ควรเข้าถึงได้ผ่าน Bastion Host หรือ VPN เท่านั้น

### 12.2 TLS/SSL และ HTTPS

TLS (Transport Layer Security) เป็นโปรโตคอลที่อยู่ระหว่าง Layer 4 (Transport) และ Layer 7 (Application) ในโมเดล OSI ให้การเข้ารหัส (Encryption), การยืนยันตัวตน (Authentication) และความสมบูรณ์ของข้อมูล (Integrity) สำหรับการสื่อสารผ่านเครือข่าย

#### 12.2.1 TLS Handshake: 1.2 vs 1.3

**TLS 1.2 Handshake (2-RTT):**
1. **ClientHello:** Client ส่ง TLS Version ที่รองรับ, Cipher Suites ที่รองรับ, และ Client Random
2. **ServerHello:** Server เลือก TLS Version และ Cipher Suite, ส่ง Server Random และ Certificate
3. **Certificate Verification:** Client ตรวจสอบ Certificate ของ Server
4. **Key Exchange:** Client สร้าง Pre-Master Secret, เข้ารหัสด้วย Public Key ของ Server, ส่งให้ Server
5. **Change Cipher Spec:** ทั้งสองฝ่ายยืนยันการเปลี่ยนไปใช้พารามิเตอร์ที่เข้ารหัส
6. **Finished:** ทั้งสองฝ่ายส่งข้อความยืนยันที่เข้ารหัส

**TLS 1.3 Handshake (1-RTT):**
1. **ClientHello:** Client ส่ง TLS Version, Key Share (สาธารณะ), Cipher Suites ที่รองรับ
2. **ServerHello:** Server เลือกพารามิเตอร์, ส่ง Key Share (สาธารณะ), Certificate, และ Finished พร้อมกัน
3. **Client Finished:** Client ยืนยันและเริ่มการสื่อสารที่เข้ารหัสได้ทันที

| คุณสมบัติ | TLS 1.2 | TLS 1.3 |
|-----------|---------|---------|
| RTT ในการ Handshake ครั้งแรก | 2-RTT | 1-RTT |
| RTT ในการต่อเชื่อมใหม่ (Session Resumption) | 1-RTT | 0-RTT (PSK) |
| ความสามารถในการต่อรอง Cipher Suite | ยืดหยุ่นสูง (เสี่ยงต่อ Downgrade Attack) | จำกัดเฉพาะชุดที่ปลอดภัยเท่านั้น |
| Forward Secrecy | ไม่บังคับ (สามารถใช้ RSA Key Exchange ที่ไม่มี PFS) | บังคับ (ต้องใช้ ECDHE หรือ DHE) |
| รายการ Cipher Suite | >300 ชุด | 5 ชุด |
| Encrypted Certificate | ไม่มี | มี (Encrypted Client Hello - ECH ใน TLS 1.3 มาตรฐาน) |
| PSK (Pre-Shared Key) | ต้องกำหนดค่าเอง | รองรับในตัวสำหรับ Session Resumption และ 0-RTT |
| ความซับซ้อนของโปรโตคอล | สูง (มีหลายขั้นตอนที่อาจมีช่องโหว่) | ต่ำ (ลดขั้นตอนที่ไม่จำเป็นออก) |

**Forward Secrecy (PFS - Perfect Forward Secrecy):**
TLS 1.3 บังคับใช้ Forward Secrecy โดยในทุกครั้งที่มีการสร้างการเชื่อมต่อ คีย์ชั่วคราว (Ephemeral Key) สำหรับการเข้ารหัสข้อมูลจะถูกสร้างขึ้นใหม่ผ่าน ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) หรือ DHE ทำให้แม้ว่าคีย์ส่วนตัว (Private Key) ของเซิร์ฟเวอร์จะถูกเปิดเผยในภายหลัง ผู้โจมตีก็ไม่สามารถถอดรหัสการสื่อสารที่ถูกบันทึกไว้ก่อนหน้าได้

**PSK (Pre-Shared Key) และ 0-RTT ใน TLS 1.3:**
TLS 1.3 รองรับการต่อเชื่อมใหม่โดยใช้ PSK ที่ได้มาจากการเชื่อมต่อก่อนหน้า ทำให้สามารถเริ่มส่งข้อมูลได้ทันทีใน 0-RTT โดยไม่ต้องรอให้ Handshake เสร็จสมบูรณ์ อย่างไรก็ตาม 0-RTT มีความเสี่ยงด้านความปลอดภัยเรื่อง Replay Attack ควรใช้เมื่อการโจมตีแบบ Replay ไม่มีผลกระทบร้ายแรงเท่านั้น

#### 12.2.2 Cipher Suites ที่ปลอดภัย

Cipher Suite คือชุดของอัลกอริทึมที่ใช้ใน TLS ประกอบด้วย 4 ส่วน:
1. **Key Exchange Algorithm:** ใช้สำหรับแลกเปลี่ยนคีย์ระหว่าง Client และ Server
2. **Authentication Algorithm:** ใช้ยืนยันตัวตนของเซิร์ฟเวอร์ (หรือ Client ใน mTLS)
3. **Bulk Encryption Cipher:** ใช้เข้ารหัสข้อมูลในการสื่อสาร
4. **Message Authentication Code (MAC):** ใช้ตรวจสอบความสมบูรณ์ของข้อมูล

**Cipher Suites ที่แนะนำสำหรับนักพัฒนาในปี 2025-2026:**

| Cipher Suite | ความปลอดภัย | ลักษณะเด่น |
|-------------|------------|-----------|
| `TLS_AES_256_GCM_SHA384` | สูงสุด | AES-256 ในโหมด GCM (Galois/Counter Mode), ใช้ในระดับความปลอดภัยสูง |
| `TLS_AES_128_GCM_SHA256` | สูง | AES-128 ในโหมด GCM, ประสิทธิภาพดี, แนะนำสำหรับการใช้งานทั่วไป |
| `TLS_CHACHA20_POLY1305_SHA256` | สูง | ChaCha20 + Poly1305, เหมาะสำหรับอุปกรณ์ที่ไม่มีฮาร์ดแวร์เร่ง AES (เช่น มือถือรุ่นเก่า) |

**Cipher Suites ที่ถูกลบและสาเหตุ:**

| Cipher Suite เดิม | สาเหตุที่ถูกลบ | ปีที่ถูกยกเลิก |
|------------------|---------------|---------------|
| RC4 | ความอ่อนแอของการเข้ารหัส — มี Bias ทางสถิติที่ทำให้สามารถถอดรหัสได้ | RFC 7465 (2015) |
| 3DES (Triple DES) | ขนาด Block 64-bit ที่ทำให้เกิด Birthday Attack ที่ 32GB ของข้อมูลที่เข้ารหัส | RFC 8925 (2020) |
| CBC Mode Ciphers | ช่องโหว่ Padding Oracle Attack (เช่น POODLE, Lucky13) | RFC 9325 (2022) |
| RSA Key Exchange | ไม่มี Forward Secrecy — หาก Private Key รั่วไหล การสื่อสารทั้งหมดในอดีตสามารถถอดรหัสได้ | RFC 9325 (2022) |
| EXPORT Grade Ciphers | ขนาดคีย์ที่อ่อนแอ (512-bit) จากการจำกัดการส่งออกของสหรัฐอเมริกา | RFC 7465 (2015) |

#### 12.2.3 การติดตั้งและจัดการ Certificate

**Let's Encrypt และ ACME Protocol:**
Let's Encrypt เป็น Certificate Authority (CA) ที่ให้บริการ Certificate ฟรีสำหรับ HTTPS โดยอัตโนมัติผ่าน ACME (Automatic Certificate Management Environment) Protocol ตามมาตรฐาน RFC 8555

ACME Protocol รองรับการยืนยันการควบคุมโดเมน (Domain Validation) หลายวิธี:
- **HTTP-01 Challenge:** สร้างไฟล์ที่มี Token ที่ URL `http://<domain>/.well-known/acme-challenge/<token>`
- **DNS-01 Challenge:** เพิ่ม TXT Record ใน DNS สำหรับ `<challenge>.example.com`
- **TLS-ALPN-01 Challenge:** ยืนยันผ่าน TLS Handshake บนพอร์ต 443

**ตัวอย่างการใช้ Certbot (ACME Client) ขอ Certificate:**

```bash
# ขอ Certificate สำหรับ Nginx ด้วย Certbot
sudo certbot --nginx -d example.com -d www.example.com

# ขอ Certificate โดยใช้ DNS Challenge (สำหรับ Wildcard Certificate)
sudo certbot certonly --manual --preferred-challenges dns -d "*.example.com"

# ต่ออายุ Certificate โดยอัตโนมัติ
sudo certbot renew --dry-run
```

**Certificate Transparency (CT):**
Certificate Transparency เป็นระบบบันทึก Certificate ทั้งหมดที่ออกโดย CA ใน Log Server ที่ตรวจสอบได้โดยสาธารณะ ตามมาตรฐาน RFC 6962 ตั้งแต่มกราคม 2023 เบราว์เซอร์ Chrome และ Safari กำหนดให้ Certificate ที่ออกใหม่ต้องมี Signed Certificate Timestamp (SCT) จาก CT Logs อย่างน้อย 2-3 รายการ มิฉะนั้นจะไม่ถูกเชื่อถือ

ประโยชน์ของ Certificate Transparency สำหรับนักพัฒนา:
- ตรวจจับ Certificate ที่ออกโดยไม่ได้รับอนุญาต (Mis-issuance)
- ตรวจสอบว่ามี Certificate สำหรับโดเมนของตนเองที่ออกโดยไม่ได้รับอนุญาตหรือไม่ (ผ่าน CT Search Tools)
- เพิ่มความโปร่งใสของห่วงโซ่ Certificate (PKI Transparency)

#### 12.2.4 mTLS สำหรับ Service-to-Service Authentication

mTLS (Mutual TLS) เป็นรูปแบบของ TLS ที่ทั้งสองฝ่าย (Client และ Server) ต้องแสดง Certificate เพื่อยืนยันตัวตนซึ่งกันและกัน ต่างจาก TLS ทั่วไปที่เฉพาะ Server เท่านั้นที่ต้องแสดง Certificate

**การทำงานของ mTLS:**
1. Client เชื่อมต่อไปยัง Server ผ่าน TLS
2. Server ส่ง Certificate ของตนเอง
3. Client ตรวจสอบ Certificate ของ Server
4. **Server ขอ Certificate จาก Client (CertificateRequest)**
5. Client ส่ง Certificate ของตนเอง
6. Server ตรวจสอบ Certificate ของ Client
7. การสื่อสารที่เข้ารหัสจึงจะเริ่มขึ้นได้

**การประยุกต์ใช้ mTLS ใน Service Mesh:**
ในสถาปัตยกรรม Service Mesh (เช่น Istio, Linkerd, Consul Connect) mTLS ถูกใช้เป็นกลไกหลักสำหรับการยืนยันตัวตนระหว่าง Service โดยอัตโนมัติ:

| ส่วนประกอบ | หน้าที่ |
|------------|--------|
| **Sidecar Proxy (Envoy)** | จัดการ mTLS แทน Application โดยไม่ต้องแก้ไขโค้ด |
| **Certificate Authority (CA)** | ออก Certificate สำหรับ Service แต่ละตัว |
| **SPIFFE Identity** | ระบุตัวตนของ Service ในรูปแบบ URI เช่น `spiffe://cluster.local/ns/default/sa/api-service` |
| **Certificate Rotation** | ต่ออายุ Certificate อัตโนมัติทุก 24 ชั่วโมง |

**ข้อควรระวังสำหรับนักพัฒนาเมื่อใช้ mTLS:**
- **โครงสร้างพื้นฐาน Certificate:** ต้องมีระบบจัดการ Certificate (Certificate Management System) ที่สามารถออก, ต่ออายุ และเพิกถอน Certificate ได้โดยอัตโนมัติ
- **การหมุนเวียน Certificate (Certificate Rotation):** กำหนดอายุของ Certificate ให้สั้น (เช่น 24 ชั่วโมง) เพื่อลดผลกระทบหาก Certificate รั่วไหล
- **การทดสอบ:** ต้องทดสอบกรณี Certificate หมดอายุ, Certificate ถูกเพิกถอน, และการเชื่อมต่อที่ไม่มี Certificate
- **Monitoring:** ตรวจสอบอัตราความสำเร็จและความล้มเหลวของการเชื่อมต่อ mTLS

#### 12.2.5 การตรวจสอบ TLS Configuration

**Qualys SSL Labs (ssllabs.com):**
SSL Labs Server Test เป็นเครื่องมือออนไลน์ฟรีที่วิเคราะห์การตั้งค่า TLS ของเว็บเซิร์ฟเวอร์ ให้คะแนนตั้งแต่ A+ ถึง F ครอบคลุม:
- Certificate Validation และ Certificate Chain
- TLS Protocol Support (1.2, 1.3)
- Cipher Suite ที่รองรับ
- ความปลอดภัยของ Key Exchange
- การโจมตีแบบ BEAST, POODLE, Heartbleed
- HSTS และ Security Headers

**testssl.sh:**
testssl.sh เป็นเครื่องมือบรรทัดคำสั่งสำหรับตรวจสอบ TLS Configuration ของเซิร์ฟเวอร์ใด ๆ ผ่าน Command Line เหมาะสำหรับการตรวจสอบภายในที่ไม่เปิดสู่อินเทอร์เน็ต:

```bash
# ตรวจสอบ TLS Configuration พื้นฐาน
testssl.sh https://example.com

# ตรวจสอบช่องโหว่เฉพาะ
testssl.sh --heartbleed example.com:443
testssl.sh --poodle example.com:443

# ตรวจสอบ Cipher Suite ที่รองรับ
testssl.sh --cipher-per-proto example.com:443

# สรุปผลลัพธ์แบบสั้น
testssl.sh --severity MEDIUM example.com:443

# ตรวจสอบ Certificate
testssl.sh --certificate example.com:443
```

**CVE-2025-68121 (เผยแพร่: 5 กุมภาพันธ์ 2026):**
ช่องโหว่ในไลบรารี `crypto/tls` ของภาษา Go คะแนน CVSS 3.1 อยู่ที่ 10.0 (CRITICAL) เกิดจากข้อบกพร่องในการตรวจสอบ Certificate ในระหว่าง Session Resumption เมื่อ `Config.Clone` ถูกเรียกใช้และ `ClientCAs` หรือ `RootCAs` ถูกเปลี่ยนแปลงภายหลังจากการเชื่อมต่อครั้งแรก การเชื่อมต่อ Session Resumption อาจสำเร็จทั้งที่ควรจะล้มเหลว ทำให้ Client ติดต่อกับเซิร์ฟเวอร์ที่ไม่ควรเชื่อถือได้ หรือ Server ติดต่อกลับกับ Client ที่ไม่ควรเชื่อถือได้ แก้ไขใน Go เวอร์ชัน 1.24.13, 1.25.7 และ 1.26.0-rc.3

**CVE-2026-2673 (เผยแพร่: 13 มีนาคม 2026):**
ช่องโหว่ใน OpenSSL เวอร์ชัน 3.5.x และ 3.6.x ที่ทำให้ TLS 1.3 Server อาจเลือกกลุ่ม Key Exchange ที่ไม่ถูกต้องเมื่อกำหนดค่ากลุ่มด้วยคีย์เวิร์ด 'DEFAULT' ข้อบกพร่องนี้ทำให้โครงสร้าง 'tuple' ของรายการกลุ่มเสียหาย ส่งผลให้ Server ไม่ส่ง Hello Retry Request (HRR) แม้ว่าจะมีกลุ่มที่ต้องการมากกว่า (เช่น กลุ่ม Post-Quantum แบบ Hybrid เช่น X25519MLKEM768) ที่รองรับร่วมกัน แก้ไขใน OpenSSL 3.5.6 และ 3.6.2 ไม่มีผลต่อ OpenSSL เวอร์ชัน 3.4, 3.3, 3.0 และ 1.x

**CVE-2026-26994 (เผยแพร่: 19 กุมภาพันธ์ 2026):**
ช่องโหว่ในไลบรารี uTLS (Go fork ที่ดัดแปลง ClientHello เพื่อต้านทาน Fingerprinting) เวอร์ชัน 1.6.7 และต่ำกว่า uTLS ไม่ได้ตรวจสอบกลไก Downgrade Protection ที่ระบุใน RFC 8446 Section 4.1.3 สำหรับ TLS 1.3 ผู้โจมตีในเครือข่ายสามารถ Downgrade การเชื่อมต่อ TLS 1.3 ไปยังเวอร์ชันที่ต่ำกว่า (เช่น TLS 1.2) โดยการแก้ไข ClientHello เพื่อลบส่วนขยาย SupportedVersions และ ServerHello จะตอบกลับด้วย TLS 1.2 พร้อม Downgrade Canary แต่ uTLS ไม่ได้ตรวจสอบ Canary นี้ ทำให้ Client ยอมรับการเชื่อมต่อที่ถูกลดระดับโดยไม่ตรวจพบ แก้ไขในเวอร์ชัน 1.7.0 CWE-693 (Protection Mechanism Failure)

### 12.3 Firewall และ Network Access Control

Firewall เป็นระบบรักษาความปลอดภัยทางเครือข่ายที่ตรวจสอบและควบคุมการรับส่งข้อมูลตามกฎที่กำหนดไว้ สำหรับนักพัฒนา การเข้าใจหลักการของ Firewall ช่วยให้สามารถออกแบบระบบที่ทำงานร่วมกับข้อจำกัดด้านเครือข่ายได้อย่างถูกต้อง

#### 12.3.1 Network Firewall vs Web Application Firewall (WAF)

| คุณสมบัติ | Network Firewall (L3/L4) | Web Application Firewall (L7) |
|-----------|-------------------------|-------------------------------|
| ชั้น OSI | Layer 3 (Network) และ Layer 4 (Transport) | Layer 7 (Application) |
| การตรวจสอบ | Source/Dest IP, Port, Protocol, TCP Flags | HTTP Headers, Request Body, URL Parameters |
| ความเร็ว | สูงมาก (ตรวจสอบเฉพาะหัวข้อแพ็กเกต) | ปานกลาง (ต้องถอดรหัส HTTP และวิเคราะห์เนื้อหา) |
| การป้องกัน SQL Injection | ไม่สามารถ | สามารถ (ด้วย Rule Set เช่น OWASP CRS) |
| การป้องกัน XSS | ไม่สามารถ | สามารถ |
| การป้องกัน DDoS (Layer 3/4) | สามารถ | จำกัด |
| การป้องกันการโจมตีที่ Protocol Level | สามารถ | ไม่สามารถ |
| บทบาท | ควบคุมการเข้าถึงเครือข่าย | ปกป้อง Web Application จาก Layer 7 Attacks |

**บทบาทที่แตกต่างแต่เสริมกัน:** Network Firewall และ WAF ทำงานในระดับที่แตกต่างกัน Network Firewall ทำหน้าที่เป็นด่านแรกที่กรองการรับส่งข้อมูลตาม IP และ Port ในขณะที่ WAF ทำหน้าที่เป็นด่านที่สองที่วิเคราะห์เนื้อหาของ HTTP/S Request อย่างละเอียดเพื่อตรวจจับการโจมตีที่ซับซ้อนใน Layer 7

#### 12.3.2 Stateful vs Stateless Firewall

**Stateless Firewall (Packet Filter):**
- ตรวจสอบแต่ละแพ็กเกตอย่างอิสระ โดยไม่พิจารณาสถานะการเชื่อมต่อ
- เร็วกว่าแต่มีความสามารถในการตรวจจับน้อยกว่า
- ต้องกำหนดกฎทั้งขาเข้าและขาออกแยกกัน
- ไม่สามารถตรวจจับแพ็กเกตที่ผิดลำดับหรือแพ็กเกตปลอมในบริบทของการเชื่อมต่อได้

**Stateful Firewall:**
- จดจำสถานะของการเชื่อมต่อที่กำลังทำงานอยู่
- ตัดสินใจอนุญาตหรือปฏิเสธแพ็กเกตโดยพิจารณาจากบริบทของการเชื่อมต่อ
- สามารถอนุญาตแพ็กเกตตอบกลับโดยอัตโนมัติเมื่อมีการเชื่อมต่อที่ถูกต้องจากภายใน
- ใช้ทรัพยากรมากกว่า Stateless Firewall

**ข้อแนะนำ:** Stateful Firewall เป็นมาตรฐานปัจจุบันสำหรับการใช้งานทั่วไป ในขณะที่ Stateless Firewall เหมาะสำหรับสภาพแวดล้อมที่ต้องการประสิทธิภาพสูงมาก เช่น Data Center หรือ Router ขอบเครือข่าย

#### 12.3.3 Security Group / Network ACL ในระบบคลาวด์

ในระบบคลาวด์ (AWS, Azure, GCP) การควบคุมการเข้าถึงเครือข่ายมี 2 ระดับหลัก:

| คุณสมบัติ | Security Group | Network ACL |
|-----------|---------------|-------------|
| ระดับ | Instance-level (Stateful) | Subnet-level (Stateless) |
| Stateful/Stateless | **Stateful** — อนุญาตตอบกลับอัตโนมัติ | **Stateless** — ต้องกำหนดกฎขากลับเอง |
| กฎ | อนุญาตเท่านั้น (Allow rules only) | อนุญาตและปฏิเสธ (Allow + Deny rules) |
| การประเมิน | กฎทั้งหมดถูกประเมินก่อนอนุญาต | กฎถูกประเมินตามลำดับหมายเลข |
| สนับสนุน | AWS, Azure (NSG), GCP (Firewall Rules) | AWS, Azure (ไม่เทียบเท่าโดยตรง), GCP (ไม่เทียบเท่า) |

**แนวปฏิบัติที่ดีสำหรับ Security Group ในระบบคลาวด์:**

```bash
# AWS CLI — สร้าง Security Group สำหรับ Web Server
aws ec2 create-security-group \
    --group-name web-server-sg \
    --description "Security group for web servers" \
    --vpc-id vpc-xxxxx

# เปิดพอร์ต 443 (HTTPS) จากอินเทอร์เน็ต
aws ec2 authorize-security-group-ingress \
    --group-id sg-xxxxx \
    --protocol tcp \
    --port 443 \
    --cidr 0.0.0.0/0

# เปิดพอร์ต 22 (SSH) จาก IP ของผู้ดูแลเท่านั้น
aws ec2 authorize-security-group-ingress \
    --group-id sg-xxxxx \
    --protocol tcp \
    --port 22 \
    --cidr 203.0.113.0/24
```

**หลักการ Default Deny:** ทั้ง Security Group และ Network ACL ควรเริ่มต้นด้วยการปฏิเสธทุกการเชื่อมต่อ (Default Deny) แล้วค่อยเปิดเฉพาะพอร์ตและแหล่งที่มาที่จำเป็นเท่านั้น

#### 12.3.4 Zero Trust Network Access (ZTNA) และ SASE

**Zero Trust Network Access (ZTNA):**
ZTNA เป็นแนวทางด้านความปลอดภัยที่ยึดหลัก "Never Trust, Always Verify" หมายถึงไม่เชื่อถืออุปกรณ์หรือผู้ใช้ใด ๆ โดยอัตโนมัติแม้ว่าจะอยู่ในเครือข่ายภายในก็ตาม การเข้าถึงทรัพยากรทุกครั้งต้องได้รับการตรวจสอบสิทธิ์และการอนุญาต

| หลักการ | คำอธิบาย |
|---------|----------|
| **Never Trust, Always Verify** | ไม่เชื่อถืออุปกรณ์หรือผู้ใช้ใด ๆ โดยปริยาย ตรวจสอบทุกครั้งที่เข้าถึง |
| **Least Privilege Access** | ให้สิทธิ์เฉพาะที่จำเป็นต่อการทำงานเท่านั้น |
| **Microsegmentation** | แบ่งเครือข่ายออกเป็นส่วนย่อยเพื่อจำกัดการเคลื่อนที่ในแนวราบ (Lateral Movement) |
| **Continuous Verification** | ตรวจสอบพฤติกรรมและความเสี่ยงอย่างต่อเนื่อง ไม่ใช่แค่ตอนเริ่มต้นเชื่อมต่อ |
| **Identity-Centric** | ควบคุมการเข้าถึงตาม Identity ไม่ใช่ตาม IP Address |

**การเปรียบเทียบ ZTNA กับ VPN แบบดั้งเดิม:**

| คุณสมบัติ | VPN แบบดั้งเดิม | ZTNA |
|-----------|---------------|------|
| การเข้าถึง | เข้าถึงเครือข่ายทั้งหมด (Network-level Access) | เข้าถึงเฉพาะ Application ที่ได้รับอนุญาต (Application-level Access) |
| ความเสี่ยง | หากเชื่อมต่อ VPN ได้ เข้าถึงทรัพยากรภายในได้ทั้งหมด | เข้าถึงได้เฉพาะ Application ที่ได้รับอนุญาตเท่านั้น |
| ตำแหน่ง | ต้องติดตั้ง Client Software | ใช้ Browser เป็นหลัก (Clientless) หรือ Agent เฉพาะ |
| ประสิทธิภาพ | การรับส่งข้อมูลทั้งหมดผ่าน VPN Gateway | Traffic ตรงไปยัง Application (Split Tunneling) |
| การปรับขนาด | ต้องขยาย VPN Gateway | ปรับขนาดได้ตามความต้องการ (Cloud-native) |

**SASE (Secure Access Service Edge):**
SASE เป็นสถาปัตยกรรมที่รวม Network Security Functions (SWG, CASB, ZTNA) และ Wide Area Network (WAN) Capabilities (SD-WAN) เป็นบริการบนคลาวด์เดียว ตัวอย่างผู้ให้บริการ: Cloudflare One, Zscaler, Netskope, Palo Alto Prisma Access

#### 12.3.5 การเปิด Port ที่จำเป็นเท่านั้น และ CAA DNS Records

**หลักการเปิด Port ที่จำเป็น:**
- เปิดเฉพาะพอร์ตที่บริการต้องการใช้งานจริง (Default Deny)
- จำกัดแหล่งที่มาที่สามารถเข้าถึงได้ (Source IP Restriction)
- ใช้ Microsegmentation เพื่อจำกัดการสื่อสารระหว่าง Service

| บริการ | พอร์ต (TCP) | หมายเหตุ |
|--------|-------------|---------|
| HTTPS | 443 | เปิดจากอินเทอร์เน็ต |
| HTTP | 80 | เปิดจากอินเทอร์เน็ต (สำหรับ Redirect ไป HTTPS เท่านั้น) |
| SSH | 22 | เปิดเฉพาะจาก Bastion Host หรือ VPN |
| Database (MySQL/PostgreSQL) | 3306/5432 | เปิดเฉพาะจาก Application Server เท่านั้น |
| Redis | 6379 | เปิดเฉพาะจาก Application Server ไม่เปิดสู่อินเทอร์เน็ต |
| Kubernetes API | 6443 | เปิดเฉพาะจากผู้ดูแลระบบ |

**CAA (Certification Authority Authorization) DNS Records:**
CAA DNS Record (RFC 6844) เป็น Resource Record ใน DNS ที่ระบุว่า Certificate Authority (CA) รายใดบ้างที่ได้รับอนุญาตให้ออก Certificate สำหรับโดเมนนั้น การตั้งค่า CAA Record ช่วยป้องกันการออก Certificate โดยไม่ได้รับอนุญาตจาก CA ที่ไม่พึงประสงค์

```bash
# CAA Record — อนุญาตเฉพาะ Let's Encrypt เท่านั้น
example.com.  CAA  0 issue "letsencrypt.org"

# CAA Record — อนุญาต CA หลายราย
example.com.  CAA  0 issue "letsencrypt.org"
example.com.  CAA  0 issue "digicert.com"

# CAA Record — ระบุรายงานการละเมิด
example.com.  CAA  0 iodef "mailto:security@example.com"
```

### 12.4 DDoS Protection

Distributed Denial of Service (DDoS) Attack เป็นการโจมตีที่ทำให้ระบบเป้าหมายไม่สามารถให้บริการได้ตามปกติ โดยใช้ทรัพยากรจากอุปกรณ์จำนวนมากที่ถูกควบคุม (Botnet)

#### 12.4.1 ประเภทของ DDoS Attack

| ประเภท | ชั้น OSI | เป้าหมาย | ตัวอย่าง |
|--------|---------|---------|----------|
| **Volume-based (Volumetric)** | L3/L4 | ทำให้ Bandwidth อิ่มตัว (Bandwidth Saturation) | UDP Flood, ICMP Flood, DNS Amplification, NTP Amplification |
| **Protocol/State-exhaustion** | L3/L4 | ทำให้ทรัพยากรของเซิร์ฟเวอร์หรืออุปกรณ์เครือข่ายหมด | SYN Flood, ACK Flood, Ping of Death, HTTP/2 Rapid Reset |
| **Application Layer** | L7 | ทำให้ทรัพยากรของ Web Application หมด | HTTP Flood, Slowloris, POST Flood, WordPress XML-RPC Flood |

**Volume-based Attacks:**
- **DNS Amplification:** ผู้โจมตีส่งคำถาม DNS แบบเปิด (Open Resolver) โดยปลอม Source IP เป็น IP ของเหยื่อ คำตอบ DNS อาจใหญ่กว่าคำถามถึง 50-100 เท่า
- **UDP Flood:** ส่ง UDP Packet จำนวนมากไปยังพอร์ตสุ่มของเหยื่อ ทำให้เซิร์ฟเวอร์ต้องตอบกลับด้วย ICMP Destination Unreachable
- **Carpet Bombing:** โจมตีหลาย IP Address พร้อมกันเพื่อหลบเลี่ยงการตรวจจับจากระบบป้องกันที่ดูเฉพาะราย IP

**Protocol/State-exhaustion Attacks:**
- **SYN Flood:** ส่ง TCP SYN Packet จำนวนมากโดยไม่ตอบกลับ ACK ทำให้ Backlog Queue เต็ม
- **HTTP/2 Rapid Reset (CVE-2023-44487):** เป็นการโจมตีที่ใช้คุณสมบัติ Multiplexed Streams ของ HTTP/2 โดย Client ส่ง Request แล้วยกเลิกทันทีด้วย RST_STREAM Frame ทำให้เซิร์ฟเวอร์ต้องทำงานหนักในการสร้างและทำลาย Stream โดยไม่กระทบกับขีดจำกัดของ Active Streams ต่อหนึ่งเชื่อมต่อ คะแนน CVSS 3.1 อยู่ที่ 7.5 (High) ถูกใช้ในการโจมตีจริงระหว่างเดือนสิงหาคมถึงตุลาคม 2023 และส่งผลกระทบต่อ Web Server ทุกรายที่ใช้งาน HTTP/2

**Application Layer Attacks:**
- **HTTP Flood:** ส่ง HTTP GET/POST Request จำนวนมากที่ดูเหมือนการเข้าชมปกติ
- **Slowloris:** ส่ง HTTP Request แบบช้า ๆ โดยการส่งข้อมูลบางส่วนแล้วหยุด ทำให้เซิร์ฟเวอร์ต้องรอการเชื่อมต่อนาน
- **WordPress XML-RPC Flood:** ใช้ฟังก์ชัน `system.multicall` ของ WordPress XML-RPC เพื่อส่ง Request จำนวนมากผ่านครั้งเดียว

#### 12.4.2 สถิติ DDoS ปี 2025-2026

ตามรายงาน Cloudflare Q4 2025 DDoS Threat Report (กุมภาพันธ์ 2026):

| สถิติ | ค่า |
|------|-----|
| จำนวนการโจมตี DDoS ทั้งปี 2025 | 47.1 ล้านครั้ง (เพิ่มขึ้น 121% จากปี 2024) |
| การเพิ่มขึ้นระหว่างปี 2023-2025 | 236% |
| ค่าเฉลี่ยการโจมตีต่อชั่วโมง | 5,376 ครั้ง (3,925 Network-layer + 1,451 HTTP-layer) |
| Network-layer DDoS ปี 2025 | 34.4 ล้านครั้ง (เพิ่มขึ้น 3 เท่าจาก 11.4 ล้านครั้งในปี 2024) |
| Hyper-volumetric Attack (Q4 2025) | เพิ่มขึ้น 40% จาก Q3 2025 |

#### 12.4.3 Aisuru-Kimwolf Botnet

Aisuru-Kimwolf Botnet เป็นเครือข่ายอุปกรณ์ที่ติดมัลแวร์ที่มีขนาดใหญ่ที่สุดเครือข่ายหนึ่งในประวัติศาสตร์ DDoS:

**ข้อมูลสำคัญ:**
- **ขนาด:** ประมาณ 1-4 ล้านอุปกรณ์ที่ติดเชื้อ โดย Kimwolf (ส่วนของ Android TV Boxes) มีประมาณ 2 ล้านอุปกรณ์
- **กำลังการโจมตี:** ทำลายสถิติ DDoS ถึง 31.4 Tbps, 14.1 Bpps และ HTTP DDoS มากกว่า 200 Mpps
- **การแพร่กระจาย:** Kimwolf เติบโตจาก 800,000 เป็น 2 ล้านอุปกรณ์ภายในไม่กี่สัปดาห์ในเดือนตุลาคม 2025
- **ประเทศที่ติดเชื้อมากที่สุด:** เวียดนาม, บราซิล, อินเดีย, ซาอุดีอาระเบีย
- **การขัดขวางโดย Lumen Black Lotus Labs:** บล็อกเซิร์ฟเวอร์ C2 มากกว่า 550 เครื่องภายใน 4 เดือน
- **ชื่อแคมเปญ:** "The Night Before Christmas" — โจมตีในช่วงไตรมาสที่ 4 ของปี 2025
- **เทคนิค:** Kimwolf ใช้ DNS over TLS เพื่อซ่อนการสื่อสาร, ใช้ Elliptic Curve Digital Signature สำหรับยืนยันคำสั่ง C2 และใช้ EtherHiding บน Blockchain Domain

#### 12.4.4 CDN Integration และ Anycast Routing

**CDN (Content Delivery Network) และ Anycast Routing:**
CDN ช่วยป้องกัน DDoS โดยกระจายปริมาณการรับส่งข้อมูลไปยังเซิร์ฟเวอร์หลายแห่งทั่วโลกผ่าน Anycast Routing ซึ่งทำให้ IP Address เดียวสามารถประกาศจากหลายตำแหน่ง และการรับส่งข้อมูลจะถูกส่งไปยังตำแหน่งที่ใกล้ที่สุด

| ผู้ให้บริการ CDN | จุดเด่นด้าน DDoS Protection |
|----------------|---------------------------|
| **Cloudflare** | ป้องกัน DDoS อัตโนมัติ, ขนาดเครือข่าย 330+ Tbps, Magic Transit สำหรับ Network-layer DDoS |
| **Akamai** | Prolexic Routed สำหรับ Network-layer และ App & API Protector สำหรับ Layer 7 |
| **AWS CloudFront** + AWS Shield | AWS Shield Standard ฟรี, AWS Shield Advanced ($3,000/เดือน) สำหรับ DDoS Mitigation ขั้นสูง |
| **Google Cloud Armor** | ปรับใช้กับ Cloud Load Balancing, รองรับ WAF และ DDoS Protection |

**ข้อแนะนำสำหรับนักพัฒนา:** การวาง CDN ไว้หน้า Web Application ช่วยดูดซับ DDoS ใน Layer 3/4 และ Layer 7 ก่อนที่การรับส่งข้อมูลจะไปถึงเซิร์ฟเวอร์จริง

#### 12.4.5 Rate Limiting และ Throttling

Rate Limiting เป็นกลไกที่จำกัดจำนวน Request ที่ Client หรือ IP Address สามารถส่งได้ภายในช่วงเวลาที่กำหนด

**ตัวอย่างการใช้งาน Rate Limiting ใน Web Application:**

```python
# Python Flask — Rate Limiting ด้วย Flask-Limiter
from flask import Flask
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

app = Flask(__name__)
limiter = Limiter(
    get_remote_address,
    app=app,
    default_limits=["200 per day", "50 per hour"]
)

@app.route("/api/login")
@limiter.limit("5 per minute")
def login():
    # API สำหรับเข้าสู่ระบบ
    return {"status": "login endpoint"}
```

```yaml
# NGINX — Rate Limiting
http {
    # กำหนด Zone สำหรับ Rate Limiting (10MB = 160,000 sessions)
    limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;
    
    server {
        location /api/login {
            limit_req zone=login burst=10 nodelay;
            proxy_pass http://backend;
        }
    }
}
```

**แนวทางปฏิบัติสำหรับ Rate Limiting:**
- ใช้ Rate Limiting ที่เข้มงวดกว่าสำหรับ API Endpoint ที่สำคัญ (Login, Register, Password Reset)
- แจ้ง Client ด้วย HTTP Header `Retry-After` และ `X-RateLimit-*` Headers
- ตอบกลับด้วย HTTP Status Code 429 (Too Many Requests) เมื่อถึงขีดจำกัด
- พิจารณาการใช้ Token Bucket หรือ Leaky Bucket Algorithm

#### 12.4.6 Auto Scaling และ Load Balancing

**Auto Scaling:**
Auto Scaling เพิ่มหรือลดจำนวนเซิร์ฟเวอร์ตามปริมาณการรับส่งข้อมูล ช่วยรับมือกับ DDoS ที่มีขนาดไม่เกินระดับที่สามารถจัดการได้ (ไม่สามารถรับมือกับ Volumetric DDoS ขนาดใหญ่ที่ทำให้โครงสร้างเครือข่ายอิ่มตัว)

**ข้อจำกัดของ Auto Scaling ในการรับมือ DDoS:**
- ไม่สามารถช่วยได้เมื่อ Bandwidth ของเครือข่ายถูกทำให้อิ่มตัว
- การเปิดเซิร์ฟเวอร์เพิ่มในช่วง DDoS อาจเพิ่มค่าใช้จ่ายโดยไม่จำเป็น
- ควรตั้งค่า Scaling Threshold ที่สูงกว่าปกติเพื่อป้องกันการเปิดปิดเซิร์ฟเวอร์บ่อยเกินไป (Thundering Herd)

**Load Balancing:**
Load Balancer กระจายปริมาณการรับส่งข้อมูลไปยังเซิร์ฟเวอร์หลายตัว มี 3 ประเภทหลัก:

| ประเภท | ชั้น | ตัวอย่าง | DDoS Protection |
|--------|------|---------|-----------------|
| Hardware Load Balancer | L4-L7 | F5 BIG-IP, Citrix ADC | มีความสามารถ DDoS Protection ในตัว |
| Cloud Load Balancer | L4-L7 | AWS ALB/NLB, GCP HTTP LB, Azure LB | บูรณาการกับ Cloud DDoS Protection |
| Software Load Balancer | L4-L7 | NGINX, HAProxy, Envoy | ต้องกำหนดค่า Rate Limiting และ Connection Limiting ด้วยตนเอง |

### 12.5 Network Monitoring และ Threat Detection

การตรวจสอบเครือข่าย (Network Monitoring) ช่วยให้นักพัฒนาและทีมปฏิบัติการสามารถตรวจจับพฤติกรรมผิดปกติและภัยคุกคามที่เกิดขึ้นบนเครือข่ายได้

#### 12.5.1 Intrusion Detection/Prevention Systems (IDS/IPS)

IDS (Intrusion Detection System) ตรวจจับการบุกรุกและแจ้งเตือน ในขณะที่ IPS (Intrusion Prevention System) สามารถบล็อกการโจมตีได้โดยอัตโนมัติ

**Suricata vs Snort:**

| คุณสมบัติ | Suricata | Snort |
|-----------|----------|-------|
| เทคโนโลยีหลัก | Multi-threaded (ใช้หลาย CPU Core) | Single-threaded |
| ประสิทธิภาพ | รองรับสูงถึง 50+ Gbps | รองรับสูงถึง 10+ Gbps |
| ความสามารถเพิ่มเติม | File Extraction, TLS Inspection, HTTP Logging | IPS Mode (Inline) |
| GPU Acceleration | รองรับ CUDA | ไม่รองรับ |
| อัปเดตกฎ | ET (Emerging Threats), ET Pro | VRT (Talos), ET |
| การตั้งโปรโตคอล | HTTP, TLS, DNS, SMTP พร้อม Logger ในตัว | ต้องใช้ Plugin เพิ่ม |
| **ข้อแนะนำ** | **แนะนำสำหรับการติดตั้งใหม่** | เหมาะสำหรับระบบที่ใช้งานอยู่แล้วและอัปเกรด |

**การติดตั้ง Suricata:**

```bash
# ติดตั้ง Suricata บน Ubuntu
sudo apt-get install suricata

# กำหนดค่า Network Interface ใน /etc/suricata/suricata.yaml
# af-packet:
#   - interface: eth0

# ดาวน์โหลด Emerging Threats Rules
sudo suricata-update

# รัน Suricata ในโหมด IDS
sudo suricata -c /etc/suricata/suricata.yaml -i eth0

# ตรวจสอบ Log
sudo tail -f /var/log/suricata/fast.log
```

**การใช้งาน Suricata สำหรับนักพัฒนา:**

Suricata สามารถถูกบูรณาการกับเครื่องมือพัฒนาและการทำงานของทีมได้:
- **Eve JSON Log:** Suricata สามารถส่งออก Log ในรูปแบบ JSON ที่เครื่องมือ SIEM และ Log Management สามารถอ่านได้
- **HTTP Log:** บันทึก HTTP Request ทั้งหมดที่ผ่านเครือข่าย ช่วยในการวิเคราะห์ API Abuse
- **TLS Log:** บันทึก TLS Handshake — ตรวจสอบ Certificate ที่ไม่ถูกต้องหรือไม่น่าเชื่อถือ
- **File Extraction:** สกัดไฟล์ที่ส่งผ่านเครือข่ายเพื่อตรวจสอบ Malware

#### 12.5.2 Network Traffic Analysis: Wireshark และ Zeek

**Wireshark:**
Wireshark เป็นเครื่องมือ Deep Packet Inspection (DPI) แบบกราฟิกที่ใช้สำหรับวิเคราะห์แพ็กเกตเครือข่ายโดยละเอียด:

| การใช้งาน | คำอธิบาย |
|-----------|----------|
| **Filter แบบ Capture** | กรองแพ็กเกตก่อนบันทึก: `tcp port 443`, `host 192.168.1.1` |
| **Filter แบบ Display** | กรองแพ็กเกตที่แสดงผล: `http.request.method == "POST"`, `tls.handshake.type == 1` |
| **Follow TCP Stream** | ดูการสื่อสารทั้งหมดในการเชื่อมต่อ TCP หนึ่ง ๆ |
| **Statistics** | Protocol Hierarchy, IO Graph, Conversations |

**ตัวอย่าง Filter ใน Wireshark สำหรับนักพัฒนา:**

```wireshark
# ดูเฉพาะ HTTP Request
http.request

# ดู TLS Handshake
tls.handshake.type == 1          # ClientHello
tls.handshake.type == 2          # ServerHello
tls.handshake.type == 11         # Certificate

# ดูเฉพาะ Traffic ของ IP ใด IP หนึ่ง
ip.addr == 192.168.1.100

# ดู DNS Query
dns.flags.response == 0

# ดู HTTP POST Request
http.request.method == "POST"

# ดู HTTP Response ที่มี Status Code 4xx หรือ 5xx
http.response.code >= 400
```

**Zeek (เดิมชื่อ Bro):**
Zeek เป็น Network Monitoring Framework ที่แตกต่างจาก IDS/IPS ตรงที่ไม่เน้นการตรวจจับตาม Signature แต่เน้นการบันทึก Log ที่มีรายละเอียดสูงสำหรับการวิเคราะห์ Forensic:

| Log ของ Zeek | คำอธิบาย |
|--------------|----------|
| `conn.log` | บันทึกทุกการเชื่อมต่อ TCP/UDP/ICMP (Source IP, Dest IP, Port, Bytes, Duration) |
| `http.log` | บันทึก HTTP Request/Response ทุกรายการ (Method, URI, Host, Status Code, User-Agent) |
| `ssl.log` | บันทึก TLS Handshake (TLS Version, Cipher Suite, Certificate, Server Name) |
| `dns.log` | บันทึก DNS Query/Response ทุกรายการ (Query, Answer, RTT) |
| `notice.log` | แจ้งเตือนกิจกรรมที่น่าสนใจ เช่น Scanning, Brute Force |
| `files.log` | บันทึกไฟล์ที่ถูกส่งผ่านเครือข่าย |

**ความแตกต่างระหว่าง Wireshark และ Zeek:**

| คุณสมบัติ | Wireshark | Zeek |
|-----------|-----------|------|
| การทำงาน | Real-time Interactive Analysis | Passive Monitoring + Log Generation |
| การจัดเก็บ | เก็บ Packet Capture (.pcap) ไฟล์ใหญ่ | เก็บ Log ที่มีขนาดเล็กกว่าและวิเคราะห์ง่ายกว่า |
| การวิเคราะห์ | Deep Packet Inspection แบบละเอียด (ทีละแพ็กเกต) | สรุป Traffic ในรูปแบบ Log ที่มีโครงสร้าง |
| เหมาะกับ | การ Debug ปัญหาเฉพาะจุด | การตรวจสอบ Forensic และการวิเคราะห์ขนาดใหญ่ |
| การใช้งานของนักพัฒนา | ทดสอบ API, ตรวจสอบ TLS Handshake, Debug | วิเคราะห์พฤติกรรมเครือข่าย, หา Indicators of Compromise |

#### 12.5.3 SIEM (Security Information and Event Management)

SIEM รวบรวม Log จากหลายแหล่ง (Server, Firewall, IDS/IPS, Application) และวิเคราะห์เพื่อตรวจจับเหตุการณ์ด้านความปลอดภัย

| เครื่องมือ SIEM | ประเภท | จุดเด่น |
|----------------|--------|---------|
| **Splunk** | Commercial (มี Free Tier) | การค้นหาและการสร้าง Dashboard ที่ยืดหยุ่น, Machine Learning Toolkit |
| **Microsoft Sentinel** | Cloud-native (Azure) | บูรณาการกับ Microsoft Ecosystem, ใช้ AI ในการตรวจจับ, ราคาตามปริมาณข้อมูล |
| **Elastic Security** (ELK Stack) | Open Source / Commercial | ใช้ Elasticsearch + Kibana + Fleet, SIEM ในตัว, ราคาถูกกว่า |
| **Wazuh** | Open Source ฟรี | SIEM + XDR, ตรวจจับการบุกรุก, การตรวจสอบ Compliance (PCI DSS, GDPR) |

**ประโยชน์ของ SIEM สำหรับนักพัฒนา:**

1. **การตรวจจับ API Abuse:** วิเคราะห์ Log การเรียก API เพื่อหารูปแบบที่ผิดปกติ
2. **การตรวจสอบ Authentication Failure:** ตรวจจับ Brute Force Attack ที่ Login Endpoint
3. **การวิเคราะห์ Error Rate ที่สูงผิดปกติ:** ตรวจจับ Application Layer DDoS หรือการโจมตีที่ทำให้ Application ผิดพลาด
4. **Correlation ของ Event:** เชื่อมโยงเหตุการณ์จากหลายแหล่ง เช่น Web Server Log + Firewall Log + IDS Alert

#### 12.5.4 NetFlow/IPFIX Analysis

NetFlow (Cisco) และ IPFIX (Internet Protocol Flow Information Export — มาตรฐาน RFC 7011) เป็นโปรโตคอลที่ใช้ในการเก็บข้อมูลสถิติของการรับส่งข้อมูลในเครือข่าย:

| ข้อมูลที่บันทึก | คำอธิบาย |
|----------------|----------|
| Source IP / Dest IP | ที่อยู่ IP ต้นทางและปลายทาง |
| Source Port / Dest Port | พอร์ตต้นทางและปลายทาง |
| Protocol | โปรโตคอลที่ใช้ (TCP, UDP, ICMP) |
| Bytes / Packets | ขนาดและจำนวนแพ็กเกตที่ส่ง |
| Timestamps | เวลาเริ่มต้นและสิ้นสุดของการรับส่ง |
| TCP Flags | Flag ของ TCP (SYN, ACK, FIN, RST) |
| Interface | อินเทอร์เฟซที่แพ็กเกตผ่านเข้าและออก |

**การประยุกต์ใช้ NetFlow สำหรับนักพัฒนา:**
- ตรวจจับ Traffic Spike ที่ผิดปกติที่อาจเป็นการโจมตี DDoS
- วิเคราะห์ Top Talkers — IP Address ที่รับส่งข้อมูลมากที่สุด
- ตรวจจับ Data Exfiltration — การส่งข้อมูลปริมาณมากออกนอกเครือข่าย
- ตรวจสอบการเชื่อมต่อไปยัง IP Address ที่น่าสงสัย

#### 12.5.5 การบูรณาการ Network Monitoring โดยนักพัฒนา

นักพัฒนาสามารถบูรณาการแนวทาง Network Monitoring ในซอฟต์แวร์ที่พัฒนาขึ้นเอง:

**Structured Logging:**
การบันทึก Log ในรูปแบบที่มีโครงสร้าง (JSON) ช่วยให้ SIEM และ Log Analysis Tool สามารถวิเคราะห์ข้อมูลได้อย่างมีประสิทธิภาพ:

```json
// ตัวอย่าง Structured Log ที่ SIEM สามารถวิเคราะห์ได้
{
  "timestamp": "2026-05-12T10:30:00.123Z",
  "event": "authentication_attempt",
  "user_id": "user-123",
  "source_ip": "203.0.113.50",
  "user_agent": "Mozilla/5.0 ...",
  "status": "failed",
  "failure_reason": "invalid_password",
  "request_id": "req-abc-123"
}
```

**Distributed Tracing:**
การติดตามคำขอข้าม Service (Distributed Tracing) ด้วยระบบเช่น OpenTelemetry, Jaeger หรือ Zipkin ช่วยให้เห็นภาพรวมของคำขอที่เดินทางผ่าน Service หลายตัว:

```yaml
# OpenTelemetry Configuration — ส่ง Trace ไปยัง Network Monitoring Tool
receivers:
  otlp:
    protocols:
      grpc:
      http:

processors:
  batch:

exporters:
  prometheus:
    endpoint: "0.0.0.0:8889"
  zipkin:
    endpoint: "http://zipkin:9411/api/v2/spans"

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [zipkin]
```

**การเพิ่ม HTTP Headers สำหรับ Network Monitoring:**

```nginx
# NGINX — การเพิ่ม Headers สำหรับติดตามการรับส่ง
server {
    listen 443 ssl;
    server_name api.example.com;
    
    # เพิ่ม Request ID สำหรับติดตาม Request
    add_header X-Request-ID $request_id;
    
    # Log รายละเอียดที่สำคัญ
    log_format detailed '$remote_addr - $remote_user [$time_local] '
                        '"$request" $status $body_bytes_sent '
                        '"$http_referer" "$http_user_agent" '
                        '$request_time $upstream_response_time';
    
    access_log /var/log/nginx/api_access.log detailed;
}
```

   ```bash
   cd testssl.sh
   ```


   - ควรเปิด OCSP Stapling

2. รายการ TLS Protocol ที่รองรับ (TLS 1.3, 1.2, 1.1, 1.0)
3. รายการ Cipher Suite ที่ปลอดภัยและไม่ปลอดภัย

### Lab 12.2: การตั้งค่า Web Application Firewall (WAF) ด้วย ModSecurity และ OWASP CRS

**วัตถุประสงค์:** เรียนรู้การติดตั้งและกำหนดค่า ModSecurity WAF ร่วมกับ OWASP Core Rule Set (CRS) เพื่อป้องกัน SQL Injection และ Cross-Site Scripting (XSS)

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** Docker, ModSecurity, OWASP CRS, ระบบ Web Application ที่อาจารย์จัดให้

**ขั้นตอน:**

1. รัน Web Application ที่มีช่องโหว่ด้วย Docker:
   ```bash
   # รัน DVWA (Damn Vulnerable Web Application)
   docker run --rm -d -p 8080:80 vulnerables/web-dvwa
   ```

2. รัน ModSecurity WAF ด้วย OWASP CRS:
   ```bash
   docker run --rm -d -p 8081:80 \
     -e PARANOIA=1 \
     -e BLOCKING_PARANOIA=1 \
     -v "$(pwd)/crs:/etc/modsecurity.d/owasp-crs" \
     --name modsec \
     coreruleset/modsecurity:latest
   ```

3. ทดสอบการโจมตี SQL Injection โดยไม่ผ่าน WAF (directly ไปที่ DVWA):
   ```bash
   # ทดสอบ SQL Injection ที่ DVWA โดยตรง (พอร์ต 8080)
   curl "http://localhost:8080/vulnerabilities/sqli/?id=1' UNION SELECT user,password FROM users--&Submit=Submit"
   ```

4. ทดสอบการโจมตี SQL Injection ผ่าน WAF (พอร์ต 8081):
   ```bash
   # ทดสอบ SQL Injection ผ่าน WAF — ควรถูกบล็อก
   curl -v "http://localhost:8081/vulnerabilities/sqli/?id=1' OR '1'='1"
   
   # สังเกต HTTP Status Code 403 (Forbidden) หรือ 406 (Not Acceptable)
   ```

5. ทดสอบการโจมตี XSS ผ่าน WAF:
   ```bash
   # ทดสอบ XSS — ควรถูกบล็อกโดย ModSecurity
   curl -v "http://localhost:8081/vulnerabilities/xss_r/?name=<script>alert(1)</script>"
   ```

6. ตรวจสอบ ModSecurity Audit Log:
   ```bash
   docker exec modsec cat /var/log/modsec_audit.log
   # ดูรายละเอียดของ Rule ที่ถูก Trigger
   ```

### Lab 12.4: การกำหนด Firewall Rules ด้วย iptables/nftables หรือ Cloud Security Group

**วัตถุประสงค์:** เรียนรู้การกำหนด Firewall Rules ด้วย iptables/nftables หรือ Security Group ในระบบคลาวด์ (AWS/GCP) สำหรับ Web Application

**เวลาที่ใช้:** 30-45 นาที

**เครื่องมือ:** Linux VM หรือ Cloud Account (AWS/GCP Free Tier) ที่ได้รับอนุญาต

**ขั้นตอน (ส่วนที่ 1 — iptables บน Linux VM):**

1. ตรวจสอบสถานะของ iptables:
   ```bash
   sudo iptables -L -n -v
   ```

2. ตั้งค่า Default Policy เป็น DROP:
   ```bash
   sudo iptables -P INPUT DROP
   sudo iptables -P FORWARD DROP
   sudo iptables -P OUTPUT ACCEPT
   ```

3. อนุญาต Local Loopback Traffic:
   ```bash
   sudo iptables -A INPUT -i lo -j ACCEPT
   ```

4. อนุญาต Established Connections (Stateful Firewall):
   ```bash
   sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
   ```

5. อนุญาต SSH จาก IP ของผู้ดูแล:
   ```bash
   sudo iptables -A INPUT -p tcp --dport 22 -s 203.0.113.0/24 -j ACCEPT
   ```

6. อนุญาต HTTP และ HTTPS จากทุกที่:
   ```bash
   sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
   sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
   ```

7. อนุญาต ICMP (Ping) เพื่อการตรวจสอบ:
   ```bash
   sudo iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT
   ```

8. ตรวจสอบกฎที่ตั้งค่า:
   ```bash
   sudo iptables -L -n -v --line-numbers
   ```

9. ทดสอบการเชื่อมต่อ:
   ```bash
   # ทดสอบจากเครื่องอื่น — ควรเชื่อมต่อ HTTPS ได้
   curl https://<IP ของ VM>
   
   # ทดสอบ Port อื่น — ควรถูกปฏิเสธ
   curl http://<IP ของ VM>:3306  # MySQL Port — ควรถูกบล็อก
   ```

10. บันทึกกฎ iptables เพื่อให้คงอยู่หลัง Reboot:
    ```bash
    sudo iptables-save > /etc/iptables/rules.v4
    ```

**ขั้นตอน (ส่วนที่ 2 — AWS Security Group — ถ้ามีสิทธิ์เข้าถึง AWS):**

1. สร้าง Security Group ใหม่:
   ```bash
   # ใช้ AWS CLI
   aws ec2 create-security-group \
       --group-name web-sg \
       --description "Web Server Security Group" \
       --vpc-id vpc-xxxxx
   ```

2. กำหนดกฎ Inbound:
   ```bash
   # SSH จาก IP ของผู้ดูแล
   aws ec2 authorize-security-group-ingress \
       --group-id sg-xxxxx \
       --protocol tcp --port 22 \
       --cidr 203.0.113.0/24
   
   # HTTP (80) และ HTTPS (443) จากทุกที่
   aws ec2 authorize-security-group-ingress \
       --group-id sg-xxxxx \
       --protocol tcp --port 80 \
       --cidr 0.0.0.0/0
   
   aws ec2 authorize-security-group-ingress \
       --group-id sg-xxxxx \
       --protocol tcp --port 443 \
       --cidr 0.0.0.0/0
   ```

3. ตรวจสอบกฎ:
   ```bash
   aws ec2 describe-security-groups --group-ids sg-xxxxx
   ```

**สิ่งที่ต้องส่ง:**

1. คำสั่ง iptables หรือ AWS CLI ที่ใช้
2. ผลลัพธ์ของ `iptables -L -n -v` หรือ `describe-security-groups` หลังการตั้งค่า
3. อธิบายว่าแต่ละกฎมีความสำคัญอย่างไร
4. ทดสอบการเชื่อมต่อไปยัง Port ที่ถูกปิด — แสดงผลลัพธ์

## คำถามท้ายบท

1. โมเดล OSI ในชั้นใดบ้างที่เกี่ยวข้องกับนักพัฒนาซอฟต์แวร์มากที่สุด จงอธิบายความเสี่ยงด้านความปลอดภัยและแนวทางป้องกันในแต่ละชั้น พร้อมยกตัวอย่างช่องโหว่ที่เกิดขึ้นจริง (ระบุ CVE ID และรายละเอียด)

2. TLS 1.3 แตกต่างจาก TLS 1.2 ในด้าน Handshake (จำนวน RTT), Forward Secrecy, Cipher Suite และ Encrypted Certificate อย่างไร จงอธิบายพร้อมยกตัวอย่างประกอบ จงอธิบายว่าทำไม RC4, 3DES และ Cipher แบบ CBC Mode จึงถูกลบออกจากรายการ Cipher Suite ที่แนะนำ

3. Network Firewall และ Web Application Firewall (WAF) แตกต่างกันอย่างไรในแง่ของชั้น OSI ที่ทำงาน, การตรวจสอบ, ความสามารถในการป้องกันการโจมตี และบทบาทที่เสริมกัน จงอธิบายพร้อมยกตัวอย่างการใช้ร่วมกัน (Defense in Depth)

4. DDoS Attack มีกี่ประเภท จงอธิบายแต่ละประเภทพร้อมตัวอย่างการโจมตีและแนวทางป้องกัน อ้างอิงสถิติจากรายงาน Cloudflare Q4 2025 (47.1 ล้านครั้ง, เพิ่มขึ้น 121%) และอธิบายว่า Aisuru-Kimwolf Botnet คืออะไรและมีผลกระทบอย่างไร

5. Suricata และ Snort แตกต่างกันอย่างไร จงอธิบายข้อดีข้อเสียและให้คำแนะนำสำหรับองค์กรที่ต้องการติดตั้ง IDS/IPS ใหม่ จากนั้นอธิบายความแตกต่างระหว่าง Wireshark กับ Zeek ในแง่ของการใช้งานและการวิเคราะห์ Network Traffic

## สรุปท้ายบท

Network Security เป็นองค์ประกอบที่สำคัญของความปลอดภัยของซอฟต์แวร์โดยรวม นักพัฒนาซอฟต์แวร์จำเป็นต้องเข้าใจหลักการพื้นฐานของเครือข่ายในโมเดล OSI โดยเฉพาะใน Layer 7 (Application), Layer 4 (Transport) และ Layer 3 (Network) เพื่อให้สามารถพัฒนาโค้ดที่ปลอดภัยและทำงานร่วมกับทีมโครงสร้างพื้นฐานได้อย่างมีประสิทธิภาพ

TCP/IP Security Issues เช่น IP Spoofing, TCP SYN Flood และ Session Hijacking ยังคงเป็นภัยคุกคามที่สำคัญ แนวทางป้องกันได้แก่ Egress Filtering (BCP 38), SYN Cookies และการใช้ CSPRNG สำหรับสร้าง Session ID ความปลอดภัยของ DNS ครอบคลุมทั้ง DNSSEC (สำหรับความสมบูรณ์ของข้อมูล) และ DNS over HTTPS/TLS/QUIC (สำหรับความเป็นส่วนตัว) ตามคำแนะนำใน NIST SP 800-81r3

TLS 1.3 เป็นโปรโตคอลที่ปลอดภัยและมีประสิทธิภาพกว่า TLS 1.2 อย่างมีนัยสำคัญ โดยบังคับใช้ Forward Secrecy และ 1-RTT Handshake การเลือก Cipher Suite ที่ปลอดภัย (AES-256-GCM, AES-128-GCM, ChaCha20-Poly1305) และการจัดการ Certificate ผ่าน Let's Encrypt, ACME Protocol และ Certificate Transparency เป็นสิ่งที่นักพัฒนาควรปฏิบัติ mTLS เป็นกลไกสำคัญสำหรับ Service-to-Service Authentication ในสถาปัตยกรรม Service Mesh

Firewall และ Network Access Control มีหลากหลายรูปแบบ ตั้งแต่ Network Firewall (L3/L4), WAF (L7), Stateful/Stateless Firewall ไปจนถึง Security Group และ Network ACL ในระบบคลาวด์ Zero Trust Network Access (ZTNA) ยึดหลัก "Never Trust, Always Verify" และ Microsegmentation เป็นแนวทางที่ได้รับความนิยมมากขึ้นแทน VPN แบบดั้งเดิม

DDoS Protection มีความสำคัญเพิ่มมากขึ้น โดยในปี 2025 มีการโจมตี DDoS ถึง 47.1 ล้านครั้ง (เพิ่มขึ้น 121%) และ Aisuru-Kimwolf Botnet ทำลายสถิติด้วยการโจมตีสูงสุดที่ 31.4 Tbps การป้องกันต้องใช้หลายมาตรการร่วมกัน ได้แก่ CDN Integration (Anycast Routing), Rate Limiting, Auto Scaling และ Load Balancing

Network Monitoring และ Threat Detection ผ่านเครื่องมือเช่น Suricata, Wireshark, Zeek และ SIEM (Splunk, Microsoft Sentinel, Elastic Security, Wazuh) ช่วยให้นักพัฒนาและทีมปฏิบัติการสามารถตรวจจับและตอบสนองต่อภัยคุกคามได้ทันท่วงที นักพัฒนาควรบูรณาการ Structured Logging, Distributed Tracing และข้อมูลที่จำเป็นสำหรับ Network Monitoring ในซอฟต์แวร์ที่พัฒนาขึ้นเอง เพื่อให้ทีมรักษาความปลอดภัยสามารถวิเคราะห์และตรวจจับเหตุการณ์ผิดปกติได้อย่างมีประสิทธิภาพ
